## 研发协作方法论

本项目采用 devpace 管理产品迭代研发，核心原则：

- **价值链驱动**：每个技术变更都追溯到产品目的（业务目标 → 产品功能 → 变更请求，双向可追溯）
- **质量门禁自动化**：自动检查（lint/test/typecheck）失败时自行修复，人工审阅是阻塞门禁——停下来呈现摘要等待决策
- **变更是常态**：需求变更不是规划失败，是学习信号。变更前做影响分析，暂停保留工作成果
- **双模式工作**：探索（读、分析、理解）不需要流程开销；推进（写代码、改状态）遵循检查点和状态机
- **会话连续性**：每个原子步骤留检查点，下次会话无缝恢复
- **分层反馈**：业务决策由人类做，产品决策人机协作，技术执行 Claude 自治到审阅门禁
- **信息分层**：默认 1 句话摘要，按需展开详情，不主动输出全量状态

## .devpace/ 目录

项目研发状态存储在 `.devpace/` 目录中：

| 文件 | 用途 | 加载时机 |
|------|------|---------|
| `state.md` | 状态快照（≤15 行） | 每次会话开始 |
| `project.md` | 业务全景 + 价值功能树 | 按需（查看全景/变更分析） |
| `backlog/CR-*.md` | 变更请求详情 | 推进模式 |
| `iterations/current.md` | 迭代计划 + 变更记录 | 按需 |
| `rules/workflow.md` | 工作流状态机 | 推进模式 |
| `rules/checks.md` | 质量检查定义 | 推进模式 |
| `metrics/dashboard.md` | 度量仪表盘 | /pace-retro |

## 开发模式

**探索**（默认）：自由读取分析，不改变 `.devpace/` 状态文件，不受状态机约束。

**推进**（改代码时）：绑定变更请求，遵循工作流状态机，每步 checkpoint。质量检查不通过自行修复；遇到人工审阅停下等待。
