单一职责原则
Claude Code plugin特别是其中包含的skills本质上是基于markdown的信息组织形式，这类以markdown信息规范模型行为的设计方式，应该遵循信息组织的原则有哪些呢？比如单向依赖，抽象和具体信息的分离，稳定部分与易变部分分离，避免循环依赖
信息组织即编程

 ┌──────────────────────────────┬───────────────────────────────────────────────┐
 │           传统软件           │              Claude Code Plugin               │
 ├──────────────────────────────┼───────────────────────────────────────────────┤
 │ 代码模块通过 import/API 调用 │ Markdown 文件通过引用和上下文加载"通信"       │
 ├──────────────────────────────┼───────────────────────────────────────────────┤
 │ 编译器确定性执行             │ LLM 概率性解释执行（有"合理化绕过"风险）      │
 ├──────────────────────────────┼───────────────────────────────────────────────┤
 │ 内存/CPU 是软约束            │ 上下文窗口是硬约束                            │
 ├──────────────────────────────┼───────────────────────────────────────────────┤
 │ 接口通过类型系统强制         │ 契约通过 Schema Markdown + 验证测试"弱强制"   │
 ├──────────────────────────────┼───────────────────────────────────────────────┤
 │ 模块间无歧义调用             │ description 语义匹配触发（可能误触发/漏触发） │
 └──────────────────────────────┴───────────────────────────────────────────────┘



| P1 | 单向依赖 | 信息从抽象流向具体——不可反向 |
| P2 | 抽象分层 | 不同抽象层级的内容分居不同文件 |
| P3 | 稳定-易变分离 | 高频变更内容不得与低频变更内容耦合 |
| P4 | 信息分类 | 不同信息类型（步骤/概念/约束/参考）不混合 |
| P5 | 按需加载 | 仅加载当前上下文所需的最小信息集 |
| P6 | 单一权威 | 每条信息有且仅有一个权威定义点 |
| P7 | 确定性分级 | 执行力度与约束的关键程度匹配 |
| P8 | 可发现性优先 | 入口信息为发现而设计，而非为完整性 |
| P9 | 认知清晰 | 面向 LLM 的指令明确无歧义，防止合理化绕过 |
| P10 | 契约隔离 | 数据格式由独立的契约层约束 |
