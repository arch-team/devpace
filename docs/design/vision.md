# devpace 项目目标

> **职责**：定义项目的愿景、业务目标和成效指标。这是项目决策的最终依据。

**定位**：docs/ 体系的**顶层文件**——回答"为谁做、做什么、值不值得"。当设计方案或需求发生冲突时，以本文件为准。

| 维度 | 说明 |
|------|------|
| 回答的问题 | 北极星是什么？目标用户是谁？目标角色有哪些？核心价值主张是什么？成功怎么衡量？ |
| 上游输入 | theory.md（理论依据） |
| 下游消费 | design.md（设计方案受愿景约束）→ requirements.md（需求从愿景拆解）→ roadmap.md（排期） |
| 更新时机 | 项目方向调整、目标用户变化、角色模型调整、新增/废弃业务目标时 |
| 类别 | 项目管理——定义项目方向，不进入产品运行时 |
| 主要读者 | 所有参与者——需要理解"为什么做这个项目" |

## 一句话概述

**devpace** 是一个 Claude Code 插件，为产品交付带来完整的 BizDevOps 研发节奏管理。它将"业务目标→产品功能→代码变更"串成一条可追溯的价值链，让业务方、产品、开发、测试、运维通过统一概念模型协作——Claude 不只是写代码的工具，而是理解业务意图的研发协作者。业务机会散落？AI 引导结构化梳理，对齐战略目标。会话中断？自动恢复上下文，团队无需重复解释。需求变更？即时评估影响范围，有序调整而非推倒重来。质量跑偏？内置门禁自动拦截，未就绪的变更无法流入下游。**需求永远在变，但从规划到交付的研发节奏，不应该因此失控。** devpace 完整覆盖 BizDevOps 三域（业务、开发、运维），Ops 域采用分阶段策略逐步深化。从 Harness Engineering 视角看，devpace 是 Claude Code 的研发节奏 harness——用规则、Schema、门禁和反馈循环约束 Agent 行为，让 AI 辅助产品交付从 vibe coding 进化为可追溯、可度量的工程实践。

## 北极星

**devpace 给 Claude Code 项目一个稳定的研发节奏——需求在变，节奏不乱。**

- 业务机会来了，结构化梳理，对齐战略目标
- 会话断了，节奏自动续上
- 需求变了，影响自动评估，有序调整
- 质量跑偏，检查自动纠正
- 上线出问题，自动追溯到引入变更，驱动修复闭环

> **需求在变是常态，失序不应该是。**

## 目标用户

**使用 Claude Code 持续迭代产品的交付团队**——不是写一次性脚本，而是在多个会话中推进有业务目标的项目。从个人开发者到企业团队，devpace 通过统一的概念模型和多角色视角适配，覆盖从规划到交付的完整链路。

### 五角色画像

| 角色 | 典型场景 | 核心痛点 | devpace 价值 |
|------|---------|---------|-------------|
| **Biz Owner** | 战略对齐、MoS 追踪、业务价值评估 | 业务机会散落，目标与交付脱节 | OPP→Epic→BR 结构化规划，MoS 达成率可视 |
| **PM** | 迭代规划、需求分解、变更管理 | 需求变更频繁，进度难以追踪 | /pace-plan 迭代规划 + /pace-change 有序变更 |
| **Dev** | CR 实现、质量门、代码推进 | 跨会话失忆，质量检查靠自觉 | 自动上下文恢复 + Gate 1-3 不可跳过 |
| **Tester** | 测试策略、覆盖分析、验收验证 | 测试范围靠猜，回归无保障 | /pace-test 影响分析 + 覆盖度追踪 |
| **Ops** | 发布管理、生产反馈、缺陷追溯 | 上线后问题难定位，发布无记录 | /pace-release 编排 + /pace-feedback 追溯闭环 |

**一人多角色**：独立开发者同时戴多顶帽子——上午梳理业务需求（Biz Owner），下午写代码（Dev），晚上发布验证（Ops）。devpace 通过 /pace-role 零成本切换视角，同一项目数据自动适配不同角色的关注维度。

> devpace 同样适用于个人开发者和开源项目——企业级能力（DORA 度量、合规追溯、发布管理）作为渐进式能力提供，不使用时零开销。

## 三个价值视角

### 客户价值

解决交付团队在使用 Claude Code 做产品研发时的三域核心痛点：

**Biz 域——业务规划**

| 痛点 | 表现 | devpace 的价值主张 | 首次可感知 |
|------|------|---------------------|-----------|
| 业务机会散落 | 灵感和需求分散在对话、文档、脑子里 | /pace-biz 引导结构化梳理：OPP→Epic→BR | 首次 /pace-biz |
| 需求与战略脱节 | 做着做着偏离业务目标 | OBJ→BR→PF 追溯链 + /pace-biz align 对齐检查 | 功能增多后 |
| 需求梳理效率低 | 从文档/代码到需求全靠手动 | /pace-biz import（文档导入）+ infer（代码推断） | 首次导入 |

**Dev 域——开发交付**

| 痛点 | 表现 | devpace 的价值主张 | 首次可感知 |
|------|------|---------------------|-----------|
| 跨会话连续性断裂 | 每次开 Claude 都要重新解释"在哪、干嘛" | 自动恢复上下文，零手动解释 | 第 2 次会话 |
| 质量不一致 | Claude 有时跳过测试、忘记检查 | 进入推进模式后质量检查自动执行，不可跳过 | 第 1 个 CR |
| 需求变更导致失序 | 需求变了不知道影响多大，进度混乱 | 变更影响分析 + 有序调整 | 首次变更时 |

**Ops 域——运维交付**

| 痛点 | 表现 | devpace 的价值主张 | 首次可感知 |
|------|------|---------------------|-----------|
| 交付过程不可见 | 组织要求可审计但 Claude 是黑盒 | 全链路追溯（OBJ→CR）+ 决策记录 + DORA 度量 | 首次 /pace-retro |
| 生产问题难追溯 | 上线后出 bug 不知道哪个变更引入的 | /pace-feedback 追溯到引入变更，自动创建修复 CR | 首次生产反馈 |
| 发布过程无管理 | 发布靠记忆，变更记录不完整 | /pace-release 管理完整生命周期（changelog + tag + verify） | 首次发布 |

### 企业价值

- Claude 的工作过程可审计、可追溯、质量可控——满足企业合规要求
- 多项目并行时不丢上下文，经验可跨项目复用
- 研发效能可度量（DORA 交付效能 + 质量门通过率 + 业务价值对齐度），可持续改进
- 发布过程有完整记录，变更可追溯到引入它的业务需求
- 多角色协作：统一概念模型（OPP→Epic→BR→PF→CR）让不同角色对齐同一语言
- 业务规划追溯：从战略目标到代码变更的完整链路，任意节点可双向追溯
- 风险管理可见性：/pace-guard 预检 + /pace-retro 度量，风险从感性判断变为数据驱动

### 生态价值

- 填补 Claude Code 生态中"BizDevOps 节奏管理"的空白
- 不替代 GitHub Issues/Jira，而是补充——管的是"Claude Code 项目级的研发节奏"
- 开源社区可贡献、可适配
- 5 角色模型为 AI 辅助团队协作提供参考范式

## 差异化定位

与纯任务管理工具（TodoWrite、GitHub Issues）的区别：

| 维度 | 任务管理工具 | devpace |
|------|------------|-----------|
| 核心模型 | 任务列表 | 价值交付链路 |
| 对变更的态度 | 计划是稳定的 | **变更是一等公民** |
| 与 Claude 的关系 | Claude 是执行者 | Claude 是自治协作者 |
| 可追溯性 | 任务 → 代码 | 业务目标 → 功能 → 变更 → 代码（始终可追溯，随迭代加深） |
| 度量 | 完成数 | 质量 + 价值对齐 |
| 业务规划 | 无 | OPP→Epic→BR 完整上游建模 |
| 角色意识 | 无 | 5 角色自动适配输出 |

### 护城河层次

| 能力层 | devpace 价值 | 被原生替代风险 |
|--------|-------------|---------------|
| 跨会话连续性 + 轻量追溯 | 用户入口 | 高——可能被原生 session persistence 替代 |
| 业务规划域（OPP→Epic→BR） | 上游差异化 | 低——需要领域概念模型和结构化引导 |
| 功能级变更管理 | 差异化所在 | 低——需要概念模型驱动的流程引擎 |
| 完整价值链 + 度量 | 不可替代的护城河 | 极低——需要完整 BR→PF→CR 链路 |
| 企业级合规（审计追溯 + DORA） | 企业用户必选 | 极低——需要完整概念模型和流程引擎 |

战略含义：跨会话连续性是获取用户的入口，业务规划域和变更管理是差异化，完整价值链 + 度量是长期不可替代的护城河，企业级合规能力确保在组织环境中不可替代。即使 Claude Code 原生增强了会话持久化，devpace 在业务规划、变更管理、价值链和企业合规层的价值不受影响。

**备选入口**：如果跨会话连续性被原生替代，devpace 的入口切换为"变更管理即时价值"——用户在单个会话内首次改变需求时，devpace 自动展示影响分析能力，让用户立即感受到差异化。

## 边界与演进

### 目标定位（愿景终态）

| 维度 | 终态描述 |
|------|---------|
| **Biz 域** | 从战略目标到业务需求的完整规划链路，多利益相关者对齐 |
| **Dev 域** | 从需求到代码的自治交付，质量门 + 变更管理 + 跨会话连续性 |
| **Ops 域** | 从发布到生产验证的闭环管理，CI/CD 集成 + 监控关联 + SRE 工作流 |
| **协作** | 多角色多项目协作，Web 可视化 dashboard，团队知识共享 |

### 当前阶段简化（v1.x）

| 维度 | 当前简化 | 目标终态 |
|------|---------|---------|
| 团队规模 | 单人操作（一人多角色） | 多角色协作（团队共享状态） |
| **Biz 域** | OPP→Epic→BR 全覆盖 | + 多利益相关者协作 |
| **Dev 域** | CR 状态机 + 4 级质量门 + 5 种变更管理 + 复杂度自适应 | + 多人并行 CR 协调 |
| **Ops 域** | 流程层 + 感知层（发布编排、CI 查询、反馈收集、DORA 代理指标） | + 执行层集成（部署/回滚/监控） |
| 可视化 | CLI 文本输出 | + Web dashboard |
| 知识共享 | 单项目经验库 | + 全局跨项目经验复用 |

### 计划扩展（v2.x+）

| 域 | 扩展层 | 方向 |
|----|--------|------|
| **Ops** | 执行层集成 | 通过 CLI 工具触发部署、回滚；与容器平台（kubectl）和云平台（aws/gcloud CLI）集成 |
| **Ops** | 监控集成 | 健康检查执行、告警关联、SRE 工作流 |
| **协作** | 多人协作 | 团队共享 .devpace/ 状态、角色分工、并行 CR 协调 |
| **可视化** | Web dashboard | 项目状态可视化、度量趋势图表、跨项目总览 |
| **跨项目** | 知识层 | 全局经验共享、跨项目模式迁移 |

### devpace 不替代什么

| 不替代 | 原因 | devpace 的角色 |
|--------|------|---------------|
| CI/CD 平台（GitHub Actions / Jenkins） | 流水线定义和执行是专业平台的职责 | 查询状态、消费结果、编排发布流程 |
| 项目管理工具（Jira / Linear） | 团队协作和看板是其职责 | 补充 Claude Code 项目级的研发节奏 |
| 监控平台（Datadog / Grafana） | 实时监控和告警是其职责 | 消费监控信号、关联变更追溯（计划中） |
| 代码质量分析 | Claude Code 原生能力已足够 | 管理流程级质量门禁 |

## 业务目标

> OBJ 可标注产品维度属性：类型（business/product/tech）+ 时间维度（短期/中期/长期）。详见 `knowledge/_schema/entity/project-format.md` "业务目标"章节。

### MVP 阶段（1-2 周）

| ID | 目标 | 成效指标（MoS） | 验证方式 |
|----|------|---------------|---------|
| OBJ-1 | 跑通完整工作流 | 1 个 CR 从 created→merged 全流程走通 | 在外部项目（非 devpace 自身）实际执行 |
| OBJ-2 | 跨会话连续性 | 连续 3 次会话，Claude 无需用户重述之前的工作即可继续 | 模拟中断恢复场景 |
| OBJ-3 | 质量检查有效 | 质量检查至少 1 次阻止了不满足准出条件的 CR 进入下一阶段 | 回顾 CR 状态流转记录，确认有拦截事件 |
| OBJ-4 | 变更管理有价值 | 需求变更后 state.md + 相关 CR 文件在同一次交互中更新完成，无残留不一致 | 模拟 3 种变更场景，检查文件一致性 |
| OBJ-5 | devpace 优于手动方案 | 对比测试：同一项目 3 次会话中断后，devpace 恢复上下文所需的用户纠正次数 < 手动 CLAUDE.md 方案 | 同一项目分别用两种方案，计数纠正次数 |
| OBJ-6 | 渐进丰富自然 | 从 1 个 CR 增长到 3+ CR 的过程中，用户未主动查询或修改 devpace 生成的结构文件 | 观察用户是否需要手动修正 state.md / project.md |

### 成熟阶段（1 个月）

| ID | 目标 | 成效指标（MoS） |
|----|------|---------------|
| OBJ-7 | 可移植性验证 | 新项目从 /pace-init 到首个 CR merged 不超过 1 小时 |
| OBJ-8 | 度量体系可用 | /pace-retro 生成有意义的回顾报告，含与基准线的对比趋势 |
| OBJ-11 | 备选入口可行 | 无 .devpace/ 初始化的项目中，/pace-change 能输出有价值的变更影响分析 |
| OBJ-12 | 发布状态可追踪 | CR merged 后到生产验证的状态对用户透明，/pace-release 管理完整 Release 生命周期 |
| OBJ-13 | 运维反馈闭环 | 生产问题通过 /pace-feedback report 追溯到引入变更，自动创建 defect/hotfix CR 驱动修复 |
| OBJ-14 | 角色视角适配 | Claude 根据用户当前角色（Biz/PM/Dev/Tester/Ops）自动调整输出视角和关注点 |
| OBJ-18 | 业务规划域端到端可用 | 从 OPP 捕获到 BR→PF 分解在 1 个会话内完成 |
