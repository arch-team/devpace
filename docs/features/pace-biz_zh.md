# 业务规划（`/pace-biz`）

devpace 一直管理着从产品功能（PF）到代码变更（CR）的交付流程，但上游的问题——*这些功能从何而来？*——始终游离在体系之外。业务机会散落在各种笔记中，专题（Epic）活在外部工具里，战略目标与日常开发之间的关联只能靠默契维系。`/pace-biz` 补上了这一环，将结构化的业务规划纳入同一条价值链，让每一行代码都能追溯到它背后的业务机会。

## 快速上手

```
1. /pace-biz opportunity "企业客户需要 SSO"        → 捕获机会 → OPP-001 创建
2. /pace-biz epic OPP-001 "企业级认证"             → 创建专题 → EPIC-001 关联 OBJ
3. /pace-biz decompose EPIC-001                   → 分解需求 → 生成 BR 和 PF
4. /pace-biz refine BR-003                        → 精炼细节 → 补充验收标准
```

或者让 Claude 交互式引导：

```
你：    /pace-biz
Claude：[扫描项目上下文，智能推荐下一步]
        检测到：工作目录下有 会议纪要.md → 推荐：/pace-biz import 会议纪要.md
        检测到：1 个 Epic 的 BR 为空 → 推荐：/pace-biz decompose EPIC-001
        或选择：opportunity / epic / decompose / refine / align / view / discover / import / infer
```

从模糊想法开始：

```
5. /pace-biz discover "我想做一个任务管理工具"   → 多轮对话发现 → 完整候选树
6. /pace-biz import 会议纪要.md                → 提取需求 → 合并到功能树
7. /pace-biz infer                            → 扫描代码库 → 差距报告
```
## 用户使用旅程

> 不确定从哪里开始？下表帮你快速定位。

| 你的情况 | 推荐旅程 | 起点子命令 |
|---------|---------|-----------|
| 有明确的业务机会或客户反馈，想从头建立规划体系 | [旅程 A](#旅程-a从零开始的业务规划) | `opportunity` → `epic` → `decompose` |
| 只有一个模糊想法，不知道该做什么 | [旅程 B](#旅程-b从模糊想法到结构化需求) | `discover` |
| 团队有散落的需求文档（会议纪要、PRD、Issue 列表） | [旅程 C](#旅程-c导入已有文档) | `import` → `align` |
| 接手一个文档缺失的遗留项目 | [旅程 D](#旅程-d遗留项目功能盘点) | `infer` → `align` |
| 项目增长后需要检查规划健康度 | [旅程 E](#旅程-e日常维护与战略对齐) | `align` + `view` + `refine` |

**lite 模式说明**：轻量模式（OBJ→PF→CR）下，`opportunity`、`epic`、`decompose` 不可用。旅程 A 需先通过 `/pace-init --upgrade-mode` 升级到完整模式。旅程 B/C/D 在 lite 模式下自动简化——跳过 OPP/Epic/BR 层，直接产出 PF 候选。旅程 E 的 `align` 简化为 OBJ→PF→CR 链路检查。

### 旅程 A：从零开始的业务规划

**触发场景**：客户反馈了一个明确的需求方向，或团队发现了新的市场机会，你想从业务机会出发建立完整的规划体系。

**完整路径**：`opportunity` → `epic` → `decompose` → `/pace-plan`

**分步操作**：

**第 1 步：捕获业务机会**

```
你：    /pace-biz opportunity 企业客户需要 SSO 登录，目前每次集成都要单独开发
Claude：已捕获业务机会：OPP-001 — 企业客户需要 SSO 登录
        来源：客户反馈（企业客户集成需求）
        状态：评估中
        → 下一步：/pace-biz epic OPP-001 评估并转化为 Epic
```

Claude 自动推断来源类型（客户反馈/竞品观察/技术发现等）。多个机会可以指向同一战略目标，形成优先级的信号积累。

**第 2 步：创建战略专题**

```
你：    /pace-biz epic OPP-001 "企业级认证体系"
Claude：这个专题服务于哪个业务目标？
        1. OBJ-1：提升企业客户留存率
        2. OBJ-2：拓展 B2B 市场份额
你：    1
Claude：如何衡量这个专题的成功？（可先跳过，后续补充）
你：    企业客户接入时间从 2 周降到 2 天；SSO 覆盖率达到 80%
Claude：已创建专题：EPIC-001 — 企业级认证体系
        关联：OBJ-1（提升企业客户留存率）← OPP-001
        MoS：
        - [ ] 企业客户接入时间从 2 周降到 2 天
        - [ ] SSO 覆盖率达到 80%
        → 下一步：/pace-biz decompose EPIC-001 分解为业务需求
```

**第 3 步：分解为可交付项**

```
你：    /pace-biz decompose EPIC-001
Claude：基于 EPIC-001 背景和 MoS，建议分解为以下业务需求：

        BR-001：SSO 协议集成        价值：高 × 成本：中 → P0
        BR-002：会话与权限管理      价值：高 × 成本：高 → P1
        BR-003：企业管理后台        价值：中 × 成本：低 → P1

        BR-002 是否依赖 BR-001？
你：    是，SSO 要先通了才能管理会话
Claude：已记录依赖：BR-002 → BR-001
        确认这个分解方案？（可调整名称、优先级或增删）
你：    确认
Claude：已分解 EPIC-001：
        ├── BR-001：SSO 协议集成 P0
        ├── BR-002：会话与权限管理 P1（依赖 BR-001）
        └── BR-003：企业管理后台 P1
        → 继续分解：/pace-biz decompose BR-001 → 产出 PF
        → 排入迭代：/pace-plan next
```

可以继续 `decompose BR-001` 将业务需求分解为产品功能（PF），产出的 PF 直接进入 `/pace-plan next` 的迭代候选池。

### 旅程 B：从模糊想法到结构化需求

**触发场景**：只有一个模糊的想法——"我想做一个任务管理工具"——不确定具体要做什么。

**完整路径**：`discover`（多轮对话）→ 候选树确认 → 写入

**分步操作**：

**第 1 步：启动发现会话**

```
你：    /pace-biz discover 我想做一个团队任务管理工具
Claude：让我们一步步理清这个想法。
        要解决什么问题？目标用户是谁？
你：    我们团队用表格管任务，经常漏掉截止日期，想要一个简单的工具
Claude：这个对用户来说有多紧迫？是锦上添花还是痛点？
        有没有现有的替代方案？
你：    是痛点，试过 Trello 但太复杂了，我们只需要看板 + 提醒
```

发现会话的中间状态自动持久化到 `.devpace/scope-discovery.md`。即使对话中断，下次可以接着上次的进度继续。

**第 2 步：功能头脑风暴**

```
Claude：基于你的描述，核心能力应该包括哪些？
        用户必须能做什么？最基本的操作是什么？
你：    创建任务、拖拽改状态、设截止日期、到期提醒
Claude：当任务过期了怎么办？有没有异常情况需要处理？
你：    过期任务标红就行，不需要自动关闭
Claude：整理中... 初步候选：
        BR-001：任务生命周期管理
        ├── PF-001：看板视图与拖拽
        ├── PF-002：截止日期设置
        └── PF-003：到期提醒通知
        方向对吗？还有其他功能想法吗？
你：    再加一个简单的成员分配
```

每轮对话后 Claude 实时整理候选，展示给你确认方向。

**第 3 步：确认候选树并写入**

```
Claude：需求发现摘要：

        目标：替代表格的轻量任务管理工具
        用户：5-10 人团队

        候选价值链：
        OBJ-1（提升团队协作效率）
        └── OPP-001：团队任务追踪需求
            └── EPIC-001：轻量看板工具
                ├── BR-001：任务生命周期管理 P0
                │   ├── PF-001：看板视图与拖拽
                │   ├── PF-002：截止日期设置
                │   └── PF-003：到期提醒通知
                └── BR-002：团队协作 P1
                    └── PF-004：成员分配

        范围：
          做：看板、截止日期、提醒、成员分配
          不做：甘特图、时间追踪、自动化工作流

        确认后将写入 .devpace/，请检查并调整。
你：    确认
Claude：已从发现会话创建：
        - 1 个业务机会（OPP-001）
        - 1 个专题（EPIC-001）
        - 2 个业务需求（BR-001 ~ BR-002）
        - 4 个产品功能（PF-001 ~ PF-004）
        → /pace-biz decompose EPIC-001 继续细化特定需求
        → /pace-biz align 检查战略对齐度
        → /pace-plan next 排入迭代
```

> **lite 模式差异**：候选树简化为 `OBJ→PF` 结构（无 OPP/Epic/BR 层），PF 直接挂在 OBJ 下。

### 旅程 C：导入已有文档

**触发场景**：团队有散落的需求文档——会议纪要、PRD、用户反馈调研、竞品分析——想把这些整合到统一的功能树中。

**完整路径**：`import` → `align`

**分步操作**：

**第 1 步：导入文档**

```
你：    /pace-biz import 会议纪要-0315.md 用户反馈汇总.md
Claude：正在分析 2 个文件...

        [会议纪要-0315.md] 检测为：会议纪要
        [用户反馈汇总.md] 检测为：用户反馈

        导入分析：

        NEW（新增 4 项）：
          + BR-005：移动端适配（来源：会议纪要-0315.md L23）
          + PF-010：离线缓存模式（来源：会议纪要-0315.md L31）
          + PF-011：推送通知优化（来源：用户反馈汇总.md L15）
          + PF-012：批量操作（来源：用户反馈汇总.md L42）

        DUPLICATE（已有 2 项）：
          = "任务提醒" ≈ PF-003（已存在，跳过）
          = "看板拖拽" ≈ PF-001（已存在，跳过）

        ENRICHMENT（丰富 1 项）：
          ~ PF-002：追加验收标准 "支持自然语言日期输入（如'下周五'）"

        逐条确认：accept all / reject all / 逐条选择
你：    accept all
Claude：导入完成（来自 2 个文件）：
        - 新增：1 个 BR + 3 个 PF
        - 丰富：1 个已有实体
        - 跳过：2 个重复项
        → /pace-biz align 检查新增内容的战略对齐度
```

Claude 自动检测文档类型（会议纪要/用户反馈/竞品分析/技术债务清单等），按 NEW/DUPLICATE/ENRICHMENT/CONFLICT 分类，避免重复导入。

**第 2 步：检查对齐度**

```
你：    /pace-biz align
Claude：战略对齐度报告
        ══════════════

        OBJ 覆盖率：[1/1] OBJ 有 Epic/BR 覆盖 ✅

        孤立实体：
        └── 孤立 BR：BR-005（移动端适配）未关联 Epic
            → /pace-change modify BR-005 关联到 EPIC-001

        MoS 完整性：
        └── Epic 级 MoS：[1/1] 已定义 ✅

        优先级分布：P0 2个 / P1 3个 / P2 1个 — 健康 ✅

        建议：
        1. 将 BR-005 关联到现有 EPIC 或创建新 EPIC
```

> **lite 模式差异**：`import` 直接提取为 PF 候选（跳过 BR 映射），`align` 简化为 OBJ→PF→CR 链路检查。

### 旅程 D：遗留项目功能盘点

**触发场景**：接手一个文档缺失的遗留项目，不清楚代码里有哪些功能，想建立功能追踪体系。

**完整路径**：`infer` → `align`

**分步操作**：

**第 1 步：扫描代码库**

```
你：    /pace-biz infer
Claude：正在扫描代码库...

        代码库分析报告：

        1. 未追踪功能（代码有，功能树无）：
           模块 [auth/]：
           + PF 候选：用户认证（routes: login, register, logout）
           + PF 候选：权限管理（middleware: checkPermission）

           模块 [api/export/]：
           + PF 候选：数据导出 API（endpoints: /export/csv, /export/json）

        2. 未实现功能（功能树有，代码无）：
           ? PF-003：到期提醒通知 — 确认状态：计划中 / 已放弃 / 其他？

        3. 技术债务（TODO/FIXME 集中区域）：
           ! src/utils/parser.js（8 个 TODO）— 纳入追踪？
           ! src/db/migration.py（5 个 FIXME）— 纳入追踪？

        [Git 增强] 热点：src/core/engine.js（30天内 42 次变更）
        [Git 增强] 耦合：auth/ ↔ session/（经常一起改动）

        逐项选择：accept / skip / 调整分组
你：    未追踪功能全部 accept，PF-003 标记计划中，技术债务 accept parser.js 那个
Claude：代码库推断完成：
        - 新增追踪：3 个产品功能
        - 技术债务：1 个待处理项
        - 未实现确认：1 个功能状态已更新
        → /pace-biz align 检查新增项的战略对齐度
        → /pace-dev 开始处理优先项
```

`infer` 通过目录结构、路由定义、API 端点、数据模型等多维度扫描，结合 Git 历史分析热点和耦合关系。技术债务 PF 附"（技术债务）"后缀，便于后续过滤。

**第 2 步：对齐检查并补全**

运行 `/pace-biz align` 检查新增项是否正确关联到 OBJ 和 Epic，发现孤立实体后按建议命令逐一修复。

> **lite 模式差异**：`infer` 扫描结果直接映射为 PF（跳过 BR 分组），追加到 OBJ 下。

### 旅程 E：日常维护与战略对齐

**触发场景**：项目已运行一段时间，功能树不断增长，需要定期检查规划健康度、精炼需求细节。

**推荐路径**：`align` → `view` → `refine`

**分步操作**：

**第 1 步：健康检查**

```
你：    /pace-biz align
Claude：战略对齐度报告
        ══════════════

        OBJ 覆盖率：[2/2] 全覆盖 ✅

        孤立实体：
        └── 空 Epic：EPIC-002（数据分析平台）— 无 BR
            → /pace-biz decompose EPIC-002

        MoS 完整性：
        ├── OBJ 级 MoS：[2/2] 已定义 ✅
        └── Epic 级 MoS：[1/2] 已定义 — EPIC-002 缺失
            → 补充：直接描述指标，或 /pace-change modify EPIC-002

        优先级分布：P0 5个 / P1 2个 / P2 1个 — ⚠️ 优先级通胀（P0 占 62%）

        MoS 达成度：
        └── EPIC-001：MoS [0/2] 勾选（BR 完成度 100%）
            → 建议 /pace-retro 评估达成度

        建议：
        1. 分解 EPIC-002 → /pace-biz decompose EPIC-002
        2. 重新评估 P0 优先级，建议将 2-3 个调降为 P1
        3. 评估 EPIC-001 MoS 是否已达成 → /pace-retro
```

`align` 覆盖 10 项检查：OBJ 覆盖率、孤立实体、MoS 完整性、分解空白、停滞检测、优先级分布、依赖健康度、MoS 达成度、需求就绪度分布、利益相关者覆盖度。每个问题附带内联修复命令。

**历史趋势追踪** — 每次 `align` 执行后，关键指标（OBJ 覆盖率、孤立实体数、P0 就绪度、MoS 完整性）追加到 `.devpace/metrics/insights.md`。后续执行时报告末尾包含趋势对比段，展示各指标相对上次的变化。某指标连续 3 次下降时追加聚焦警告。

**第 2 步：查看全景视图**

```
你：    /pace-biz view
Claude：业务全景（TaskFlow）
        ══════════════════

        [OBJ-1：提升团队协作效率]
        ├── [EPIC-001：轻量看板工具]（进行中）← OPP-001
        │   MoS：0/2 达成
        │   ├── BR-001：任务生命周期管理 P0 进行中
        │   │   ├── PF-001 → CR-001 ✅
        │   │   ├── PF-002 → CR-002 ✅
        │   │   └── PF-003 → (待创建 CR)
        │   └── BR-002：团队协作 P1 待开始
        │       └── PF-004 → (待创建 CR)
        └── [EPIC-002：数据分析平台]（规划中）
            └── （待分解 → /pace-biz decompose EPIC-002）

        [统计]
        ├── Epic：2 总计（1 进行中 / 1 规划中）
        ├── BR：3 总计
        ├── PF：7 总计（2 完成）
        └── CR：2 总计（2 已完成）
```

**第 3 步：精炼需求细节**

```
你：    /pace-biz refine BR-002
Claude：当前 BR-002：团队协作
        ├── 用户故事：未定义
        ├── 验收标准：未定义
        ├── 优先级：P1
        └── 关联：EPIC-001 → OBJ-1

        这个需求怎么算做完了？有哪些可验证的条件？
你：    任务可以分配给团队成员，被分配人收到通知
Claude：有哪些业务约束？比如一个任务能分配给多人吗？
你：    目前只支持单人，以后再考虑多人
Claude：精炼预览 BR-002：
          + 新增用户故事：作为团队负责人，我希望将任务分配给成员，以便明确责任人
          + 新增验收标准：
            - [ ] 任务可分配给一名团队成员
            - [ ] 被分配人收到通知（站内 + 邮件）
            - [ ] 任务列表显示当前负责人
          + 新增业务规则：单人分配（v1），多人分配标记为后续需求

        确认写入？
你：    确认
Claude：已精炼 BR-002：团队协作
        + 新增验收标准 3 条
        + 补充用户故事和业务规则
        → /pace-biz decompose BR-002 继续分解为 PF
        → /pace-dev 开始开发
```

`refine` 只补充和深化已有需求的细节（验收标准、边界条件、用户故事），不改变需求方向。如需调整范围或优先级，使用 `/pace-change modify`。

## 子命令

### `opportunity` — 捕获业务机会

业务机会是规划流程的原始输入——市场信号、客户反馈、利益相关者请求或竞争观察。`/pace-biz opportunity` 用结构化元数据捕获它们，确保没有遗漏。

每个机会包含：
- **自动编号**（OPP-001、OPP-002……）
- **来源分类** — 客户反馈、市场调研、利益相关者请求、竞争分析、技术债务、内部倡议
- **初步评估** — Claude 建议与现有 OBJ 的关联度和潜在影响级别
- **持久化记录** — 写入 `.devpace/opportunities.md`，含创建日期和状态（new / evaluating / accepted / declined / deferred）

多个机会可以指向同一战略目标，自然形成需求优先级的信号积累。

### `epic` — 创建战略专题

专题是原始机会与可执行业务需求之间的桥梁。`/pace-biz epic` 创建结构化专题，将相关工作归入清晰的业务意图之下。

专题包含：
- **OBJ 对齐** — 服务于哪些战略目标？Claude 根据专题描述和关联机会建议映射
- **成功度量（MoS）** — 具体的、可验证的业务级完成标准
- **利益相关者**（可选）— 关键角色、关切和影响程度，在 discover/decompose 过程中渐进丰富
- **范围边界** — 明确纳入和排除的内容
- **独立文件** — 每个专题存储为 `epics/EPIC-xxx.md`，支持追溯和交叉引用

专题可从现有机会创建（`/pace-biz epic OPP-001 "标题"`）或直接创建（`/pace-biz epic "标题"`）。从机会创建时，双向链接自动记录。

### `decompose` — 分解为可交付项

分解是战略意图与执行规划的交汇点。`/pace-biz decompose` 支持两条分解路径：

1. **Epic → BR** — 将专题分解为业务需求，每个代表一项独立的业务能力或成果
2. **BR → PF** — 将业务需求分解为产品功能，每个可在一到两个迭代内交付

Claude 分析父级实体并建议分解方案——子项数量、建议标题、各自范围。用户审查、调整并确认后才创建文件。分解遵循现有价值链：新 BR 链接回所属专题，新 PF 链接回所属 BR。

**优先级框架** — 提供三种互补方法：

| 方法 | 适用场景 | 运作方式 |
|------|---------|---------|
| **Value x Effort**（默认） | 通用分解 | 高价值+低成本=P0，以此类推 |
| **MoSCoW**（`--moscow`） | 大量 BR 需分类 | Must/Should/Could/Won't → P0/P1/P2/排除 |
| **Kano**（`--kano`） | 面向用户的产品功能 | 基本型/期望型/兴奋型 → P0/P1/P2 |

默认仍为 Value x Effort，确保向后兼容。Claude 会在上下文合适时建议 MoSCoW 或 Kano（如需要分类大量 BR 或面向终端用户的功能）。用户始终可直接指定优先级。

**依赖追踪与可视化** — Epic → BR 分解时，Claude 会询问每个新 BR 是否依赖同 Epic 下的已有 BR。依赖关系记录在 Epic 文件中，在输出中可视化展示（`BR-002 ──依赖──→ BR-001`），并在 `/pace-biz align` 健康检查时浮现。自动提供基于拓扑排序的建议实施顺序。

### `refine` — 精炼已有需求

`decompose` 将实体分解为子项，而 `refine` 则深化已有 BR 或 PF 的细节——补充验收标准、边界条件、用户故事、异常处理和非功能需求。

精炼流程：
1. **定位实体并展示就绪度** — 读取当前内容，计算就绪度评分（0-100%），基于维度覆盖度（用户故事、验收标准、优先级、上游关联、异常场景、NFR）。>= 80% 为"就绪"，60-79% 为"基本就绪"，< 60% 为"需精炼"
2. **缺口分析** — Claude 识别哪些维度缺失或不完善（验收标准、用户故事、异常场景、NFR 等）
3. **引导追问** — 每轮 2-3 个针对性问题，仅针对需要补充的维度；已完善的维度自动跳过
4. **预览确认** — 变更以 diff 风格预览展示，确认后才写入
5. **建设性退出** — 用户跳过所有维度时，Claude 提供建设性引导（查看全景了解上下文、在开发中自然补充细节、稍后再回来精炼）而非简单的"无变更"信息

当 BR 涉及多步骤操作、审批流程或条件分支时，`refine` 检测到流程关键词后引导记录关键流程——编号步骤+条件分支+异常路径——写入 BR 溢出文件的可选"关键流程"section。

`refine` 与 `/pace-change modify` 的区别：refine 深化同一需求方向（内容更丰富，意图不变），modify 改变需求方向（重命名、调整范围、变更优先级）。

### `align` — 战略对齐检查

随着项目增长，业务规划容易偏移。`/pace-biz align` 对整个上游价值链做健康检查：

| 检查项 | 检测内容 |
|--------|---------|
| **OBJ 覆盖率** | 没有专题或 BR 映射的战略目标 |
| **孤立实体** | 未链接到任何上游实体的专题、BR 或 PF |
| **MoS 完整性** | 缺少可度量成功标准的专题或 BR |
| **分解空白** | 没有 BR 的专题，或没有 PF 的 BR |
| **停滞检测** | 长时间停留在"new"状态的机会 |
| **优先级分布** | 检测优先级通胀（P0 > 60%）或核心优先级缺失（无 P0），附健康比例参考 |
| **依赖健康度** | 循环依赖、关键路径瓶颈、就绪风险（BR 已排期但依赖项未完成） |
| **MoS 达成度** | 反向回溯 Epic MoS 完成度与 BR/PF 进度的对应关系；标记 BR 已完成但 MoS 未勾选的 Epic |
| **需求就绪度分布** | 计算所有 BR/PF 的就绪度评分；P0 需求平均就绪度 < 70% 时警告；推荐最需精炼的 Top 3 实体 |
| **利益相关者覆盖度** | 对有利益相关者数据的 Epic，检查 BR 是否覆盖所有高影响利益相关者的关切；标记进行中但未识别利益相关者的 Epic |

输出为简洁的对齐报告，每个问题附带内联修复命令（如 `→ /pace-biz decompose EPIC-002`、`→ /pace-change modify EPIC-001`）。

### `view` — 业务全景视图

`/pace-biz view` 渲染从机会到 CR 的完整价值流，鸟瞰业务意图如何贯穿整个系统：

```
OPP-001 "企业 SSO 需求"
  └─ EPIC-001 "企业级认证"  [OBJ-1]
       MoS：2/3 达成
       ├─ BR-003 "SSO 集成" P0
       │    ├─ PF-007 "SAML 提供商"      → CR-012 (developing)
       │    └─ PF-008 "OIDC 提供商"      → CR-013 (created)
       └─ BR-004 "会话管理" P1
            └─ PF-009 "Token 生命周期"    → CR-014 (created)
```

支持按 OBJ、专题、状态或层级深度过滤。无过滤条件时展示完整树状结构及状态指示。BR 和 PF 在有就绪度数据时展示就绪度标记（如 `BR-001：任务生命周期 P0 [就绪度 85%]`）。

**问题优先模式** — 当 3 个以上实体需要关注（空 MoS、未分解、孤立实体等）时，视图自动切换为问题优先布局：需要操作的实体归入顶部"需要关注"段并附内联修复命令，健康实体折叠展示在下方。

**内联操作引导** — 处于可操作状态的实体会显示下一步提示：空 MoS 提示补充定义、未分解的专题/BR 建议 `/pace-biz decompose`、评估中的机会建议 `/pace-biz epic`、低就绪度项建议 `/pace-biz refine`。

**角色适配展示** — 通过 `/pace-role` 设置首选角色后，view 在不改变基础结构的前提下增加角色相关列（如 Biz Owner 看每个 Epic 的 MoS 进度，PM 看 PF 完成率和依赖信息，Tester 看验收标准数量和 Gate 2 状态）。

### `discover` — 交互式需求发现

当只有一个模糊想法——"我想做一个智能客服系统"——`/pace-biz discover` 启动多轮引导对话，逐步将想法塑造为结构化的价值链。

**智能路由** — 进入发现流程前，Claude 检查用户输入是否更适合其他子命令：文件路径重定向到 `import`，代码相关关键词重定向到 `infer`。用户可选择继续 `discover`。

流程分阶段展开：
1. **目标框定**（1-2 轮）——要解决什么问题？目标用户是谁？可选地识别关键利益相关者及其关切
2. **功能头脑风暴**（2-4 轮）——用户必须能做什么？异常情况怎么办？
3. **边界定义**（1-2 轮）——明确不做什么？有什么约束？
4. **验证确认**（1 轮）——审查结构化候选树（OPP→Epic→BR→PF），调整后写入

会话状态持久化到 `.devpace/scope-discovery.md`，发现过程可跨多次对话。确认后所有候选写入对应 `.devpace/` 文件，临时会话文件删除。

**角色适配追问** — 设置首选角色后，头脑风暴阶段调整追问侧重：Biz Owner 侧重营收影响和市场拓展，PM 侧重用户场景和竞品对比，Dev 侧重技术可行性和架构约束，Tester 侧重可测试性和边界条件，Ops 侧重部署需求和基础设施影响。核心流程和产出结构不因角色而改变。

### `import` — 多源文档导入

团队在很多地方积累需求——会议纪要、用户反馈调研、竞品分析、技术债务清单。`/pace-biz import` 读取这些文档，提取需求实体，合并到现有功能树。

支持的源类型（自动检测）：
- **会议纪要** — Action Items 转为 BR/PF 候选
- **用户反馈** — 痛点转为 BR，功能请求转为 PF
- **竞品分析** — 差距功能转为 PF 候选
- **技术债务清单** — 债务项转为 PF（标记技术债务）
- **Issue 导出**（CSV/JSON）— Issues 映射为 PF/CR 候选
- **PRD / API 规格** — 同 `/pace-init --from` 的解析规则

每个提取的实体按 NEW/DUPLICATE/ENRICHMENT/CONFLICT 分类。合并计划包含来源交叉引用——每项标注来源（文件+行号）、相关已有实体、重复项相似度百分比、丰富项前后对比——帮助做出知情的 accept/reject 决策。相似度阈值默认 80%，可通过 `--threshold` 调整（如 `import doc.md --threshold 0.7`）。两层判断机制结合快速关键词筛选和语义分析处理边界情况。import 在 OPP/Epic/BR/PF 层面操作——不创建 CR。

### `infer` — 代码库功能推断

对于遗留项目或文档滞后的项目，`/pace-biz infer` 扫描代码库，反向推导功能地图：

- **结构分析** — 目录、路由、API 端点、数据模型、UI 组件
- **信号挖掘** — TODO/FIXME 密度、README 与代码漂移、构建脚本
- **Git 增强分析**（可用时）— 文件热点、共变耦合、贡献者分布

输出为三段式差距报告：
1. **未追踪功能** — 代码存在但功能树中无对应 PF
2. **未实现功能** — 功能树有 PF 但无代码实现
3. **技术债务** — TODO/FIXME 高密度区域

用户选择哪些项纳入功能树。技术债务 PF 附"（技术债务）"后缀，便于过滤。

## 角色感知

`/pace-biz` 根据通过 `/pace-role` 设置的首选角色调整行为。角色感知影响三个维度，但不改变核心工作流和产出结构：

| 维度 | 角色如何影响行为 |
|------|-----------------|
| **追问方向** | `discover` 和 `decompose` 调整追问焦点（如 Biz Owner → 营收影响，Dev → 技术约束，Tester → 验收标准） |
| **展示内容** | `view` 增加角色相关列（如 PM 看依赖信息，Ops 看发布状态） |
| **分解提示** | `decompose` 在 BR/PF 创建时追加角色相关考量 |

默认角色（Dev）不产生任何行为变化——角色适配是纯粹的增量叠加。

## 写入范围保护

`/pace-biz` 是规划域 Skill，不得修改源代码或非规划文件。运行时通过 PreToolUse Hook 强制执行：

- **允许目标**：`.devpace/` 下的所有文件（opportunities.md、epics/、project.md、requirements/、scope-discovery.md、state.md 等）
- **阻断目标**：`.devpace/` 以外的所有文件——Hook 以 exit code 2 退出并输出描述性错误信息
- **Epic 质量门**：prompt Hook 验证即将写入的 Epic 文件是否包含有效 OBJ 引用和可度量的 MoS 定义

此保护自动运行，用户无需配置或手动调用。

## 向后兼容

在 `/pace-biz` 引入之前初始化的项目——没有 `opportunities.md`、`epics/` 目录或上游 BR 链接——继续正常工作，不受任何影响。现有的 PF → CR 流程完全不变。`/pace-biz` 的能力渐进式可用：在你准备好时捕获第一个机会或创建第一个专题，价值链自然向上延伸。

## 与其他命令的集成

| 命令 | 与 `/pace-biz` 的关系 |
|------|---------------------|
| `/pace-init` | 初始化 `.devpace/` 结构。`/pace-biz` 可用后，`pace-init` 可选择性地创建 `opportunities.md` 和 `epics/` 目录。 |
| `/pace-change` | 处理 PF 级需求变更。`/pace-biz` 在上游操作——创建 BR 和 PF，由 `/pace-change` 后续管理。 |
| `/pace-plan` | 通过选择 PF 规划迭代。`/pace-biz decompose` 产出的 PF 直接进入 `/pace-plan next` 的候选池。 |
| `/pace-status` | 展示完整项目状态。有 `/pace-biz` 数据时，状态视图可包含上游上下文（专题进度、OBJ 覆盖率）。 |
| `/pace-trace` | 追溯价值链连接。`/pace-biz` 在现有 BR → PF → CR 链之上增加 OPP → EPIC → BR 层级，丰富追溯能力。 |
| `/pace-role` | 设置首选角色视角。`/pace-biz` 读取此设置，按角色调整追问方向、展示列和分解提示。 |

## 架构：统一发现引擎

三个发现型子命令（`discover`、`import`、`infer`）共享一套统一的管道架构，定义在 `biz-procedures-discovery-engine.md` 中。每个子命令扮演**输入适配器**角色——负责领域特定的提取逻辑——而共享引擎处理公共操作：

- **§1 公共前置检查**：`.devpace/` 验证、已有实体基准表构建、模式检测
- **§2 候选实体标准格式**：所有适配器在进入管道前必须产出的标准实体格式
- **§3 分析管道**：策略路由——`pass-through`（discover）、`merge`（import）、`gap`（infer）
- **§4 编号分配**：统一的 OPP/EPIC/BR/PF 编号分配，含冲突重试
- **§5 写入管道**：功能树追加、溯源标记、溢出检查、适配器收尾钩子、git commit
- **§6 下游引导**：模板化输出，附适配器特定统计行
- **§7 降级模式**：`.devpace/` 缺失或 `project.md` 为桩时的优雅降级
- **§8 lite 模式适配**：统一约束——候选限制为 PF 类型，父实体限制为 OBJ

此架构确保一致性（如溯源标记格式、编号分配算法），同时保持各适配器的领域特定逻辑（discover 的多轮对话、import 的源类型检测、infer 的代码结构分析）。
