# 迭代规划（`/pace-plan`）

`/pace-plan` 是 devpace 的迭代规划 Skill。它管理迭代的完整生命周期——从关闭当前迭代到规划下一个迭代——补齐产品闭环中的"规划→执行→回顾"循环。与处理产品功能（PF）级需求变更的 `/pace-change` 不同，`/pace-plan` 在迭代范围层级操作，决定每个迭代纳入哪些 PF 并管理容量约束。

## 快速上手

```
1. /pace-plan              --> 评估当前迭代状态,建议下一步操作
2. /pace-plan next         --> 规划新迭代(含 Plan Proposal)
3. /pace-plan close        --> 关闭当前迭代(自动回顾)
4. /pace-plan adjust       --> 迭代中途范围调整
5. /pace-plan health       --> 展示迭代健康度指标
```

## 子命令

| 子命令 | 使用场景 | 功能 |
|--------|---------|------|
| `(空)` | 检查迭代状态 | 评估完成率并建议下一步(关闭或继续) |
| `next` | 启动新迭代 | 规划下个迭代,含引导流程和 Plan Proposal |
| `close` | 结束当前迭代 | 关闭迭代,自动轻量回顾并归档 |
| `adjust` | 迭代中途 | 增删 PF 或调整当前迭代优先级 |
| `health` | 监控进展 | 展示健康度指标(完成率 vs 时间,范围稳定性) |

## 核心特性

### 空功能树引导式规划

当 `project.md` 处于桩状态(无活跃或待做 PF)时，`/pace-plan next` 进入引导规划模式：

- **用户目标获取**：询问"这个迭代想做什么？用 1-2 句话描述目标"
- **PF 推断**：从用户描述提取关键词并推断 PF(类似 `pace-init --from` 逻辑)
- **自动填充**：将推断的 PF 写入 `project.md`，附带 `<!-- source: claude -->` 溯源标记
- **确认调整**：展示推断的 PF 供用户确认或调整后继续

这消除了冷启动问题——无需手动编写 PF 就能直接进入迭代规划。

### Plan Proposal（智能规划建议）

不再要求用户逐项手动选择 PF，Claude 生成完整的迭代规划建议方案：

**生成逻辑**：
1. **目标归纳**：从候选 PF 的业务需求归属自动推导迭代目标(如"完成用户认证相关功能")
2. **PF 选择策略**(基于优先级排序)：
   - a. 上迭代延后的 PF(最高优先级)
   - b. 依赖已满足的 PF
   - c. 遵循 `project.md` 中的排列顺序(隐含优先级)
   - d. 容量上限：不超过上一迭代实际完成数(首次迭代默认 3-5 个 PF)
3. **工作量估算**：为每个 PF 附加工作量估算(有历史数据用历史数据,无则用启发式)
4. **容量评估**：汇总预估值并与迭代周期(如有)对比

**建议方案格式**：
```
建议方案：
  目标：[自动归纳的 1 句话目标]
  纳入 PF（共 N 个）：
    [P0] PF-001 xxx — 预估 M（2-4 CR）[上迭代延后]
    [P1] PF-003 xxx — 预估 S（1-2 CR）
    [P2] PF-005 xxx — 预估 L（4-7 CR）⚠️ 高风险
  总预估工作量：约 X 个工作会话
  周期：[起止日期，如用户之前有设定]

确认这个方案？还是需要调整？
```

**用户响应分支**：
- "好 / 确认 / 可以" → 直接采用建议方案
- "调整：去掉 X，加上 Y" → 增量修改后重新展示
- 提供完整替代方案 → 使用用户自定义方案
- 指定优先级调整 → 更新 P0/P1/P2 标记

### 启发式估算(无历史数据)

对于首个迭代或无历史数据时，估算回退到启发式规则：

| PF 复杂度 | 启发式规则 | 预估 CR 数 | 工作量(会话) |
|-----------|-----------|-----------|-------------|
| **S**(简单) | 无验收标准或 <1 条标准 | 1-2 CR | ~1-2 会话 |
| **M**(中等) | 1-2 条验收标准 | 2-4 CR | ~2-4 会话 |
| **L**(复杂) | ≥3 条验收标准 | 4-7 CR | ~4-7 会话 |

- **触发条件**：无 `dashboard.md` 且无 `iterations/iter-*.md` 文件
- **首次迭代上限**：默认 3-5 个 PF 避免过度承诺
- **自动升级**：首个迭代完成后自动切换为历史数据估算模式

启发式估算会明确标注："启发式估算(无历史数据)。首个迭代结束后自动切换为历史数据估算"。

### 历史速度限制

当存在历史数据时：

- **数据源**：`.devpace/metrics/dashboard.md` 中的平均 CR 周期和迭代速度
- **容量上限**：若速度 < 1.0，建议纳入不超过 `上一迭代实际完成数` 个 PF
- **警告**："历史迭代速度 X%，建议纳入不超过 N 个 PF"
- **超出预警**：若预估工作量超过迭代周期，附加建议："预估工作量可能超出迭代周期，建议缩减 N 个 PF 或延长迭代"

### 候选 PF 按业务需求分组

当可选 PF 数量 ≥5 时：

- **分组**：按 `project.md` 中 PF 归属的父级业务需求(BR)分组
- **摘要输出**：展示分组摘要(每组 PF 数 + 概要说明)
- **下钻查看**：用户可指定 BR 展开查看该组内详细 PF 列表
- **平铺列表**：当 <5 个 PF 时直接全部列出无需分组

这降低了从大量 PF 池中选择时的认知负荷。

### PF 依赖校验

如果 PF 规格中有 `dependencies` 字段：

- **检查**：验证依赖的 PF 是否已完成或在当前迭代中排序更前
- **提醒**：若依赖未满足 → "PF-002 依赖 PF-001([原因])。建议先纳入 PF-001 或确认可并行"
- **跳过**：若无 `dependencies` 字段则跳过校验

这防止规划具有未满足前置条件的 PF。

### 风险集成(pace-guard)

当 `.devpace/risks/` 存在时：

- **扫描**：通过 `affected_pf` 字段匹配候选 PF 与 open 风险条目
- **高风险**：标注 "⚠️ 高风险" + 估算增加 30% buffer(如 M 3CR → 4CR)
- **中风险**：标注 "⚡ 中风险"(不调整估算)
- **降级**：若无匹配风险或 `risks/` 不存在则静默跳过

风险信息在规划过程中自动展示无需手动查询。

### 上迭代回顾建议直联

规划新迭代时：

- **数据源**：读取最新归档迭代 `iter-N.md`(N=最新)的回顾 section
- **展示**：若存在"下迭代建议"section，展示给用户："上一迭代回顾建议：[内容]"
- **跳过**：若无回顾 section 或无建议则静默跳过

这闭合了从回顾到规划的反馈循环。

### 关闭时自动轻量回顾

关闭迭代时(`/pace-plan close`)：

1. **归档**：将 `current.md` 重命名为 `iter-N.md` 并填写完成快照
2. **提取指标**(自动,无需用户输入)：
   - PF 完成率(计划 vs 实际)
   - 平均 CR 周期(created→merged 天数)
   - 迭代速度(实际完成 PF 数 / 计划 PF 数)
3. **更新仪表板**：在 `.devpace/metrics/dashboard.md` 追加或更新这 3 项指标
4. **输出**："本迭代：X% 完成率，N 天平均 CR 周期，Y 速度"
5. **建议完整回顾**：推荐运行 `/pace-retro` 做全面分析(不自动调用)

这确保 `dashboard.md` 基线数据永不缺失，同时完整回顾仍为可选。

### 迭代中途范围调整(adjust)

规划后对迭代范围的变更：

**与 /pace-change 的边界**：
- `/pace-change`：PF 级需求变更(add/pause/resume/modify 功能特性)
- `/pace-plan adjust`：迭代范围级调整(本迭代纳入哪些 PF)

**操作流程**：
1. **展示当前状态**：目标、PF 列表(含完成状态)、剩余容量
2. **调整操作**(通过 `AskUserQuestion`)：
   - **增加 PF**：从 `project.md` 待做 PF 中选择,附带工作量估算
   - **移除 PF**：将未开始的 PF 移回待做池(已开始的建议用 `/pace-change pause`)
   - **调整优先级**：重排当前迭代内的 P0/P1/P2 优先级
3. **每次操作后重算容量**：
   - 汇总当前迭代所有 PF 的预估工作量
   - 与迭代周期(如有)对比
   - **超出警告**："调整后工作量 X 会话,超出剩余容量 Y。建议移除低优先级 PF"
4. **更新 `current.md`**：PF 列表 + 优先级 + 变更记录追加调整原因
5. **不关闭**：迭代保持打开,不归档

### 迭代健康度指标(health)

展示当前迭代的健康指标：

**展示指标**：
1. **完成率 vs 时间进度**：PF 完成率 / 迭代时间消耗比(若设定周期);否则仅报告 PF 完成率
2. **范围变更次数**：`current.md` 变更记录条数(反映范围稳定性)
3. **PF 状态分布**：✅ 完成 / 🔄 进行中 / ⏳ 待开始 / ⏸️ 暂停 各多少
4. **速度趋势**：与上一迭代速度对比(若 `dashboard.md` 存在)

**健康度评估**：
- **健康**："迭代正常：X% 完成率,按计划推进"
- **风险**："可能延期：X% 完成但时间已过 Y%。建议 `/pace-plan adjust` 缩减范围"
- **范围不稳定**："范围变更 N 次。关注需求稳定性"

**pace-pulse 集成**：健康度低于阈值(完成率/时间进度 < 0.5)时,标记为 pulse 可检测信号。

## 使用场景

### 场景 1：首次迭代规划(空功能树)

```
你：    /pace-plan next
Claude：未找到活跃或待做 PF。让我们从你的目标开始。
        这个迭代想做什么？用 1-2 句话描述目标。

你：    构建用户认证,包括登录和注册
Claude：基于你的描述,我推断了以下功能：
        - PF-001：邮箱密码登录
        - PF-002：邮箱验证注册
        - PF-003：密码重置流程
        确认还是需要调整？

你：    看起来不错
Claude：PF 已添加到 project.md。

        建议方案：
          目标：完成用户认证功能
          纳入 PF（共 3 个）：
            [P0] PF-001 用户登录 — 预估 M（2-4 CR）
            [P0] PF-002 用户注册 — 预估 M（2-4 CR）
            [P1] PF-003 密码重置 — 预估 S（1-2 CR）
          总预估工作量：约 7-10 会话
          周期：[接下来 2 周,如已指定]

        确认这个方案？

你：    好
Claude：迭代已规划并保存到 .devpace/iterations/current.md。
        最高优先级：PF-001 用户登录。现在就开始？

你：    开始吧
Claude：建议运行：/pace-dev "实现 PF-001"
```

### 场景 2：基于历史数据规划

```
你：    /pace-plan next
Claude：上一迭代完成 4/5 个 PF（80% 速度）。
        2 个 PF 从上迭代延后。
        上迭代回顾建议："聚焦小型 PF 以提高完成率"。

        建议方案：
          目标：完成延后功能 + 结算流程
          纳入 PF（共 4 个）：
            [P0] PF-005 用户资料(延后) — 预估：2.5 会话(历史均值)
            [P0] PF-008 邮件通知(延后) — 预估：1.8 会话
            [P1] PF-012 购物车 — 预估：3.2 会话
            [P2] PF-014 支付集成 — 预估：4.5 会话 ⚠️ 高风险(+30%)
          总计：~12 会话 vs 容量 15 会话

        确认还是调整？

你：    去掉 PF-014,风险太高
Claude：已调整。移除 PF-014。

        更新方案：
          纳入 PF（共 3 个）：
            [P0] PF-005 用户资料 — 2.5 会话
            [P0] PF-008 邮件通知 — 1.8 会话
            [P1] PF-012 购物车 — 3.2 会话
          总计：~7.5 会话(远低于容量)

        确认？

你：    好
Claude：迭代已保存。从 PF-005 开始？
```

### 场景 3：迭代中途调整

```
你：    /pace-plan adjust
Claude：当前迭代状态：
          目标：完成用户认证
          进度：2/4 个 PF 完成（50%）
          剩余：PF-003(进行中), PF-004(待做)
          容量：已用约 5 会话,剩余约 3 会话

        需要什么调整？

你：    去掉 PF-004,换成 PF-007
Claude：检查容量...
        - 移除 PF-004 释放 ~2 会话
        - 增加 PF-007 需要 ~3 会话
        - 新总计：剩余约 6 会话

        调整可行。确认？

你：    确认
Claude：已更新 current.md。
        变更记录："迭代中途调整：PF-004 → PF-007(优先级调整)"
```

### 场景 4：迭代健康检查

```
你：    /pace-plan health
Claude：迭代健康度报告：
          时间进度：60%（10 天迭代中已过 6 天）
          PF 完成率：40%（5 个中完成 2 个）
          状态分布：✅2 🔄1 ⏳2
          范围变更：0 次(稳定)

        ⚠️ 有风险：完成率(40%)落后于时间进度(60%)。
        需要 1.5 倍当前速度才能按时完成。

        建议：
        1. 运行 /pace-plan adjust 缩减范围
        2. 检查进行中的 CR 是否受阻
        3. 考虑延长迭代周期
```

### 场景 5：迭代关闭

```
你：    /pace-plan close
Claude：正在关闭当前迭代...

        最终状态：
          计划：5 个 PF
          完成：4 个 PF（80%）
          延后：1 个 PF（PF-015）

        自动回顾指标：
          - PF 完成率：80%
          - 平均 CR 周期：2.3 天
          - 迭代速度：0.8

        指标已更新到 dashboard.md。
        迭代已归档到 iter-03.md。

        建议运行 /pace-retro 做完整回顾分析。

        准备好规划下一个迭代了吗？
```

## 与其他命令的集成

| 命令 | 与 `/pace-plan` 的关系 |
|------|----------------------|
| `/pace-init` | 初始化迭代运行所在的 `.devpace/` 结构 |
| `/pace-change` | 处理 PF 级需求变更;容量超出时可能触发 `/pace-plan adjust` 建议 |
| `/pace-dev` | 执行阶段;`/pace-plan` 规划后引导到 `/pace-dev` |
| `/pace-retro` | 完整回顾;`/pace-plan close` 执行轻量自动回顾,建议 `/pace-retro` 深度分析 |
| `/pace-status` | 展示当前迭代进度及整体项目状态 |
| `/pace-guard` | 风险扫描集成到规划;高风险 PF 获得估算 buffer |
| `/pace-pulse` | 健康度指标反馈 pulse 信号;低健康度触发 pulse 检测 |
| `/pace-next` | 迭代结束或无活跃工作时可能建议 `/pace-plan next` |

## 降级行为

当某些数据缺失时，`/pace-plan` 优雅降级：

| 缺失数据 | 降级处理 |
|---------|---------|
| `project.md` 空 | 引导规划模式:获取用户目标→推断 PF→填充功能树 |
| 无历史数据 | 启发式估算(基于验收标准数量 S/M/L 分级) |
| 无迭代周期 | 容量检查跳过时间基准警告,仅展示 PF 数量限制 |
| 无 `dashboard.md` | 跳过速度基准限制,用启发式上限(首次迭代 3-5 个 PF) |
| 无上迭代回顾 | 跳过建议展示,继续正常规划 |
| 无 `.devpace/risks/` | 跳过风险集成,无高风险标记 |
| `current.md` 空 | `/pace-plan adjust` / `health` 提示"无活跃迭代,请先运行 `/pace-plan next`" |

## Plan Proposal vs 手动选择

**传统迭代规划**(手动)：
1. 列出所有候选 PF
2. 用户手动选择纳入哪些 PF
3. 用户设置优先级
4. 用户估算工作量(或跳过)
5. 工具生成 `current.md`

**Plan Proposal**(devpace)：
1. 工具分析候选 PF + 历史数据 + 约束
2. 工具生成完整建议方案(含推理过程)
3. 用户用自然语言确认或调整
4. 工具应用调整并重新生成方案
5. 批准后工具生成 `current.md`

**优势**：
- **更快**：一次确认 vs 多次选择
- **有引导**：用户看到具体方案,调整比从头构建容易
- **有依据**：方案自动反映速度限制、依赖、风险
- **灵活**：仍支持完全自定义方案
