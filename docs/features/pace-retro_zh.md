# 迭代回顾与度量（`/pace-retro`）

`/pace-retro` 是 devpace 的回顾与度量分析 Skill。它生成迭代回顾报告、更新度量仪表盘，并提供多粒度分析视角——从迭代中期快速检查到完整回顾（含经验沉淀和跨迭代趋势分析）。通过 `pace-analyst` Agent 在 fork 上下文中运行，Agent 具备项目级记忆，跨会话加速分析。

`/pace-plan close` 执行的是轻量自动回顾（3 项基准指标），而 `/pace-retro` 提供深度多维分析——含改进建议、经验沉淀和知识闭环。

## 快速上手

```
1. /pace-retro              --> 完整迭代回顾（行动摘要 + 维度详情）
2. /pace-retro update       --> 仅刷新度量数据（带变化反馈）
3. /pace-retro focus quality --> 聚焦某维度深入分析
4. /pace-retro compare      --> 当前迭代 vs 上一迭代对比
5. /pace-retro history      --> 跨 3+ 迭代趋势总览
6. /pace-retro mid          --> 迭代中期检查（轻量，不更新基准线）
7. /pace-retro accept       --> 确认回顾建议的操作（MoS 更新等）
```

## 子命令

| 子命令 | 使用场景 | 功能 |
|--------|---------|------|
| `(空)` | 迭代末尾 | 完整回顾：收集数据、更新 dashboard、生成报告、沉淀经验、生成传递清单 |
| `update` | 任何时候 | 仅刷新 dashboard 度量数据；输出变化摘要，展示哪些指标变了、变了多少 |
| `focus <维度>` | 需要深入分析 | 单维度分析：`quality` / `delivery` / `dora` / `defects` / `value` / `knowledge` |
| `compare` | 迭代关闭后 | 当前迭代 vs 上一迭代的关键指标对比 |
| `history` | 趋势分析 | 基于 dashboard 历史快照的 3+ 迭代趋势总览 |
| `mid` | 迭代中期 | 轻量检查；不更新 dashboard 基准线 |
| `accept` | 回顾之后 | 执行上次回顾建议的操作（如标记 MoS 为已达标） |

## 核心特性

### 两层输出结构

每份回顾报告采用两层设计，解决信息过载问题：

- **第一层——行动摘要（约 10 行，必输出）**：关键指标 + 趋势箭头 + 关注项 + 亮点 + 核心建议 + 待确认操作。人类用户看完摘要即可获得 80% 的关键信息。
- **第二层——维度详情（紧跟摘要）**：交付、质量、缺陷、价值、周期维度，加上条件段（DORA/风险趋势/知识驱动改进/知识库健康度），仅在数据充分时出现。

### 多粒度分析

从最轻到最深，按需选择分析层级：

| 粒度 | 命令 | 范围 | 更新 Dashboard? |
|------|------|------|----------------|
| 度量刷新 | `update` | 仅数据，不生成报告 | 是 |
| 中期检查 | `mid` | 轻量报告，不含经验沉淀 | 否 |
| 聚焦深入 | `focus <维度>` | 单维度分析 | 是 |
| 完整回顾 | `(空)` | 全维度 + 经验 + 传递清单 | 是 |
| 跨迭代对比 | `compare` | 两迭代 delta 对比 | 否 |
| 趋势总览 | `history` | 3+ 迭代趋势 | 否 |

### 经验沉淀透明化

完整回顾后输出「本次学习」透明段，展示提取了哪些经验：

```
### 本次回顾经验提取

**提交给知识库的 pattern**：
1. 🆕 [改进] "边界条件测试可预防 60% 的缺陷"（置信度 0.5，待验证）
2. ✅ [验证] "Auth 变更需要集成测试"（置信度 0.7→0.8，+1 次验证）
3. ⚡ [冲突] "快速迭代优先于全面测试"——与已有 pattern #5 存在张力

处理状态：已提交 pace-learn 管道（去重检查 + 写入）
```

学习过程透明可见——用户一目了然看到新 pattern、验证结果和冲突。

### 跨迭代趋势分析

dashboard 历史快照跨迭代保留，支持 `compare` 和 `history` 子命令：

- **compare**：当前迭代与上一迭代的关键指标并排对比，高亮改善和退步项
- **history**：3+ 迭代趋势线，含趋势判定（持续改善/改善中/波动/需关注/稳定）

### 迭代传递清单

完整回顾生成结构化传递清单，写入 `iterations/current.md`，供 `/pace-plan next` 消费：

- 继承到下个迭代的改进项（含数据支撑）
- 度量基线（当前迭代速度、平均周期）
- 未完成 PF 及延期原因
- 下个迭代建议关注维度

用结构化数据（而非自由文本）闭合 retro→plan 反馈循环。

### 角色适配输出

五种角色（Biz Owner / PM / Dev / Tester / Ops）根据各自关注点重新组织摘要和详情：

| 角色 | 摘要侧重 | 措辞风格 |
|------|---------|---------|
| Biz Owner | MoS 达成率 + 价值交付 + OBJ 进度 | 业务术语，关联目标编号 |
| PM | 迭代完成率 + CR 周期 + 范围变更次数 | 功能维度，关注交付节奏 |
| Dev（默认） | 质量门通过率 + 技术债 + 代码质量 | 技术术语，关注实现质量 |
| Tester | 缺陷分布 + 逃逸率 + 打回率 + 覆盖率趋势 | 质量维度，关注缺陷预防 |
| Ops | DORA 代理度量 + 部署频率 + MTTR + 稳定性 | 运维术语，关注交付可靠性 |

底层数据收集所有角色完全一致——仅调整报告排序和措辞。

### DORA 代理度量

当项目使用 Release 管理（`.devpace/releases/` 存在）时，报告包含 DORA 段，基于 `.devpace/` 数据计算代理值：

| 指标 | 代理计算方式 | 分级 |
|------|------------|------|
| 部署频率 | Release 关闭频率 | Elite(>=1次/周) / High(1次/2周) / Medium(1次/月) / Low |
| 变更前置时间 | CR 创建到发布的平均时间 | Elite(<=1天) / High(<=3天) / Medium(<=7天) / Low |
| 变更失败率 | 有缺陷 Release / 总 Release | Elite(<=15%) / High(<=30%) / Medium(<=45%) / Low |
| MTTR | 缺陷 CR 创建到合并的平均时间 | Elite(<=1天) / High(<=3天) / Medium(<=7天) / Low |

每项指标含趋势箭头（与上次回顾对比）。数据不足的指标标注具体原因（如"未使用 Release 管理"），而非省略。

### 报告质量自评

完整回顾末尾附加三维质量自评：

| 维度 | 评估内容 |
|------|---------|
| 数据充分度 | 底层 CR 事件数据的完整程度（5/5 = 所有 CR 有完整事件表） |
| 趋势可信度 | 历史迭代数据量（5/5 = 5+ 迭代历史） |
| 建议可执行度 | 建议的具体性和可实施程度（5/5 = 所有建议有具体行动 + 量化目标） |

帮助用户校准对报告结论的信任度。

### Agent 记忆利用

`pace-analyst` Agent 使用项目级持久化记忆：

- **启动时**：读取上次关键发现、累计趋势方向和项目分析模型——加速定位本次变化
- **结束时**：保存本次关键发现（最多 3 条）、更新趋势方向和分析模型

每次回顾在上次基础上递进，减少冷启动分析时间。

### MoS 确认流程

`/pace-retro accept` 执行上次回顾建议的操作——通常是在 `project.md` 中标记已达标的 MoS（成效指标）。回顾会识别已达到目标值的指标并建议更新，但不会在没有人类确认的情况下自动修改业务数据。

### update 模式反馈

`/pace-retro update` 不只是静默刷新——它输出变化摘要：

```
✅ dashboard.md 已更新（3 项变化）：
- 一次通过率：70% → 85% ↑
- 平均 CR 周期：3.0 天 → 1.5 天 ↑
- 迭代速度：3 PF → 5 PF ↑

无变化的指标已保持原值。
```

首次度量时输出基准线建立消息。

### retro-learn 双向闭环

回顾与学习形成知识闭环：

- **retro -> learn**：批量验证本迭代引用过的 pattern（确认或质疑置信度）
- **learn -> retro**：defense pattern 汇总反馈宏观改进方向，输出到报告的"知识驱动改进"段

### 知识库元分析

当知识库积累 5+ 条 pattern 时，回顾报告包含"知识库健康度"段：

- 按类型和状态统计 pattern 数量（Active / Dormant / Archived）
- 各类型平均置信度
- Top-3 高产标签和高效标签
- 未解决冲突对列表
- 增长区和衰退区（哪些类型/标签在增长或衰减）

### 缺陷根因分类

缺陷按 6 种基础分类（逻辑错误、边界条件遗漏、集成问题、回归、配置错误、未知）加上项目自定义分类（来自 `project.md` 或 `insights.md`）。当"未知"占比超过 30% 时，报告建议用户细化分类体系。

### 验证-审批一致率分析

当自动验证（pace-test accept）与人类审批（Gate 3）的一致率低于 80% 时，报告包含逐 case 的差异分析，推断不一致原因，并将改进建议反馈给验证策略。

## 输出示例

### 完整回顾——行动摘要

```
## 迭代回顾摘要：Sprint 3

📊 交付 3/5（60%）| 质量 85% ↑ | 周期 1.5 天 ↑
⚠️ 关注：打回率 15% ↑（上迭代 8%），2 个 major defect 未闭合
✅ 亮点：一次通过率 85% 创新高，DORA 前置时间 Elite 级
💡 建议：下迭代聚焦缺陷预防（边界条件遗漏占 60%）

📌 **待确认操作**（2 项）：
- MoS 达标更新：2 个指标建议勾选（`/pace-retro accept` 确认）
  · "Gate 一次通过率 > 80%"→ 当前 85%，已达标
  · "平均 CR 周期 < 2 天"→ 当前 1.5 天，已达标

详情见下方各段...
```

### update 模式——变化反馈

```
✅ dashboard.md 已更新（2 项变化）：
- 一次通过率：80% → 85% ↑
- 缺陷数量：3 → 5 ↑（需关注）

无变化的指标已保持原值。
```

### 中期检查

```
## 中期检查：Sprint 3

📊 交付 2/5（40%），时间已过 60%
⚠️ 关注：完成率落后于时间进度，可能需要调整范围
✅ 质量 90% 一次通过（健康）

快速建议：
1. 检查进行中 CR 是否受阻
2. 考虑运行 /pace-plan adjust 缩减范围
3. 剩余时间优先处理 P0 功能
```

### compare 模式

```
## 迭代对比：Sprint 2 vs Sprint 3

| 指标 | Sprint 2 | Sprint 3 | 变化 | 趋势 |
|------|----------|----------|------|------|
| PF 完成率 | 4/5（80%） | 3/5（60%） | -20% | 需关注 |
| 一次通过率 | 80% | 85% | +5% | 改善中 |
| 平均 CR 周期 | 2.5 天 | 1.5 天 | -1.0 天 | 改善中 |
| 打回率 | 8% | 15% | +7% | 需关注 |

关键变化：
- 交付速度放缓（完成 PF 减少），但单 CR 质量提升
- 建议：保持质量改善，排查交付瓶颈
```

## 使用场景

### 场景 1：迭代末尾完整回顾

```
你：    /pace-retro
Claude：## 迭代回顾摘要：Sprint 3

        📊 交付 3/5（60%）| 质量 85% ↑ | 周期 1.5 天 ↑
        ⚠️ 关注：打回率 15%，2 个 major defect 未闭合
        ✅ 亮点：一次通过率 85% 创新高
        💡 建议：下迭代聚焦缺陷预防

        📌 待确认操作（2 项）：
        - MoS："Gate 一次通过率 > 80%"→ 当前 85%，已达标
        - MoS："平均 CR 周期 < 2 天"→ 当前 1.5 天，已达标

        [维度详情、DORA 度量、经验沉淀、传递清单...]

你：    /pace-retro accept
Claude：已确认 2 项操作：
        - project.md：已勾选"Gate 一次通过率 > 80%"
        - project.md：已勾选"平均 CR 周期 < 2 天"
```

```

        Gate 一次通过率：85%（趋势：连续 3 迭代改善）
        打回率：15%（趋势：需关注，上迭代 8%）
        测试基准线：94% 通过率，12 秒执行时间

        - 60% 边界条件遗漏
        - 25% 集成不匹配
        - 15% 逻辑错误

```

```
Claude：## 趋势总览（Sprint 1-3）

        | 指标 | Sprint 1 | Sprint 2 | Sprint 3 | 趋势 |
        |------|----------|----------|----------|------|
        | 一次通过率 | 70% | 80% | 85% | ↑↑ 持续改善 |
        | 平均 CR 周期 | 3.0 天 | 2.5 天 | 1.5 天 | ↑↑ 持续改善 |

        | 迭代速度 | 3 PF | 4 PF | 5 PF | ↑↑ 持续提升 |

        关键洞察：
        - 质量和速度持续改善
        - 打回率上升——需检查 Review 标准是否对齐
```

## 与其他命令的集成

| 命令 | 与 `/pace-retro` 的关系 |
|------|----------------------|
| `/pace-plan close` | 执行轻量自动回顾（3 项基准指标）；`/pace-retro` 提供完整深度分析 |
| `/pace-plan next` | 消费 `/pace-retro` 写入的迭代传递清单 |
| `/pace-learn` | 双向：retro 通过 pace-learn 管道沉淀经验；retro 批量验证已有 pattern |
| `/pace-status` | 展示当前度量快照（只读）；retro 做分析并生成建议 |
| `/pace-test` | 验证-审批一致率分析的改进建议反馈给 pace-test 验证策略 |
| `/pace-role` | 角色设置适配 retro 输出侧重和措辞 |
| `/pace-next` | 策略信号（回顾提醒、MoS 达成）指向 `/pace-retro` |
| `/pace-pulse` | retro 度量反馈 pulse 信号，用于会话启动检测 |
| `/pace-guard` | 风险数据反馈回顾报告的"风险趋势"条件段 |

## 条件段（渐进暴露）

多个报告段仅在数据充分时出现：

| 段落 | 出现条件 | 缺失时 |
|------|---------|--------|
| DORA 代理度量 | `.devpace/releases/` 存在且有 Release 文件 | 静默跳过 |
| 风险趋势 | `.devpace/risks/` 存在 | 静默跳过 |
| 知识驱动改进 | `insights.md` 存在且有 defense pattern | 静默跳过 |
| 知识库健康度 | `insights.md` 有 >= 5 条 pattern | 静默跳过 |
| 学习效能 | 有 pattern 引用数据 | 静默跳过 |
| 验证-审批一致率分析 | 一致率 < 80% | 静默跳过 |
| 缺陷根因报告 | 有已合并的 defect CR 且含根因数据 | 静默跳过 |

不报错、不显示空段——数据不足的段落不可见。
