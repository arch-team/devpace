# Test Strategy and Quality Management (`/pace-test`)

devpace testing goes beyond "run tests and check results." `/pace-test` manages a **requirements-driven** testing lifecycle: Product Feature (PF) acceptance criteria define what to verify, test strategies map criteria to test types, coverage analysis measures requirements coverage (not just code coverage), and AI-powered acceptance verification provides per-criterion evidence with code-level references. The result is a testing process that traces back to business intent at every step.

## Prerequisites

| Requirement | Purpose | Required? |
|-------------|---------|:---------:|
| `.devpace/` initialized | Project structure, PF definitions, CR tracking | Yes |
| `.devpace/rules/checks.md` | Configured test commands for `run` and `dryrun` | Recommended |
| `.devpace/rules/test-strategy.md` | PF-to-test mapping (generated by `strategy`) | Recommended |

> **Graceful degradation**: Without `.devpace/`, the skill falls back to code-only mode (auto-detects test commands). Without `checks.md`, common test commands are auto-detected from `package.json`, `pyproject.toml`, `go.mod`, or `Cargo.toml`. Without `test-strategy.md`, requirements-level analysis still works by reading PF criteria directly from `project.md`.

## Quick Start

```
1. /pace-test strategy              --> Map PF acceptance criteria to test types
2. /pace-test generate PF-001 --full --> Generate test cases from criteria
3. /pace-test coverage              --> Analyze requirements coverage gaps
4. /pace-test                       --> Run all configured tests
5. /pace-test accept                --> AI acceptance verification
6. /pace-test report                --> Generate review-ready summary
```

Daily workflow: most sessions only need `/pace-test` (run) + `/pace-test accept`. Use `impact --run` during iterations to quickly execute affected tests.

## Command Reference

### Execution Layer

#### `run` (default, no arguments)

Execute all configured test commands and produce a structured report.

**Syntax**: `/pace-test`

Reads `checks.md` for test commands (falls back to auto-detection), executes them respecting dependency order, and produces a pass/fail report grouped by Gate. When a test fails, analyzes the recent `git diff` to infer probable root causes. Automatically appends CI run status when `.devpace/integrations/config.md` is configured. See [test-procedures-core.md](../../skills/pace-test/test-procedures-core.md) for detailed steps.

**Output example**:
```
| # | Check    | Gate   | Status | Time | Notes               |
|---|----------|--------|--------|------|---------------------|
| 1 | npm test | Gate 1 | PASS   | 3.2s | --                  |
| 2 | eslint . | Gate 1 | FAIL   | 1.1s | 3 errors in auth.ts |
Summary: 1/2 passed
Suggestion: run /pace-test dryrun 1 to pre-check Gate 1
```

#### `generate`

Create test cases from PF acceptance criteria.

**Syntax**: `/pace-test generate [PF-title] [--full]`

Default (skeleton) generates scaffolding with `// TODO` placeholders. `--full` generates complete implementations with assertions, boundary conditions, and error paths (marked `// REVIEW: AI-generated`). Auto-registers into `test-strategy.md`. In TDD contexts, appends a Red-Green-Refactor reminder. See [test-procedures-generate.md](../../skills/pace-test/test-procedures-generate.md) for steps.

**Output example**:
```
Generated 4 test cases [full] for PF "User Authentication" (4 criteria):
1. Users can log in --> test_login_email_password [3 assertions + 2 boundary]
2. Failed login error --> test_login_failure_message [2 assertions]
File: tests/test_auth.py | Mode: full (review REVIEW markers)
```

#### `dryrun`

Simulate a Gate check without triggering state transitions.

**Syntax**: `/pace-test dryrun [1|2|4]`

Executes the full Gate check flow (command checks, intent checks, adversarial review for Gate 2) in read-only mode. No CR state changes, no event log entries. See [test-procedures-advanced.md](../../skills/pace-test/test-procedures-advanced.md) for steps.

**Output example**:
```
Gate 1 Dry-Run: 1 PASS / 1 FAIL
Prediction: Gate will FAIL
Fix: resolve eslint errors, then re-run /pace-test dryrun 1
```

### Strategy Layer

#### `strategy`

Generate a systematic test strategy from PF acceptance criteria.

**Syntax**: `/pace-test strategy`

For each acceptance criterion, recommends a primary test type (unit, integration, E2E, performance, security, accessibility, manual) plus 0-2 auxiliary types. Matches existing test files using name and content analysis. Outputs a test pyramid health assessment and implementation guidance. Persists to `.devpace/rules/test-strategy.md`. See [test-procedures-strategy-gen.md](../../skills/pace-test/test-procedures-strategy-gen.md) for steps.

**Output example**:
```
Strategy: 3 PFs, 12 criteria --> 5 unit / 3 integration / 2 E2E / 1 perf [+security] / 1 manual
Covered: 7 | To build: 5
Pyramid health: needs attention (unit 42%, below 50% threshold)
Next: /pace-test generate [PF] to create tests for uncovered criteria
```

#### `coverage`

Analyze how many PF acceptance criteria have corresponding tests.

**Syntax**: `/pace-test coverage`

Cross-references PF criteria against `test-strategy.md`, `checks.md`, and scanned test files. Optionally collects code coverage as a supplementary signal (Jest, pytest, go test, cargo tarpaulin). When `test-strategy.md` contains a threshold section, checks values against targets. See [test-procedures-coverage.md](../../skills/pace-test/test-procedures-coverage.md) for steps.

**Output example**:
```
| PF     | Feature        | Criteria | Covered | Rate |
|--------|----------------|----------|---------|------|
| PF-001 | Authentication | 5        | 3       | 60%  |
| PF-002 | Data Import    | 3        | 0       | 0%   |
Requirements coverage: 3/8 (38%)
Code coverage (supplementary): 72% line, 58% branch (Jest)
```

#### `impact`

Analyze change impact and recommend regression scope.

**Syntax**: `/pace-test impact [CR-ID] [--run]`

Extracts changed files from `git diff`, builds a file-to-PF reverse mapping, identifies directly and indirectly affected PFs, and assigns a risk level. With `--run`, automatically executes "must-run" tests after analysis. See [test-procedures-impact.md](../../skills/pace-test/test-procedures-impact.md) for steps.

**Output example**:
```
CR-005 "Add CSV export" | Scope: 6 files / 2 modules | Risk: MEDIUM
| PF     | Feature     | Impact  | Suggested Tests       |
|--------|-------------|---------|------------------------|
| PF-002 | Data Import | Direct  | test_import, test_csv  |
| PF-003 | Reports     | Indirect| Spot-check recommended |
Must-run: test_import, test_csv
```

### Analysis Layer

#### `report`

Generate a human-readable test summary for review or release.

**Syntax**: `/pace-test report [CR-ID|REL-xxx]`

**CR mode** (default): aggregates Layer 1 (test execution), Layer 2 (requirements coverage), Layer 3 (AI acceptance verification) and produces a merge/reject recommendation. **Release mode** (`REL-xxx`): aggregates all CRs in a release with per-CR quality summary and ship/defer recommendation. Uses a "report what you have" principle -- missing layers are noted but do not block generation. See [test-procedures-report.md](../../skills/pace-test/test-procedures-report.md) for detailed steps.

**Output example** (CR mode):
```
CR-005 | L1: 8/8 passed | L2: 3/5 covered (60%) | L3: 4/5 passed, 1 manual
Recommendation: supplement tests before merge
```

#### `flaky`

Detect unstable tests and proactive maintenance issues.

**Syntax**: `/pace-test flaky`

Scans CR event history for intermittent/environment/order-dependent failures and timeout fluctuations. Performs proactive maintenance detection: empty assertions, time bloat, long-untouched tests, and skipped tests. Persists to `insights.md` and downgrades unstable tests in `test-strategy.md`. See [test-procedures-advanced.md](../../skills/pace-test/test-procedures-advanced.md) for steps.

**Output example**:
```
Unstable: e2e-login 2/5 (40%) intermittent [CR-003, CR-005]
Maintenance: test_utils::helper (empty assert) | lint (+217% bloat)
Priority: fix empty assertions first (false security)
```

#### `baseline`

Establish or update the test execution baseline for trend tracking.

**Syntax**: `/pace-test baseline`

Runs the full test suite, records pass rates and execution times, compares against the previous baseline. Persists to `.devpace/rules/test-baseline.md`. Consumed by `/pace-retro` for metrics. See [test-procedures-advanced.md](../../skills/pace-test/test-procedures-advanced.md) for steps.

**Output example**:
```
Baseline updated: pass rate 85%->92% (+7%) | exec 12.3s->10.1s (-2.2s) | checks 8->10 (+2)
```

### Verification Layer

#### `accept`

AI-powered acceptance verification against PF criteria.

**Syntax**: `/pace-test accept [CR-ID]`

The core differentiator of devpace testing. For each PF acceptance criterion, Claude selects a verification level -- L1 dynamic (execute tests/CLI), L2 static semantic (read code with line references), L3 manual (generate human checklists) -- and produces per-criterion evidence. Also performs a Test Oracle Check: reviews whether existing tests actually verify what they claim, downgrading weak or false coverage in `test-strategy.md`. See [verify-procedures.md](../../skills/pace-test/verify-procedures.md) for steps.

**Output example**:
```
CR-005 "Add CSV export" (PF: Data Import)
| # | Criterion                  | Status  | Level  | Evidence                        |
|---|----------------------------|---------|--------|---------------------------------|
| A1| CSV parses all columns     | PASS    | L1     | test_csv_parser passed          |
| A2| Error on malformed rows    | PASS    | L2     | src/parser.ts:45 validates rows |
| A3| Progress bar during upload | PARTIAL | L2     | Complex async, needs runtime    |
| A4| Accessibility              | MANUAL  | L3     | Checklist generated             |
Summary: 2 passed, 1 needs supplement, 1 needs manual check
```

**Degradation**: Without PF association, falls back to CR-intent acceptance criteria (noted in the report). Without either, exits with a clear prompt.

## Key Differentiator: Requirements-Driven Testing

Traditional tools measure **code coverage** ("what percentage of lines are executed"). devpace measures **requirements coverage** ("what percentage of PF acceptance criteria have verification"). A project can have 95% code coverage yet 0% requirements coverage. `/pace-test` closes this gap through the strategy-generate-coverage-accept-report pipeline, where every test traces back to a PF acceptance criterion.

## Usage Scenarios

### Scenario 1: New Feature Test Planning (TDD)

```
/pace-test strategy         --> PF-003: 6 criteria, 3 unit / 2 integration / 1 E2E
/pace-test generate PF-003 --full --> 6 test cases. TDD: run to confirm Red phase.
/pace-test coverage         --> PF-003 requirements: 6/6 (100%). Code: 0% (expected).
```

### Scenario 2: Pre-Merge Quality Check

```
/pace-test accept CR-005    --> 4/5 passed, 1 manual. Oracle: weak coverage found.
/pace-test report CR-005    --> 3-layer report. Recommendation: supplement, then merge.
```

### Scenario 3: Release Readiness

```
/pace-test report REL-001   --> 5 CRs, 3 PFs. Risk: LOW. Can ship.
```

## Integration with Other Commands

| Command | Integration Point |
|---------|-------------------|
| `/pace-dev` | Gate 1/2 consumes `checks.md` (same commands `run` executes). `accept` reports are Gate 2 evidence. |
| `/pace-review` | Gate 2 consumes `accept` verification reports as structured review evidence. |
| `/pace-release` | `report REL-xxx` generates release-level quality reports. `dryrun 4` validates pre-release. |
| `/pace-retro` | `baseline` feeds retrospective metrics. `flaky` findings feed `insights.md`. |
| `/pace-change` | `impact` uses CR change scope to determine regression recommendations. |
