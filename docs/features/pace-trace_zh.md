# 决策轨迹查询（`/pace-trace`）

devpace 决策轨迹是三层渐进透明体系的"深入层"——唯一暴露 AI 决策完整轨迹和溯源标记的入口。表面层（推理后缀）和中间层（追问展开）提供快速解释，`/pace-trace` 则提供完整的审计级决策重建。

## 前置条件

| 需求 | 用途 | 必要? |
|------|------|:-----:|
| `.devpace/` 已初始化 | 项目结构、CR 追踪 | 是 |
| 活跃或近期的 CR | 决策数据来源 | 是 |

> **优雅降级**：无 `.devpace/` 时输出初始化引导。无匹配 CR 时建议 `/pace-status detail` 浏览所有 CR。数据不完整时降低置信度输出，而非直接失败。

## 快速开始

```
1. /pace-trace CR-001 gate1     --> 完整的 Gate 1 质量检查轨迹
2. /pace-trace CR-001 intent    --> 意图推断过程（含溯源标记）
3. /pace-trace CR-001 timeline  --> 全生命周期决策时间线
4. /pace-trace                   --> 当前 CR 的最新决策轨迹
```

典型工作流：看到推理后缀（"Gate 1 通过——测试 92%, lint 0 error"）后追问"为什么？"获得中间层展开，再用 `/pace-trace` 查看完整审计轨迹。

## 决策类型

| 类型 | 重建内容 | 关键数据源 |
|------|---------|-----------|
| `gate1` | Gate 1 质量检查判定过程 | CR 检查项、命令输出、checkpoint 证据 |
| `gate2` | Gate 2 集成验证 + 验收比对 | CR 验收条件 vs 实际变更 |
| `gate3` | Gate 3 审批前摘要生成 | CR 状态、review 摘要 |
| `intent` | 意图检查点推断过程 | CR 意图 section、溯源标记 |
| `change` | 变更影响分析推理 | CR 关联、PF/迭代影响链 |
| `risk` | 风险评估判断依据 | 5 维扫描结果、风险文件 |
| `autonomy` | 自主级别如何影响 Gate 行为 | project.md 自主级别、交互模式差异 |
| `timeline` | CR 全生命周期决策时间线 | 所有事件和 checkpoint 按时间排序 |

## 输出结构

pace-trace 使用**结论先行**的三段叙事，符合审计思维：

1. **判定结论** — 最终判定 + 关键因素 + 重建置信度（高/中/低）
2. **评估过程** — 逐项检查 + 规则匹配 + 经验影响
3. **决策上下文** — 读取的信息 + 溯源标记

### 重建置信度

置信度反映轨迹重建的数据完整性，而非决策本身的正确性：

| 数据完整性 | 置信度 | 条件 |
|-----------|--------|------|
| 完整 | 高 | 事件表 + 溯源标记 + checkpoint 证据 + insights 引用记录全部存在 |
| 较完整 | 中 | 事件表 + checkpoint 证据存在，溯源标记部分缺失 |
| 部分 | 低 | 仅事件表存在，checkpoint 证据和溯源标记均缺失 |

### 时间线视图

`timeline` 类型将所有 checkpoint 和状态转换聚合为时间线总览：

```
| 时间  | 决策点       | 结果         | 关键因素                 |
|-------|-------------|-------------|-------------------------|
| 02-21 | 意图检查     | 复杂度 M     | 3 文件，标准模式          |
| 02-21 | Gate 1      | ✅ 通过      | tsc 0 error, 8/8 tests  |
| 02-21 | Gate 2      | ✅ 通过      | 3/3 验收满足             |
| 02-22 | Gate 3      | ❌ 打回      | 用户：错误提示不清晰       |
| 02-23 | Gate 2（重审）| ✅ 通过      | 1/1 修复项确认           |
```

## 错误处理

| 场景 | 输出 | 引导 |
|------|------|------|
| 项目未初始化 | "未找到 .devpace/ 目录" | 建议 `/pace-init` |
| CR 未找到 | "未找到匹配 '[input]' 的变更请求" | 建议 `/pace-status detail` |
| 决策类型无数据 | "该 CR 无 [type] 类型的决策记录" | 列出该 CR 可用的决策类型 |
| 溯源标记缺失 | "有事件记录但溯源标记不完整" | 说明基于事件表重建 |
| 未知类型 | "未知决策类型 '[input]'" | 列出支持的类型 |

## 上下文导航

每次输出末尾附加基于决策类型的导航建议：

| 决策类型 | 导航 |
|---------|------|
| gate1/gate2 | 经验模式、质量趋势 |
| intent | 设计理论、变更历史 |
| change | 测试影响、风险扫描 |
| risk | 风险总览、相关经验 |
| autonomy | 角色视角、理论基础 |
| timeline | 深入某节点轨迹、当前状态 |

## 与相似功能的区别

- **`/pace-trace [CR] [type]`**："Claude 为什么这样判断？"——单个决策的审计轨迹
- **`/pace-status trace <name>`**："这个目标完成到哪了？"——价值链完成度追踪（OBJ→BR→PF→CR）
- **`/pace-theory why`**："devpace 为什么这样设计？"——系统方法论解释

## 三层透明体系

```
表面层：推理后缀（系统行为后附 ≤15 字）
        "Gate 1 通过——测试 92%, lint 0 error。"

中间层：追问展开（问"为什么？"→ 2-5 行）
        判断依据 + 具体数据
        → 完整决策轨迹：/pace-trace [CR] [类型]

深入层：/pace-trace（完整审计轨迹）
        判定结论 + 评估过程 + 决策上下文 + 溯源标记
```
