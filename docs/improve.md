```

已验证项目：DAF（已有）、devpace-verify-newproject（新建）

七项差距（按优先级）：
G1 🔴 核心验证未闭环   — V2.4 质量拦截延后，OBJ-3 无直接验证
```

## G1：核心验证未闭环（Phase 2 内）

**现状**：V2.4（质量拦截验证）在 T9 中被延后，OBJ-3（质量检查有效）缺少直接验证证据。

**风险等级**：🔴 高

**分析**：质量门是 devpace 三大核心承诺之一（跨会话连续性 ✅、变更管理 ⏳ T10、质量检查 ❌）。T9 完成了会话连续性和会话保存验证，但质量拦截（"故意引入不符合质量检查的代码 → 质量检查拦截"）未执行。OBJ-3 的 MoS 是"质量检查至少 1 次阻止了不满足准出条件的 CR 进入下一阶段"，当前没有这样的证据。

**建议**：新增 T15（V2.4 质量拦截验证），独立测试 Gate 1/2 自动拦截能力。

**影响**：不影响 T10-T14 的执行，可并行推进。

## G2：差异化未证明（Phase 2 内）

**现状**：T13（devpace vs 手动方案对比验证）已定义但未执行。

**风险等级**：🔴 高

**分析**：OBJ-5 的 MoS 是"devpace 恢复上下文所需的用户纠正次数 < 手动 CLAUDE.md 方案"。这是 devpace 价值主张的核心证据——"我们确实比手动方式好"。没有这个数据：
- 无法向潜在用户证明差异化
- 无法写出有说服力的 README
- 无法在 marketplace 描述中给出量化承诺

**建议**：T13 应在 Phase 2 完成前执行，且结果应记录为可引用的数据。

## G3：UX 承诺未验证（Phase 2 内）

**现状**：T14（渐进丰富自然度观察）已定义但未执行。

**风险等级**：🟡 中

**分析**：OBJ-6 验证的是 UX 原则 P3（副产物非前置）的真实表现。如果用户在 CR 从 1 增长到 3+ 的过程中需要手动修正 state.md 或 project.md，说明"零摩擦"承诺不成立，需要调整设计或降低承诺。

**建议**：可在 T10 执行过程中顺带观察（T10 涉及多个 CR 的变更操作），降低独立验证成本。

## G4：用户入口缺失（Phase 3 前置）

**现状**：项目只有面向开发者的 CLAUDE.md，缺少面向用户的入口。

**风险等级**：🟡 中

**差距明细**：

| 缺失 | 影响 | 所属 Phase |
|------|------|-----------|
| 用户 README.md | 新用户看到 plugin 后不知道它是什么、怎么用 | Phase 3（M3.2） |
| Quickstart 路径 | OBJ-9 要求"5 分钟跑通"，当前无引导 | Phase 3（M3.2） |
| 安装方式 | `--plugin-dir` 是调试路径，非用户路径 | Phase 4（M4.3） |

**建议**：README 和 quickstart 应在 Phase 3 可移植性验证前完成——第 2 个项目的验证者需要这些材料。

## G5：被替代风险未缓解（Phase 3）

**现状**：vision.md 已识别"跨会话连续性可能被 Claude Code 原生 session persistence 替代"的风险，并设计了备选入口（变更管理即时价值）。但备选入口未经测试。

**风险等级**：🟡 中

**分析**：

- design.md 声称"/pace-change 不依赖跨会话状态"，但未在无 state.md 的场景下验证
- 当前 onboarding 假设用户在"第 2 次会话"感知跨会话连续性。如果这个被原生替代，需要一个"第 1 次变更"的价值感知路径
- 这不是当前的紧迫问题，但应在 Phase 3 中验证，因为它直接影响 Phase 4 的市场定位

**建议**：Phase 3 新增验证项——在无 .devpace/ 初始化的场景下测试 /pace-change 的独立价值。

## G6：度量体系不可用（Phase 3）

**现状**：`knowledge/metrics.md` 有指标定义，但 pace-retro 的数据收集机制不明确。

**风险等级**：🟢 低（当前阶段）

**差距明细**：

- **数据源**：回顾报告需要的数据（质量门通过率、CR 周期时间、变更影响范围等）如何从 CR 文件和 state.md 中自动提取？
- **基准数据**：没有"没有 devpace 时"的基准度量，无法证明效能提升
- **输出格式**：OBJ-8 要求"/pace-retro 生成有意义的回顾报告"，"有意义"的具体标准待定义

**建议**：Phase 3 中通过第 2 个项目的完整迭代收集度量数据，同时明确 pace-retro 的数据提取逻辑。

## G7：社区发布未准备（Phase 4）

**现状**：Phase 4 任务定义为粗略的 4 个 bullet point。

**风险等级**：🟢 低（当前阶段）

**差距明细**：

| 缺失 | 优先级 | 说明 |
|------|--------|------|
| LICENSE | 必须 | 无 license = 不是开源项目 |
| CONTRIBUTING.md | 必须 | 阻碍社区贡献 |
| Changelog | 重要 | 用户需要知道版本间变化 |
| 示例项目 | 重要 | 新用户学习的最快路径 |
| 用户指南 | 重要 | 超越 README 的完整使用文档 |
| Marketplace 正式提交 | 重要 | marketplace.json 当前是本地开发配置 |
| 规模验证 | 建议 | 10+ CR 时 state.md 是否仍 ≤15 行（NF2） |

## 技术加固项（跨 Phase）

以下问题不属于特定 Phase，但影响产品可靠性：

| 项目 | 现状 | 建议时机 |
|------|------|---------|
| 质量门 Hook 自动化 | session-start/stop hook 存在，但 Gate 1/2/3 的自动拦截未完整实现 | Phase 3（M3.1） |
| 异常中断容错 | 验证覆盖正常中断，未测试异常场景（写入中断、网络断连） | Phase 3 |
| state.md 规模控制 | NF2 要求 ≤15 行，未在 10+ CR 场景下验证 | Phase 3（M3.3） |
| Hook 脚本跨平台 | 当前 bash 脚本仅 macOS 验证，Linux/WSL 未测试 | Phase 4 |

## 推荐推进顺序

```
Phase 2 收尾（当前）
  T10 → 变更管理 + 辅助功能验证（核心差异化）
  T15 → V2.4 质量拦截验证（补齐 OBJ-3）
  T13 → 对比验证（量化差异化证据）
  T14 → 渐进丰富观察（可与 T10 合并观察）

Phase 3（可移植性 + 加固）
  M3.1 → 战略风险缓解：变更管理独立入口验证 + 质量门 Hook 自动化
  M3.2 → 用户入口：README + Quickstart
  M3.3 → 可移植性验证：第 2 个项目 + 度量收集 + 规模验证

Phase 4（社区发布）
  M4.1 → 开源合规：LICENSE + CONTRIBUTING + Changelog
  M4.2 → 社区引导：示例项目 + 用户指南
  M4.3 → 分发准备：Marketplace 提交 + 跨平台验证
```
