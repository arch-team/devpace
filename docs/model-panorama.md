## 二、模型定义与关联关系图

### 模型 ①：BizDevOps 三要素（元模型）

所有其他模型的理论基础。三要素相互作用构成价值交付过程本身：

```
                    ┌─────────────────┐
                    │   概念模型三要素  │
                    │   （元模型）      │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
     ┌────────────┐  ┌────────────┐  ┌────────────┐
     │  作业对象   │  │  作业空间   │  │  作业规则   │
     │ "交付什么"  │  │ "在哪协作"  │  │ "怎么流转"  │
     └─────┬──────┘  └─────┬──────┘  └─────┬──────┘
           │               │               │
           │  ┌────────────┘               │
           │  │  作业对象在作业空间中        │
           │  │  按照作业规则流转            │
           │  │  ＝ 价值交付过程             │
           ▼  ▼                             ▼
    ┌──────────────────────────────────────────┐
    │            价值交付过程                    │
    │   OBJ → BR → PF → CR → Release          │
    │   （在产品线/团队/应用空间中）              │
    │   （按工作流+质量检查规则流转）             │
    └──────────────────────────────────────────┘
```

**三要素的实例化**：

| 要素 | 实例化为 | 对应模型 |
|------|---------|---------|
| 作业对象 | OBJ, BR, PF, CR, Release, Defect | ② 价值交付链、③ 价值追溯树 |
| 作业空间 | 产品线, 交付团队, 应用 | 文件架构（project.md, .devpace/） |
| 作业规则 | 工作流, 质量检查, 分支策略 | ⑤ CR 状态机、⑦ 质量检查模型 |

### 模型 ③④⑤⑥⑦⑧⑨⑩ 之间的关联关系

```
                        ① 三要素（元模型）
                       ╱        │        ╲
                      ╱         │         ╲
                     ▼          ▼          ▼
              ② 价值交付链   ④ 三个闭环   ⑤ CR 状态机
               (对象分解)   (角色划分)    (规则实例)
                  │    ╲       │╱           │  ╲
                  │     ╲      │            │   ╲
                  ▼      ╲     ▼            ▼    ╲
             ③ 价值追溯树  ⑥ 双模式     ⑦ 质量检查  ╲
              (可视化)    (行为切换)    (检查点)     ╲
                  │         │             │          ╲
                  │    ┌────┘             │           ▼
                  ▼    ▼                  ▼       ⑩ 信息分层
             ⑧ 变更管理              ⑨ DIKW 度量   (加载策略)
              (应变机制)             (持续改进)
```

**关联关系明细表**：

| 关联 | 说明 |
|------|------|
| ① → ② | 三要素中的"作业对象"实例化为价值交付链 (OBJ→BR→PF→CR) |
| ① → ④ | 三要素框架决定了三个闭环各自操作不同层级的作业对象 |
| ① → ⑤ | 三要素中的"作业规则"实例化为 CR 状态机的工作流 |
| ② → ③ | 价值交付链的分解关系可视化为追溯树 |
| ② ← ⑧ | 变更管理修改交付链上的对象（新增/暂停/调整 PF 和 CR） |
| ④ → ⑥ | 三个闭环中，技术闭环对应推进模式，业务/产品闭环对应探索模式 |
| ⑤ → ⑦ | 状态机的状态转换由质量检查把关 |
| ⑤ → ⑧ | 暂停（paused）状态是变更管理的核心机制 |
| ⑥ → ⑤ | 推进模式激活时绑定 CR，遵循状态机规则 |
| ⑦ → ⑨ | 质量检查通过率等数据成为 DIKW 度量的数据源 |
| ⑧ → ③ | 变更执行后同步更新追溯树 |
| ⑨ → ④ | 度量结果反馈到三个闭环，驱动持续改进 |
| ⑩ → ②③ | 信息分层决定何时加载交付链/追溯树的哪个层级 |

## 三、CR 状态机详图（模型 ⑤）

```
                                  ┌───────────────────────────────────────────────┐
                                  │          CR 状态机（核心引擎）                 │
                                  └───────────────────────────────────────────────┘

                                             ┌──────────┐
                            /pace-advance    │ created  │  自动创建或 /pace-advance
                            或自动创建  ────▶ │  已创建   │
                                             └────┬─────┘
                                                  │
                                                  ▼
          ┌───────────────────────────────── ┌──────────┐ ──── 质量检查失败 ────┐
          │  人类驳回                         │developing│                      │
          │  (附原因)                         │ 开发中   │ ◀───── 自动修复 ──────┘
          │                                  └────┬─────┘
          │                                       │  代码提交 + 质量检查通过
          │                                       ▼
          │                                  ┌──────────┐ ──── 质量检查失败 ────┐
          │                                  │verifying │                      │
          │                                  │ 验证中   │ ◀───── 自动修复 ──────┘
          │                                  └────┬─────┘
          │                                       │  集成测试通过 + 质量检查通过
          │                                       ▼
          │                                  ┌──────────┐
          └────────────────────────────────▶ │in_review │  生成摘要，Claude 停止
                                             │ 待审阅   │  等待人类决策
                                             └────┬─────┘
                                                  │  人类批准
                                                  ▼
                                             ┌──────────┐
                                             │ approved │
                                             │ 已批准   │
                                             └────┬─────┘
                                                  │  git merge + 更新追溯树
                                                  ▼
                                             ┌──────────┐
                                             │  merged  │  触发：更新 PF 状态
                                             │ 已合并   │  更新 state.md
                                             └──────────┘  更新 project.md


              任意状态 ◀────────────────▶ ┌──────────┐
              (需求变更时)                 │  paused  │  保留：分支、代码、
                                          │  已暂停   │  质量检查进度、事件记录
                                          └──────────┘  记录：暂停前状态（便于恢复）
```

**各转换的执行者与门禁**：

| 转换 | 门禁条件 | 执行者 |
|------|---------|--------|
| created → developing | 关联 PF + 指定应用 | 自动 |
| developing → verifying | 代码提交到分支 + 质量检查通过 | Claude 自治 |
| verifying → in_review | 集成测试通过 + 质量检查通过 | Claude 自治 |
| in_review → approved | **人类 Code Review 批准** | **人类（阻塞门禁）** |
| in_review → developing | 人类驳回（附原因） | 人类决策 |
| approved → merged | git merge 到主分支 | Claude 执行 |
| 任意 → paused | 需求变更触发 | 人类决策 |
| paused → 暂停前状态 | 人类决策恢复 | 人类决策 |

## 八、变更管理模型详图（模型 ⑧）

```
          需求变更触发
          ━━━━━━━━━━
          "别做了" / "加一个" / "先做那个" / "改一改"
                            │
                            ▼
          ┌─────────────────────────────────────┐
          │         Step 1: 影响分析              │
          │                                     │
          │  读取 project.md 追溯树               │
          │        │                            │
          │        ├── 直接影响：哪些 PF/CR       │
          │        ├── 依赖链：间接影响的 CR       │
          │        ├── 迭代目标：容量/进度         │
          │        └── MoS 影响：成效指标变化      │
          │                                     │
          │  → 自然语言汇报影响                    │
          └──────────────┬──────────────────────┘
                         ▼
          ┌─────────────────────────────────────┐
          │         Step 2: 调整方案              │
          │                                     │
          │  ┌───────────┬───────────┬─────────┐│
          │  │ 新增需求   │ 暂停/取消  │ 调优先级 ││
          │  │           │           │         ││
          │  │创建 PF+CR │CR→paused  │重排计划  ││
          │  │评估容量   │解除依赖   │更新下一步 ││
          │  │建议排期   │标记追溯树  │         ││
          │  └───────────┴───────────┴─────────┘│
          │  ┌───────────┐                      │
          │  │ 修改需求   │                      │
          │  │评估返工量  │                      │
          │  │重置质量检查│                      │
          │  └───────────┘                      │
          │                                     │
          │  → 等待人类确认                       │
          └──────────────┬──────────────────────┘
                         ▼
          ┌─────────────────────────────────────┐
          │         Step 3: 执行 + 记录           │
          │                                     │
          │  更新 CR 文件（状态/质量检查/事件）     │
          │  更新 project.md 追溯树               │
          │  更新 iterations/current.md 计划      │
          │  记录到迭代变更记录表                   │
          │  更新 state.md 快照                   │
          └─────────────────────────────────────┘
```

**关键原则**：
- 变更不是异常，是常态（专题模式核心理念）
- 暂停 ≠ 删除（paused 状态保留所有工作成果）
- 变更可审计（所有变更记录在迭代变更记录表）
- 追溯树必须保持最新（变更后立即同步）

## 九、DIKW 度量模型详图（模型 ⑨）

```
  数据源                      DIKW 层级                    devpace 实现
  ━━━━━                      ━━━━━━━━                    ━━━━━━━━━━━

  CR 事件表  ───────────▶   Data（数据）  ──────────▶   CR 文件 + Git 历史
  Git 历史                   原始事实                    自动产生，零额外负担

        │
        ▼

  质量检查首次通过率 ──▶  Information（信息） ──────▶   dashboard.md
  人类驳回率                 加工后数据                  /pace-retro update
  交付周期
  MoS 达成率

        │
        ▼

  质量趋势分析 ──────▶   Knowledge（知识） ──────▶   /pace-retro 回顾报告
  做得好的/需改进的          理解与规律                  团队自省工具
  瓶颈识别

        │
        ▼

  下个迭代改进建议 ──▶    Wisdom（智慧）  ──────▶   反馈到下轮迭代规划
```

**度量原则**：
- 基于客观数据自动生成，不增加额外填写负担
- 服务于团队自省，不用于个人绩效评比
- 关注趋势而非绝对值
- 仅在 `/pace-retro` 时更新 dashboard，不在每次会话更新
