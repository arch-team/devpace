# devpace 需求

> **职责**：定义用户场景、功能需求和非功能需求。需求变更时更新本文件。

**定位**：docs/ 体系的**需求清单**——回答"具体要实现哪些功能"。将 vision.md 的目标和 design.md 的方案拆解为可验收的功能条目。

| 维度 | 说明 |
|------|------|
| 回答的问题 | 有哪些用户场景？每个场景需要什么功能？非功能约束是什么（性能、兼容性等）？ |
| 上游输入 | vision.md（业务目标拆解为场景）、design.md（设计方案拆解为功能点） |
| 下游消费 | roadmap.md（需求条目排入实施阶段） |
| 更新时机 | 新增用户场景、功能需求变更、发现遗漏的非功能需求时 |
| 类别 | 项目管理——功能验收规格，不进入产品运行时 |
| 主要读者 | 实现者——需要知道"做哪些功能、做到什么程度" |

## 用户场景

### S1：首次初始化

**角色**：开发者
**场景**：开始一个新项目，想用 devpace 管理研发流程
**期望**：运行一个命令，通过对话式引导完成初始化，得到完整的项目状态目录

**验收标准**：
- [x] `/pace-init` 通过对话收集项目信息
- [x] 自动生成 .devpace/ 目录和所有必需文件
- [x] 生成的价值功能树结构合理

### S2：日常推进

**角色**：开发者
**场景**：新开一个 Claude Code 会话，继续昨天的工作
**期望**：Claude 自动知道做到哪了，直接继续，不需要我重新解释

**验收标准**：
- [x] Claude 会话开始自动读取 state.md
- [x] 用 1 句自然语言报告当前状态
- [x] 用户说"继续"就能从上次断点恢复

### S3：查看进度

**角色**：开发者/项目经理
**场景**：想快速了解项目整体进展
**期望**：不同粒度的进度报告，从 1 句话概览到完整功能树

**验收标准**：
- [x] `/pace-status` 默认 1-3 行概览
- [x] `/pace-status detail` 展开到产品功能级别
- [x] `/pace-status tree` 展示完整价值功能树

### S4：需求插入

**角色**：开发者
**场景**：做到一半，发现需要加一个新功能
**期望**：告诉 Claude 新需求，它自动分析影响、调整计划

**验收标准**：
- [x] Claude 自动识别"新需求"意图
- [x] 执行影响分析：影响哪些 PF/CR，迭代容量够不够
- [x] 提出调整方案，等待确认
- [x] 确认后自动更新所有相关文件（功能树、迭代计划、state）

### S5：需求暂停/砍掉

**角色**：开发者
**场景**：某个功能暂时不做了，或者优先级降低
**期望**：已有工作成果不丢失，依赖关系自动调整

**验收标准**：
- [x] CR 标记为 paused，工作成果保留（分支、代码、质量检查进度）
- [x] 依赖此 CR 的其他 CR 自动解除阻塞或标记受影响
- [x] 功能树中用 ⏸️ 标记暂停项
- [x] 恢复时从暂停前状态继续

### S6：优先级重排

**角色**：开发者
**场景**：某个功能需要提前做
**期望**：调整优先级后迭代计划自动重排

**验收标准**：
- [x] state.md 的"下一步"更新为新的最高优先项
- [x] 迭代计划反映调整后的顺序
- [x] 变更记录保留调整原因

### S7：需求修改

**角色**：开发者
**场景**：已有功能的范围或要求变了
**期望**：Claude 评估变更对已有工作的影响

**验收标准**：
- [x] 识别已完成工作中需要返工的部分
- [x] 质量检查重置为未检查（需要重新验证）
- [x] CR 事件记录变更原因

### S8：Code Review

**角色**：开发者
**场景**：Claude 做完一个 CR，等待人类审核
**期望**：清晰的变更摘要，一键审批或打回

**验收标准**：
- [x] 自动生成变更摘要（改了什么、为什么、影响）
- [x] 审批后自动合并和状态更新
- [x] 打回后回到 developing 并记录原因

### S9：迭代回顾

**角色**：开发者/项目经理
**场景**：一个阶段结束，复盘进展
**期望**：基于数据的回顾报告，包括变更造成的偏差分析

**验收标准**：
- [x] 自动收集质量和交付数据
- [x] 生成回顾报告（交付/质量/价值/变更影响）
- [x] 首次回顾时建立度量基准线，后续回顾展示与基准的对比趋势
- [x] 更新度量仪表盘

### S10：会话结束

**角色**：开发者
**场景**：工作告一段落，准备关闭 Claude Code 会话
**期望**：Claude 自动保存当前进度和下一步建议，下次会话无缝衔接

**验收标准**：
- [x] Claude 在会话结束前自动更新 state.md（当前工作 + 下一步建议）
- [x] 进行中的 CR 文件更新到最新状态
- [x] 输出 3-5 行会话总结（做了什么、质量状态、下一步）

### S11：自由探索

**角色**：开发者
**场景**：想了解代码、分析问题、评估方案，但不打算修改代码
**期望**：自由读取分析，不触发任何流程约束，不修改状态文件

**验收标准**：
- [x] 默认进入探索模式，无需显式声明
- [x] 探索过程中不创建 CR、不触发质量检查
- [x] 可随时通过修改代码或显式命令切换到推进模式

### S12：理论参考

**角色**：开发者
**场景**：想了解 devpace 背后的方法论，或查阅 BizDevOps 理论指导
**期望**：获得与当前工作上下文相关的理论参考，而非通篇文档

**验收标准**：
- [x] `/pace-theory` 返回与当前项目阶段相关的理论指导
- [x] 只读操作，不修改任何状态文件
- [x] 输出简洁，聚焦实用建议而非学术概念

### S13：新用户入门

**角色**：新用户
**场景**：在 Marketplace 或 GitHub 上发现 devpace，想了解它是什么、是否适合自己、如何开始使用
**期望**：清晰的说明文档和快速上手路径，5 分钟内跑通第一个命令

**验收标准**：
- [x] README.md 包含：问题定义、价值主张、安装方式、Quickstart 步骤、命令列表
- [x] Quickstart 路径从安装到 `/pace-init` 首次输出不超过 5 分钟
- [x] 新用户无需阅读 design.md 或 theory.md 即可开始使用
- [x] README 中引用 T13 对比验证数据作为量化证据

### S14：无初始化变更管理

**角色**：开发者
**场景**：未运行 /pace-init，项目中没有 .devpace/ 目录，但在对话中提出了需求变更
**期望**：Claude 仍能提供有价值的变更影响分析，作为 devpace 差异化能力的即时展示

**验收标准**：
- [x] /pace-change 在无 .devpace/ 时不报错，以降级模式运行
- [x] 降级模式基于代码库分析和对话上下文，输出变更影响评估
- [x] 不创建 .devpace/ 目录或状态文件（仅即时分析，无持久化）
- [x] 输出末尾自然引导用户了解完整初始化后的增强能力

### S15：迭代规划

**角色**：开发者
**场景**：当前迭代接近尾声或已结束，需要关闭当前迭代并规划下一个迭代的范围
**期望**：Claude 引导完成迭代关闭和新迭代的 PF 选择，生成迭代文件

**验收标准**：
- [x] `/pace-plan` 展示当前迭代 PF 完成率
- [x] `/pace-plan close` 关闭当前迭代，归档为 iter-N.md
- [x] `/pace-plan next` 展示待选 PF 列表，引导用户选择纳入范围
- [x] 生成的 iterations/current.md 符合 iteration-format schema

### S16：主动节奏提醒

**角色**：开发者
**场景**：在推进模式中长时间工作，Claude 主动检测节奏异常
**期望**：Claude 自动提醒 CR 滞留、Review 积压、迭代偏移等问题

**验收标准**：
- [x] 推进模式中 Claude 主动检测 6 种信号
- [x] 建议以 1-2 行自然语言输出，不阻断工作流
- [x] 同一信号在同一会话中只提醒一次

### S17：自动知识积累

**角色**：开发者
**场景**：CR 完成 merged 后，Claude 自动提取经验
**期望**：insights.md 自动积累可复用 pattern，无需手动触发

**验收标准**：
- [x] CR merged 后自动提取 pattern
- [x] 质量门修复后提取防御 pattern
- [x] 已有 pattern 被验证时追加记录

### S18：上线后反馈收集

**角色**：开发者
**场景**：上线后收到用户反馈或发现 bug
**期望**：反馈关联到价值链，自动创建修复 CR 或记录改进建议

**验收标准**：
- [x] /pace-feedback 分类反馈（缺陷/改进/新需求）
- [x] 自动关联到最相关的 PF 和 BR
- [x] 缺陷自动创建修复 CR

### S19：发布管理

**角色**：开发者/运维
**场景**：一批 CR 已 merged，准备打包发布到生产环境
**期望**：收集候选 CR、追踪部署状态、验证后关闭 Release，全程可追溯

**验收标准**：
- [x] `/pace-release create` 自动收集未关联 Release 的 merged CR
- [x] `/pace-release deploy` 记录部署操作，Release 状态 staging → deployed
- [x] `/pace-release verify` 执行验证清单，通过后 deployed → verified
- [x] `/pace-release close` 执行关闭连锁更新：关联 CR 批量 merged → released + 功能树 🚀
- [x] 不使用 Release 流程时所有 Release 行为静默跳过，不影响核心流程

### S20：运维反馈与缺陷管理

**角色**：开发者/运维
**场景**：上线后发现生产问题，需要追溯问题来源并创建修复 CR
**期望**：自动追溯到引入变更的 CR 和 PF，创建 defect/hotfix CR 驱动修复闭环

**验收标准**：
- [x] `/pace-feedback report` 引导式问题摄入（问题描述 + 影响范围 + 紧急程度）
- [x] 自动追溯到最相关的 PF 和 Release
- [x] 根据影响范围×紧急程度自动建议严重度和 CR 类型（defect/hotfix）
- [x] hotfix + critical 可走加速路径（跳过 in_review）
- [x] 创建的 defect/hotfix CR 遵循同一状态机

### S22：渐进教学

**角色**：新用户
**场景**：用户首次使用 devpace，对 Claude 的系统行为（自动创建 CR、质量检查、等待审批等）不理解原因
**期望**：Claude 在首次触发每个系统行为时，附加简短解释帮助理解，但不重复解释已知行为

**验收标准**：
- [x] 6 个核心行为首次触发时各附加 ≤1 句解释
- [x] 同一行为在项目生命周期内只解释一次（通过 state.md 教学标记去重）
- [x] 教学标记缺失时等同于"全部未教"（向后兼容）
- [x] 基于成熟度的功能发现：第 3/5 个 CR merged 后提示进阶命令
- [x] 首个 CR merged 后附加 3 行 devpace 行为回顾

### S21：角色视角切换

**角色**：开发者（兼多角色）
**场景**：同一个人在不同阶段需要以不同角色视角查看项目信息
**期望**：Claude 自动推断当前角色，或通过命令显式切换，输出内容随角色变化

**验收标准**：
- [x] Claude 根据对话关键词自动推断角色（Biz/PM/Dev/Tester/Ops）
- [x] `/pace-role <角色>` 显式切换角色视角
- [x] 同一 Skill（如 /pace-status）在不同角色下输出不同维度信息
- [x] 默认 Dev 角色，角色变化时 1 句话自然过渡

### S23：从需求文档自动初始化项目

**角色**：开发者/PM
**场景**：已有现成的 PRD 或需求文档，想快速初始化 devpace 项目
**期望**：传入文档路径，AI 自动解析并生成 BR→PF→CR 功能树的初始结构，用户确认后写入

**验收标准**：
- [x] `/pace-init --from <文档路径>` 读取并解析需求文档
- [x] 自动提取业务目标、功能模块、用户场景并映射为 BR→PF→CR 结构
- [x] 生成的功能树展示给用户确认后才写入 project.md
- [x] 与手动初始化（对话式）相比，减少初始化交互轮数

### S24：决策轨迹查询

**角色**：开发者
**场景**：Claude 做了一个系统判断（Gate 失败、高风险评估、简化审批），用户想了解判断依据
**期望**：按需查看不同深度的推理信息——从简短后缀到完整决策轨迹

**验收标准**：
- [x] 系统行为输出默认附加 ≤15 字推理后缀（表面层）
- [x] 用户追问"为什么"时展开 2-5 行结构化推理链（中间层）
- [x] `/pace-trace` 展示完整决策轨迹，包含溯源标记（深入层）
- [x] 深入层展示用户输入（source: user）与 Claude 推断（source: claude）的区分

### S25：自主级别配置

**角色**：开发者
**场景**：用户根据对 devpace 的熟悉程度，想调整 Claude 在推进模式中的自主程度
**期望**：可在辅助（更多询问）、标准（当前行为）、自主（更少干预）之间切换

**验收标准**：
- [x] project.md 支持 `自主级别:` 字段配置（辅助/标准/自主）
- [x] 辅助模式下 Gate 失败时询问而非自动修复
- [x] 标准模式保持当前所有行为不变（默认）
- [x] 自主模式下简化审批条件放宽（≤5 文件）
- [x] Gate 3 在任何自主级别下都是人类阻塞门禁

### S26：纠正即学习

**角色**：开发者
**场景**：Claude 做了一个判断（简化审批、PF 分解、影响评估），用户纠正了这个判断
**期望**：devpace 记住这个偏好，以后同类场景自动应用纠正后的行为

**验收标准**：
- [x] Claude 检测到纠正行为时主动提议记录（1 句话）
- [x] 用户确认后写入 insights.md 偏好条目
- [x] 用户拒绝时不记录，无副作用
- [x] 后续 §12 引用时偏好条目优先级高于统计 pattern
- [x] 同一类纠正同一会话内只提议一次

| ID | 需求 | 标准 |

| NF1 | 零摩擦交互 | 用户全程自然语言，不需要学术语或操作文件 |
