# devpace v2.0 路线图

> **制定日期**：2026-02-25 | **基于**：v1.4.0 用户价值增强战略分析
> **核心原则**：先服务独立开发者，再向企业扩展；90 天内暂停新功能，聚焦发布和用户获取

---

## 战略前提

1. **"不发布 = 不存在"**：v1.4.0 功能完备度已远超同类竞品，最大瓶颈不是功能缺失而是市场缺位
2. **先独立开发者，再企业**：主要画像是 2-3 年经验的 Claude Code 日常用户，无敏捷背景
3. **90 天发布优先**：暂停新功能开发，100% 聚焦发布、UX 修复和用户获取
4. **Layer 1 获客，Layer 2 留存**：跨会话连续性做无摩擦入口，变更管理做差异化粘性
5. **数据驱动迭代**：每个 Phase 设定可衡量指标，根据真实用户反馈调整优先级

---

## Phase 18: 发布就绪（优先级 P0）

**目标**：Marketplace 上架 + GitHub 公开，从"不存在"到"可发现"

**用户价值**：潜在用户首次能发现和安装 devpace

**来源**：Agent 3 增长杠杆 L1（★★★★★）+ Agent 2 差异化窗口

| # | 任务 | 产出 | 完成标准 |
|---|------|------|---------|
| 18.1 | GitHub 仓库公开 | 公开的 devpace 仓库 | 仓库可访问、Topics 已设置（claude-code, claude-code-plugin, ai-development, devops） |
| 18.2 | 英文 README（双语） | 中英双语 README | 英文为主 + 中文摘要，含 Hero section + Quick Start + GIF 占位 |
| 18.3 | marketplace.json 描述优化 | 面向用户的产品描述 | 痛点驱动文案："Never re-explain your project to Claude again..." |
| 18.4 | 录制演示 GIF | README 视觉演示 | 3 个场景：初始化 → 开发 → 跨会话恢复 |
| 18.5 | 提交 Marketplace 审核 | 审核申请 | 按 Anthropic Plugin 审核标准提交 |
| 18.6 | plugin.json author 信息完善 | 可信的作者信息 | 真实 GitHub 用户名/组织 |

**关键指标**：Marketplace 审核通过 + GitHub 仓库可访问

**依赖**：无（可立即启动）

## Phase 19: 首次体验优化（优先级 P0）

**目标**：30 秒内价值感知，首次→二次使用转化率最大化

**用户价值**：新用户安装后立即感受到"这个工具有用"

**来源**：UX 审计 C1/M1 + Agent 3 增长杠杆 L2

| # | 任务 | 产出 | 完成标准 |
|---|------|------|---------|
| 19.1 | 重写首次 opt-in 提问 | 具象化的推进模式引导 | "我来管理进度和质量检查"替代"跟踪变更" |
| 19.2 | Skill 渐进暴露优化 | init 后只推荐 1 个下一步 | "说你想做什么，我来帮你推进"替代命令列表 |
| 19.3 | 推进模式进度感知 | 每阶段 1 句话进度输出 | 状态转换时附加简短信号 |
| 19.4 | 跨会话价值提前告知 | 会话结束时预告 | "下次开会话我会记住进度——不用重新解释" |
| 19.5 | 简化审批附加原因 | 审批时 1 句话说明为什么 | "变更很小且质量门通过，自动合并还是要看看？" |

**关键指标**：首次使用→二次使用转化率 >50%（间接通过 GitHub 活跃度评估）

**依赖**：可与 Phase 18 并行

## Phase 20: Skill 命名与简化（优先级 P1）

**目标**：减少认知负荷，提升功能可发现性

**用户价值**：想做风险分析时能找到命令，想知道"为什么"时不用翻文档

**来源**：UX 审计 C2/C3 + Agent 3"认知负担非吸引力"

| # | 任务 | 产出 | 完成标准 |
|---|------|------|---------|
| 20.1 | /pace-guard → /pace-risk | Skill 目录改名 + 内部引用更新 | 新名称生效，旧名称作为说明保留在 description 中 |
| 20.2 | /pace-trace → /pace-why | Skill 目录改名 + 内部引用更新 | 同上 |
| 20.3 | /pace-test 子命令简化 | 默认暴露 3 个（空参/accept/coverage） | strategy 合并到 coverage，impact 合并到空参，help 不展示兼容旧名 |
| 20.4 | /pace-release 子命令分组 | 空参引导更突出 | changelog/version/tag/notes/branch 合并到 close 流程 |
| 20.5 | /pace-status 去掉角色参数 | 依赖 /pace-role 自动适配 | 去掉 biz/pm/dev/tester/ops 参数 |
| 20.6 | 术语统一 | "质量检查"统一面向用户使用 | 消除 Gate/质量门/质量检查混用 |
| 20.7 | plugin.json 同步 | 更新 Skill 声明 | 改名后 plugin.json 与文件系统一致 |

**关键指标**：Skill 命名评分平均 >4.0/5（当前专项 Skill 平均 3.2）

**依赖**：Phase 18 完成后（已发布版本需要考虑向后兼容）

**风险评估**：改名涉及 rules/devpace-rules.md、knowledge/ 引用、SKILL.md description 等多处更新。需要系统性排查所有引用点。

## Phase 21: 技术债务清理（优先级 P1）

**目标**：提升可维护性，为社区贡献者降低门槛

**用户价值**：更快的会话启动（减少上下文消耗）、更可靠的 Hook 行为

**来源**：Agent 4 技术债 TD-1/TD-2/TD-3

| # | 任务 | 产出 | 完成标准 |
|---|------|------|---------|
| 21.1 | Rules 模块化拆分 | devpace-rules.md → 4 个文件 | 核心（SS0-SS3）、推进（SS4-SS9）、节奏（SS10-SS12）、扩展（SS13-SS15） |
| 21.2 | Schema 共享定义 | `_schema/common.md` | 抽取状态枚举、复杂度、关联类型等共享定义 |
| 21.3 | Hook 测试框架 | `lib/utils.mjs` 单元测试 + 关键 Hook 集成测试 | pre-tool-use.mjs、intent-detect.mjs 有测试覆盖 |
| 21.4 | Skill 依赖文档化 | design.md 附录 B 补充数据依赖图 | pace-guard→pace-pulse、pace-guard→pace-retro 等依赖明确 |

**关键指标**：`devpace-rules.md` 单文件缩减 >60%（拆分后核心文件 <300 行）

**依赖**：Phase 20 完成后（改名后再拆分，避免双重变更）

**约束**：Claude Code Plugin 的 rules/ 目录自动加载所有 .md 文件，拆分后无需额外配置

## Phase 22: 用户反馈驱动迭代（优先级 P1）

**目标**：基于真实用户反馈优化，验证核心假设

**用户价值**：产品按用户需求而非假设演进

**来源**：Agent 3 Phase 3 验证迭代

**条件触发**：首批 20+ 用户后启动

| # | 任务 | 产出 | 完成标准 |
|---|------|------|---------|
| 22.1 | 用户反馈收集机制 | GitHub Issue 模板 + 反馈引导 | 结构化的 bug/feature/feedback 模板 |
| 22.2 | Top 3 问题修复 | 产品迭代 | 基于用户反馈的高优先级修复 |
| 22.3 | 留存路径验证 | 间接数据分析 | 通过 GitHub 活动（Issue/Star/讨论）评估留存 |
| 22.4 | 社区反馈内容 | "我们听到了反馈并改进"帖子 | 展示响应速度和迭代能力 |
| 22.5 | 90 天复盘报告 | 数据驱动的策略调整 | 增长数字 vs 预期对比 + 下一阶段计划 |

**关键指标**：
- 收到 10+ 条用户反馈
- Top 3 问题修复后无再次报告
- D7 留存率 >30%（间接评估）

**依赖**：Phase 18/19 完成（需要有用户才能收集反馈）

## Phase 23: Ops 执行层 MVP（优先级 P2）

**目标**：补齐 merged→deployed 自动化缺口

**用户价值**：CR 合并后一键部署，闭合 DevOps 最后一环

**来源**：Agent 4 v2.x 可行性矩阵（技术可行性：高）

**条件触发**：Phase 22 完成 + 用户反馈确认 Ops 需求

| # | 任务 | 产出 | 完成标准 |
|---|------|------|---------|
| 23.1 | /pace-release deploy 支持 CLI 部署 | 新增 deploy 子命令执行配置的部署命令 | 通过 integrations/config.md 配置 kubectl/aws/gcloud 命令 |
| 23.2 | 部署确认门禁 | Gate 4 级别的人类确认 | 生产部署必须人类确认（铁律） |
| 23.3 | 部署状态追踪 | CR 状态新增 deployed | 支持 merged→deployed 转换 |
| 23.4 | 回滚增强 | /pace-release rollback 支持部署回滚 | 执行配置的回滚命令 + 状态记录 |

**关键指标**：端到端 CR 闭环（created→...→merged→deployed）可演示

**约束**：
- 部署命令必须非交互式
- 长时间命令需合理 timeout
- 生产操作人类确认不可绕过

## Phase 24: 企业能力探索（优先级 P3）

**目标**：验证企业市场需求，为 Open Core 做准备

**用户价值**：企业团队获得可审计、可追溯的 AI 辅助研发流程

**来源**：Agent 2 Layer 4 评估 + Agent 3 商业模式 Phase B

**条件触发**：Stars >500 + 月活 >200 + 3+ 企业询问

| # | 任务 | 产出 | 完成标准 |
|---|------|------|---------|
| 24.1 | 审计日志导出 | 结构化审计记录导出功能 | 导出 CR 全生命周期事件为 JSON/CSV |
| 24.2 | 团队协作探索 | 多人共享 .devpace/ 的方案设计 | 技术方案文档，含 Git 冲突处理策略 |
| 24.3 | 自定义质量门 | 企业自定义 Gate 检查项 | 通过配置文件扩展默认 Gate 规则 |
| 24.4 | 企业案例文档 | 1-2 个企业使用案例 | 展示可审计/可追溯价值的真实场景 |

**关键指标**：至少 1 个企业用户完成试用并给出正式反馈

## 成功定义

**90 天后的成功画面**：
- devpace 在 Marketplace 上线并可被全球 Claude Code 用户发现
- 首次使用体验从"17 个 Skill 列表"变为"说你想做什么就行"
- Skill 命名从"guard/trace"变为用户直觉能理解的"risk/why"
- 有 40-80 个月活用户，收到 10+ 条真实反馈
- 基于反馈的迭代正在进行，产品在真实需求驱动下演进

**非目标**：
- 不追求功能数量增长（17 个 Skill 已经足够）
- 不追求企业客户（等条件触发）
- 不追求多平台支持（聚焦 Claude Code）
- 不追求收入（纯开源阶段）
