## 三、改进项

### 3.1 P0 - 高优先级

#### 改进 1：缺少利益相关者分析能力

**问题**：有 preferred-role 角色适配（影响追问方向），但不等同于对项目利益相关者的结构化分析。识别"谁关心/受影响/有决策权"是优先级和范围决策的关键输入。

**方案**：
- Epic/BR 创建时引导标注关键利益相关者（可选字段）
- discover Step 1 增加可选追问："涉及哪些角色？各自关切？"
- align 增加检查维度：利益相关者覆盖度
- 不新增子命令，作为现有流程渐进丰富

**文件影响**：`epic-format.md`(+字段)、`biz-procedures-discover.md`(Step 1)、`biz-procedures-decompose.md`(Step 3E)、`biz-procedures-align.md`(+2.8)

#### 改进 2：需求优先级方法论单一

**问题**：仅 Value x Effort 矩阵，"价值/成本"的高/中/低判断缺结构化评估标准。缺 MoSCoW 和 Kano 模型等互补方法。

**方案**：
- Value 维度关联 Epic MoS 贡献度（定量化）
- 新增 `--method` 参数：`--moscow`(四分法) / `--kano`(基本/期望/兴奋型)
- 默认仍 Value x Effort（向后兼容）

**文件影响**：`biz-procedures-decompose.md`(Step 3E/3P)

### 3.2 P1 - 中高优先级

#### 改进 3：发现型子命令入口混淆

**问题**：discover/import/infer 需用户预知机制才能选择。"帮我梳理需求"无法明确对应。

**方案**：
- 空参数引导增强：检测 `.md` 文件 -> 推荐 import；检测 `src/` -> 推荐 infer
- discover Step 0 增加智能路由：提供文件路径 -> import 逻辑；说"看代码" -> infer；文本描述 -> discover 对话
- "统一入口 + 智能路由"降低决策负担

**文件影响**：`SKILL.md`(空参数引导)、`biz-procedures-discover.md`(Step 0)

#### 改进 4：refine 缺少精炼完整度指标

**问题**：用户无法快速判断哪些 BR/PF "需要精炼"、"精炼到什么程度足够"。

**方案**：
- 定义 BR/PF "就绪度"评分（Readiness Score, 0-100%），基于维度覆盖度实时计算
- view 展示就绪度标记：`BR-001：需求名 P0 [就绪度 60%]`
- align 增加就绪度分布检查：P0 需求平均就绪度 <70% 则警告

**文件影响**：`biz-procedures-view.md`(展示)、`biz-procedures-align.md`(+2.8)、`biz-procedures-refine.md`(Step 1 展示)

### 3.3 P2 - 中等优先级（本次不实施）

- 改进 6：缺少业务流程建模支持
- 改进 7：align 缺少历史趋势对比

## 五、架构级建议

### A. 统一发现引擎 + 输入适配器

当前 import/infer 各自独立实现合并分析逻辑。统一引擎可共享去重、编号分配、溯源标记等公共逻辑。**建议**：不做破坏性重构，未来版本中可将合并分析逻辑提取为共享 procedures。

### B. 需求成熟度模型

L0 信号 -> L1 机会 -> L2 专题 -> L3 需求 -> L4 就绪。不需要用户记住"该用 epic 还是 decompose 还是 refine"，系统根据成熟度自动建议。**建议**：在空参数引导中融入成熟度思想（低成本实现）。

### C. 需求验证与基线管理

覆盖了发现/分析/规格三阶段，缺少需求验证和基线管理。**建议**：渐进融入，不阻塞当前改进。
