# 业务分析抽象模型

> **职责**：定义 devpace 业务规划域背后的四个抽象模型。这些模型是 /pace-biz 智能行为的理论锚点，驱动生命周期感知推荐、合并检测和质量评估。
>
> **运行时消费者**：`/pace-theory` Skill 在运行时可读取本文件扩展业务分析方法论说明。
>
> **设计原则**：模型是内部智能，不是外部复杂度——用户不需要知道模型存在，只需要感受到更智能的引导和更精准的推荐。

**类别**：方法论知识库

## §0 速查卡片

```
四模型体系：
  Process Model  — 需求从感知到就绪的六阶段生命周期 + 三级反馈循环
  Data Model     — 六层实体层级 × 四级成熟度矩阵 + 五种关系类型
  Discovery Model — 三类输入源 × 六阶段统一管道 + 智能路由
  Quality Model  — 四维度评估（就绪度 × 对齐度 × 完整度 × 健康度）

设计原则：通用性（独立于 devpace 特定实现）· AI 原生（明确自动/人类活动边界）· 渐进使用
消费方：/pace-biz（智能行为驱动）· /pace-theory（方法论扩展）
```

### 模型导航

| 模型 | 章节 | 回答的问题 | pace-biz 主要映射 |
|------|------|-----------|------------------|
| Process Model | §1 | 需求经历哪些阶段？何时流转？ | 空参引导、子命令推荐 |
| Data Model | §2 | 实体有多丰富？还缺什么？ | 成熟度提示、创建输出 |
| Discovery Model | §3 | 输入从哪来？怎么处理？ | discover/import/infer 管道 |
| Quality Model | §4 | 质量怎么样？哪里有问题？ | refine/align/view 评估 |

## §1 Process Model（业务需求生命周期）

需求从模糊信号到开发就绪的六个阶段，以及驱动迭代改进的三级反馈循环。

### 六阶段定义

```
Sense → Ideate → Structure → Refine → Validate → Ready
 感知     构思     结构化      精炼      验证       就绪
```

| 阶段 | 目标 | 输入 | 输出 | 关键活动 |
|------|------|------|------|---------|
| **Sense** | 捕获外部信号，识别潜在业务价值 | 用户反馈、竞品观察、技术发现、市场趋势 | Opportunity 候选 | 信号识别、机会记录 |
| **Ideate** | 从机会展开探索，产出需求候选 | Opportunity、模糊想法、文档、代码库 | BR/PF 候选列表 | 头脑风暴、文档提取、代码推断 |
| **Structure** | 组织需求层级，建立分解关系 | BR/PF 候选 | Epic→BR→PF 层级树 | 分组、分解、优先级排序 |
| **Refine** | 补充细节，深化需求定义 | 骨架级实体 | 就绪级实体（验收标准、边界条件等） | 交互式精炼、维度补充 |
| **Validate** | 检查一致性，评估整体质量 | 功能树 + 实体详情 | 对齐度报告、改进建议 | 战略对齐检查、完整度审查 |
| **Ready** | 需求达到开发就绪标准 | 就绪度 >= 80% 的实体 | 可进入开发的 PF/BR | 就绪度确认、开发移交 |

### 阶段流转规则

- **顺序推进**：默认按 Sense→...→Ready 顺序推进
- **跳跃允许**：已有明确需求时可跳过 Sense/Ideate 直接进入 Structure
- **并行允许**：Structure 和 Refine 可对不同实体并行（分解 Epic A 的同时精炼 BR B）
- **回退触发**：Validate 发现问题时按反馈循环回退（见下文）

### 三级反馈循环

| 级别 | 名称 | 范围 | 触发条件 | 示例 |
|------|------|------|---------|------|
| **L1** | 微循环 | 阶段内迭代 | 单轮追问不足 | Ideate 内多轮头脑风暴 |
| **L2** | 中循环 | 相邻阶段回退 | 精炼时发现分解不合理 | Refine → Structure（重新分解） |
| **L3** | 宏循环 | 跨阶段调整 | 验证时发现方向偏差 | Validate → Sense（重新评估机会） |

### pace-biz 映射

| 阶段 | 对应子命令 |
|------|-----------|
| Sense | `opportunity`、`discover` Step 1（目标框定） |
| Ideate | `discover` Step 2-3、`import` Step 2（提取）、`infer` Step 1-2（分析） |
| Structure | `epic`、`decompose` |
| Refine | `refine` |
| Validate | `align` |
| Ready | 就绪度 >= 80% → 移交 `/pace-dev` |

## §2 Data Model（需求信息模型）

需求实体的层级结构和成熟度演进框架。

### 六层实体层级（纵轴）

```
Vision → OBJ → Theme(Epic) → BR → PF → CU(CR)
 愿景    目标    专题          业务   产品   变更
                              需求   功能   单元
```

每一层是对上一层的具体化分解：Vision 回答"为什么"，OBJ 回答"达成什么"，Theme 回答"围绕什么主题"，BR 回答"业务上要什么"，PF 回答"产品上怎么做"，CU 回答"代码上改什么"。

### 四级成熟度（横轴）

实体从创建到就绪经历四级成熟度演进：

| 级别 | 名称 | 特征 | 典型信息 |
|------|------|------|---------|
| **M0** | Skeleton（骨架） | 仅有名称和编号 | 标题、ID |
| **M1** | Intentional（有意图） | 有用户故事/描述和优先级 | 用户故事、优先级标记 |
| **M2** | Defined（已定义） | 有验收标准、边界条件、关联关系 | 验收标准 >= 2 条、上游关联、异常场景 |
| **M3** | Ready（就绪） | 就绪度 >= 80%，可进入开发 | 全维度覆盖，NFR 考量 |

**成熟度与 Process Model 的映射**：
- Sense/Ideate → 通常产出 M0 实体
- Structure → 通常产出 M0-M1 实体
- Refine → 将实体从 M1 提升到 M2-M3
- Ready → M3 确认

**重要**：M0-M3 是内部评估框架，不暴露给用户。用户侧用自然语言描述：骨架级（仅名称）、基本级（有描述和优先级）、详细级（有验收标准）、就绪级（可开发）。

### 五种关系类型

| 关系 | 方向 | 说明 | 示例 |
|------|------|------|------|
| **Decomposition** | 上→下 | 分解关系 | Epic → BR → PF |
| **Association** | 同层 | 同层依赖 | PF-001 depends-on PF-002 |
| **Traceability** | 下→上 | 价值追溯 | CR → PF → BR → OBJ |
| **Conflict** | 同层 | 定义矛盾 | BR-A 与 BR-B 需求冲突 |
| **Enrichment** | 外→内 | 新信息补充已有实体 | 文档中的详情丰富已有 PF |

### 变更追溯

实体变更通过溯源标记（`<!-- source: ... -->`）和迭代变更记录表追踪。每次变更记录：时间、类型（add/pause/resume/modify）、影响范围。

## §3 Discovery Model（需求发现模型）

三种输入源通过各自的交互模式进入共享的处理管道。

### 三类输入源

| 输入源 | 交互模式 | 特点 | 对应子命令 |
|--------|---------|------|-----------|
| **Conversational**（对话式） | 多轮对话，Claude 引导 | 渐进深入、适合模糊想法 | `discover` |
| **Document**（文档式） | 批量处理，结构化提取 | 高效、适合已有文档 | `import` |
| **Artifact**（工件式） | 自动分析，反向推断 | 无需输入、适合已有代码 | `infer` |

**关键设计决策**：三类输入源保留各自的交互差异（对话式的多轮追问 vs 文档式的批量展示 vs 工件式的自动报告），不做交互层统一。统一体现在处理管道层面。

### 六阶段统一处理管道

无论输入源是什么，实体候选都经过相同的六阶段处理：

```
Ingest → Extract → Reconcile → Confirm → Persist → Guide
 摄入     提取      调和         确认      持久化    引导
```

| 阶段 | 职责 | 对话式实现 | 文档式实现 | 工件式实现 |
|------|------|-----------|-----------|-----------|
| **Ingest** | 获取原始输入 | 用户回答 | 读取文件 | 扫描代码 |
| **Extract** | 识别实体候选 | 整理对话为候选 | 按映射规则提取 | 模块→功能映射 |
| **Reconcile** | 与已有实体对比 | 轻量快筛 | 完整四分类合并 | 差距分析 |
| **Confirm** | 用户裁定 | 候选树确认 | 合并计划 accept/reject | 逐项 accept/skip |
| **Persist** | 写入 .devpace/ | 创建文件 | 追加功能树 | 补充追踪 |
| **Guide** | 推荐下一步 | 成熟度提示+命令建议 | 成熟度提示+命令建议 | 成熟度提示+命令建议 |

### 智能路由

空参数调用 `/pace-biz` 时，根据项目上下文自动推荐入口：

```
检测 .md/.txt 文档 → 推荐 import
检测 src/lib/ 代码 → 推荐 infer
有活跃 Epic 待分解 → 推荐 decompose
其他             → 推荐 discover
```

`discover` Step 0 内还有二次路由：检测到文件路径→建议 import，检测到代码关键词→建议 infer。

### Reconcile 覆盖差异

三个子命令的 Reconcile 阶段实现深度不同：

| 子命令 | Reconcile 实现 | 合并分类 |
|--------|---------------|---------|
| `import` | 完整合并分析 | NEW/DUPLICATE/ENRICHMENT/CONFLICT（四分类+两层判断+阈值） |
| `discover` | 轻量快筛 | 标题关键词重叠检测，提醒但不阻断 |
| `infer` | 差距分析 | 未追踪/未实现/技术债务/文档漂移（四分类） |

## §4 Quality Model（质量评估模型）

四个互补维度评估需求的整体质量。

### 四维度定义

| 维度 | 名称 | 评估对象 | 回答的问题 | 主要消费者 |
|------|------|---------|-----------|-----------|
| **D1** | Readiness（就绪度） | 单个 BR/PF | "这个需求够不够详细？" | `refine`、`align` |
| **D2** | Alignment（对齐度） | 实体间关系 | "需求和目标是否对齐？" | `align` |
| **D3** | Completeness（完整度） | 价值链各层 | "哪个层级有覆盖空白？" | `view` |
| **D4** | Health（健康度） | 时间趋势 | "质量在变好还是变差？" | `align`（趋势追踪） |

### D1 就绪度

六维度加权评分，定义见 `_schema/readiness-score.md`（权威源）：
- 用户故事/描述、验收标准、优先级、上游关联、异常/边界、NFR 考量
- 阈值：>= 80% 就绪 | 60-79% 基本就绪 | < 60% 需精炼

### D2 对齐度

`align` 子命令的九维检查（权威实现在 `biz-procedures-align.md`）：
- OBJ 覆盖度、Epic-OBJ 关联、MoS 定义、Epic-BR 分解、BR-PF 分解
- 孤立实体检测、优先级一致性、需求就绪度分布、利益相关者覆盖

### D3 完整度

价值链各层的覆盖率统计：
- OBJ→Epic 覆盖率：每个 OBJ 是否有对应 Epic
- Epic→BR 分解率：每个 Epic 是否已分解出 BR
- BR→PF 分解率：每个 BR 是否已分解出 PF
- PF→CR 关联率：多少 PF 已进入开发

### D4 健康度

时间维度的质量趋势（`align` 趋势追踪写入 `metrics/insights.md`）：
- 滚动保留近期评估快照
- 三次连续退化触发警告
- 支持"在变好还是变差"的趋势判断

### 维度间协同

```
D1（单实体就绪度）→ 汇聚为 → D2（跨实体对齐度）
                                    ↑
D3（覆盖完整度）→ 补充空白发现 ─────┘
                                    ↓
D4（时间健康度）← 追踪趋势 ─── D2 快照历史
```

D1 是基础——单实体不就绪，跨实体对齐也无意义。D3 发现结构空白（某层缺失），D2 检查已有实体的一致性，D4 追踪这些指标随时间的变化。

## §5 模型间协同

四个模型不独立运作，而是协同驱动 /pace-biz 的智能行为：

### 空参数引导场景

```
Process Model: 判断当前所处阶段 → 推荐对应子命令
Data Model:    扫描实体成熟度分布 → 优先推荐精炼低成熟度项
Quality Model: 检查各维度得分 → 优先推荐最薄弱维度
```

### 创建后引导场景

```
Discovery Model: 完成 Persist 阶段后进入 Guide
Data Model:      评估新实体成熟度 → 提示骨架级实体数量
Process Model:   判断下一阶段 → 推荐 Structure 或 Refine
```

### 精炼反馈场景

```
Quality Model D1: 评估就绪度变化 → 展示前后对比
Process Model:    就绪度达标 → 推荐进入 Ready/开发
                  就绪度不达标 → 推荐继续 Refine 或回退 Structure
Quality Model D2: 距上次 align 过久 → 建议 Validate
```
