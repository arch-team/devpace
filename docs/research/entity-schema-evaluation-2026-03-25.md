### 关键发现 Top 5

1. **P0（系统性）**：4 个 schema（opportunity/epic/br/pf）缺失 Consumers section——Schema 变更时无法快速评估影响范围
2. **P1（系统性）**：状态命名三种风格（中文/英文/emoji+中文）缺少统一映射表——影响自动聚合的确定性
3. **P1（系统性）**：4 个 schema 缺少渐进阶段定义——与"渐进丰富"核心设计原则不一致
4. **P1（单点）**：cr-format（最复杂/最高 fan-in）和 project-format（最大）缺少独立容错章节
5. **P2（单点）**：MoS 聚合路径（BR→Epic→OBJ→Vision）缺少形式化计算规则

### 突出优势

- **渐进丰富模式**：零摩擦的桩→初始→成长→成熟路径，超越行业标准的"一次性完整定义"假设
- **双维度 MoS**：客户价值+企业价值渗透 OBJ/Epic/BR 三层，行业标准中无对应物
- **三层验收体系**：BR 成效指标 / PF 验收标准 / CR 验收条件，层次分明
- **溯源标记系统**：区分人类输入 vs Claude 推断，解决 AI 辅助需求管理的信任问题
- **cr-format 事件表**：17 种结构化事件 + 分钟级时间戳 + 操作者 + 交接列，审计级变更追踪

### 3.4 标准对照综合发现

#### 覆盖良好的领域

1. **需求分层体系**（三标准共同认可）：8 层实体模型（Vision→OBJ→OPP→Epic→BR→PF→CR→Project）提供了比 IEEE 830（单层 SRS）、BABOK（3 层）、SAFe（4 层）都更精细的需求分层
2. **可追溯性**（IEEE A1.5 / BABOK A2.5 / SAFe A3.8）：价值功能树+每个实体的交叉引用字段 = 完整双向追溯。溯源标记进一步增强审计能力
3. **验收标准体系**（IEEE A1.6 / BABOK A2.6）：业界罕见的三层验收——BR 业务结果 / PF 实现正确性 / CR 代码正确性
4. **变更管理**（IEEE A1.8 / BABOK A2.8）：CR 事件表的 17 种结构化事件+操作者+交接列超越所有三个标准
5. **需求生命周期**（BABOK A2.3）：每个实体有完整的渐进阶段/状态机，CR 的 7+1 态门禁机制是行业参考实现

#### 存在差距的领域

| 差距 | 涉及标准 | 影响度 | 说明 |
|------|---------|:------:|------|
| 接口需求无专门字段 | IEEE A1.2 | 中 | 多系统集成项目中 API 契约无结构化记录位置。缓解：devpace 定位单项目/小团队 |
| 利益相关者分析不系统 | BABOK A2.2 | 中 | 仅 Epic 有可选利益相关者表。缓解：用户故事格式隐含利益相关者视角 |
| WSJF 优先级框架缺失 | SAFe A3.6 | 低-中 | 有定性 P0/P1/P2，缺定量排序公式。缓解：OBJ MoS 已提供输入维度 |
| 业务规则缺少统一载体 | BABOK A2.6 | 低-中 | 散布在 BR 关键流程/CR 约束/Vision 假设中。缓解：BR 关键流程已是良好载体 |
| 多迭代对齐不足 | SAFe A3.7 | 低 | 缺少 PI 目标和迭代间依赖。缓解：devpace 目标为单团队敏捷 |

#### devpace 超越标准的创新设计

1. **渐进丰富模式（Progressive Enrichment）**：从最小桩状态自然生长到完整状态，解决行业标准"要求完整填写导致前期阻力"的核心痛点
2. **双维度 MoS**：客户价值+企业价值在 OBJ/Epic/BR 三层统一，防止价值偏斜
3. **溢出模式（Overflow Pattern）**：BR/PF 从内联自动溢出为独立文件，解决"小需求不值独立文件，大需求放不下"的矛盾
4. **溯源标记系统**：`<!-- source: user -->` / `<!-- source: claude -->`，AI 辅助需求管理的信任保障
5. **CR 事件表审计轨迹**：17 种事件 + 操作者(Claude/用户/系统) + 交接列 + 度量用途字段，超越所有三个标准

**详细说明**：

- **C1 全部 ✅**：每个 Schema 的字段覆盖度与其实体职责匹配，信息维度齐全
- **C2 两处 ⚠️**：pf-format 用"必填/否"替代"核心/渐进"，语义有差异；project-format 缺少系统性字段级标注表
- **C3 三处 ⚠️**：vision 的北极星值无类型约束（数字？百分比？文本？）；epic 的利益相关者"影响程度"未量化；pf 的状态用 emoji 未定义纯文本等价值

- **C5 一处 ⚠️**：opportunity 与 OBJ 无直接关联（OPP→OBJ 需经 Epic 间接追溯）

### 5.2 业务流程覆盖度（C6-C9）

#### C6 价值链追溯

**主链路**：Vision → OBJ → Epic → BR → PF → CR — **完整**

每个层级都有正向（向下关联表）和反向（向上追溯字段）追溯能力。

**侧链路**：Opportunity → Epic — **完整但单向**

OPP 采纳后关联 Epic（正向✅），Epic 来源字段记录 OPP-ID（反向✅），但 OPP→OBJ 需间接追溯。

**弱环节**：
- PF 未溢出时在 project.md 内联，正向追溯 CR 仅通过纯文本（`→ CR-001 🔄`），非结构化
- Vision→OBJ 的"北极星贡献"是渐进字段，可能长期为空

#### C7 状态流转

| 实体 | 状态数 | 计算方式 | 完整性 |
|------|:-----:|---------|:------:|
| Vision | 4 阶段 | 触发时机驱动 | ✅ |
| OBJ | 3 态 | 人类确认 | ⚠️ "已达成"阈值模糊 |
| Opportunity | 4 态 | 人工转换 | ✅ |
| Epic | 4 态 | BR 状态聚合计算 | ✅ |
| BR | 4 态 | PF/CR 状态聚合计算 | ✅ |
| PF | 5 态 | CR 状态聚合计算 | ✅ |
| CR | 8 态 | 门禁驱动（Gate 1/2/3） | ✅ 最完整 |
| Project | 渐进阶段 | 7 触发条件 | ✅ |

**关键发现**：自动聚合链（CR→PF→BR→Epic）跨越三种状态命名风格（英文→emoji+中文→中文→中文），缺少统一的跨实体状态映射表。

#### C8 生命周期

所有实体覆盖了创建→活跃→变更→暂停/恢复→归档/废弃的完整生命周期，有两个 Gap：
- **Vision 缺归档机制**：产品 pivot 时无"已废弃"或"历史版本"概念
- **Project 缺关闭状态**：项目终止时 Schema 未定义关闭流程

#### C9 闭环反馈

| 反馈机制 | 支撑字段 | 评估 |
|---------|---------|:----:|
| MoS 度量回顾 | OBJ/Epic/BR 的 MoS 进度字段 | ✅ |
| 北极星追踪 | Vision 当前值/目标值 | ✅ |
| CR 一次通过率 | 事件类型 gate1_pass/fail | ✅ |
| 核心假设验证 | Vision 假设状态 | ✅ |
| 变更审计 | CR 事件表 + PF history 注释 | ✅ |
| 经验积累 | insights-format（auxiliary） | ✅ |

**唯一弱点**：MoS 从 BR 聚合到 Epic 再到 OBJ 的计算方式未形式化定义，依赖 /pace-retro 时 Claude 推断。

### 5.3 非功能性评估（C10-C12）

#### C10 一致性

| 维度 | 现状 | 评估 |
|------|------|:----:|
| 状态命名 | 中文（OBJ/Epic/BR/OPP）/ 英文（CR）/ emoji+中文（PF） | ⚠️ |
| 字段分类 | 核心/渐进（6个）/ 必填/否（pf）/ 无统一表（project） | ⚠️ |
| 存储模式 | 独立/集中/内联+溢出三种 | ✅ 差异有设计理由 |
| 编号体系 | XXX-NNN 三位补零统一 | ✅ |
| §0 速查卡片 | 8/8 都有 | ✅ |
| 容错章节 | 6/8 有独立章节，cr/project 缺失 | ⚠️ |
| 更新时机 | 6/8 有结构化表格，cr 散落、project 段落式 | ⚠️ |

**主要不一致**：
1. 状态命名三种风格混合——设计理由存在（CR 技术闭环用英文/上层面向人类用中文），但 PF 的 emoji 引入了第三种风格
2. 字段分类标注不统一——pf-format 的"必填/否"与其他的"核心/渐进"语义不等价

#### C11 可维护性

| 机制 | 评估 |
|------|:----:|
| 字段缺失不影响功能 | ✅ 8/8 |
| 有/无独立文件双模式 | ✅ vision/obj/epic |
| 内联/溢出双模式 | ✅ br/pf |
| 旧格式仍合法 | ✅ cr 向后兼容规则 |
| 占位文字→实际内容渐进 | ✅ vision/project |
| 默认值明确 | ✅ project 配置 |

向后兼容设计是整套 Schema 的显著强项，渐进丰富模式优雅解决了"字段存在但内容可以很简单"的需求。

**潜在维护风险**：cr-format（447行，fan-in=10）修改影响面广；design.md 与 Schema 的渐进阶段双重定义存在 IA-6 单一权威风险。

#### C12 扩展性

| 扩展点 | 机制 | 评估 |
|--------|------|:----:|
| 新增 CR/OBJ 类型 | 枚举扩展 | ✅ |
| 新增自主级别 | 矩阵新增行 | ✅ |
| 新增外部集成 | integration/ 子目录 | ✅ |
| 溢出模式复用 | br/pf 模式可推广 | ✅ |

**扩展限制**：
- Opportunity 单文件集中模式，大量 OPP 时无溢出机制
- Vision 单一性假设（每项目一个），多产品线场景受限
- CR 事件表无归档/裁剪机制，长期运行可能过长
