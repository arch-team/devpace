# devpace 项目目标

> **职责**：定义项目的愿景、业务目标和成效指标。这是项目决策的最终依据。

## 北极星

**让任何 Claude Code 用户都能在人机协作下有序进行产品迭代研发——即使需求在变、优先级在调、范围在扩。**

## 目标用户

Claude Code 社区的开发者——任何使用 Claude Code 进行产品开发的个人或团队。

## 三个价值视角

### 客户价值

解决开发者在使用 Claude Code 做产品研发时的四个核心痛点：

| 痛点 | 表现 | devpace 的价值主张 |
|------|------|---------------------|
| 跨会话连续性断裂 | 每次开 Claude 都要重新解释"在哪、干嘛" | 自动恢复上下文，零手动解释 |
| 质量不一致 | Claude 有时跳过测试、忘记检查 | 质量检查体系自动执行，不可跳过 |
| 价值与技术脱节 | 做着做着偏离业务目标 | 端到端追溯，技术决策锚定业务价值 |
| 需求变更导致失序 | 需求变了不知道影响多大，进度混乱 | 变更影响分析 + 有序调整 |

### 企业价值

- Claude 的工作过程可审计、可追溯、质量可控
- 多项目并行时不丢上下文
- 研发效能可度量、可改进

### 生态价值

- 填补 Claude Code 生态中"研发流程管理"的空白
- 不替代 GitHub Issues/Jira，而是补充——管的是"Claude Code 会话级的研发节奏"
- 开源社区可贡献、可适配

## 差异化定位

与纯任务管理工具（TodoWrite、GitHub Issues）的区别：

| 维度 | 任务管理工具 | devpace |
|------|------------|-----------|
| 核心模型 | 任务列表 | 价值交付链路 |
| 对变更的态度 | 计划是稳定的 | **变更是一等公民** |
| 与 Claude 的关系 | Claude 是执行者 | Claude 是自治协作者 |
| 可追溯性 | 任务 → 代码 | 业务目标 → 需求 → 功能 → 变更 → 代码 |
| 度量 | 完成数 | 质量 + 价值对齐 |

## 业务目标

### MVP 阶段（1-2 周）

| ID | 目标 | 成效指标（MoS） | 验证方式 |
|----|------|---------------|---------|
| OBJ-1 | 跑通完整工作流 | 1 个 CR 从 created→merged 全流程走通 | 在 diagnostic-agent-framework 实际执行 |
| OBJ-2 | 跨会话连续性 | 连续 3 次会话零手动解释 | 模拟中断恢复场景 |
| OBJ-3 | 质量检查质量保障 | 所有 CR 的质量检查自动执行，无跳过 | 检查 CR 文件的质量检查 checkbox |
| OBJ-4 | 需求变更有序处理 | 需求插入/暂停/重排后功能树自动调整，已有工作不丢失 | 模拟 3 种变更场景 |

### 成熟阶段（1 个月）

| ID | 目标 | 成效指标（MoS） |
|----|------|---------------|
| OBJ-5 | 可移植性验证 | 新项目从 /pace-init 到首个 CR merged 不超过 1 小时 |
| OBJ-6 | 度量体系可用 | /pace-retro 生成有意义的回顾报告 |

### 生态阶段（长期）

| ID | 目标 | 成效指标（MoS） |
|----|------|---------------|
| OBJ-7 | 社区可用 | 新用户 5 分钟内跑通 /pace-init |
| OBJ-8 | 需求变更管理成为差异化卖点 | 用户反馈中"变更处理"出现频率 > 30% |
