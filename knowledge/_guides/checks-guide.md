# 质量检查编写指南

> **职责**：意图检查和对抗审查的编写最佳实践。格式契约见 `knowledge/_schema/process/checks-format.md`。

## §0 速查卡片

```
对抗审查：质疑式心态，必须找出问题，假阳性由人类过滤
意图检查：确认式心态，验证是否满足规则，Claude 可自修复
对抗审查范围：具体但不过窄（边界条件、安全隐患、性能问题）
意图检查三要素：可判定 · 可定位 · 可修复
```

## 对抗审查 vs 意图检查的区别

| 维度 | 意图检查 | 对抗审查 |
|------|---------|---------|
| 心态 | 确认式（验证是否满足规则） | 质疑式（假设存在问题去发现） |
| 零发现 | 正常（通过） | 异常（需重新分析） |
| 失败处理 | Claude 自修复 | 记录到 Review 摘要，由人类过滤 |
| 假阳性 | 不应有 | 预期存在，人类负责过滤 |

## 对抗审查编写指南

好的对抗审查范围：
- "检查边界条件处理和错误路径"
- "检查安全隐患和输入验证"
- "检查性能问题和资源泄露"
- "检查与现有代码的一致性和集成风险"

不推荐的范围（太泛或太窄）：
- "检查代码质量"（太泛，无法聚焦）
- "检查第 42 行"（太窄，失去探索价值）

## 意图检查编写指南

好的意图检查应具备：
- **可判定**：Claude 阅读代码后能明确判断通过/失败（非模糊的"代码质量好"）
- **可定位**：违反时能指向具体文件/函数（非泛泛的"需要改进"）
- **可修复**：Claude 能根据规则自行修复（非需要人类创意判断的）

好的示例：
- "所有公共函数必须有 docstring"
- "单个函数不超过 50 行"
- "不允许使用 eval() 或 exec()"
- "所有 API 端点必须有错误处理"
- "新增数据库查询必须有索引说明注释"

不推荐的示例（太模糊，不可判定）：
- "代码质量要好"
- "架构要合理"
- "性能要优化"
