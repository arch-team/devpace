# 分级输出参考指南

> **职责**：分级输出的示例表和格式规范。Rules §5 保留行为指令，本文件提供按需参考的示例和格式细节。

## §0 速查卡片

```
表面层：推理后缀（系统行为后附 ——[推理] ≤15 字）
中间层：追问展开（用户问"为什么" → 2-5 行推理链）
深入层：/pace-trace（完整决策轨迹 + 溯源标记）
```

## 分级输出场景表

| 场景 | 输出量 | 示例 |
|------|--------|------|
| 完成一个步骤 | 1 句话 | "标注完成，框架占比 75%。" |
| 会话结束 | 3-5 行 | 做了什么 + 质量检查概况 + 下一步 |
| 用户问详情 | 按需展开 | 质量检查清单、进度百分比 |
| `/pace-status detail` | 完整状态 | 全部 PF/CR 状态 |
| `/pace-retro` | 度量仪表盘 | dashboard.md 内容 |

| 系统行为 | 无推理（旧） | 有推理（新） |

## 中间层格式规范（追问展开）

**触发词**："为什么""怎么判断的""详细说说""依据是什么""解释一下"

**输出格式**（2-5 行）：

```
判断依据：
  1. [主要因素]——[具体数据/事实]
  2. [次要因素]——[具体数据/事实]
  建议：[下一步行动建议]
  → 完整决策轨迹：/pace-trace [当前CR] [对应类型]
```

**导航提示规则**：末尾的 `→ 完整决策轨迹` 行是从中间层到深入层的桥梁。仅在以下条件时附加：
- 追问内容对应已知决策类型（gate1/gate2/gate3/intent/change/risk）
- 当前处于推进模式且有活跃 CR

**示例**：

```
用户："为什么高风险？"
Claude："判断依据：
  1. auth/middleware.ts 是公共接口——被 3 个下游模块直接 import
  2. 涉及 JWT 验证逻辑变更——影响所有受保护路由
  3. 无现有测试覆盖这个代码路径
  建议：先补充集成测试再推进。"
```

## 深入层说明（/pace-trace）

用户通过 `/pace-trace` 主动查询完整决策轨迹。

**输出包含**（三段叙事结构）：
- 判定结论（结论先行 + 重建置信度）
- 评估过程（逐项检查 + 规则匹配 + 经验影响）
- 决策上下文（读取的信息 + 溯源标记）
- 上下文感知的导航建议

## 分层输出约定（SSOT）

子命令输出详细度的统一定义。各 Skill 的 `*-common.md` 引用本节而非各自重复定义。

| 层级 | 触发 | 内容 | 适用场景 |
|------|------|------|---------|
| 简要 | `--brief` | 1-3 行核心结论（关键指标、等级、下一步建议） | 自动化消费、快速确认 |
| 标准 | （默认） | 结构化表格 + 汇总 + 建议（各规程定义的输出格式） | 日常使用 |
| 详细 | `--detail` | 完整输出含历史对比、实施指导、扩展分析 | 深度审查、首次使用 |

**通用规则**：
- 未指定参数时使用"标准"层级（向后兼容）
- `--brief` 输出可被其他 Skill 或 report 子命令程序化消费
- 各子命令的"详细"内容由各规程文件定义
- 简要行统一风格：`类型：关键指标1 · 指标2 · 下一步/趋势`
- 各 Skill 可定义自动升级规则（特定上下文自动提升输出层级）
