# 质量检查格式契约

> **职责**：定义 .devpace/rules/checks.md 的结构。/pace-init 生成时遵循此格式。

## §0 速查卡片

```
三种检查类型：命令检查（bash 执行，exit code 判定）| 意图检查（自然语言，Claude 判定）| 对抗审查（强制找问题，Claude 判定）
Gate 归属：developing→verifying（Gate 1）| verifying→in_review（Gate 2，含对抗审查）
内置检查不可删除：需求完整性（Gate 1）+ 意图一致性（Gate 2）+ 对抗审查（Gate 2）
意图检查格式：`检查方式：Claude 检查 [自然语言规则]`
命令检查格式：`检查方式：[bash 命令]`
对抗审查格式：`检查方式：Claude 对抗审查 [必须找出至少 1 个问题或改进建议]`
验证证据：Gate 通过时 checkpoint 附带证据摘要（≤30 字），无新鲜证据不可声称通过
可选字段：依赖（短路逻辑）| 阈值（数值合格线）
安全检查：推荐按技术栈添加（npm audit / bandit / gosec）——非强制
```

## 检查项格式

```
- [ ] **[检查名称]**：[描述]
      检查方式：[命令检查 或 意图检查]
```

### 可选增强字段

检查项支持以下可选增强字段，提升检查流程的智能化：

#### 依赖（depends-on）

指定前置检查项。前置检查未通过时，自动跳过被依赖项（短路逻辑），避免无效检查噪声。

格式：在检查项描述行后添加 `依赖：[前置检查名称]`

```
- [ ] **测试覆盖率**：覆盖率不低于指定阈值
      检查方式：pytest --cov --cov-fail-under=80
      依赖：编译通过
```

#### 阈值（threshold）

数值型检查的合格阈值。与命令检查配合使用。

```
- [ ] **测试覆盖率**：覆盖率不低于指定阈值
      检查方式：pytest --cov --cov-fail-under=80
      阈值：80%
```

#### 敏感范围（sensitivity）

标注检查项敏感的文件/目录范围。`/pace-change modify` 时用于精确判断哪些检查需要重置（OPT-05）。

格式：在检查项描述行后添加 `敏感范围：[glob 模式]`

```
- [ ] **单元测试**：所有测试通过
      检查方式：pytest -v
      敏感范围：src/, tests/
- [ ] **类型检查**：TypeScript 类型正确
      检查方式：tsc --noEmit
      敏感范围：src/**/*.ts
- [ ] **API 文档**：API 文档与实际接口一致
      检查方式：Claude 检查 API 文档覆盖所有公开接口
      敏感范围：src/api/, openapi/
```

**重置判断规则**（`/pace-change modify` 使用）：
- 有 `敏感范围` 字段的检查项：对比变更文件列表与敏感范围 glob 模式，有交集才重置
- 无 `敏感范围` 字段的检查项：保守策略——任何功能性变更均重置
- 非功能性变更（仅文案/注释调整）：允许跳过重置（需 Claude 判定并说明理由）

**短路逻辑规则**：
- 配置了 `依赖` 字段的检查项，其前置检查失败时自动跳过
- 跳过的检查项标记为 `⏭️ 跳过（依赖 [前置检查] 未通过）`
- 目的：编译失败时跳过测试和 lint，避免一屏错误淹没根因

### 命令检查（command）

适用于有确定性工具的检查——编译、测试、lint、typecheck 等。

- **判定依据**：bash 命令的 exit code（0=通过，非0=失败）
- **失败处理**：Claude 分析输出 → 自修复代码 → 重跑命令
- **格式**：`检查方式：[bash 命令]`（如 `pytest -v`、`npm run lint`）
- **修复分类**：自修复分为 cosmetic（格式性）和 structural（结构性）两类，影响简化审批判定

#### 修复分类（影响简化审批）

| 类别 | 含义 | 示例 | 简化审批影响 |
|------|------|------|-------------|
| cosmetic | 格式性修复，不改变逻辑 | lint 修复、import 排序、format 对齐、尾随空白 | 不计入"非首次通过" |
| structural | 结构性修复，改变逻辑或行为 | 编译错误修复、测试逻辑错误、类型错误、缺失返回值 | 计入"非首次通过" |

判定规则：
- 命令检查失败后的自修复，由 Claude 根据修改内容分类
- 仅 `structural` 类自修复影响简化审批的"一次通过"判定
- 意图检查（claude-judgment）的自修复始终视为 `structural`（语义问题无"格式性"之分）

### 意图检查（claude-judgment）

适用于需要语义理解的检查——架构规范、文档完整性、设计模式遵循等。

- **判定依据**：Claude 阅读变更代码 → 对照自然语言规则 → 输出通过/失败+理由
- **失败处理**：Claude 分析未达标原因 → 自修复代码 → 重新判定
- **格式**：`检查方式：Claude 检查 [自然语言规则描述]`
- **输出格式**：失败时附加 1 句原因（如"函数 parse_config 有 67 行，超过 50 行限制"）

### 对抗审查（adversarial）

适用于 Gate 2 的深度审查——强制发现问题，对抗 Claude 的确认偏差。

- **判定依据**：Claude 假设代码存在问题，主动搜寻至少 1 个真实问题或改进建议
- **核心机制**：不是"检查是否有问题"，而是"必须找出问题"——利用 AI 被要求找问题时一定能找到的特性
- **失败处理**：找到的问题由人类 reviewer 过滤假阳性，Claude 不自动修复对抗审查发现的问题
- **格式**：`检查方式：Claude 对抗审查 [审查范围描述]`
- **输出格式**：`🔍 对抗审查发现 N 项：[问题列表]`

编写指南和示例详见 `knowledge/_guides/checks-guide.md`。

## 验证证据格式

Gate 1/2 通过时，在 CR 事件表备注列写入**验证摘要**——将瞬时验证结果持久化为可审计证据。

**铁律：无新鲜验证证据，不可声称门禁通过。**

### 证据格式指引

| 检查类型 | 证据格式 | 示例 |
|---------|---------|------|
| 命令检查 | `[命令关键字] [关键数据]` | `tsc 0 error`、`12/12 tests passed`、`lint 0 warnings` |
| 意图检查 | `[判定结论] [1 句依据]` | `意图一致——5/5 验收条件满足`、`范围匹配——无范围外改动` |

### Checkpoint 证据摘要

门禁通过时的 checkpoint 标记应附带验证摘要：

```
[checkpoint: gate1-passed] tsc 0 error, 12/12 tests passed, lint 0 warnings
[checkpoint: gate2-passed] 5/5 验收条件满足, 无范围外改动
```

规则：
- 每条证据摘要 ≤30 字
- 只记关键数据（错误数/通过率/覆盖率），不记完整输出
- 命令检查记录 exit code 关键输出，意图检查记录判定依据 1 句摘要
- 证据必须来自**本次**验证运行，不可引用之前的结果

### 常见声称与所需证据

| 声称 | 所需证据 | 不充分的证据 |
|------|---------|------------|
| "测试通过" | 完整 test output: 0 failures | "应该能通过"、之前的运行结果 |
| "编译成功" | build output: exit 0 | linter 通过（编译 ≠ lint） |
| "验收条件满足" | 逐条 checklist 对比 | "测试通过了"（测试通过 ≠ 需求满足） |
| "无范围外改动" | git diff 文件列表与意图范围对比 | "我只改了目标文件"（需独立验证） |

## 内置检查（不可删除）

| 检查名称 | Gate | 类型 | 描述 |
|---------|------|------|------|
| 需求完整性 | Gate 1 | 意图检查 | CR 意图 section 与变更复杂度匹配 |
| 意图一致性 | Gate 2 | 意图检查 | 实际变更与 CR 意图 section 一致 |
| 对抗审查 | Gate 2 | 对抗审查 | 假设代码存在问题，必须找出至少 1 个问题或改进建议 |

## 安全检查推荐

以下安全检查为推荐项（非强制），由 /pace-init 根据项目技术栈自动建议。

| 技术栈 | 推荐安全检查 | 检查方式 | Gate |
|--------|-------------|---------|------|
| Node.js | 依赖漏洞扫描 | `npm audit --audit-level=high` | Gate 1 |
| Python | 静态安全分析 | `bandit -r src/` 或 `safety check` | Gate 1 |
| Go | 安全扫描 | `gosec ./...` | Gate 1 |
| Rust | 依赖审计 | `cargo audit` | Gate 1 |
| 通用 | 密钥泄露检测 | `git secrets --scan` 或 Claude 检查"代码中不含硬编码密钥" | Gate 1 |

说明：
- 安全检查归入 Gate 1（代码质量），在编译和测试之后执行
- 项目可根据需要在 checks.md 中添加或移除
- /pace-init 生成 checks.md 时会将匹配的安全检查作为注释项包含（`<!-- 推荐：... -->`）
