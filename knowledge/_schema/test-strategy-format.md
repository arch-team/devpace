# 测试策略格式契约

> **职责**：定义 `.devpace/rules/test-strategy.md` 的结构。`/pace-test strategy` 生成时遵循此格式。

## §0 速查卡片

```
用途：将 PF 验收条件系统化地映射到测试策略
生成命令：/pace-test strategy
消费者：/pace-test coverage（覆盖率分析）| /pace-test impact（变更影响分析）| /pace-test report（汇总报告）
核心结构：策略总览表 + 每个 PF 的验收条件→测试映射表 + 建议列表
测试类型：unit | integration | E2E | performance | security | accessibility | manual（需人工验证）
状态标记：✅ 已有 | ❌ 待建 | ⚠️ 部分覆盖
优先级：P0 核心流程 | P1 重要功能 | P2 辅助功能
```

---

## 文件结构

```markdown
# 测试策略

> 由 /pace-test strategy 生成于 [日期]。基于 project.md 功能规格。

## 策略总览

| PF | 功能名 | 验收条件数 | 已覆盖 | 覆盖率 | 代码覆盖率 | 优先级 |
|----|--------|-----------|--------|--------|----------|--------|
| PF-001 | [功能名] | N | M | X% | —（待采集） | P0 |

## PF-001：[功能名]

### 验收条件→测试映射

| # | 验收条件 | 测试类型 | 工具/框架 | 测试文件/位置 | 状态 |
|---|---------|---------|----------|-------------|------|
| 1 | [条件文本] | unit | pytest | tests/test_xxx.py | ✅ 已有 |
| 2 | [条件文本] | E2E | playwright | — | ❌ 待建 |

### 建议

- [优先实现的测试及理由]

## 技术栈信息

- **语言**：[检测到的语言]
- **测试框架**：[检测到的框架]
- **来源**：[.devpace/context.md | 自动检测]
```

## 字段定义

### 策略总览表

| 字段 | 必填 | 说明 |
|------|------|------|
| PF | 是 | PF 编号，与 project.md 价值功能树一致 |
| 功能名 | 是 | PF 标题 |
| 验收条件数 | 是 | 该 PF 下可测试的验收条件总数 |
| 已覆盖 | 是 | 有对应测试（✅ 已有 或 ⚠️ 部分覆盖）的条件数 |
| 覆盖率 | 是 | 已覆盖 / 验收条件数，百分比（需求覆盖率） |
| 代码覆盖率 | 否 | 行覆盖率百分比（由 /pace-test coverage 采集，辅助信号） |
| 优先级 | 是 | P0（核心流程）/ P1（重要功能）/ P2（辅助功能） |

### 验收条件→测试映射表

| 字段 | 必填 | 说明 |
|------|------|------|
| # | 是 | 验收条件序号（与 PF 内验收标准的顺序一致） |
| 验收条件 | 是 | 验收条件文本（从 project.md 提取） |
| 测试类型 | 是 | 主类型：`unit` / `integration` / `E2E` / `performance` / `security` / `accessibility` / `manual`；可含辅助类型后缀：`unit [+security]`（0-2 个辅助类型） |
| 工具/框架 | 是 | 具体测试框架（pytest / jest / playwright / vitest 等）；manual 类型填 `—` |
| 测试文件/位置 | 否 | 已有测试的文件路径；待建时填 `—` |
| 状态 | 是 | `✅ 已有` / `❌ 待建` / `⚠️ 部分覆盖` |

### 建议列表

每个 PF section 末尾的建议列表，按以下优先级排序：

1. P0 功能的 ❌ 待建 测试（最高优先）
2. P1 功能的 ❌ 待建 测试
3. ⚠️ 部分覆盖的测试补全建议
4. P2 功能的测试建议

### 技术栈信息

| 字段 | 必填 | 说明 |
|------|------|------|
| 语言 | 是 | 项目主要编程语言 |
| 测试框架 | 是 | 项目使用的测试框架 |
| 来源 | 是 | 技术栈信息来源：`.devpace/context.md` 或 `自动检测`（注明检测文件） |

## 测试类型推荐规则

### 主类型判定

| 验收条件特征 | 推荐主类型 | 理由 |
|-------------|-----------|------|
| 数据校验、计算逻辑、格式转换 | unit | 无外部依赖，可精确断言 |
| 模块间交互、API 调用、数据库操作 | integration | 需要组件协作验证 |
| 用户交互流、页面跳转、端到端场景 | E2E | 需模拟真实用户操作 |
| 响应时间、吞吐量、并发负载、资源消耗 | performance | 需基准测量和阈值断言 |
| 认证、授权、输入校验、数据保护、密钥管理 | security | 需安全扫描工具或渗透测试 |
| 屏幕阅读器、键盘导航、色彩对比、WCAG 合规 | accessibility | 需自动化可访问性审计（axe/pa11y） |
| 视觉效果、用户体验、主观判断 | manual | 无法自动化验证 |

### 辅助类型触发条件

辅助类型叠加到主类型之上（每条 0-2 个），输出格式如 `unit [+security]`：

| 触发条件 | 辅助类型 | 示例 |
|---------|---------|------|
| 含安全关键词（认证/授权/加密/敏感数据） | +security | "用户能安全上传 CSV" → `unit [+security]` |
| 含性能关键词（响应时间/吞吐量/超时/并发） | +performance | "批量导入 1000 条记录在 5s 内完成" → `integration [+performance]` |
| 含用户交互 + 数据操作（且主类型为 unit） | +integration | "用户填写表单后数据正确保存" → `unit [+integration]` |

## 状态标记规则

| 状态 | 条件 | 说明 |
|------|------|------|
| ✅ 已有 | 找到对应测试文件且包含相关测试用例 | 该验收条件已有自动化测试覆盖 |
| ❌ 待建 | 未找到对应测试 | 该验收条件需要新建测试 |
| ⚠️ 部分覆盖 | 有测试但未完全覆盖验收条件的所有方面 | 需要补充测试用例 |
