# 经验驱动决策参考

> **职责**：经验引用时机的详细描述和纠正检测规则。Rules §12 保留核心行为指令，本文件提供按需参考的完整规则。

## §0 速查卡片

```
5 个引用时机：推进前 | 质量门 | 变更管理 | 迭代规划 | 会话开始
纠正检测：否决审批 | 修改 PF | 修改影响评估 | 否决 Gate 2 | 修改意图
交互："记住这个偏好？以后 [场景] 时我会 [调整后的行为]。" → 用户确认才记录
```

## 引用时机详述

### 时机 1：进入推进模式前

1. 读取 `.devpace/metrics/insights.md`（不存在则跳过）
2. 匹配当前 CR 类型（功能/修复/重构）与已有 pattern 的类型标签
3. 有匹配 pattern 时：
   - 成功 pattern → 在意图检查点中采用推荐策略
   - 防御 pattern → 预先告知风险（"根据历史经验，此类变更容易 [问题]，建议 [预防措施]"）
   - 改进 pattern → 提前纳入注意事项

### 时机 2：质量门执行时

1. Gate 执行前读取 insights.md 中与质量相关的 pattern
2. 有相关防御 pattern → 额外关注该 pattern 描述的问题领域
3. Gate 失败后记录原因到 insights.md（通过 pace-learn 自动提取）

### 时机 3：变更管理时（§9 / pace-change）

1. 执行影响分析前读取 insights.md 中与变更相关的 pattern
2. 有同类变更 pattern → 在影响评估中引用历史数据（"历史上类似范围的变更平均影响 N 个模块"）
3. 有变更回滚 pattern → 在风险评估中提升风险等级并引用原因

### 时机 4：迭代规划时（pace-plan）

1. 评估工时/复杂度前读取 insights.md 中与任务规模相关的 pattern
2. 有同类任务 pattern → 引用历史 checkpoint 数作为估算参考（"同类任务历史平均 N 个 checkpoint"）
3. 有迭代回顾 pattern → 提示上次迭代的关键经验

### 时机 5：会话开始时（§1）

1. §1 节奏检测时读取 insights.md（不存在则跳过）
2. 匹配当前进行中 CR 的类型与已有 pattern
3. 有高优先级防御 pattern → 在 1 句话报告后附加 1 句提醒（"注意：此类变更历史上容易 [问题]"）
4. 每会话最多附加 1 条经验提醒（不计入 §10 的 3 条上限）

## 纠正检测表

| 纠正行为 | Claude 的原始判断 | 检测方式 |
|---------|------------------|---------|
| 用户否决简化审批（"看看"而非"合并"） | Claude 判断满足简化审批条件 | 简化审批流程中用户选择"看看" |
| 用户手动修改 Claude 的 PF 分解 | Claude 推断了 PF 名称/结构 | 溯源标记检测：`source: claude` 内容被用户编辑 |
| 用户修改 Claude 的变更影响评估 | Claude 判断为"低/中/高风险" | 用户在变更确认时修改风险等级 |
| 用户否决 Gate 2 验收比对结果 | Claude 判断验收条件满足 | in_review 阶段用户打回并指出验收不满足 |
| 用户修改 CR 意图 section | Claude 自动生成的意图描述 | 溯源标记检测：`source: claude` 的意图内容被用户修改 |

## 纠正交互流程

1. **检测到纠正行为**（基于上表的检测方式）
2. **Claude 主动提问**（1 句话）：`"记住这个偏好？以后 [场景] 时我会 [调整后的行为]。"`
3. **用户确认** → 写入 insights.md 偏好条目（格式见 `knowledge/_schema/insights-format.md`）
4. **用户拒绝** → 不记录，无副作用
5. **后续引用**：§12 五个引用时机中，偏好类型条目优先级高于模式类型
