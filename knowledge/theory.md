## §1 BizDevOps 定义

BizDevOps 是数字化时代的产品交付及业务创新方法体系。它通过业务和技术的有机融合和有效协同机制，打通从业务到开发再到运维和运营的价值交付链和反馈闭环，并在此基础上实现全链路的数字化运作，保障并持续改进产品和业务交付的效率、质量和有效性。

### vs-devops：与 DevOps 的关系

DevOps 打通了开发与运维的壁垒，BizDevOps 在此基础上向前延伸到业务侧，形成"业务-开发-运营"三位一体。

| 维度 | DevOps | BizDevOps |
|------|--------|-----------|
| 关注点 | 开发-运维效率 | 业务价值端到端交付 |
| 起点 | 代码提交 | 业务需求/机会 |
| 度量 | 部署频率、MTTR 等 | + 业务价值对齐、MoS 达成率 |
| 协同范围 | Dev + Ops | Biz + Dev + Ops |
| 核心差异 | 技术效能优化 | 业务价值驱动的全链路协同 |

### 三大核心能力

1. **业务与技术的协同能力**：打通业务目标→技术实现的映射，确保技术工作锚定业务价值
2. **持续交付能力**：从需求到上线的自动化、流程化、可度量的交付链路
3. **业务运营能力**：产品上线后的业务反馈闭环，数据驱动的持续改进

## §2 概念模型三要素（model）

BizDevOps 核心主张：**概念模型是一切的基础**——没有统一的概念模型，就无法对齐目标、打通链路、产生可用数据、持续改进。

三要素：

| 要素 | 定义 | 回答的问题 |
|------|------|-----------|
| **作业对象** | 价值交付链路上的基本单元 | "交付什么？" |
| **作业空间** | 容纳并操作作业对象的空间 | "在哪里协作？" |
| **作业规则** | 流程、规范、约束的集合 | "怎么流转？" |

三者的关系：作业对象在作业空间中，按照作业规则流转。流转的过程就是价值交付的过程。

## §3 作业对象详解（objects）

作业对象是价值交付链路上的基本单元，随时间发生状态迁移和流转。

### 3.0c 产品愿景（Vision）

- **定义**：产品的长期方向锚点，描述"为谁解决什么问题，为什么选择我们"
- **核心要素**：目标用户 + 核心问题 + 差异化 + 成功图景
- **特点**：
  - 愿景是战略层面的锚定——所有 OBJ 和 Epic 应当回溯到愿景
  - 包含北极星指标（单一核心量化指标）作为愿景的量化锚点
  - 包含战略上下文（核心假设 + 外部约束）辅助决策
- **核心观点**：愿景必须来自人类——它代表产品方向的人类意志，不可由 AI 推断。愿景的价值在于提供一个稳定的方向参照，使得所有下游决策（OBJ、Epic、BR）都能回溯到"我们为什么做这件事"

### 3.0d 业务目标（OBJ - Objective）

- **定义**：从愿景分解出的可度量的业务目标，描述"要达成什么具体成果"
- **来源**：愿景分解、战略规划、业务回顾
- **类型**：6 类——`business`（收入盈利）、`product`（产品体验）、`tech`（技术基础设施）、`growth`（用户增长）、`efficiency`（运营效率）、`compliance`（合规安全）
- **特点**：
  - 每个 OBJ 有双维度成效指标（MoS）——客户价值和企业价值
  - 北极星贡献描述 OBJ 与 Vision 北极星指标的关系
  - OBJ:Epic 为 1:N 关系，一个 Epic 有主 OBJ + 可选副 OBJ
  - 3 态状态：活跃 → 已达成 / 已废弃
- **核心观点**：OBJ 是连接"产品愿景"和"专题规划"的中间层。OBJ 的 MoS 双维度设计确保每个目标同时关注"对谁有价值"——客户价值（用户体验、满意度）和企业价值（收入、效率、合规）

### 3.0a 业务机会（Opportunity）

- **定义**：外部或内部信号中识别到的潜在业务价值点
- **来源**：用户反馈、竞品观察、技术发现、市场趋势、内部洞察
- **特点**：是原始输入信号，需要经过评估才能转化为具体的 Epic 或 BR
- **核心观点**：业务机会是价值链的最上游——它连接了外部世界（客户、市场、技术变化）和内部规划（Epic、BR）。不是所有机会都值得追踪，但被捕获的机会提供了需求的源头追溯

### 3.0b 专题（Epic）

- **定义**：为实现某个业务目标而组织的一组相关业务需求的集合
- **来源**：业务机会评估、战略规划、业务目标分解
- **特点**：
  - 比单个 BR 粒度更大，通常跨多个迭代
  - 有自己的成效指标（MoS），用于衡量专题级别的业务成果
  - 是"拥抱不确定性"的载体——专题用 MoS 衡量成果而非固定范围
- **核心观点**：专题是连接"业务目标"和"具体需求"的中间层。一个业务目标可以分解为多个专题，每个专题聚焦一个业务主题

### 3.1 业务需求（BR - Business Requirement）

- **定义**：从业务视角提出的需求，描述"业务上要什么"
- **来源**：Epic 分解、业务机会、日常需求（快速注入）
- **特点**：面向业务目标，由业务团队/产品经理提出
- **核心观点**：业务需求来源于业务机会，是产品和交付团队为了抓住业务机会而采取的具体行动

### 3.2 产品需求/产品功能（PF - Product Feature）

- **定义**：BR 分解后的产品功能点，描述"产品上怎么做"
- **来源**：对 BR 的产品化分解
- **特点**：可独立交付和验证的功能单元
- **核心观点**：产品需求是基于业务需求视角的产品解决方案，一个业务需求可以被分解为多个产品需求
- **关键洞察**：PF 是连接"业务语言"和"技术语言"的翻译层

### 3.3 变更请求（CR - Change Request）

- **定义**：为实现特定的产品需求或修复缺陷，某个应用所需完成的工作
- **来源**：对 PF 的技术分解
- **关键特征**：
  - 变更请求包含了设计、编码、提交和部署等在内的完整过程
  - 与分散的技术任务不同，变更请求聚合了为响应特定产品需求，应用上所发生的所有技术活动
  - 变更请求是连接产品需求和技术实现的桥梁
- **特点**：
  - 每个变更对应一个代码分支
  - 有明确的状态流转（开发→测试→预发→生产）
  - 一个 PF 可分解为多个 CR（按应用拆分）

### 3.4 发布（Release）

- **定义**：将开发完毕的功能分发到目标环境中，使之对外部可见和可用的过程
- **特点**：可包含多个变更请求的组合发布，有独立的集成验证流程

### 3.5 缺陷（Defect）

- **定义**：产品上线后发现的问题
- **特点**：缺陷也是一种作业对象，有自己的生命周期

### 对象间的分解关系

```
Vision（产品愿景）
  │ 北极星指标
  ↓
OBJ（业务目标）─── 北极星贡献 ───→ Vision
  │ 双维度 MoS（客户价值 + 企业价值）
  │ 1:N（主 OBJ + 可选副 OBJ）
  ↓
Epic（专题）←─── 评估转化 ─── Opportunity（业务机会）
  │ 双维度 MoS          （侧入口：外部信号经评估后转化为 Epic）
  │ 1:N
  ↓
BR（业务需求）── 1:N ──→ PF（产品功能）── 1:N ──→ CR（变更请求）── N:1 ──→ Release（发布）
  │                                                  │
  ↑                                                  ↓
双维度 MoS                                      Application（应用）
```

Vision 是战略起点，分解为多个 OBJ；每个 OBJ 通过北极星贡献回溯到 Vision。OBJ:Epic 为 1:N 关系，一个 Epic 有主 OBJ + 可选副 OBJ。业务机会经评估可转化为 Epic；一个 Epic 可分解为多个 BR；一个 BR 可分解为多个 PF；一个 PF 可分解为多个 CR（当涉及多个应用时）；多个 CR 可组合为一次 Release。日常需求可跳过 Opportunity/Epic 直接创建 BR。MoS 双维度（客户价值 + 企业价值）渗透到 OBJ/Epic/BR 三层，确保价值链每一层都关注"对谁有价值"。

## §4 作业空间详解（spaces）

作业空间是容纳并操作作业对象的空间，在其中定义作业的规则、流程、计划，提供或链接作业所需的资源。

### 4.1 产品线（Product Line）

- **定义**：业务规划和管理的空间，包含业务目标、需求全景、价值追溯
- **核心定位**：产品线是业务和产品规划的载体，承载从业务机会到产品需求的完整链路
- **包含**：业务愿景、业务目标、成效指标、业务需求、产品需求

### 4.2 交付团队（Delivery Team）

- **定义**：跨职能的产品交付团队，负责产品需求的技术实现
- **核心定位**：交付团队是一种虚拟的组织形式，围绕价值交付而组建
- **特点**：团队成员来自不同职能，共同负责一组应用的交付

### 4.3 应用（Application）

- **定义**：云原生环境下最基本的部署单元和运维对象
- **核心定位**：
  - 应用拥有自己的代码、配置和数据，可以被单独变更、部署
  - 应用是研发团队日常工作的第一入口
  - 是资源聚合和治理维度
  - 是团队工程效能度量和分析的主要视角
- **特点**：每个应用有唯一的负责团队；由 1 到多个组件构成

### 4.4 发布单元（Release Unit）

- **定义**：发布的最小粒度，可能是单个应用或多个应用的组合

### 空间间的关系

```
产品线 ───包含───→ 业务需求 ───分解为───→ 产品需求
                                            │
                                        分配给
                                            ↓
交付团队 ───负责───→ 应用 ←───归属─── 变更请求
                      │
                  组成
                      ↓
                  发布单元
```

## §5 作业规则详解（rules）

作业规则是流程、规范、约束的集合，定义作业对象如何在作业空间中流转。

### 5.1 工作流（Workflow）

- **定义**：作业对象的状态流转规则
- **特点**：
  - **可配置化**：不同项目/团队可自定义工作流
  - 有明确的**准入和准出条件**
  - 支持多种**分支策略**
- **变更的完整生命周期**：
  1. 开发阶段：每个变更对应一个代码分支，在隔离环境中工作
  2. 集成验证阶段：多个变更的集成验证，失败的变更从集成列表移除
  3. 发布阶段：合入发布分支，构建和发布前验证
  4. 部署完成后：变更分支自动合入主干并关闭

### 5.2 门禁（Gates / Checkpoints）

- **定义**：阶段流转的质量检查点
- **核心原则**：
  - 每个阶段有明确的准入和准出条件
  - 变更进入预发阶段前必须通过测试阶段的验证
  - 门禁可配置为**自动检查**或**人工审批**
  - 门禁是保障质量的核心机制

### 5.3 规则的配置化

BizDevOps 强调规则应该是**可配置的**，而非硬编码的：
- 工作流阶段可增减
- 门禁检查项可按项目定制
- 分支策略可按团队选择
- 审批流程可按风险等级调整

## §6 价值交付链路与双向追溯（trace）

这是 BizDevOps 区别于纯 DevOps 的核心能力。

### 端到端链路

```
产品愿景(Vision) → 业务目标(OBJ) → 专题(Epic) → 业务需求(BR) → 产品功能(PF) → 变更请求(CR) → 发布(Release)
    北极星指标        北极星贡献      主/副OBJ       双维度MoS
                    双维度MoS       双维度MoS
```

Vision 作为战略起点，通过北极星指标锚定方向。OBJ 通过北极星贡献回溯到 Vision。MoS 双维度（客户价值 + 企业价值）从 OBJ 渗透到 Epic 和 BR，确保价值链每一层都明确"对谁有价值"。

### 度量链

MoS 沿价值树向上聚合：CR 完成推动 PF 完成，PF 完成推动 BR 的 MoS 达成，BR 的 MoS 聚合到 Epic 的 MoS，Epic 的 MoS 贡献到 OBJ 的 MoS，最终体现在 Vision 的北极星指标上。日常操作中度量链隐式存在（通过树结构推断），/pace-retro 时 Claude 自动生成贡献分析。

### 双向追溯

**从业务看技术**（自上而下）：
- 站在业务的角度，从业务需求追溯与之相关的所有产品需求
- 以及为了实现这些产品需求所涉及的所有应用变更及变更的发布情况
- **价值**：保证工程技术团队总是聚焦在最重要的业务价值上

**从技术看业务**（自下而上）：
- 站在技术的角度，从变更请求追溯其实现的产品需求
- 以及这些产品需求所承载的业务需求
- **价值**：使得技术在服务于业务价值的基础上，加速业务价值的验证

### 价值追溯树的体现

价值追溯树就是这条链路的可视化表达：

```
OBJ-1（业务目标）
├── BR-001：[业务需求]
│   ├── PF-001：[产品功能] → CR-001 🔄
│   └── PF-002：[产品功能] → CR-002 ✅
└── BR-002：[业务需求]
    └── PF-003：[产品功能] → (待创建 CR)
```

### 端到端分解示例：Todo App

以一个 Todo 应用为例，展示从业务目标到代码变更的完整分解过程：

**业务目标（OBJ）**：构建一个好用的个人任务管理工具

**成效指标（MoS）**：
- 用户可以完成基本的增删改查操作
- 支持到期日提醒

**分解为业务需求（BR）→ 产品功能（PF）→ 变更请求（CR）**：

```
OBJ-1（个人任务管理工具）
├── BR-001：任务基本操作
│   ├── PF-001：添加任务（作为用户，我可以创建新任务）
│   │   └── CR-001：实现 add 命令 + 数据存储 ✅
│   ├── PF-002：查看任务列表
│   │   └── CR-002：实现 list 命令 + 格式化输出 ✅
│   └── PF-003：删除任务
│       └── CR-003：实现 delete 命令 ✅
├── BR-002：任务增强功能
│   ├── PF-004：到期日设置
│   │   └── CR-004：添加 --due 参数 + 到期高亮 🔄
│   └── PF-005：任务优先级（变更新增）
│       └── (待创建 CR)
└── BR-003：数据持久化
    └── PF-006：本地文件存储
        └── CR-005：JSON 文件读写 ✅
```

**分解中的关键时刻**：
1. `/pace-init`：用户描述"做个 Todo 应用"→ Claude 推断 OBJ + BR + 初始 PF
2. 首个 CR 创建：PF-001 从"功能标题"丰富为"用户故事 + 验收标准"
3. `/pace-change add`：用户说"加个优先级功能"→ 插入 PF-005 到 BR-002 下
4. `/pace-change pause`：用户说"先不做优先级"→ PF-005 暂停 ⏸️
5. `/pace-retro`：MoS 达成率 1/2（基本操作 ✅，到期提醒 🔄）

这个过程中，用户只需说自然语言，BR→PF→CR 的概念模型由 Claude 自动维护。

每次 CR 状态变更时自动更新追溯树，保持业务-技术视图的实时同步。

## §7 专题模式与成效指标（topic）

BizDevOps 提出"专题"模式作为对传统"项目"模式的补充和超越。

### 专题 vs 项目

| 维度 | 项目模式 | 专题模式 |
|------|---------|---------|
| 确定性假设 | 范围、时间、成本预先确定 | **拥抱不确定性** |
| 考核标准 | 按期交付预先确定的内容 | **成效指标（MoS）达成** |
| 迭代方式 | 按计划执行 | **持续迭代、反馈、调整** |
| 失败定义 | 未按计划交付 | 未达成或未超越成效指标 |
| 需求变更态度 | 变更是风险，需控制 | **变更是常态，需拥抱** |
| 团队心态 | 完成任务 | 达成成效 |

### 成效指标（MoS - Measures of Success）

- **定义**：专题在启动时确定的、衡量其结果的标准
- **核心观点**：考核专题成功与否的并非是按期交付预先确定的内容，而是达成或超越所定义的成果
- **特点**：
  - 在启动时就确定"什么算成功"
  - 关注数据的可获取性
  - 逐步形成自动化的获取、处理和呈现机制
  - 不是固定范围，而是固定目标

### 专题的关键启示

专题模式的核心理念：拥抱不确定性，基于专题的定位和成果定义，团队动态规划专题的具体内容，并在过程中持续迭代、反馈和调整。

## §8 度量体系（metrics）

### DIKW 模型

BizDevOps 度量的理论基础：

| 层级 | 含义 | 在研发中的体现 |
|------|------|--------------|
| **Data**（数据）| 原始事实 | Git commits、CI 日志、门禁结果 |
| **Information**（信息）| 加工后的数据 | 变更周期、门禁通过率 |
| **Knowledge**（知识）| 理解和模式 | "A 类门禁失败率高是因为 X" |
| **Wisdom**（智慧）| 决策和预见 | "下个迭代应调整门禁策略" |

### 三维度量

BizDevOps 定义三个维度的度量指标。devpace 选用的具体指标定义见 [metrics.md](metrics.md)（单一来源）。

#### 维度一：持续交付效能（源自 DORA 四指标）

变更前置时间、部署频率、变更失败率、恢复时间（MTTR）。devpace 核心不强制纳入此维度；启用 Release 管理后 DORA 指标自动生效（见 metrics.md）。

#### 维度二：质量保障效能

质量检查一次通过率、人类打回率、缺陷逃逸率。

#### 维度三：业务价值对齐

成效指标达成率（MoS）、价值链完整率、需求交付周期。

### 度量应用原则

BizDevOps 强调度量不是为了考核，而是为了改进：
- 度量应该基于**客观数据自动生成**，不增加额外填写负担
- 度量应该服务于**团队自省**，不用于个人绩效评比
- 度量应该关注**趋势**而非绝对值
- 数据从已有的作业对象状态和 Git 历史中提取，**零额外录入**

## §9 三个反馈闭环（loops）

BizDevOps 的价值在于打通三个层次的反馈闭环：

### 业务闭环（Biz Loop）

- **主导方**：业务团队 / 产品经理
- **关注**：业务目标是否达成、MoS 是否满足
- **周期**：专题 / 季度级

### 产品闭环（Product Loop）

- **主导方**：产品经理 + 技术团队
- **关注**：产品功能是否按预期交付、用户反馈如何
- **周期**：迭代级

### 技术闭环（Dev+Ops Loop）

- **主导方**：技术团队
- **关注**：代码质量、测试通过、门禁合规
- **周期**：变更级（每个 CR）

### 闭环间的协同

```
业务闭环
  │ MoS 定义 ↓         ↑ MoS 达成反馈
产品闭环
  │ PF 拆解 ↓           ↑ 交付进度反馈
技术闭环
  └─ CR 状态机运转 ──────┘
```

业务闭环定义"做什么"和"怎样算成功"，产品闭环将其拆解为可交付的功能，技术闭环负责高质量实现。各闭环的反馈向上传递，驱动调整和改进。

## §10 变更管理的理论依据（change）

### BizDevOps 的立场

BizDevOps 通过"专题模式"明确表态：**变更不是异常，是常态。**

核心理念：拥抱不确定性，在过程中持续迭代、反馈和调整。

### 变更在概念模型中的位置

变更发生在所有层级：
- **业务层变更**：业务目标调整、MoS 重定义 → 影响所有下游
- **产品层变更**：PF 新增/暂停/修改 → 影响 CR 和迭代计划
- **技术层变更**：CR 返工/暂停 → 影响门禁状态和分支

### 变更的有序处理原则

1. **影响分析先行**：变更前必须评估对追溯链路的影响（双向追溯能力使这成为可能）
2. **保留工作成果**：暂停不等于删除，已完成的工作应可恢复（paused 状态）
3. **追溯树同步**：变更后追溯树必须反映最新状态，保持业务-技术视图一致
4. **变更可审计**：所有变更记录到迭代的变更记录表，支持回顾分析

## §14 规格驱动开发参考（sdd）

Spec Kit（GitHub 71K star）代表了"规格驱动开发"（Specification-Driven Development）范式——在编码前通过结构化模板强制深度思考。其核心阶段与 devpace 概念的映射：

| SDD 阶段 | 核心内容 | devpace 对应概念 | 映射方式 |
|----------|---------|-----------------|---------|
| Constitution | 项目范围 + 核心原则 + 技术栈 | project.md 范围 + 项目原则 | 渐进式桩填充（非前置模板） |
| Specify | 功能规格 + 验收标准 + 边界定义 | PF 用户故事 + CR 验收条件 + 功能规格 | 复杂度自适应格式（S/M/L） |
| Plan | 实施计划 + 架构决策 + 风险评估 | CR 意图 section + 事件表决策记录 | 意图检查点自动生成 |
| Tasks | 原子任务 + 依赖关系 + 验收标准 | CR 拆分建议 + 关联关系 + 质量门 | Gate 1/2/3 结构化验收 |

### devpace 如何适配 SDD

SDD 是静态前置模板范式（先填完再编码），devpace 通过以下设计实现同等的"结构化思考纪律"，同时保持 BizDevOps 的"拥抱变更"原则：

| SDD 特点 | devpace 适配 | 设计理由 |
|----------|-------------|---------|
| 前置模板（Constitution 必填） | 渐进式桩填充（范围/原则在自然时机填充） | P3 副产物非前置——不阻断启动 |
| 静态格式（Specify 固定模板） | 复杂度自适应格式（简单=自由文本，复杂=Given/When/Then） | 简单任务不加负担，复杂任务强制结构化 |
| 人工驱动（Plan 需人类编写） | AI 自动生成（意图检查点 Claude 自主完成） | 技术闭环 Claude 自治 |
| 确定性假设（Tasks 预先确定） | 拥抱变更（/pace-change + paused 状态） | 专题模式核心理念 |
| 歧义显式化（Specify 阶段标记） | 歧义标记 `[待确认]`（意图检查点自动添加） | Gate 2 前必须解决 |

### 关键启示

SDD 的核心价值在于"结构化思考纪律"——迫使团队在编码前想清楚做什么、怎么做、怎样算完成。devpace 通过三个机制实现同等纪律：

1. **意图检查点**：created→developing 时强制明确范围和验收条件（对应 Specify）
2. **验收标准结构化**：复杂度自适应格式确保复杂 CR 有精确的验收定义（对应 Specify + Tasks）
3. **歧义标记**：`[待确认]` 机制将隐式假设显式化，Gate 2 前强制解决（对应 Specify 的完备性）

区别在于：SDD 是"先写完规格再开始"，devpace 是"边做边结构化，但关键门禁前必须到位"——这是 BizDevOps"拥抱不确定性"与 SDD"结构化纪律"的融合。
