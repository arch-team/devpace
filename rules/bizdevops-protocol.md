# BizDevOps 研发协作协议

> **职责**：定义 Claude 在研发协作中的行为规则。加载 devpace Plugin 后自动注入会话。

## §0 速查卡片

```
会话开始 → 读 bizdevops/state.md → 1 句话报告 → 等用户指令
探索模式（默认）→ 自由读取分析，不改状态文件
推进模式（改代码时）→ 绑定 CR，走状态机，记 checkpoint
门禁不过 → 自行修复，不请示用户
human_review → 停下等待，不自行推进
需求变更 → 影响分析 → 调整方案 → 等确认 → 执行更新
会话结束 → 更新 state.md + CR → 3-5 行摘要
```

## §1 会话开始

1. 检查项目根目录是否存在 `bizdevops/state.md`
2. 存在 → 读取，用 **1 句自然语言** 报告当前状态和建议
3. 不存在 → 正常工作，不强制初始化（用户需要时可运行 `/pace-init`）
4. 等待用户指令（不自作主张开始工作）

**输出示例**：
> "上次停在 analyzer 标注阶段，继续创建框架版本？"

**不输出**：ID（CR-001）、状态机术语（developing）、门禁列表、表格。

## §2 双模式

### 探索模式（默认）

- 自由读取、分析、解释任何文件
- 不修改 `bizdevops/` 下的状态文件
- 不受作业规则（状态机、阻塞关系）约束
- 触发词："看看""分析一下""评估""了解"

### 推进模式

进入条件（满足任一）：
- 用户使用 `/pace-advance`
- 用户说"开始做""帮我改""提取""实现""修复"等明确修改意图
- Claude 开始写入或修改项目代码文件

推进模式行为：
- 绑定到一个变更请求（已有的或自动创建的）
- 遵循 `bizdevops/rules/workflow.md` 状态机
- 每完成一个原子步骤：git commit + 更新 CR + 更新 state.md
- 阶段流转前运行 `bizdevops/rules/gates.md` 中的门禁
- 门禁不通过 → 自行修复重试，不请示用户
- 遇到 human_review 门禁 → 生成摘要，通知用户，**停下等待**

退出：会话结束，或用户明确切换话题。

不确定时问用户："要正式开始改代码，还是先看看？"

## §3 自然语言映射

| 规则 | 说明 |
|------|------|
| 用户 → 作业对象 | 用标题关键词匹配，不要求用户说 ID |
| 作业对象 → 用户 | 用自然语言标题，不暴露 ID（CR-001） |
| 状态机 → 用户 | 不说"developing"，说"在做"；不说"review_ready"，说"等你 review" |
| 用户主动问 ID | 才展示 ID 和详细状态 |

## §4 自动创建变更请求

当用户说"帮我做 X"且 `bizdevops/backlog/` 中无对应 CR 时：
1. 自动创建 CR-xxx.md（格式参考 Plugin 的 `knowledge/_schema/cr-format.md`）
2. 自动关联到 `product-line.md` 中最匹配的产品功能
3. ID 自增（扫描现有 CR 文件的最大编号 +1）
4. 对用户只说"开始做 X"，不提 CR 文件的创建

## §5 分级输出

| 场景 | 输出量 | 示例 |
|------|--------|------|
| 完成一个步骤 | 1 句话 | "标注完成，框架占比 75%。" |
| 会话结束 | 3-5 行 | 做了什么 + 门禁概况 + 下一步 |
| 用户问详情 | 按需展开 | 门禁清单、进度百分比 |
| `/pace-status detail` | 完整状态 | 全部 PF/CR 状态 |
| `/pace-retro` | 度量仪表盘 | dashboard.md 内容 |

## §6 中断恢复

- 每个"原子步骤"后 checkpoint：git commit + 更新 CR 事件 + 更新 state.md
- 原子步骤粒度：一个有意义的工作成果（如标注完成、文件创建、一项门禁通过）
- 下次会话从 state.md 的"下一步"定位恢复点
- 门禁检查是幂等的——重跑即可，无副作用

## §7 Git 协作

- CR 文件只记 **分支名**，不记 commit hash
- 需要 commit 详情时通过 `git log <branch>` 查询
- 门禁结果只记 pass/fail（checkbox），不记完整输出
- 度量数据从 CR 事件表 + Git 历史聚合计算

## §8 状态文件维护规则

- state.md 由 Claude 在会话结束时自动更新
- CR 文件由 Claude 在推进模式中自动维护
- product-line.md 的价值追溯树在 CR 创建/完成时自动更新
- iterations/current.md 在 CR 状态变更时自动更新进度
- dashboard.md 仅在 `/pace-retro` 时更新，不在每次会话更新
- 人类可以手动编辑任何文件（如调整优先级、修改目标）

## §9 需求变更管理

变更不是异常，是常态。Claude 必须能识别和有序处理需求变更。

### 变更意图识别

触发词："不做了""先不搞""加一个""改一下""优先级调""延后""提前""砍掉""插入""新增需求""先做这个"

当识别到变更意图时，**不立即执行**，而是按以下三步处理。

### Step 1：影响分析

1. 读取 `bizdevops/product-line.md` 价值追溯树
2. 识别受影响的 PF 和 CR（直接影响 + 依赖链）
3. 评估对迭代目标的影响（容量、进度）
4. 评估对成效指标（MoS）的影响

用自然语言向用户报告影响范围，不使用技术术语。

### Step 2：提出调整方案

根据变更类型，提出具体方案：

| 变更类型 | 调整动作 |
|---------|---------|
| 插入新需求 | 创建 PF + CR，评估迭代容量，建议排期或延迟其他项 |
| 暂停/砍掉 | CR 标记 paused，解除依赖阻塞，追溯树标 ⏸️ |
| 优先级调整 | 更新 state.md 下一步，重排迭代计划 |
| 修改已有需求 | 评估返工范围，重置受影响的门禁 checkbox |

**必须等待用户确认**后再执行。不自作主张调整计划。

### Step 3：执行并记录

确认后执行：
1. 更新相关 CR 文件（状态、门禁、事件）
2. 更新 `product-line.md` 追溯树（新增/暂停/状态变更）
3. 更新 `iterations/current.md` 计划和变更记录表
4. 更新 `state.md` 快照

变更事件记录到迭代文件的"变更记录"表：

```markdown
## 变更记录

| 日期 | 类型 | 描述 | 影响 |
|------|------|------|------|
```

### CR 的 paused 状态

`paused` 表示"工作成果保留，暂时不推进"：
- 任何状态都可以转到 paused
- paused 可以恢复到暂停前的状态
- 暂停时保留：分支、代码、门禁进度、事件记录
- 追溯树中用 ⏸️ 标记

### 变更后的追溯树标记

```
⏸️ — 暂停
🔄 — 进行中
✅ — 完成
⏳ — 待开始
(新增 MM-DD) — 变更新增
(提前) — 优先级提升
```
