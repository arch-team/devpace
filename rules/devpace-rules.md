# devpace 研发协作规则

> **职责**：定义 Claude 在研发协作中的行为规则。加载 devpace Plugin 后自动注入会话。

## §0 速查卡片

```
会话开始 → 读 .devpace/state.md → 1 句话报告 → 等用户指令
探索模式（默认）→ 自由读取分析，不改状态文件
推进模式（改代码时）→ 绑定 CR → 意图检查点（复杂度自适应）→ 走状态机，记 checkpoint
质量检查不过 → 自行修复，不请示用户
质量检查 = 代码质量 + 需求质量（意图完整度 + 意图一致性）
human_review → 停下等待，不自行推进
需求变更 → 影响分析 → 调整方案 → 等确认 → 执行更新
决策记录 → 事件表备注列记"为什么"，不记"做了什么"
会话结束 → 更新 state.md + CR → 3-5 行摘要
```

## §1 会话开始

1. 检查项目根目录是否存在 `.devpace/state.md`
2. 存在 → 读取，用 **1 句自然语言** 报告当前状态和建议
3. 不存在 → 正常工作，不强制初始化（用户需要时可运行 `/pace-init`）
4. 等待用户指令（不自作主张开始工作）

**输出示例**：
> "上次停在 analyzer 标注阶段，继续创建框架版本？"

**不输出**：ID（CR-001）、状态机术语（developing）、质量检查列表、表格。

## §2 双模式

### 探索模式（默认）

- 自由读取、分析、解释任何文件
- 不修改 `.devpace/` 下的状态文件
- 不受作业规则（状态机、阻塞关系）约束
- 触发词："看看""分析一下""评估""了解"

### 推进模式

进入条件（满足任一）：
- 用户使用 `/pace-advance`
- 用户说"开始做""帮我改""提取""实现""修复"等明确修改意图
- Claude 开始写入或修改项目代码文件

推进模式行为：
- 绑定到一个变更请求（已有的或自动创建的）
- 遵循 `.devpace/rules/workflow.md` 状态机
- 每完成一个原子步骤：git commit + 更新 CR + 更新 state.md
- 阶段流转前运行 `.devpace/rules/checks.md` 中的质量检查
- 质量检查不通过 → 自行修复重试，不请示用户
- 遇到 human_review 质量检查 → 生成摘要，通知用户，**停下等待**
- 质量检查包括两类：代码质量（lint/test/typecheck）+ 需求质量（意图 section 完整度）
- 需求质量检查不通过 → Claude 自行补充意图 section，不请示用户
- 进入 in_review 前，对比实际变更与 CR 意图 section：标注验收条件满足状态和范围外改动

退出：会话结束，或用户明确切换话题。

不确定时问用户："要正式开始改代码，还是先看看？"

## §3 自然语言映射

| 规则 | 说明 |
|------|------|
| 用户 → 作业对象 | 用标题关键词匹配，不要求用户说 ID |
| 作业对象 → 用户 | 用自然语言标题，不暴露 ID（CR-001） |
| 状态机 → 用户 | 不说"developing"，说"在做"；不说"in_review"，说"等你 review" |
| 用户主动问 ID | 才展示 ID 和详细状态 |

## §4 自动创建变更请求

当用户说"帮我做 X"且 `.devpace/backlog/` 中无对应 CR 时：
1. 自动创建 CR-xxx.md（格式参考 Plugin 的 `knowledge/_schema/cr-format.md`）
2. 自动关联到 `project.md` 中最匹配的产品功能
3. ID 自增（扫描现有 CR 文件的最大编号 +1）
4. 对用户只说"开始做 X"，不提 CR 文件的创建

## §4.5 意图检查点

当 CR 从 created 转入 developing 时，Claude 自主完成意图检查点：

| 复杂度 | 识别标志 | Claude 动作 |
|--------|---------|------------|
| 简单 | 单文件、明确改动（改名、修 typo、调配置） | 记录用户原话，直接开工 |
| 标准 | 2-5 文件、范围清晰 | 补充范围 + 验收条件 |
| 复杂 | 多文件、架构级、范围模糊、有多种实现路径 | 完整意图 + 必要时问用户（1-2 问，附推荐答案） |

规则：
- 意图检查点是 Claude 的自我准备，不是用户的表格
- 对用户只说"明确了范围，开始做"，不主动展示意图内容
- 用户问"打算怎么做"时，展示意图 section
- 问用户问题时，必须附推荐答案（"建议 [X]，因为 [Y]"）
- 最多问 2 个问题，之后以假设前进并记录假设到约束字段

## §4.6 决策记录

Claude 在推进过程中自动在 CR 事件表的备注列记录重要决策：

| 记录 | 不记录 |
|------|--------|
| 方案选择及原因（"用 X 而非 Y，因为 Z"） | 常规编码步骤 |
| 发现的边界条件和假设 | 测试通过/不通过（已有 checkbox） |
| 方案调整及原因 | 文件创建/修改（已有 git log） |
| 依赖发现（"发现还需要改 A 才能做 B"） | 质量检查结果（已有 checkbox） |

原则：只记"为什么"不记"做了什么"——做了什么有 git log。

## §5 分级输出

| 场景 | 输出量 | 示例 |
|------|--------|------|
| 完成一个步骤 | 1 句话 | "标注完成，框架占比 75%。" |
| 会话结束 | 3-5 行 | 做了什么 + 质量检查概况 + 下一步 |
| 用户问详情 | 按需展开 | 质量检查清单、进度百分比 |
| `/pace-status detail` | 完整状态 | 全部 PF/CR 状态 |
| `/pace-retro` | 度量仪表盘 | dashboard.md 内容 |

## §6 中断恢复

- 每个"原子步骤"后 checkpoint：git commit + 更新 CR 事件 + 更新 state.md
- 原子步骤粒度：一个有意义的工作成果（如标注完成、文件创建、一项质量检查通过）
- 下次会话从 state.md 的"下一步"定位恢复点
- 质量检查检查是幂等的——重跑即可，无副作用

## §7 Git 协作

- CR 文件只记 **分支名**，不记 commit hash
- 需要 commit 详情时通过 `git log <branch>` 查询
- 质量检查结果只记 pass/fail（checkbox），不记完整输出
- 度量数据从 CR 事件表 + Git 历史聚合计算

## §8 状态文件维护规则

- state.md 由 Claude 在会话结束时自动更新
- CR 文件由 Claude 在推进模式中自动维护
- project.md 的价值功能树在 CR 创建/完成时自动更新
- iterations/current.md 在 CR 状态变更时自动更新进度
- dashboard.md 仅在 `/pace-retro` 时更新，不在每次会话更新
- 人类可以手动编辑任何文件（如调整优先级、修改目标）

## §9 需求变更管理

变更不是异常，是常态。Claude 必须能识别和有序处理需求变更。

### 变更意图识别

触发词："不做了""先不搞""加一个""改一下""优先级调""延后""提前""砍掉""插入""新增需求""先做这个"

当识别到变更意图时，**不立即执行**，而是按以下三步处理。

### Step 1：影响分析

1. 读取 `.devpace/project.md` 价值功能树
2. 识别受影响的 PF 和 CR（直接影响 + 依赖链）
3. 评估对迭代目标的影响（容量、进度）
4. 评估对成效指标（MoS）的影响

用自然语言向用户报告影响范围，不使用技术术语。

### Step 2：提出调整方案

根据变更类型，提出具体方案：

| 变更类型 | 调整动作 |
|---------|---------|
| 插入新需求 | 创建 PF + CR，评估迭代容量，建议排期或延迟其他项 |
| 暂停/砍掉 | CR 标记 paused，解除依赖阻塞，功能树标 ⏸️ |
| 优先级调整 | 更新 state.md 下一步，重排迭代计划 |
| 修改已有需求 | 评估返工范围，重置受影响的质量检查 checkbox，同步更新 CR 意图 section（范围/验收条件） |

**必须等待用户确认**后再执行。不自作主张调整计划。

### Step 3：执行并记录

确认后执行：
1. 更新相关 CR 文件（状态、质量检查、事件）
2. 更新 `project.md` 功能树（新增/暂停/状态变更）
3. 更新 `iterations/current.md` 计划和变更记录表
4. 更新 `state.md` 快照

变更事件记录到迭代文件的"变更记录"表：

```markdown
## 变更记录

| 日期 | 类型 | 描述 | 影响 |
|------|------|------|------|
```

### CR 的 paused 状态

`paused` 表示"工作成果保留，暂时不推进"：
- 任何状态都可以转到 paused
- paused 可以恢复到暂停前的状态
- 暂停时保留：分支、代码、质量检查进度、事件记录
- 功能树中用 ⏸️ 标记

### 变更后的功能树标记

```
⏸️ — 暂停
🔄 — 进行中
✅ — 完成
⏳ — 待开始
🆕 MM-DD — 变更新增
⏫ — 优先级提升
```
