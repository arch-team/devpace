# refine 子命令 procedures

> **职责**：交互式精炼已有 BR 或 PF，补充验收标准、边界条件、用户故事等细节。

## 触发

`/pace-biz refine <BR-xxx|PF-xxx>` 或用户要"精炼/细化/补充"某个需求。

## 与 pace-change modify 的区别

- **refine**：深化同一需求方向（补充验收标准、边界条件、异常场景）——内容更丰富，方向不变
- **modify**：改变需求方向（修改名称、调整范围、变更优先级）——方向变了

## 步骤

### Step 0：模式检查

读取 project.md 的 `mode` 字段。

- **lite 模式**：仅支持 PF 精炼（无 BR 层）。`BR-xxx` 参数 → 提示"轻量模式无 BR 层，请指定 PF 编号"，终止
- **完整模式**：BR 和 PF 均支持

### Step 1：定位实体

根据参数定位目标实体：

1. `BR-xxx` → 从 project.md 功能树 + `requirements/BR-xxx.md`（如有溢出文件）读取
2. `PF-xxx` → 从 project.md 功能树读取对应行
3. 无参数 → 列出最近创建或待精炼的实体，引导选择

展示当前实体内容（含就绪度评分）：

```
当前 [BR-xxx|PF-xxx]：[名称] [就绪度 N%]
├── 用户故事：[已有内容 或 "未定义"] [✓/✗]
├── 验收标准：[N 条 或 "未定义"] [✓/✗]
├── 优先级：[Px] [✓/✗]
├── 关联：[上游 Epic/OBJ] [✓/✗]
└── 建议精炼：[缺失维度列表]
```

**就绪度评分计算**（Readiness Score）：

| 维度 | BR 权重 | PF 权重 | 条件 |
|------|:-------:|:-------:|------|
| 用户故事/描述 | 20% | 25% | 非空且 >10 字 |
| 验收标准 | 25% | 30% | >= 2 条 |
| 优先级 | 15% | 15% | 已评估（非"—"） |
| 上游关联 | 15% | 10% | 有 Epic/OBJ 关联 |
| 异常/边界 | 15% | 10% | 有异常场景或边界条件 |
| NFR 考量 | 10% | 10% | 有非功能需求描述 |

就绪度 >= 80% 标记"就绪"，60-79% 标记"基本就绪"，<60% 标记"需精炼"

### Step 2：精炼引导

根据实体类型，Claude 基于已有信息提出补充问题：

**BR 精炼维度**：

| 维度 | 问题示例 | 适用场景 |
|------|---------|---------|
| 验收标准 | "这个需求怎么算做完了？有哪些可验证的条件？" | 验收标准为空或不足 3 条 |
| 业务规则 | "有哪些业务约束？例如数据保留期限、权限规则？" | 涉及数据/权限的 BR |
| 异常场景 | "如果 [关键操作] 失败了怎么办？" | 包含关键流程的 BR |
| 用户故事 | "能用'作为...我希望...以便...'描述这个需求吗？" | 用户故事为空 |
| 非功能需求 | "有性能、安全、可用性方面的特殊要求吗？" | 未提及 NFR |
| 关键流程 | "这个需求涉及哪些关键步骤？有分支或异常路径吗？" | 检测到流程关键词（"步骤"、"流程"、"审批"、"跳转"、"条件"） |

**PF 精炼维度**：

| 维度 | 问题示例 | 适用场景 |
|------|---------|---------|
| 边界条件 | "输入的最大/最小值是什么？空值怎么处理？" | 涉及数据输入的 PF |
| 输入输出 | "这个功能接收什么输入？期望什么输出？" | 规格不明确 |
| 错误处理 | "操作失败时用户看到什么？" | 用户交互类 PF |
| 安全约束 | "谁可以使用这个功能？需要什么权限？" | 涉及权限的 PF |
| 可测试性 | "怎么验证这个功能正常工作？测试数据是什么？" | 验收标准不具体 |

**精炼策略**：
- 仅提问缺失或不足的维度（已有完善内容的维度跳过）
- 每轮 2-3 个问题，避免信息过载
- 用户回答"不确定"或"以后再说" → 跳过该维度，不强制

### Step 3：写入更新

用户回答后，生成更新预览：

```
精炼预览 [BR-xxx]：
  + 新增验收标准：
    - [ ] [标准 1]
    - [ ] [标准 2]
  + 新增异常场景：[描述]
  ~ 更新用户故事：[新内容]

确认写入？
```

确认后执行写入：

1. **BR 更新路径**：
   - 有溢出文件（`requirements/BR-xxx.md`）→ 更新溢出文件对应 section
   - 无溢出文件且内容超过功能树行内容量 → 触发溢出（创建 `requirements/BR-xxx.md`）
   - 功能树行内描述更新（如用户故事缩写）
   - **关键流程**（如有）→ 写入溢出文件的"关键流程"section（无溢出文件时触发溢出）
2. **PF 更新路径**：
   - 更新 `project.md` 功能树中 PF 行的描述/用户故事
   - 验收标准过多时追加到关联 BR 的溢出文件中
3. 所有内容标记溯源：`<!-- source: claude, refined -->`

### Step 4：输出

```
已精炼 [BR-xxx|PF-xxx]：[名称]
变更摘要：
  + 新增验收标准 N 条
  + 补充异常场景 M 个
  ~ 更新用户故事描述

→ 下一步：/pace-biz decompose BR-xxx 继续分解 | /pace-dev 开始开发
```

## 容错

| 异常 | 处理 |
|------|------|
| BR/PF 编号不存在 | 提示无效编号，列出可用选项 |
| 实体已非常完善 | 提示"当前内容已较完善"，展示内容让用户决定是否仍要补充 |
| 用户跳过所有维度 | 提示"未做任何变更"，不写入文件。附建设性反馈："当前 [BR/PF] 可能已经足够清晰，或者您可能需要更多上下文才能精炼。建议：1) 先用 `/pace-biz view` 查看全景了解上下文 2) 在开发过程中通过 `/pace-dev` 自然补充细节 3) 稍后再回来精炼——随项目推进，需求会自然清晰" |
| requirements/ 目录不存在 | 触发溢出时自动创建 |
