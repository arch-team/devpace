# 需求变更处理规程

> **职责**：需求变更的详细处理流程。Rules §9 检测到变更意图或用户调用 /pace-change 后，Claude 按需读取本文件。

## 降级模式（无 .devpace/）

当项目未初始化 `.devpace/` 时，/pace-change 以降级模式运行——不创建文件，仅输出即时影响评估。

### 分析步骤

1. **识别变更范围**：用 Glob/Grep/Read 扫描项目代码结构，理解模块划分
2. **依赖分析**：基于 import/require/include 关系，识别变更涉及的模块和文件
3. **影响评估**：评估变更对现有模块的波及范围、可能的副作用
4. **风险识别**：标注高风险区域（核心模块、公共接口、共享状态）

### 输出格式

```
## 即时影响评估

**变更**：[用户描述的变更]
**涉及模块**：[受影响的代码模块/目录]
**依赖关系**：[与其他模块的依赖]
**风险点**：[需要注意的风险]
**建议**：[执行建议]
```

### 引导

评估结束后自然提及："完整初始化后可获得持久化追溯、质量门禁和度量能力。运行 /pace-init 开始。"

---

## Step 1：影响分析（正常模式）

### 加载上下文

1. 读取 `.devpace/project.md` 价值功能树
   - 无 project.md → 读取 `.devpace/state.md` 功能概览行（早期项目仅有 state.md）
2. 读取 `.devpace/iterations/current.md` 当前迭代
   - 无迭代文件 → 跳过迭代容量评估，仅分析功能影响
3. 读取相关 `.devpace/backlog/` CR 文件
   - 无 CR 文件 → 记录"当前无进行中的变更请求"
4. 评估对成效指标（MoS）的影响（有 MoS 数据时）

### 按变更类型分析

**add（插入新需求）**：
- 新需求属于哪个现有功能？需要新建功能分组吗？
- 需要几个 CR？
- 当前迭代容量够吗？需要延后哪些现有功能？

**pause（暂停/砍掉）**：
- 目标功能下有哪些 CR？各自什么状态？
- 有哪些其他 CR 依赖它们？
- 暂停后对成效指标（MoS）有影响吗？

**resume（恢复已暂停）**：
- 暂停时是什么状态？（读取 CR 的 `暂停前状态` 字段）
- 暂停期间代码库是否有变化？质量检查是否仍然有效？
- 恢复后对当前迭代容量的影响？

**reprioritize（优先级调整）**：
- 哪个功能/CR 需要提前？
- 当前正在进行的工作要不要暂停让路？

**modify（修改已有需求）**：
- 已有 CR 的哪些工作需要返工？
- 哪些质量检查需要重置为未检查？
- CR 意图 section 的哪些字段需要更新？

### 报告格式

用自然语言向用户报告影响范围，遵循 §3 自然语言映射规则：

- 用功能名称而非 ID（说"用户认证功能"而非"PF-002"）
- 用状态描述而非术语（说"正在做"而非"developing"）
- 先说结论再说细节（"这个变更会影响 2 个功能"→ 具体列出）

## Step 2：提出调整方案

根据变更类型，提出具体方案：

| 变更类型 | 调整动作 |
|---------|---------|
| 插入新需求 | 创建 PF + CR，评估迭代容量，建议排期或延后其他项 |
| 暂停/砍掉 | CR 标记 paused + 记录暂停前状态，解除依赖阻塞，功能树标 ⏸️ |
| 恢复 | CR 从 paused 恢复到暂停前状态，评估质量检查有效性，功能树恢复原标记 |
| 优先级调整 | 更新 state.md 下一步，重排迭代计划 |
| 修改已有需求 | 评估返工范围，重置受影响的质量检查 checkbox，更新 CR 意图 section（范围/验收条件） |

**必须等待用户确认**后再执行。不自作主张调整计划。

## Step 3：执行并记录

确认后执行：

1. 更新相关 CR 文件（状态、意图、质量检查、事件表）
   - pause：增加 `暂停前状态` 字段，状态改为 paused
   - resume：移除 `暂停前状态` 字段，状态恢复为暂停前值
   - modify：更新意图 section，重置受影响的质量检查 checkbox 为 `[ ]`
2. 更新 `project.md` 功能树（新增/暂停/恢复/状态变更）
   - 无 project.md → 更新 state.md 功能概览行
3. 更新 `iterations/current.md` 计划和变更记录表
   - 无迭代文件 → 跳过此步
4. 更新 `state.md` 快照（当前工作 + 下一步）

变更事件记录到迭代文件的"变更记录"表：

```markdown
## 变更记录

| 日期 | 类型 | 描述 | 影响 |
|------|------|------|------|
```

无迭代文件时，变更事件记录到 CR 文件的事件表。

## paused 状态规则

paused 状态的完整定义见 `knowledge/_schema/cr-format.md`。操作要点：

- 进入 paused：记录 `暂停前状态` 字段，保留分支、代码、质量检查进度、事件记录
- 功能树中用 ⏸️ 标记暂停项
- 解除依赖此 CR 的其他 CR 的阻塞关系
- 恢复时回到暂停前状态；代码库在暂停期间有变化时重新验证质量检查

## 变更后的功能树标记

```
⏸️ — 暂停       🔄 — 进行中     ✅ — 完成
⏳ — 待开始     🆕 MM-DD — 变更新增    ⏫ — 优先级提升
```
