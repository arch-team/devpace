# 降级场景规程

> **职责**：无 .devpace/ 时的降级模式处理流程。自包含，按 SKILL.md 路由表按需加载。

当项目未初始化 `.devpace/` 时，/pace-change 以降级模式运行——不创建文件，仅输出即时影响评估。

## Triage 降级

无 .devpace/ 时，Triage 简化为口头确认："这个变更现在要处理还是先记下来？"用户确认后再决定是否进入即时影响评估。

## 影响分析降级

### 分析步骤

1. **识别变更范围**：用 Glob/Grep/Read 扫描项目代码结构，理解模块划分
2. **依赖分析**：基于 import/require/include 关系，识别变更涉及的模块和文件
3. **影响评估**：评估变更对现有模块的波及范围、可能的副作用
4. **风险识别**：标注高风险区域（核心模块、公共接口、共享状态）
5. **Git 历史增强**（OPT-18，git 可用时）：
   - **co-change 分析**：`git log --follow --name-only` 识别经常一起改动的文件组——变更其中一个文件时提醒关注同组其他文件
   - **hotspot 识别**：`git log --since="30 days" --name-only` 统计近期变更热点——变更热点文件时提升风险等级

### 输出格式

```
## 即时影响评估

**变更**：[用户描述的变更]
**涉及模块**：[受影响的代码模块/目录]
**依赖关系**：[与其他模块的依赖]
**变更热点**：[近期频繁变更的相关文件（git 分析）]
**风险点**：[需要注意的风险]
**建议**：[执行建议]
```

### 引导

评估结束后自然提及："完整初始化后可获得持久化追溯、质量门禁和度量能力。运行 /pace-init 开始。"

## 风险量化降级

无 .devpace/ 时，风险评估基于代码库结构分析：
- 波及模块数：从 import/require 依赖图推算 + git co-change 分析
- 进行中 CR：不可用（标注"无法评估"）
- 质量检查重置：不可用（标注"无法评估"）

## 降级增量丰富路径

> OPT-12：降级到完整初始化之间的增量丰富。

**可选持久化**：降级分析结果可选存储到 `.devpace-lite/changes.md`（不创建完整 .devpace/ 结构）。

**升级引导**：累计 3+ 次降级使用后，升级引导语变为更具体：
- 第 1-2 次："完整初始化后可获得持久化追溯、质量门禁和度量能力。运行 /pace-init 开始。"（标准引导）
- 第 3+ 次："你已使用 [N] 次变更分析，初始化只需 2 分钟就能获得完整的变更追溯和影响分析——运行 /pace-init。"（强化引导）

**计数方式**：降级模式每次执行时检查 `.devpace-lite/changes.md` 的条目数。无此文件时视为第 1 次。

## Git 历史补充维度（正常模式也可用）

> OPT-18：正常模式下 git 历史作为影响分析的补充维度。

即使在正常模式下，Step 2 影响分析也可利用 git 历史作为补充：
- **co-change 分析**：识别受影响文件的"常见同伴"文件，提醒关注
- **hotspot 权重**：近期高频变更的模块在风险评估中权重提升
- 此维度为补充，不替代基于 .devpace/ 的主分析
