# 需求变更处理规程

> **职责**：需求变更的详细处理流程。Rules §9 检测到变更意图或用户调用 /pace-change 后，Claude 按需读取本文件。经验引用时机详见 `knowledge/experience-reference.md`。

## 降级模式（无 .devpace/）

当项目未初始化 `.devpace/` 时，/pace-change 以降级模式运行——不创建文件，仅输出即时影响评估。

### 分析步骤

1. **识别变更范围**：用 Glob/Grep/Read 扫描项目代码结构，理解模块划分
2. **依赖分析**：基于 import/require/include 关系，识别变更涉及的模块和文件
3. **影响评估**：评估变更对现有模块的波及范围、可能的副作用
4. **风险识别**：标注高风险区域（核心模块、公共接口、共享状态）

### 输出格式

```
## 即时影响评估

**变更**：[用户描述的变更]
**涉及模块**：[受影响的代码模块/目录]
**依赖关系**：[与其他模块的依赖]
**风险点**：[需要注意的风险]
**建议**：[执行建议]
```

### 引导

评估结束后自然提及："完整初始化后可获得持久化追溯、质量门禁和度量能力。运行 /pace-init 开始。"

---

## Step 0.5：Triage 分流

在进入完整影响分析前，先对变更请求做快速分流判断，避免所有变更都走完整流程。

### 分流选项

| 决策 | 含义 | 后续 |
|------|------|------|
| **Accept** | 接受，进入完整影响分析 | → Step 1 影响分析 |
| **Decline** | 拒绝，记录原因并归档 | → 记录到迭代变更记录（如存在）或 CR 事件表，结束 |
| **Snooze** | 延后，标记提醒条件 | → 记录延后原因和触发条件，结束 |

### 判断依据

Claude 根据以下信号自动建议分流决策（用户可覆盖）：

| 信号 | 建议 |
|------|------|
| 变更与当前迭代目标高度相关 | Accept |
| 变更明确是低优先级且无紧迫性 | Snooze |
| 变更与项目方向冲突或已被否决 | Decline |
| 变更为 hotfix/critical 级别 | Accept（跳过 Triage，直接进入影响分析） |
| 无法判断 | 询问用户："接受并分析影响、延后处理、还是不做？" |

### Decline 记录格式

```
**变更请求**：[描述]
**决策**：Decline
**原因**：[拒绝原因]
**日期**：[日期]
```

### Snooze 记录格式

```
**变更请求**：[描述]
**决策**：Snooze
**原因**：[延后原因]
**触发条件**：[满足什么条件时重新评估，如"下一迭代开始时"或"X 功能完成后"]
**日期**：[日期]
```

### 降级模式下的 Triage

无 .devpace/ 时，Triage 简化为口头确认："这个变更现在要处理还是先记下来？"用户确认后再决定是否进入即时影响评估。

## Step 1：影响分析（正常模式）

### 加载上下文

1. 读取 `.devpace/project.md` 价值功能树
   - 无 project.md → 读取 `.devpace/state.md` 功能概览行（早期项目仅有 state.md）
2. 读取 `.devpace/iterations/current.md` 当前迭代
   - 无迭代文件 → 跳过迭代容量评估，仅分析功能影响
3. 读取相关 `.devpace/backlog/` CR 文件
   - 无 CR 文件 → 记录"当前无进行中的变更请求"
4. 评估对成效指标（MoS）的影响（有 MoS 数据时）

### Release 影响评估

如果 `.devpace/releases/` 中存在活跃 Release（staging/deployed），额外评估：

1. 变更是否涉及已纳入 Release 的 CR？
2. 涉及 → 评估对 Release 稳定性的影响：
   - **staging 中的 CR**：可修改或移出，影响范围可控
   - **deployed 中的 CR**：不可修改，需创建新 defect/hotfix CR
3. 在影响报告中附加 Release 影响段

### 按变更类型分析

**add（插入新需求）**：
- 新需求属于哪个现有功能？需要新建功能分组吗？
- 需要几个 CR？
- 当前迭代容量够吗？需要延后哪些现有功能？

**pause（暂停/砍掉）**：
- 目标功能下有哪些 CR？各自什么状态？
- 有哪些其他 CR 依赖它们？（遍历所有关联关系：阻塞→直接影响、前置→时序调整、关联→通知参考）
- 暂停后对成效指标（MoS）有影响吗？

**resume（恢复已暂停）**：
- 暂停时是什么状态？（读取 CR 的 `暂停前状态` 字段）
- 读取 CR 事件表中最近的 `[checkpoint: ...]` 标记，对比暂停后的代码变更（`git log --since="<暂停日期>"`），判断已通过的门禁是否仍然有效
- 暂停期间代码库是否有变化？质量检查是否仍然有效？
- 恢复后对当前迭代容量的影响？

**reprioritize（优先级调整）**：
- 哪个功能/CR 需要提前？
- 当前正在进行的工作要不要暂停让路？

**modify（修改已有需求）**：
- 已有 CR 的哪些工作需要返工？
- 哪些质量检查需要重置为未检查？
- CR 意图 section 的哪些字段需要更新？

### 报告格式

用自然语言向用户报告影响范围，遵循 §3 自然语言映射规则：

- 用功能名称而非 ID（说"用户认证功能"而非"PF-002"）
- 用状态描述而非术语（说"正在做"而非"developing"）
- 先说结论再说细节（"这个变更会影响 2 个功能"→ 具体列出）
- 报告末尾附加追溯说明："（影响分析覆盖了从业务目标到代码的完整链路。）"

## Step 1.5：风险量化

影响分析完成后，输出半定量风险概要：

### 评估维度

| 维度 | 低 | 中 | 高 |
|------|:--:|:--:|:--:|
| 波及模块数 | ≤2 | 3-5 | >5 |
| 进行中 CR 受影响数 | 0 | 1-2 | ≥3 |
| 质量检查需重置数 | 0 | 1-3 | >3 |

### 风险等级计算

- **低**：三个维度均为"低" → 可直接执行
- **中**：任一维度为"中"且无"高" → 建议分步执行
- **高**：任一维度为"高" → 建议分步执行 + 重点测试

### 输出格式

```
**风险概要**：
- 波及模块：N 个（低/中/高）
- 受影响 CR：N 个（低/中/高）
- 检查重置：N 项（低/中/高）
- **综合风险：低/中/高**
```

高风险变更额外输出：
- 建议测试重点区域（列出受波及的核心模块）
- 建议分步执行策略（将大变更拆为 2-3 个可独立验证的小步骤）

### 降级模式下的风险量化

无 .devpace/ 时，风险评估基于代码库结构分析：
- 波及模块数：从 import/require 依赖图推算
- 进行中 CR：不可用（标注"无法评估"）
- 质量检查重置：不可用（标注"无法评估"）

## Step 2：提出调整方案

根据变更类型，提出具体方案：

| 变更类型 | 调整动作 |
|---------|---------|
| 插入新需求 | 创建 PF + CR，评估迭代容量，建议排期或延后其他项 |
| 暂停/砍掉 | CR 标记 paused + 记录暂停前状态，解除依赖阻塞，功能树标 ⏸️ |
| 恢复 | CR 从 paused 恢复到暂停前状态，评估质量检查有效性，功能树恢复原标记 |
| 优先级调整 | 更新 state.md 下一步，重排迭代计划 |
| 修改已有需求 | 评估返工范围，重置受影响的质量检查 checkbox，更新 CR 意图 section（范围/验收条件） |

**必须等待用户确认**后再执行。不自作主张调整计划。

## Step 3：执行并记录

确认后执行：

1. 更新相关 CR 文件（状态、意图、质量检查、事件表）
   - pause：增加 `暂停前状态` 字段，状态改为 paused
   - resume：移除 `暂停前状态` 字段，状态恢复为暂停前值
   - modify：更新意图 section，重置受影响的质量检查 checkbox 为 `[ ]`
2. 更新 `project.md` 功能树（新增/暂停/恢复/状态变更）
   - 无 project.md → 更新 state.md 功能概览行
3. 更新 `iterations/current.md` 计划和变更记录表（格式参考 Plugin `knowledge/_schema/iteration-format.md`）
   - 无迭代文件 → 跳过此步
4. 更新 `state.md` 快照（当前工作 + 下一步）

变更事件记录到迭代文件的"变更记录"表：

```markdown
## 变更记录

| 日期 | 类型 | 描述 | 影响 |
|------|------|------|------|
```

无迭代文件时，变更事件记录到 CR 文件的事件表。

## paused 状态规则

paused 状态的完整定义见 `knowledge/_schema/cr-format.md`。操作要点：

- 进入 paused：记录 `暂停前状态` 字段，保留分支、代码、质量检查进度、事件记录
- 功能树中用 ⏸️ 标记暂停项
- 解除依赖此 CR 的其他 CR 的阻塞关系
- 恢复时回到暂停前状态；代码库在暂停期间有变化时重新验证质量检查

## 变更后的功能树标记

```
⏸️ — 暂停       🔄 — 进行中     ✅ — 完成
⏳ — 待开始     🆕 MM-DD — 变更新增    ⏫ — 优先级提升
```
