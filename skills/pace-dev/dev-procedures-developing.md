# 开发阶段执行规程

> **职责**：CR 处于 developing 状态时的步骤隔离、漂移检测和 checkpoint 规则。

## 步骤隔离（L/XL CR）

L/XL CR 的状态转换严格逐阶段执行，防止 AI 在长上下文中提前行动或合并步骤。

### 隔离规则

| 当前阶段 | 聚焦范围 | 不可预读 |
|---------|---------|---------|
| created→developing | 意图检查点 + 复杂度评估 + 执行计划 + 方案确认 | Gate 1 检查内容、Gate 2 策略 |
| developing | 按执行计划逐步实现 + Gate 1 准备 | Gate 2 验收对比、Review 摘要格式 |
| developing→verifying | Gate 1 执行 + 自修复 | Gate 2 细节、合并策略 |
| verifying→in_review | Gate 2 执行 + 对抗审查 + Review 摘要 | 合并操作、连锁更新步骤 |
| in_review→approved→merged | 等待人类审批 → 执行合并和连锁更新 | 下一个 CR 的内容 |

### 执行方式

1. **进入新阶段时**：仅加载该阶段的相关 procedures section，不读取后续阶段
2. **阶段完成时**：Git commit + checkpoint 后，再加载下一阶段的 procedures
3. **context: fork 场景**：当 Skill 在子 agent 中运行时，天然实现上下文隔离——每个子 agent 仅接收当前阶段的上下文

### L/XL 步骤级 Checkpoint 与进度通知

L/XL CR 在 developing 阶段按执行计划逐步推进时，每完成一个步骤：

1. **事件记录**：在 CR 事件表写入一行 `| [时间戳] | step_N_done | Claude | [步骤描述] | — |`（N 为步骤序号）
2. **执行快照更新**：更新执行快照的"步骤进度"为 `N/M`，"恢复建议"为步骤 N+1 的简述
3. **state.md 步骤定位**：更新"当前工作"为 `→ 步骤 N/M：[下一步描述]`（M 为总步骤数），跨会话恢复时可精确定位到下一步
4. **进度通知**：输出 1 行进度——`[步骤 N/M] [步骤描述] ✅`

示例：
```
[步骤 3/5] middleware 实现完毕 ✅
```

**规则**：
- S/M CR 不触发步骤级 checkpoint 和进度通知（无执行计划）
- 进度通知为 1 行，不展开为详细说明
- 跨会话恢复时：读取 state.md 的步骤标记 + CR 事件表最后一个 `step_N_done` 事件 → 精确定位到下一步

### Developing 中期 Compact 建议（L/XL 专属）

L/XL CR 执行计划完成 >60% 时，如果当前步骤与前序步骤无直接代码依赖（如前序步骤处理模块 A，当前步骤处理模块 B），输出非强制 compact 建议：

```
💡 建议 /compact — 执行计划已完成 N/M 步，前序步骤上下文已持久化到 git 和 CR，压缩可释放空间给后续步骤。
```

**触发条件**（全部满足）：
1. L/XL CR 且执行计划存在
2. 当前步骤序号 > 总步骤数 × 60%
3. 当前步骤与最近完成的步骤无直接文件依赖（不涉及相同文件）
4. 本 CR 内未触发过此建议（同一 CR 仅建议 1 次）

**规则**：
- 纯建议，不阻断工作流
- S/M CR 不触发
- 与 Gate 1 后 compact 建议互补（§6 在 Gate 阶段，本建议在 developing 中期）

### S/M CR 不受约束

步骤隔离仅对 L/XL CR 强制执行。S/M CR 流程简短，Claude 可自然流畅地推进各阶段。

## 意图漂移检测

推进模式中，每次 checkpoint（git commit + 更新 CR）时执行轻量漂移检测：

### 检测步骤

1. 获取当前 CR 的意图 section 中描述的**范围**（涉及的模块/目录/文件）
2. 执行 `git diff --name-only` 获取本次 checkpoint 涉及的文件列表
3. 对比 diff 文件与意图范围：
   - 计算范围外文件占比 = 范围外文件数 / diff 总文件数
   - 范围外文件 = diff 中不属于意图声明的模块/目录的文件

### 触发条件

- 范围外文件占比 > 30% → 在 checkpoint 输出中附加提示
- 范围外文件占比 ≤ 30% → 不提示，正常继续

### 提示格式

```
checkpoint 提示：当前变更涉及意图范围外的文件：[文件列表]。
这是合理扩展还是需要调整 CR 范围？
```

### 规则

- **不阻断**：漂移检测仅提醒，不阻断工作流
- **不累积**：每次 checkpoint 独立计算，不与之前的 checkpoint 累积
- **简单 CR 跳过**：复杂度为"简单"（单文件）的 CR 跳过漂移检测
- **用户确认后记录**：用户确认为合理扩展时，更新 CR 意图 section 的范围字段

## 复杂度漂移检测（运行时升级守卫）

开发过程中持续监控实际复杂度是否超出初始评估，提供运行时升级建议。

### 检测步骤

每次 checkpoint 时（与意图漂移检测同时执行），额外检查：
1. 获取 CR 的"复杂度"字段（初始评估）
2. 统计当前 CR 累积涉及的文件数（`git diff --name-only main..HEAD`）和目录数
3. 对比实际指标与初始评估对应的阈值：

| 初始评估 | 文件数预警阈值 | 目录数预警阈值 |
|---------|-------------|-------------|
| S | > 3 文件 | > 1 目录 |
| M | > 7 文件 | > 3 目录 |

### 触发条件

- 实际指标超出初始评估的预警阈值 → 触发升级建议
- L/XL 不触发（已是最高级别）

### 提示格式

```
⚠️ 复杂度漂移：初始评估 [S/M]，实际涉及 N 文件 / M 目录。
建议：升级为 [M/L] 并[补充执行计划 / 考虑拆分]。继续还是调整？
```

### 规则

- **同一 CR 只提醒一次**：首次触发后记录到 CR 事件表，后续不重复
- **不阻断**：用户选择继续则按原路推进
- **与意图漂移互补**：意图漂移检测"做了什么不该做的"，复杂度漂移检测"规模比预期大"

## 测试策略自动生成（Strategy 副产物）

CR 进入 developing 阶段时，自动检测是否需要生成测试策略——将 strategy 从"需要用户主动调用"降级为"开发启动的自然副产物"。

### 触发条件（全部满足）

1. CR 状态变为 developing（created→developing 转换完成后）
2. CR 关联了至少一个 PF（产品功能）
3. `.devpace/rules/test-strategy.md` 不存在

### 执行方式

- 在 created→developing 转换完成、意图检查点通过后：
  1. 检查上述 3 个条件是否全部满足
  2. 全部满足 → 输出 1 行提示："📝 检测到 PF 关联且无测试策略，自动生成中..."
  3. 自动生成测试策略文件（输出格式见 `knowledge/_schema/process/test-strategy-format.md`，或委托 `/pace-test strategy`）
  4. 生成成功 → 输出"测试策略已生成（.devpace/rules/test-strategy.md），后续 coverage/accept/impact 将自动使用"
  5. 继续正常的 developing 开发流程
- 条件不全满足 → 静默跳过，不提示

### 规则

- **不阻断开发**：strategy 生成失败不影响 developing 阶段的正常推进（输出"策略生成跳过：[原因]"后继续）
- **仅首次**：strategy 已存在时不重新生成（由 accept Step 5.5 和 flaky 维护更新）
- **用户可覆盖**：用户后续可随时执行 `/pace-test strategy` 重新生成（覆盖自动版本）

## 测试预检建议（Dryrun 推荐）

developing 阶段接近完成时，主动建议执行 dryrun 预检——在正式 Gate 1 前发现质量问题。

### 触发条件

在 developing 阶段的 checkpoint 中（意图漂移检测同时执行），当满足以下条件之一：
1. 执行计划存在（L/XL CR）且当前步骤为最后一步
2. 执行计划不存在（S/M CR）且用户明确表示"做完了"/"提交"等完成意图
3. 所有意图 section 的验收条件对应的文件已有变更（通过 git diff 检测）

### 建议格式

```
💡 开发接近完成。建议执行 `/pace-test dryrun 1` 预检 Gate 1，提前发现质量问题。
```

### 规则

- **同一 CR 仅建议 1 次**：首次触发后在 CR 事件表备注 `[dryrun-suggested]`，后续不重复
- **不阻断**：纯建议，用户可忽略
- **与智能推荐互补**：pace-test 无参数运行时的智能推荐（§1.6）在 developing 状态也推荐 dryrun，此处是 pace-dev 侧的主动触达

## 语义漂移检测（持续需求对齐）

在 developing 阶段的 checkpoint 中，检测代码变更是否持续与验收标准语义对齐——补充意图漂移（"做了什么不该做的"）和复杂度漂移（"规模比预期大"）之外的第三维度："做的东西还是需求要的吗"。

### 检测步骤

每次 checkpoint（与意图/复杂度漂移检测同时执行），额外检查：

1. 读取 CR 关联 PF 的验收标准列表（从 project.md 或 PF 溢出文件）
2. 读取当前 CR 的 `git diff --stat` 获取变更文件和行数统计
3. 对每个验收标准，评估当前代码变更是否有相关实现：
   - **已覆盖**：变更文件中有直接实现该标准的代码
   - **部分覆盖**：有相关代码但不完整
   - **未触及**：尚无相关变更（可能是后续步骤）
   - **偏离**：代码实现了标准未要求的功能，可能偏离需求

### 触发条件

同时满足以下条件时输出语义漂移提示：
- CR 开发进度 ≥ 50%（L/XL 按步骤比例，S/M 按变更文件与预期比例）
- 存在"偏离"分类的验收标准 ≥ 1 个
- 或"未触及"分类的验收标准占比 > 50% 且进度 ≥ 70%

### 提示格式

```
📊 语义对齐检查：N/M 验收标准已覆盖 · X 项部分覆盖 · Y 项未触及 · Z 项可能偏离
偏离项：[验收标准描述] — 当前实现 [实际方向]
→ 建议：确认需求理解或调整实现方向
```

### 规则

- **不阻断**：纯提示，不影响开发流程
- **轻量执行**：仅读取 PF 验收标准（3-10 条）和 git diff stat，不做深度代码分析
- **S CR 跳过**：单文件 CR 验收标准通常明确，不需要中间检查
- **与 Gate 2 互补**：Gate 2 做完整逐条比对，语义漂移检测是开发过程中的早期预警
- **与 accept 协同**：语义漂移检测结果可作为后续 `/pace-test accept` 的优先级参考
