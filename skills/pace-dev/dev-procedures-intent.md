# 意图检查点与计划规程

> **执行流程**：意图检查点 → 复杂度评估 → [S: 直接开工 | M: 可选计划 | L/XL: 技术方案评审 → 执行计划 + 反思 + 方案确认] → 风险预扫描（条件） → 进入 developing

## Phase A：意图明确（始终执行）

### 意图检查点

当 CR 从 created 转入 developing 时，Claude 自主完成意图检查点：

| 复杂度 | 识别标志 | Claude 动作 |
|--------|---------|------------|
| 简单 | 单文件、明确改动（改名、修 typo、调配置） | 记录用户原话 + 验收条件（自由文本），直接开工 |
| 标准 | 2-5 文件、范围清晰 | 补充范围 + 验收条件（编号清单格式）+ 歧义标记 `[待确认]` |
| 复杂 | 多文件、架构级、范围模糊、有多种实现路径 | 完整意图 + 验收条件（Given/When/Then 格式）+ 歧义标记 + **执行计划** + **方案确认** + 必要时问用户（1-2 问，附推荐答案） |

规则：
- 意图检查点是 Claude 的自我准备，不是用户的表格
- 对用户只说"明确了范围，开始做"，不主动展示意图内容
- 用户问"打算怎么做"时，展示意图 section（含执行计划）
- 问用户问题时，必须附推荐答案（"建议 [X]，因为 [Y]"）
- 最多问 2 个问题，之后以假设前进并记录假设到约束字段

### 探索→推进上下文继承

从探索模式切换到推进模式创建 CR 时，Claude 自动回溯本会话最近对话，提取方案相关信息预填入 CR 意图 section：

**提取内容**：
- 讨论过的方案选择及排除理由
- 发现的技术约束或依赖
- 达成的结论或偏好（如"用 X 而非 Y"）
- 已分析过的范围和边界

**执行规则**：
1. 仅在本会话存在探索阶段对话时触发（直接 `/pace-dev` 无前序探索不触发）
2. 提取的内容写入 CR 意图 section 的对应字段（范围/方案/约束），带溯源标记 `<!-- source: claude, 从探索阶段对话提取 -->`
3. 用户原话字段仍记录触发推进的原始请求，不与探索内容混合
4. 对用户不展示提取过程，仅在"明确了范围"的输出中自然体现已继承的上下文

### 复杂度量化评估

意图检查点完成后，Claude 自动评估 CR 复杂度并写入 CR 文件的"复杂度"字段。

#### 评估维度

复杂度等级的量化阈值见 `knowledge/_schema/entity/cr-format.md` "复杂度"章节。

#### 评分规则

1. 逐维度评估，取**最高等级**作为最终复杂度
2. 写入 CR 文件元信息的"复杂度"字段
3. L/XL → 自动进入下方"CR 拆分建议"流程
4. S/M → 跳过拆分评估，直接开工

#### 与已有拆分建议的关系

复杂度评估取代了原有的启发式触发条件（>5 文件 / >3 目录 / >3 验收条件），提供统一的量化基础。拆分建议的评估步骤和输出格式不变。

### 歧义标记

歧义标记语法和规则定义见 `knowledge/_schema/entity/cr-format.md` "歧义标记"章节。

**生命周期**（补充行为逻辑）：
1. **添加**：意图检查点发现歧义时自动添加（附推荐值和理由）
2. **解决**：用户确认或 Claude 以假设前进后，替换为确认内容或移至约束字段
3. **检查**：Gate 2 前检查 CR 意图 section，若存在未解决的 `[待确认]` 标记则 Gate 2 失败

### CR 拆分建议

意图检查点完成后，对标准和复杂 CR 进行拆分评估：

#### 触发条件

- 复杂度评估结果为 **L** 或 **XL**（即：涉及 >7 文件、>3 目录、>3 验收条件，或存在跨模块双向依赖）

#### 评估步骤

1. 分析意图 section 中的范围和验收条件
2. 识别可独立交付的功能切面（按模块、层次或功能维度）
3. 评估各切面之间的依赖关系

#### 输出格式

满足触发条件时，在"明确了范围，开始做"之前附加建议：

```
建议将此变更拆分为 N 个独立 CR：
- CR-A：[模块/功能]（[文件范围]）
- CR-B：[模块/功能]（[文件范围]）
依赖关系：CR-A 应先于 CR-B。
要拆分还是继续作为一个 CR 推进？
```

#### 规则

- **不阻断**：用户选择不拆分则继续作为单个 CR 推进
- **简单 CR 跳过**：复杂度为"简单"的 CR 不做拆分评估
- **记录决策**：用户选择不拆分时，在 CR 事件表备注"评估拆分后决定保持单 CR，因为 [用户理由]"

## Phase B：执行计划（M+ 复杂度）

### 复杂度自适应路径

意图检查点完成后，根据复杂度评估结果提供差异化的工作流路径：

**快速模式（S + 单文件）**：
- 触发条件：复杂度 S 且涉及 ≤ 1 文件
- 省略步骤：跳过详细意图 section（仅记录用户原话+验收条件）、跳过执行计划、跳过计划反思
- 直接进入编码 + 测试
- CR 状态机仍完整执行（created→developing→verifying→in_review→...），不绕过状态机

**标准模式（S 多文件 / M）**：
- 按现有流程执行
- M 可选生成执行计划（Claude 判断）

**完整模式（L/XL）**：
- 按现有流程执行（意图+执行计划+计划反思+方案确认，全部必须）

**升级守卫**：
- 如果初始评估为 S/M，但意图检查点中发现多组件依赖、架构级语言或多种实现路径 → 主动建议升级：
  - 轻度升级（S→M）："涉及范围比预期大（[原因]），建议补充范围和验收条件。"
  - 重度升级（S/M→L）："涉及多模块依赖（[原因]），建议生成执行计划。要升级还是继续？"
- 升级建议不阻断——用户选择继续则按原复杂度推进
- 已完成的工作不丢失——升级后作为更完整计划的输入保留

### 技术方案评审（L/XL 必须）

在生成执行计划之前，先评审技术方案选型：

**Step B0：方案分析**

1. 基于意图检查点的范围和约束，Claude 提出 2-3 个备选技术方案
2. 每个方案包含：
   - **方案名称**：1 行描述
   - **核心思路**：2-3 行说明实现路径
   - **优势**：关键优点
   - **劣势/风险**：主要缺点或技术风险
3. 将方案对比写入 CR 文件的"方案"字段

**Step B0.5：方案确认**

展示方案对比，等待用户选择：

```
技术方案（[CR 标题]）：

方案 A：[名称]
  思路：[实现路径]
  优势：[优点] | 劣势：[缺点]

方案 B：[名称]
  思路：[实现路径]
  优势：[优点] | 劣势：[缺点]

推荐方案 A，因为 [理由]。选哪个？
```

- 用户选择后 → 更新 CR 方案字段为选定方案
- 用户说"都行/你定" → 采用推荐方案

**ADR 自动触发**：如果方案涉及以下任一条件，自动建议创建 ADR（通过在 merged 后引导提醒，不阻断当前流程）：
- 新引入依赖（框架/库）
- 改变已有架构模式（如从同步改异步）
- 影响 3+ 个模块的接口
- 用户明确说"记录这个决策"

**规则**：
- **S/M CR 跳过本步骤**——直接进入执行计划生成
- 方案评审与执行计划生成合并在同一个用户交互中展示（方案确认 → 立即生成执行计划），不增加额外交互轮次
- 如果 CR 只有 1 个可行方案（如 bug 修复），标注"单一方案，无需对比"并跳过

### 执行计划生成（L/XL 必须）

复杂度 L/XL 的 CR 在意图检查点完成后，自动生成执行计划写入 CR 意图 section。

**生成规则**：
- 每步是一个原子动作（对应一次有意义的 commit）
- 必须包含：精确文件路径 + 做什么 + 可验证的预期结果
- "预期"字段描述可验证的结果（"运行 `pytest` 通过"而非"测试写好了"）
- 步骤间标注依赖关系（如"依赖步骤 1 完成"）
- **测试先行引导**（非强制）：当 `.devpace/rules/test-strategy.md` 存在且当前 CR 关联的验收条件有 `❌ 待建` 测试时，在执行计划中优先安排"写测试骨架"步骤，再安排"实现功能"步骤。格式：`步骤 N: 生成 [PF] 测试骨架（/pace-test generate）` → `步骤 N+1: 实现 [功能]`。无 test-strategy.md 时不生成此引导

**触发条件**：复杂度自适应触发规则见 `knowledge/_schema/entity/cr-format.md` "执行计划"章节（S 不生成 | M 可选 | L/XL 必须）。

### 计划反思（L/XL 必须）

L/XL CR 的执行计划生成后、方案确认前，Claude 自主完成计划反思：

**反思维度（4 个）**：
1. **需求覆盖度**：执行计划是否覆盖了所有验收条件？遗漏哪些？
2. **过度设计风险**：是否引入了验收条件未要求的复杂性？可否简化？
3. **拆分必要性**：步骤间耦合度如何？是否应该拆为多个 CR？
4. **技术假设合理性**：计划依赖了哪些未验证的假设？风险可控吗？

**输出**：1-3 行观察写入 CR 事件表备注列（`[plan-reflection: ...]`），格式：
- 无问题："计划反思：4 维度无异常"
- 有问题："计划反思：需求覆盖遗漏验收条件 3；过度设计风险低；建议保持单 CR"

**触发条件**：与执行计划生成相同——S 不执行，M 可选，L/XL 必须。

### 方案确认门禁（L/XL 强制）

L/XL CR 的执行计划生成后，**必须主动展示并等待用户确认**，才能进入 developing 实现阶段。

**流程**：
1. Claude 生成执行计划后，**主动展示**给用户（不默认隐藏）
2. 附加："这是实现方案，确认开始还是要调整？"
3. 用户确认 → 进入 developing
4. 用户调整 → 支持自然语言修改指令 → 修改后重新展示确认

**自然语言编辑指令**（步骤 4）：
- "删除步骤 N" → 移除指定步骤，后续步骤重新编号
- "合并步骤 N 和 M" → 合并为单个步骤，保留两者的文件范围和预期结果
- "在步骤 N 后加一步：[描述]" → 插入新步骤，后续步骤重新编号
- "调整步骤 N 的范围" → 修改指定步骤的文件范围或预期结果
- "交换步骤 N 和 M 的顺序" → 交换，自动检查依赖关系是否仍然成立
- 修改后 Claude 重新展示完整执行计划 + 再次询问确认

**步骤暂停点**（可选）：
- 方案确认阶段，用户可设定暂停点："在步骤 N 后暂停"
- 到达暂停点时：输出当前进度摘要 + 已完成步骤列表 + "继续还是要调整后续步骤？"
- 用户说"继续" → 继续执行后续步骤
- 用户调整 → 修改剩余步骤（同上自然语言编辑指令）→ 确认后继续
- 默认无暂停点——不设定则全程自主推进
- 暂停点记录在 CR 事件表备注列：`[pause-point: after-step-N]`

**S/M 不受影响**——直接进入 developing。

**设计决策**：设计和实现的混合是质量问题的根源。L/XL CR 通过方案确认门禁将"设计阶段"和"实现阶段"显式分离，确保设计错误在编码前发现。

### 验收条件格式指引

验收条件格式根据 CR 复杂度自适应选择（S=自由文本 | M=编号清单 | L/XL=Given/When/Then）。格式定义和示例见 `knowledge/_schema/entity/cr-format.md` "验收条件格式"章节。

**行为规则**：
- 简单 CR 不加格式负担，Claude 直接写自由文本
- 标准/复杂由 Claude 在意图检查点自动生成对应格式
- Gate 2 按条目逐一比对验收条件与实际变更

## Phase C：风险预扫描（条件触发）

### 风险预扫描（scan 触发）

意图检查点完成后，根据复杂度评估自动触发风险预扫描：

| 复杂度 | 触发条件 | 行为 |
|--------|---------|------|
| S（单文件） | 不触发 | — |
| S（多文件）/ M | insights.md 有匹配 defense pattern（置信度 ≥ 0.5） | 轻量扫描（仅历史教训维度） |
| L / XL | 必须触发 | 完整 5 维扫描 |

执行 5 维风险扫描（维度定义和输出格式见 `knowledge/_schema/auxiliary/risk-format.md` 风险预评估章节，或委托 `/pace-guard scan`）。扫描结果写入 CR "风险预评估" section（格式见 `knowledge/_schema/entity/cr-format.md`）。

若综合风险等级为 High：
- 执行计划中增加对应防护步骤（如："增加 E2E 测试"、"架构审查"等具体动作）
- 输出提醒："⚠️ 风险预评估为 High（[维度: 原因]），已在执行计划中增加防护步骤。"

## Anti-Rationalization 守卫（L/XL 铁律）

| 铁律 | Claude 可能的合理化 | 预防反驳 |
|------|-------------------|---------|
| 意图明确 | "用户催得急，先写代码" | 催促不改变复杂度。L/XL 不做意图明确 = 返工概率极高 |
| 意图明确 | "需求已经在聊天中说清楚了" | 聊天内容可能被推出上下文窗口。意图 section 是持久化的唯一保证 |
| 意图明确 | "之前的会话已经讨论过了" | 跨会话时上下文归零。执行计划确保恢复到步骤级精度 |
| 意图明确 | "先写代码再补文档" | 执行计划不是文档——是设计。先设计再实现，不可反过来 |
| 执行计划 | "需求已经很清楚了" | 清楚的需求 ≠ 清楚的实现路径。执行计划解决"怎么做"而非"做什么" |
| 执行计划 | "太简单不需要计划" | 如果真的简单，复杂度不会评为 L/XL。按评估结果执行 |
| 执行计划 | "写计划浪费时间" | 计划暴露步骤间依赖和风险，减少返工时间 |
| 计划反思 | "计划已经很完善了" | 反思是审视完善度的机制，跳过反思 ≠ 计划完善 |
| 计划反思 | "反思浪费时间" | 1-3 行观察 < 30 秒。发现问题后避免的返工远超此成本 |
