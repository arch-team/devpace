# 推进模式执行规程

> **职责**：推进模式中的详细执行规则。Rules 检测到推进意图后，Claude 按需读取本文件。

## 意图检查点

当 CR 从 created 转入 developing 时，Claude 自主完成意图检查点：

| 复杂度 | 识别标志 | Claude 动作 |
|--------|---------|------------|
| 简单 | 单文件、明确改动（改名、修 typo、调配置） | 记录用户原话，直接开工 |
| 标准 | 2-5 文件、范围清晰 | 补充范围 + 验收条件 |
| 复杂 | 多文件、架构级、范围模糊、有多种实现路径 | 完整意图 + 必要时问用户（1-2 问，附推荐答案） |

规则：
- 意图检查点是 Claude 的自我准备，不是用户的表格
- 对用户只说"明确了范围，开始做"，不主动展示意图内容
- 用户问"打算怎么做"时，展示意图 section
- 问用户问题时，必须附推荐答案（"建议 [X]，因为 [Y]"）
- 最多问 2 个问题，之后以假设前进并记录假设到约束字段

## CR 拆分建议

意图检查点完成后，对标准和复杂 CR 进行拆分评估：

### 触发条件

- 范围涉及 > 5 个文件，或
- 范围涉及 > 3 个目录，或
- 意图 section 中列出 > 3 个独立验收条件

### 评估步骤

1. 分析意图 section 中的范围和验收条件
2. 识别可独立交付的功能切面（按模块、层次或功能维度）
3. 评估各切面之间的依赖关系

### 输出格式

满足触发条件时，在"明确了范围，开始做"之前附加建议：

```
建议将此变更拆分为 N 个独立 CR：
- CR-A：[模块/功能]（[文件范围]）
- CR-B：[模块/功能]（[文件范围]）
依赖关系：CR-A 应先于 CR-B。
要拆分还是继续作为一个 CR 推进？
```

### 规则

- **不阻断**：用户选择不拆分则继续作为单个 CR 推进
- **简单 CR 跳过**：复杂度为"简单"的 CR 不做拆分评估
- **记录决策**：用户选择不拆分时，在 CR 事件表备注"评估拆分后决定保持单 CR，因为 [用户理由]"

## 决策记录

Claude 在推进过程中自动在 CR 事件表的备注列记录重要决策：

| 记录 | 不记录 |
|------|--------|
| 方案选择及原因（"用 X 而非 Y，因为 Z"） | 常规编码步骤 |
| 发现的边界条件和假设 | 测试通过/不通过（已有 checkbox） |
| 方案调整及原因 | 文件创建/修改（已有 git log） |
| 依赖发现（"发现还需要改 A 才能做 B"） | 质量检查结果（已有 checkbox） |

原则：只记"为什么"不记"做了什么"——做了什么有 git log。

## 意图漂移检测

推进模式中，每次 checkpoint（git commit + 更新 CR）时执行轻量漂移检测：

### 检测步骤

1. 获取当前 CR 的意图 section 中描述的**范围**（涉及的模块/目录/文件）
2. 执行 `git diff --name-only` 获取本次 checkpoint 涉及的文件列表
3. 对比 diff 文件与意图范围：
   - 计算范围外文件占比 = 范围外文件数 / diff 总文件数
   - 范围外文件 = diff 中不属于意图声明的模块/目录的文件

### 触发条件

- 范围外文件占比 > 30% → 在 checkpoint 输出中附加提示
- 范围外文件占比 ≤ 30% → 不提示，正常继续

### 提示格式

```
checkpoint 提示：当前变更涉及意图范围外的文件：[文件列表]。
这是合理扩展还是需要调整 CR 范围？
```

### 规则

- **不阻断**：漂移检测仅提醒，不阻断工作流
- **不累积**：每次 checkpoint 独立计算，不与之前的 checkpoint 累积
- **简单 CR 跳过**：复杂度为"简单"（单文件）的 CR 跳过漂移检测
- **用户确认后记录**：用户确认为合理扩展时，更新 CR 意图 section 的范围字段
