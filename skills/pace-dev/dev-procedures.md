# 推进模式执行规程

> **职责**：推进模式中的详细执行规则。Rules 检测到推进意图后，Claude 按需读取本文件。经验引用时机详见 `knowledge/experience-reference.md`。

## §0 速查卡片

- **意图检查点**：CR 进入 developing 时的自我准备——按简单/标准/复杂分级记录意图和验收条件
- **复杂度自适应路径**：S 快速模式→M 标准模式→L/XL 完整模式，含步骤隔离
- **复杂度量化评估**：5 维评分（范围/依赖/不确定/测试/架构）→ S/M/L/XL 等级
- **CR 拆分建议**：XL CR 自动建议拆分，含拆分策略和关联关系
- **意图漂移检测**：每次文件变更后比对意图，检测范围蔓延
- **复杂度漂移检测**：运行时升级守卫，触发时暂停并重新评估
- **Gate 通过反思**：Gate 1/2 通过后的自我审视，防止过度自信
- **风险预扫描**：L/XL 必须 + S 多文件/M 按 insights 匹配触发（guard-procedures.md scan）
- **功能发现与执行透明摘要**：嵌入式新功能捕获 + Human Transparency 输出
- **PF 溢出检查**：CR 创建/merged 时检查关联 PF 是否触发溢出条件（>15 行 | 3+ CR | modify 历史）
- **L/XL 步骤定位与进度**：checkpoint 标记 `[checkpoint: step-N-done]` + state.md 步骤级定位 + 每步完成 1 行进度通知
- **探索→推进上下文继承**：mode 切换时回溯对话提取方案信息预填 CR 意图
- **快速 CR 切换**：`#N` 编号匹配 + `--last` 上一个 CR

## 智能 context.md 自动生成

首次推进时（CR created→developing），如果 `.devpace/context.md` 不存在：

1. **静默扫描项目特征**：检查 `tsconfig.json`、`.eslintrc*`、`package.json`、`pyproject.toml`、`go.mod`、`Cargo.toml`、`.editorconfig` 等配置文件
2. **提取技术约定**：从配置文件中提取技术栈、编码规范、项目约定
3. **阈值检查**：提取到 ≥3 条约定 → 自动创建 `context.md`（格式遵循 `knowledge/_schema/context-format.md`）；< 3 条 → 跳过，不创建
4. **教学触发**：首次创建时标记 `context_generated`，附教学："（根据项目配置自动生成了技术约定，推进时会参考这些规则。）"
5. **零摩擦**：不询问用户确认，不阻断推进流程

**与 /pace-init 的关系**：/pace-init Step 6 在初始化时扫描生成 context.md。本逻辑是补充路径——用户跳过 init 或 init 时项目太简单（<3 条），但后续推进时项目已有足够配置文件。两个路径生成的 context.md 格式完全一致。

## 同步关联提议（sync-mapping.md 存在时）

CR 创建完成后，如果 `.devpace/integrations/sync-mapping.md` 存在：

1. 自然语言提议："是否要为 CR-{id} 创建 GitHub Issue 并关联？"
   - 用户同意 → 执行 `/pace-sync create CR-{id}`（复用 sync-procedures §7）
   - 用户拒绝 → 静默跳过
2. 首次提议后标记教学 `sync_create`（每项目仅提议前 3 次，之后静默跳过或自动创建）

**自主级别分化**：
- 辅助级：每次询问
- 标准级：前 3 次询问，之后静默跳过
- 自主级：自动创建并关联（不询问）

**规则**：sync-mapping.md 不存在时完全跳过，不提醒配置同步。

## 意图检查点

当 CR 从 created 转入 developing 时，Claude 自主完成意图检查点：

| 复杂度 | 识别标志 | Claude 动作 |
|--------|---------|------------|
| 简单 | 单文件、明确改动（改名、修 typo、调配置） | 记录用户原话 + 验收条件（自由文本），直接开工 |
| 标准 | 2-5 文件、范围清晰 | 补充范围 + 验收条件（编号清单格式）+ 歧义标记 `[待确认]` |
| 复杂 | 多文件、架构级、范围模糊、有多种实现路径 | 完整意图 + 验收条件（Given/When/Then 格式）+ 歧义标记 + **执行计划** + **方案确认** + 必要时问用户（1-2 问，附推荐答案） |

规则：
- 意图检查点是 Claude 的自我准备，不是用户的表格
- 对用户只说"明确了范围，开始做"，不主动展示意图内容
- 用户问"打算怎么做"时，展示意图 section（含执行计划）
- 问用户问题时，必须附推荐答案（"建议 [X]，因为 [Y]"）
- 最多问 2 个问题，之后以假设前进并记录假设到约束字段

### 探索→推进上下文继承

从探索模式切换到推进模式创建 CR 时，Claude 自动回溯本会话最近对话，提取方案相关信息预填入 CR 意图 section：

**提取内容**：
- 讨论过的方案选择及排除理由
- 发现的技术约束或依赖
- 达成的结论或偏好（如"用 X 而非 Y"）
- 已分析过的范围和边界

**执行规则**：
1. 仅在本会话存在探索阶段对话时触发（直接 `/pace-dev` 无前序探索不触发）
2. 提取的内容写入 CR 意图 section 的对应字段（范围/方案/约束），带溯源标记 `<!-- source: claude, 从探索阶段对话提取 -->`
3. 用户原话字段仍记录触发推进的原始请求，不与探索内容混合
4. 对用户不展示提取过程，仅在"明确了范围"的输出中自然体现已继承的上下文

### 风险预扫描（scan 触发）

意图检查点完成后，根据复杂度评估自动触发风险预扫描：

| 复杂度 | 触发条件 | 行为 |
|--------|---------|------|
| S（单文件） | 不触发 | — |
| S（多文件）/ M | insights.md 有匹配 defense pattern（置信度 ≥ 0.5） | 轻量扫描（仅历史教训维度） |
| L / XL | 必须触发 | 完整 5 维扫描 |

扫描规则详见 `skills/pace-guard/guard-procedures.md` scan 章节。扫描结果写入 CR "风险预评估" section（格式见 `knowledge/_schema/cr-format.md`）。

若综合风险等级为 High：
- 执行计划中增加对应防护步骤（如："增加 E2E 测试"、"架构审查"等具体动作）
- 输出提醒："⚠️ 风险预评估为 High（[维度: 原因]），已在执行计划中增加防护步骤。"

### 复杂度自适应路径

意图检查点完成后，根据复杂度评估结果提供差异化的工作流路径：

**快速模式（S + 单文件）**：
- 触发条件：复杂度 S 且涉及 ≤ 1 文件
- 省略步骤：跳过详细意图 section（仅记录用户原话+验收条件）、跳过执行计划、跳过计划反思
- 直接进入编码 + 测试
- CR 状态机仍完整执行（created→developing→verifying→in_review→...），不绕过状态机

**标准模式（S 多文件 / M）**：
- 按现有流程执行
- M 可选生成执行计划（Claude 判断）

**完整模式（L/XL）**：
- 按现有流程执行（意图+执行计划+计划反思+方案确认，全部必须）

**升级守卫**：
- 如果初始评估为 S/M，但意图检查点中发现多组件依赖、架构级语言或多种实现路径 → 主动建议升级：
  - 轻度升级（S→M）："涉及范围比预期大（[原因]），建议补充范围和验收条件。"
  - 重度升级（S/M→L）："涉及多模块依赖（[原因]），建议生成执行计划。要升级还是继续？"
- 升级建议不阻断——用户选择继续则按原复杂度推进
- 已完成的工作不丢失——升级后作为更完整计划的输入保留

**铁律：L/XL CR 不可跳过意图明确步骤（完整意图 + 执行计划 + 方案确认）。**

| Claude 可能的合理化 | 预防反驳 |
|-------------------|---------|
| "用户催得急，先写代码" | 催促不改变复杂度。L/XL 不做意图明确 = 返工概率极高 |
| "需求已经在聊天中说清楚了" | 聊天内容可能被推出上下文窗口。意图 section 是持久化的唯一保证 |
| "之前的会话已经讨论过了" | 跨会话时上下文归零。执行计划确保恢复到步骤级精度 |
| "先写代码再补文档" | 执行计划不是文档——是设计。先设计再实现，不可反过来 |

### 执行计划生成（L/XL 必须）

复杂度 L/XL 的 CR 在意图检查点完成后，自动生成执行计划写入 CR 意图 section。

**生成规则**：
- 每步是一个原子动作（对应一次有意义的 commit）
- 必须包含：精确文件路径 + 做什么 + 可验证的预期结果
- "预期"字段描述可验证的结果（"运行 `pytest` 通过"而非"测试写好了"）
- 步骤间标注依赖关系（如"依赖步骤 1 完成"）
- **测试先行引导**（非强制）：当 `.devpace/rules/test-strategy.md` 存在且当前 CR 关联的验收条件有 `❌ 待建` 测试时，在执行计划中优先安排"写测试骨架"步骤，再安排"实现功能"步骤。格式：`步骤 N: 生成 [PF] 测试骨架（/pace-test generate）` → `步骤 N+1: 实现 [功能]`。无 test-strategy.md 时不生成此引导

**触发条件**：
- S：不生成
- M：可选（Claude 判断有多步骤依赖时可主动生成）
- L/XL：必须生成

**铁律：L/XL CR 不可跳过执行计划生成。**

| Claude 可能的合理化 | 预防反驳 |
|-------------------|---------|
| "需求已经很清楚了" | 清楚的需求 ≠ 清楚的实现路径。执行计划解决"怎么做"而非"做什么" |
| "太简单不需要计划" | 如果真的简单，复杂度不会评为 L/XL。按评估结果执行 |
| "写计划浪费时间" | 计划暴露步骤间依赖和风险，减少返工时间 |

### 计划反思（L/XL 必须）

L/XL CR 的执行计划生成后、方案确认前，Claude 自主完成计划反思：

**反思维度（4 个）**：
1. **需求覆盖度**：执行计划是否覆盖了所有验收条件？遗漏哪些？
2. **过度设计风险**：是否引入了验收条件未要求的复杂性？可否简化？
3. **拆分必要性**：步骤间耦合度如何？是否应该拆为多个 CR？
4. **技术假设合理性**：计划依赖了哪些未验证的假设？风险可控吗？

**输出**：1-3 行观察写入 CR 事件表备注列（`[plan-reflection: ...]`），格式：
- 无问题："计划反思：4 维度无异常"
- 有问题："计划反思：需求覆盖遗漏验收条件 3；过度设计风险低；建议保持单 CR"

**触发条件**：与执行计划生成相同——S 不执行，M 可选，L/XL 必须。

**铁律：L/XL CR 不可跳过计划反思。**

| Claude 可能的合理化 | 预防反驳 |
|-------------------|---------|
| "计划已经很完善了" | 反思是审视完善度的机制，跳过反思 ≠ 计划完善 |
| "反思浪费时间" | 1-3 行观察 < 30 秒。发现问题后避免的返工远超此成本 |

### 方案确认门禁（L/XL 强制）

L/XL CR 的执行计划生成后，**必须主动展示并等待用户确认**，才能进入 developing 实现阶段。

**流程**：
1. Claude 生成执行计划后，**主动展示**给用户（不默认隐藏）
2. 附加："这是实现方案，确认开始还是要调整？"
3. 用户确认 → 进入 developing
4. 用户调整 → 支持自然语言修改指令 → 修改后重新展示确认

**自然语言编辑指令**（步骤 4）：
- "删除步骤 N" → 移除指定步骤，后续步骤重新编号
- "合并步骤 N 和 M" → 合并为单个步骤，保留两者的文件范围和预期结果
- "在步骤 N 后加一步：[描述]" → 插入新步骤，后续步骤重新编号
- "调整步骤 N 的范围" → 修改指定步骤的文件范围或预期结果
- "交换步骤 N 和 M 的顺序" → 交换，自动检查依赖关系是否仍然成立
- 修改后 Claude 重新展示完整执行计划 + 再次询问确认

**步骤暂停点**（可选）：
- 方案确认阶段，用户可设定暂停点："在步骤 N 后暂停"
- 到达暂停点时：输出当前进度摘要 + 已完成步骤列表 + "继续还是要调整后续步骤？"
- 用户说"继续" → 继续执行后续步骤
- 用户调整 → 修改剩余步骤（同上自然语言编辑指令）→ 确认后继续
- 默认无暂停点——不设定则全程自主推进
- 暂停点记录在 CR 事件表备注列：`[pause-point: after-step-N]`

**S/M 不受影响**——直接进入 developing。

**设计决策**：设计和实现的混合是质量问题的根源。L/XL CR 通过方案确认门禁将"设计阶段"和"实现阶段"显式分离，确保设计错误在编码前发现。

### 验收条件格式指引

验收条件格式根据 CR 复杂度自适应选择（与 `knowledge/_schema/cr-format.md` 一致）：

**简单（S）— 自由文本**：
```
- **验收条件**：修复 typo 并确认页面显示正确
```

**标准（M）— 编号清单**：
```
- **验收条件**：
  1. 支持邮箱+密码登录
  2. 密码强度校验（≥8 位，含大小写和数字）
  3. 登录失败显示错误提示
```

**复杂（L/XL）— Given/When/Then**：
```
- **验收条件**：
  1. Given 用户已注册, When 输入正确凭据, Then 登录成功并跳转首页
  2. Given 用户输入错误密码 3 次, When 再次尝试, Then 账户锁定 15 分钟
```

规则：简单不加格式负担；标准/复杂由 Claude 自动生成对应格式；Gate 2 按条目逐一比对。

### 歧义标记

当意图检查点中发现歧义时，Claude 在 CR 意图 section 添加歧义标记：

```
[待确认: 具体疑问？建议 X，因为 Y]
```

**生命周期**：
1. **添加**：意图检查点发现歧义时自动添加（附推荐值和理由）
2. **解决**：用户确认或 Claude 以假设前进后，替换为确认内容或移至约束字段
3. **检查**：Gate 2 前检查 CR 意图 section，若存在未解决的 `[待确认]` 标记则 Gate 2 失败

**示例**：
```
- **范围**：实现用户登录功能
  [待确认: 是否需要"记住我"功能？建议暂不实现，因为首版聚焦核心登录流程]
```

## 复杂度量化评估

意图检查点完成后，Claude 自动评估 CR 复杂度并写入 CR 文件的"复杂度"字段。

### 评估维度

| 维度 | S (≤) | M | L | XL (>) |
|------|-------|---|---|--------|
| 涉及文件数 | 3 | 4-7 | 8-15 | 15 |
| 涉及目录数 | 1 | 2-3 | 4-5 | 5 |
| 独立验收条件数 | 1 | 2-3 | 4+ | 多组 |
| 跨模块依赖 | 无 | 单向 | 双向 | 架构级 |

### 评分规则

1. 逐维度评估，取**最高等级**作为最终复杂度
2. 写入 CR 文件元信息的"复杂度"字段
3. L/XL → 自动进入下方"CR 拆分建议"流程
4. S/M → 跳过拆分评估，直接开工

### 与已有拆分建议的关系

复杂度评估取代了原有的启发式触发条件（>5 文件 / >3 目录 / >3 验收条件），提供统一的量化基础。拆分建议的评估步骤和输出格式不变。

## CR 拆分建议

意图检查点完成后，对标准和复杂 CR 进行拆分评估：

### 触发条件

- 复杂度评估结果为 **L** 或 **XL**（即：涉及 >7 文件、>3 目录、>3 验收条件，或存在跨模块双向依赖）

### 评估步骤

1. 分析意图 section 中的范围和验收条件
2. 识别可独立交付的功能切面（按模块、层次或功能维度）
3. 评估各切面之间的依赖关系

### 输出格式

满足触发条件时，在"明确了范围，开始做"之前附加建议：

```
建议将此变更拆分为 N 个独立 CR：
- CR-A：[模块/功能]（[文件范围]）
- CR-B：[模块/功能]（[文件范围]）
依赖关系：CR-A 应先于 CR-B。
要拆分还是继续作为一个 CR 推进？
```

### 规则

- **不阻断**：用户选择不拆分则继续作为单个 CR 推进
- **简单 CR 跳过**：复杂度为"简单"的 CR 不做拆分评估
- **记录决策**：用户选择不拆分时，在 CR 事件表备注"评估拆分后决定保持单 CR，因为 [用户理由]"

## 步骤隔离（L/XL CR）

L/XL CR 的状态转换严格逐阶段执行，防止 AI 在长上下文中提前行动或合并步骤。

### 隔离规则

| 当前阶段 | 聚焦范围 | 不可预读 |
|---------|---------|---------|
| created→developing | 意图检查点 + 复杂度评估 + 执行计划 + 方案确认 | Gate 1 检查内容、Gate 2 策略 |
| developing | 按执行计划逐步实现 + Gate 1 准备 | Gate 2 验收对比、Review 摘要格式 |
| developing→verifying | Gate 1 执行 + 自修复 | Gate 2 细节、合并策略 |
| verifying→in_review | Gate 2 执行 + 对抗审查 + Review 摘要 | 合并操作、连锁更新步骤 |
| in_review→approved→merged | 等待人类审批 → 执行合并和连锁更新 | 下一个 CR 的内容 |

### 执行方式

1. **进入新阶段时**：仅加载该阶段的相关 procedures section，不读取后续阶段
2. **阶段完成时**：Git commit + checkpoint 后，再加载下一阶段的 procedures
3. **context: fork 场景**：当 Skill 在子 agent 中运行时，天然实现上下文隔离——每个子 agent 仅接收当前阶段的上下文

### L/XL 步骤级 Checkpoint 与进度通知

L/XL CR 在 developing 阶段按执行计划逐步推进时，每完成一个步骤：

1. **Checkpoint 标记**：在 CR 事件表备注列写入 `[checkpoint: step-N-done]`（N 为步骤序号），用于跨会话精确恢复
2. **state.md 步骤定位**：更新"当前工作"为 `→ 步骤 N/M：[下一步描述]`（M 为总步骤数），跨会话恢复时可精确定位到下一步
3. **进度通知**：输出 1 行进度——`[步骤 N/M] [步骤描述] ✅`

示例：
```
[步骤 3/5] middleware 实现完毕 ✅
```

**规则**：
- S/M CR 不触发步骤级 checkpoint 和进度通知（无执行计划）
- 进度通知为 1 行，不展开为详细说明
- 跨会话恢复时：读取 state.md 的步骤标记 + CR 事件表最后一个 `step-N-done` checkpoint → 精确定位到下一步

### Developing 中期 Compact 建议（L/XL 专属）

L/XL CR 执行计划完成 >60% 时，如果当前步骤与前序步骤无直接代码依赖（如前序步骤处理模块 A，当前步骤处理模块 B），输出非强制 compact 建议：

```
💡 建议 /compact — 执行计划已完成 N/M 步，前序步骤上下文已持久化到 git 和 CR，压缩可释放空间给后续步骤。
```

**触发条件**（全部满足）：
1. L/XL CR 且执行计划存在
2. 当前步骤序号 > 总步骤数 × 60%
3. 当前步骤与最近完成的步骤无直接文件依赖（不涉及相同文件）
4. 本 CR 内未触发过此建议（同一 CR 仅建议 1 次）

**规则**：
- 纯建议，不阻断工作流
- S/M CR 不触发
- 与 Gate 1 后 compact 建议互补（§6 在 Gate 阶段，本建议在 developing 中期）

### S/M CR 不受约束

步骤隔离仅对 L/XL CR 强制执行。S/M CR 流程简短，Claude 可自然流畅地推进各阶段。

## 决策记录

Claude 在推进过程中自动在 CR 事件表的备注列记录重要决策：

| 记录 | 不记录 |
|------|--------|
| 方案选择及原因（"用 X 而非 Y，因为 Z"） | 常规编码步骤 |
| 发现的边界条件和假设 | 测试通过/不通过（已有 checkbox） |
| 方案调整及原因 | 文件创建/修改（已有 git log） |
| 依赖发现（"发现还需要改 A 才能做 B"） | 质量检查结果（已有 checkbox） |

原则：只记"为什么"不记"做了什么"——做了什么有 git log。

## 意图漂移检测

推进模式中，每次 checkpoint（git commit + 更新 CR）时执行轻量漂移检测：

### 检测步骤

1. 获取当前 CR 的意图 section 中描述的**范围**（涉及的模块/目录/文件）
2. 执行 `git diff --name-only` 获取本次 checkpoint 涉及的文件列表
3. 对比 diff 文件与意图范围：
   - 计算范围外文件占比 = 范围外文件数 / diff 总文件数
   - 范围外文件 = diff 中不属于意图声明的模块/目录的文件

### 触发条件

- 范围外文件占比 > 30% → 在 checkpoint 输出中附加提示
- 范围外文件占比 ≤ 30% → 不提示，正常继续

### 提示格式

```
checkpoint 提示：当前变更涉及意图范围外的文件：[文件列表]。
这是合理扩展还是需要调整 CR 范围？
```

### 规则

- **不阻断**：漂移检测仅提醒，不阻断工作流
- **不累积**：每次 checkpoint 独立计算，不与之前的 checkpoint 累积
- **简单 CR 跳过**：复杂度为"简单"（单文件）的 CR 跳过漂移检测
- **用户确认后记录**：用户确认为合理扩展时，更新 CR 意图 section 的范围字段

## 复杂度漂移检测（运行时升级守卫）

开发过程中持续监控实际复杂度是否超出初始评估，提供运行时升级建议。

### 检测步骤

每次 checkpoint 时（与意图漂移检测同时执行），额外检查：
1. 获取 CR 的"复杂度"字段（初始评估）
2. 统计当前 CR 累积涉及的文件数（`git diff --name-only main..HEAD`）和目录数
3. 对比实际指标与初始评估对应的阈值：

| 初始评估 | 文件数预警阈值 | 目录数预警阈值 |
|---------|-------------|-------------|
| S | > 3 文件 | > 1 目录 |
| M | > 7 文件 | > 3 目录 |

### 触发条件

- 实际指标超出初始评估的预警阈值 → 触发升级建议
- L/XL 不触发（已是最高级别）

### 提示格式

```
⚠️ 复杂度漂移：初始评估 [S/M]，实际涉及 N 文件 / M 目录。
建议：升级为 [M/L] 并[补充执行计划 / 考虑拆分]。继续还是调整？
```

### 规则

- **同一 CR 只提醒一次**：首次触发后记录到 CR 事件表，后续不重复
- **不阻断**：用户选择继续则按原路推进
- **与意图漂移互补**：意图漂移检测"做了什么不该做的"，复杂度漂移检测"规模比预期大"

## Defect/Hotfix CR 创建规则

当 CR 类型为 defect 或 hotfix 时，意图检查点有额外步骤：

### Defect CR 创建

1. 自动添加"根因分析"section（格式参考 `knowledge/_schema/cr-format.md`）
2. 现象字段：从用户描述自动填充
3. 根因字段：初始为"待调查"，developing 阶段定位后填充
4. 引入点字段：尝试追溯到引入问题的 CR（通过 git blame 或 Release 追溯）
5. 分支名前缀：`fix/` 而非 `feature/`

### Hotfix CR 创建

1. 同 defect 规则，额外检查 severity
2. severity:critical → 告知用户可走加速路径（跳过 in_review）
3. 用户确认加速路径后：
   - 在 CR 事件表记录"加速路径：跳过 in_review，原因 [描述]"
   - developing → verifying → merged（跳过 in_review + approved）
   - merged 后提醒补充事后审批
4. 分支名前缀：`hotfix/` 而非 `feature/`

### Defect/Hotfix 修复后处理

1. merged 后自动触发 pace-learn 提取根因 pattern
2. 更新 CR 根因分析 section：填充根因、预防措施
3. 如该 defect 关联了 Release → 更新 Release 文件的缺陷记录
4. 更新 dashboard.md 缺陷修复周期指标

## Gate 通过反思

Gate 通过后，Claude 附加 1-2 行轻量反思观察，记录到 CR 事件表备注列。不阻断流程，仅积累质量信号。

### Gate 1 通过反思

Gate 1（代码质量检查）通过后，反思以下维度：
- **技术债观察**：本次变更是否引入了临时方案或硬编码？
- **测试覆盖观察**：关键路径的测试是否充分？新增代码是否有对应测试？
- **测试先行回顾**（仅当 test-strategy.md 存在时）：新增实现代码对应的验收条件，是否在编码前已有测试骨架？若无，在反思中标注"建议下次对 [条件] 先写测试"

输出格式：`[gate1-reflection: 技术债观察 + 测试覆盖观察 + 测试先行]`
示例：`[gate1-reflection: 无新增技术债；核心路径测试覆盖充分；验收条件 2 无前置测试——建议下次先 /pace-test generate]`

### Gate 2 通过反思

Gate 2（需求质量检查）通过后，反思以下维度：
- **边界场景观察**：验收条件是否覆盖了异常和边界情况？
- **验收全面性观察**：是否有隐含的用户期望未在验收条件中体现？

输出格式：`[gate2-reflection: 边界场景观察 + 验收全面性观察]`
示例：`[gate2-reflection: 空输入场景已覆盖；用户可能期望错误提示国际化——超出当前 CR 范围]`

### 规则

- **不阻断**：反思仅记录，不影响 Gate 通过/失败判定
- **简单 CR 精简**：S 复杂度 CR 可省略 Gate 通过反思（变更范围小，反思价值低）
- **与 pace-learn 配合**：反思内容在 CR merged 后纳入 pace-learn 的经验提取范围

## 功能发现（嵌入式触发，从 devpace-rules.md §11 迁入）

根据用户使用进度，在自然时机**直接执行**进阶功能的核心价值输出（而非推荐命令名称）。每条仅出现一次（使用 §15 教学标记去重）。

**"隐形 AI"原则**：用户感受到的是"devpace 自动帮我做了 X"，而非"devpace 教我用一个命令"。

| 条件 | 嵌入式行为（直接执行） | 标记值 |
|------|----------------------|--------|
| 第 3 个 CR merged | 直接输出迷你回顾——"3 个变更完成：Gate 一次通过率 N%，平均 M 个 checkpoint。" | `discover_retro` |
| 首次变更管理成功完成 | 直接输出影响摘要——"本次变更涉及 N 个模块，已同步更新关联状态。" | `discover_change` |
| 第 5 个 CR merged | 直接输出迭代进度——"已完成 N/M 个功能点，迭代完成率 X%。" | `discover_plan` |
| 首个 CR merged 且 releases/ 目录已存在 | 直接检查 Release 纳入——"此变更可纳入当前 Release。是否纳入？" | `discover_release` |
| 首个 defect/hotfix CR 创建 | 直接输出追溯——"追溯到引入点：[CR/commit]，影响范围：[模块]。" | `discover_feedback_report` |

**执行规则**：检查 state.md 教学标记 → 未出现过 → 在自然时机直接执行功能的核心价值输出 → 更新标记。用户如果问"怎么手动触发"，再说明对应命令名称。

## 执行透明摘要（§13.5 Human Transparency）

pace-dev 完成实现后（Gate 1 通过、代码已提交），**必须**向用户输出结构化变更摘要：

```
📋 变更摘要
- **文件**：[新增 N / 修改 M / 删除 D 个文件]
- **关键决策**：[实现中做出的技术选择，如"选择 X 方案而非 Y，因为..."]
- **质量**：[Gate 反思浓缩，≤20 字]
- **CR 状态**：[当前状态] → [下一状态]
- **注意**：[风险/遗留/需要用户关注的事项，无则省略此行]
```

**规则**：
- 简单 CR（S）：1-2 行即可（文件列表 + 状态变化），省略"质量"行
- 标准/复杂 CR（M/L/XL）：完整 5 项，"质量"行浓缩 Gate 反思为 ≤20 字（如"无新技术债，核心路径测试充分"）
- 不透明动作禁令：写入 .devpace/ 的任何文件必须在摘要中列出——用户不应在不知情的情况下发现文件被修改

## PF 溢出检查（Overflow Check）

CR 创建或状态变更（尤其是 merged）时，检查关联 PF 是否满足溢出条件，满足时自动创建独立文件。

### 检查时机

| 事件 | 检查行为 |
|------|---------|
| CR 创建（新关联 PF） | 计算该 PF 关联的 CR 数（含本次），≥3 则触发 |
| CR merged（§11 连锁更新时） | 综合检查三个条件 |
| `/pace-change modify` 涉及 PF | 由 change-procedures.md 负责检查并触发 |

### 溢出条件（满足任一）

1. **功能规格行数 >15**：计算 project.md 功能规格 section 中该 PF 段落的行数（从 `### PF-xxx` 到下一个 `###` 或 section 末尾）
2. **关联 CR 数 ≥3**：计算 project.md 价值功能树中该 PF 行关联的 CR 数（含已完成和进行中）
3. **经历过 modify**：检查迭代变更记录或 CR 事件表中是否有该 PF 的 `/pace-change modify` 记录

### 溢出执行步骤

检测到溢出条件后，按 `knowledge/_schema/pf-format.md` 定义的步骤执行：

1. 创建 `.devpace/features/` 目录（如不存在）
2. 从 project.md 功能规格 section 提取该 PF 的验收标准、边界等内容
3. 从 project.md 价值功能树提取 BR/OBJ 关联和 CR 列表
4. 从 backlog/ CR 文件提取各 CR 状态和类型
5. 聚合信息创建 `features/PF-xxx.md`
6. 更新 project.md：
   - 功能规格 section 中移除该 PF 段落
   - 价值功能树中该 PF 行追加 `→ [详情](features/PF-xxx.md)`
7. 在执行透明摘要中报告溢出："PF-xxx 信息已迁移到独立文件 features/PF-xxx.md"

### 规则

- **零摩擦**：溢出自动发生，不询问用户确认（对齐 P1 原则）
- **幂等性**：已存在 `features/PF-xxx.md` 时不重复溢出，仅更新已有文件
- **向后兼容**：无 `features/` 目录的项目不触发任何溢出逻辑
- **容错**：溢出失败（如文件写入异常）不阻断主流程，在摘要中标注失败原因
