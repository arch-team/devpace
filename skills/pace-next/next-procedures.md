# 下一步导航规程

> **职责**：/pace-next 的 12 级优先级决策算法、数据源读取、输出格式和角色适配规则。

## 未初始化项目输出

当 `.devpace/state.md` 不存在时，直接输出：

```
💡 建议：初始化项目研发管理 → /pace-init
   原因：启用追踪后可获得上下文恢复、质量门禁、变更管理等能力
```

不做进一步信号采集，直接结束。

## Step 2：信号采集

按以下顺序读取数据源（目录/文件不存在时静默跳过）：

| 数据源 | 读取内容 | 用于优先级 |
|--------|---------|-----------|
| `.devpace/backlog/*.md` | 所有 CR 的状态字段 | P1/P2/P4/P7/P9/P11 |
| `.devpace/state.md` | 当前工作、下一步 | P2 |
| `.devpace/releases/*.md` | Release 状态字段 | P3 |
| `.devpace/iterations/current.md` | PF 完成率、变更记录 | P5/P10 |
| `.devpace/metrics/dashboard.md` | 最近更新日期 | P6 |
| `.devpace/project.md` | MoS checkbox、PF 列表 | P8/P9 |
| `.devpace/metrics/insights.md` | 经验 pattern（可选增强） | Step 4 |

**读取策略**：
- 使用 Glob 扫描 `backlog/*.md` 和 `releases/*.md`，再用 Grep 提取状态字段
- 文件不存在或目录为空时，相关优先级信号视为"未命中"
- 不全量读取文件内容，只提取决策所需字段

## Step 3：12 级优先级矩阵

从高到低遍历，取**首个命中**的信号。

| 优先级 | 信号条件 | 建议模板 |
|:------:|---------|---------|
| 1 | `backlog/` 中 `in_review` 状态 CR > 0 | "有 N 个变更等你审批——审批后才能合并推进" |
| 2 | `backlog/` 中存在 `developing` 状态 CR | "继续 [CR 标题]——上次停在 [state.md 进行中摘要]" |
| 3 | `releases/` 中存在 `deployed` 未 `verified` 的 Release | "[Release 名] 已部署待验证 → /pace-release" |
| 4 | `backlog/` 中 `paused` CR 且其阻塞原因已不存在 | "[CR 标题] 之前暂停，现在可以继续" |
| 5 | `iterations/current.md` PF 完成率 > 80% | "迭代接近完成——回顾后规划下一迭代" |
| 6 | `dashboard.md` 最近更新距今 > 7 天 + `backlog/` 有 merged CR | "有新完成的工作，建议做一次回顾" |
| 7 | `backlog/` 中 defect 类型 CR 占比 > 30% | "缺陷比例较高，建议优先修复" |
| 8 | `project.md` MoS 达成率 > 80% | "目标接近达成，回顾成效指标" |
| 9 | `project.md` 有 PF 无对应 CR（功能树中 PF 下无任何 CR） | "功能 [PF 名] 还没开始——可以着手推进" |
| 10 | 上一迭代已关闭 + `iterations/current.md` 不存在 | "上一迭代已结束——规划新迭代" |
| 11 | `backlog/` 全部 CR 为 merged 或 released | "所有任务完成——开始新功能或回顾" |
| 12 | 无信号命中 | "当前无紧急事项，可以自由探索或规划新目标" |

**优先级 4 判断规则**：读取 paused CR 的暂停原因（事件表最后一条 paused 事件备注），判断阻塞是否已解除：
- 原因为等待某 CR → 检查该 CR 是否已 merged
- 原因为外部阻塞 → 无法自动判断，跳过此条
- 无法判断 → 跳过此条

**优先级 6 日期判断**：读取 `dashboard.md` 首行或 blockquote 中的"最近更新"日期字段。字段缺失时跳过此条。

**优先级 9 判断规则**：遍历 `project.md` 功能树中的 PF 行，如果 PF 下没有关联任何 CR（无 CR 缩进行），视为"未开始的功能"。

## Step 4：经验增强

如果 `.devpace/metrics/insights.md` 存在，按当前命中的优先级信号类型匹配经验 pattern：

| 命中优先级 | 匹配策略 | 增强方式 |
|:---------:|---------|---------|
| 2（继续 developing） | 按 CR 类型或涉及模块匹配 | 附加"上次类似变更 [经验摘要]" |
| 7（缺陷占比高） | 匹配防御型 pattern | 附加具体改进建议 |
| 其他 | 匹配任何相关 pattern | 附加"经验参考：[pattern 摘要]" |

**规则**：
- 只读引用，不写入 insights.md
- 置信度 > 0.7 的 pattern 才引用
- 无匹配时不附加，保持简洁
- 增强内容 ≤ 1 行

## Step 5：角色适配

根据 §13 角色意识调整建议措辞（不改变优先级顺序）：

| 角色 | 视角调整 |
|------|---------|
| Dev（默认） | 技术行动为主："继续实现 [功能]""修复 [问题]" |
| PM | 功能交付为主："[功能] 交付进度需关注""迭代目标完成 N%" |
| Biz | 业务目标为主："[MoS 指标] 接近达标""业务价值交付 N 项" |
| Tester | 质量指标为主："N 个缺陷待修复""质量门通过率 N%" |
| Ops | 部署发布为主："[Release] 待验证""部署健康度需确认" |

角色读取方式：从 `project.md` 的角色字段或当前会话推断。无法推断时默认 Dev。角色适配通用原则见 `skills/pace-role/role-procedures-dimensions.md`（权威源）。

## 输出格式

### 默认输出（≤ 3 行）

```
💡 建议：[行动建议]
   原因：[1 句推理]
   操作：[对应命令或自然语言指引]
```

**命令引导规则**：
- 优先级 1 → "→ /pace-review"
- 优先级 2 → "→ /pace-dev 或说'继续'"
- 优先级 3 → "→ /pace-release"（由 pace-release 空参向导接管）
- 优先级 5 → "→ /pace-retro 后 /pace-plan"
- 优先级 6 → "→ /pace-retro"
- 优先级 9 → "→ 说'帮我实现 [PF 名]'"
- 优先级 10 → "→ /pace-plan"
- 其他 → 用自然语言描述操作

### detail 输出（≤ 8 行）

```
💡 建议：[最高优先级行动]
   原因：[推理]
   操作：[命令]

📋 候选（按优先级）：
  2. [次优先行动] —— [简述]
  3. [第三行动] —— [简述]
```

detail 模式遍历全部 12 级，收集所有命中信号，输出 top-3。只有 1 条命中时不展示候选段。

### 输出约束

- 自然语言为主，不暴露 CR ID、状态机术语（遵循 §3）
- "操作"行引导到对应 Skill 或自然语言操作，不引导到具体子命令（如引导到 /pace-release 而非 /pace-release verify）
- 经验增强内容附加在"原因"行末尾，用"——"连接
