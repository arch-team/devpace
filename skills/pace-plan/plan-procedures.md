# 规划新迭代详细规程

> **职责**：规划新迭代的详细处理流程（Step 3 + 4）。经验引用时机详见 `knowledge/experience-reference.md`。

## Step 3：规划新迭代

> `$ARGUMENTS` 为 `next` 时直接进入；否则在 Step 2 完成后或无当前迭代时进入。

### 3.1 读取候选 PF

读取 `.devpace/project.md` 功能树，列出所有待开始（⏳）和进行中（🔄）的 PF。

**空功能树降级**（project.md 桩状态或无待做/进行中 PF 时）：
1. 使用 `AskUserQuestion`："这个迭代想做什么？用 1-2 句话描述目标"
2. 从用户回答推断 PF（参考 pace-init `--from` 的推断逻辑：关键词提取→功能点识别→PF 映射）
3. 将推断的 PF 写入 project.md 功能树（添加 `<!-- source: claude -->` 溯源标记）
4. 展示推断结果供用户确认/调整："基于你的描述，我推断了以下功能点：[PF 列表]。确认还是需要调整？"
5. 用户确认后继续正常规划流程（3.2+）

**PF 数量 ≥5 时按 BR 分组**：
- 按 project.md 中 PF 归属的 BR 分组，输出分组摘要（每组 PF 数 + 概要说明）
- 用户可指定 BR 后展开该组详细 PF 列表
- PF 数量 <5 时直接全部列出

### 3.2 延后 PF 优先列出

如有上一迭代延后的 PF，优先列出并标注"上迭代延后"。

### 3.3 范围估算

**有历史数据时**：
- 读取 `.devpace/metrics/dashboard.md` 获取历史平均变更周期和迭代速度
- 读取已归档迭代（`iterations/iter-*.md`）统计历史 CR 完成天数
- 按每个候选 PF 的预估 CR 数 × 平均变更周期计算工作量
- 引用历史迭代速度：如速度 <1.0，建议纳入不超过 `上一迭代实际完成数` 个 PF——"历史迭代速度 X%，建议纳入不超过 N 个 PF"
- 预估工作量超过迭代周期时，附加建议："预估工作量可能超出迭代周期，建议缩减 N 个 PF 或延长迭代"

**无历史数据时**（启发式估算）：
- 触发条件：无 `dashboard.md` 且无 `iterations/iter-*.md`
- 启发式规则：
  - PF 有验收标准 ≥3 条 → **L**（预估 4-7 个 CR）
  - PF 有验收标准 1-2 条 → **M**（预估 2-4 个 CR）
  - PF 无验收标准 → **S**（预估 1-2 个 CR）
- 每 CR 约 1 个工作会话（参考值）
- 明确标注："启发式估算（无历史数据），首个迭代结束后自动切换为历史数据估算"
- 首次迭代默认容量上限：3-5 个 PF（避免过度承诺）

### 3.4 业务引导增强

- 如有上一迭代 `.devpace/metrics/dashboard.md` 数据 → 分析上迭代 MoS 达成情况，输出风险预测
- 基于 project.md 现有 BR/PF 结构推断功能分组，主动建议 MoS 指标

### 3.5 PF 依赖校验

功能规格有"依赖"字段时：
- 检查候选 PF 的依赖是否已满足（依赖的 PF 已完成或在本迭代中排序更前）
- 未满足 → 附加建议："PF-002 依赖 PF-001（[原因]），建议先纳入 PF-001 或确认可并行"
- 无依赖字段 → 跳过校验

**风险集成**（`.devpace/risks/` 存在时）：
- 扫描候选 PF 关联的 open 风险条目（通过 PF 编号匹配风险的 `affected_pf` 字段）
- 高风险 PF → 标注 "⚠️ 高风险" + 估算增加 30% buffer（如 M 3CR → 4CR）
- 中风险 PF → 标注 "⚡ 中风险"（不调整估算）
- 无匹配风险或 risks/ 不存在 → 静默跳过

### 3.5.1 回顾建议直联

读取上一迭代 `iter-N.md`（N 为最新归档）的回顾 section（如有"下迭代建议"段）：
- 有建议 → 展示给用户："上一迭代回顾建议：[内容]"
- 无回顾 section 或无建议 → 静默跳过

### 3.6 智能规划建议（Plan Proposal）

取代原有的逐项填空模式，Claude 先生成建议方案，用户确认或调整。

**生成 Plan Proposal**：

1. **目标自动归纳**：从候选 PF 的 BR 归属自动归纳迭代目标（如"完成用户认证相关功能"）
2. **推荐 PF 选择策略**（按优先级排序选取）：
   a. 延后 PF 优先（上迭代遗留）
   b. 依赖已满足的 PF 优先
   c. 按 project.md 中的排列顺序（用户隐含的优先级）
   d. 容量限制：不超过上一迭代实际完成量（首次用 Step 3.3 启发式估算默认上限）
3. **预估**：每个 PF 附带工作量估算（有历史数据用 Step 3.3 结果，无则用启发式）
4. **总容量评估**：汇总预估，与迭代周期（如有）对比

**展示 Proposal**：

```
建议方案：
  目标：[自动归纳的 1 句话目标]
  纳入 PF（共 N 个）：
    [P0] PF-001 xxx — 预估 M（2-4 CR）[上迭代延后]
    [P1] PF-003 xxx — 预估 S（1-2 CR）
    [P2] PF-005 xxx — 预估 L（4-7 CR）⚠️ 高风险
  总预估工作量：约 X 个工作会话
  周期：[起止日期，如用户之前有设定]

确认这个方案？还是需要调整？
```

**用户响应分支**：
- "好 / 确认 / 可以" → 直接采用 Proposal
- "调整：去掉 X，加上 Y" / 具体修改指令 → 增量修改后重新展示
- 提供完整替代方案（自己列 PF） → 使用用户方案
- 指定优先级调整 → 更新 P0/P1/P2 标记

### 3.7 生成迭代文件

1. 生成 `iterations/current.md`（格式遵循 `knowledge/_schema/iteration-format.md`，含优先级列）
2. 更新 `.devpace/state.md`：反映新迭代信息

## Step 4：确认与输出

1. 展示新迭代规划摘要（目标 + PF 列表 + 周期 + 工作量估算）
2. **衔接引导**：
   - 识别 Proposal 中最高优先级 PF
   - 展示："迭代已规划。最高优先级是 [PF 名]，现在就开始？"
   - 用户确认 → 建议执行 `/pace-dev`（plan 作为 fork agent 不能直接调用 pace-dev，通过建议引导）
   - 用户说"晚点" / 不回应 → 正常结束
