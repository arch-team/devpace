# pace-pulse 详细规则

> **职责**：定义节奏心跳检查的信号评估标准和建议模板。由 `SKILL.md` 引用。

## 信号评估表

| 信号 | 检测方式 | 阈值 |
|------|---------|------|
| CR 滞留 | 当前 CR 在 developing 的 checkpoint 计数 | > 5 次 |
| 迭代接近尾声 | current.md PF 完成率 | > 80% |
| Review 积压 | backlog/ 中 in_review 状态 CR 数 | > 2 个 |
| 检查项重复失败 | 当前 CR 质量检查同一项失败次数 | > 2 次 |
| 会话高产出 | 本会话已 merged 的 CR 数 | ≥ 2 个 |
| 迭代频繁变更 | current.md 变更记录条数 | > 3 条 |
| 上下文密集 | 本会话已有 3+ 个 Gate 1 通过事件 | ≥ 3 个 |
| 风险积压 | `.devpace/risks/` 中 open 状态风险 > 3 或 High 严重度风险 > 0（数据格式见 pace-guard 风险文件 Schema） | Medium |
| 同步滞后 | sync-mapping.md 存在时，已关联 CR 最后同步距今时间 | > 24h |
| 迭代时间紧迫 | current.md 周期字段距结束日期 <2 天且 PF 完成率 <50% | Medium |
| 迭代健康度低 | current.md PF 完成率 / 迭代时间消耗比 | < 0.5 |
| 目标回顾逾期 | dashboard.md "最近更新"日期距今，且 project.md MoS 有未勾选项 | > 14 天 |
| Snooze 唤醒 | CR 事件表或迭代变更记录中 Snooze 条目的触发条件已满足 | 条件匹配 |
| PF 验收漂移 | PF 实际范围（CR 变更文件集合）与 PF 验收标准覆盖范围的偏差 | > 30% |
| 重复失败模式 | 同一 PF 下连续 2+ CR 在同一检查项失败 | ≥ 2 次 |
| 需求冲突 | 2+ 进行中 CR 修改同一文件区域（通过 git diff 交集检测） | ≥ 2 CR |
| 运维关键词 | 用户对话中出现运维关键词（"线上问题""生产报错""告警""监控异常""部署后""上线后发现"） | 关键词匹配 |

## 建议模板

无信号触发 → 静默，不输出任何内容。
有信号触发 → 输出 1-2 行自然语言建议，格式：

```
💡 [建议内容]
```

| 信号 | 建议 |
|------|------|
| CR 滞留 | "当前变更范围较大（已 N 个 checkpoint），考虑拆分为更小的变更？需要我帮你分析拆分方案吗？" |
| 迭代接近尾声 | "迭代完成率已超 80%，建议规划下一迭代——`/pace-plan next`" |
| Review 积压 | "有 N 个变更等待 review——`/pace-review`" |
| 检查项重复失败 | "质量检查「[项目]」连续失败 N 次，建议调整检查策略。需要我分析失败模式吗？" |
| 会话高产出 | "本次会话已完成 N 个变更，建议做一次增量回顾——`/pace-retro update`" |
| 迭代频繁变更 | "本迭代已有 N 次范围变更，注意目标偏移风险——`/pace-plan health` 查看健康度" |
| 上下文密集 | "本次会话上下文较重（N 个变更实现），建议在逻辑边界压缩——`/compact`" |
| 风险积压 | "有 [N] 个未处理风险（含 [M] 个高级别）——`/pace-guard report`" |
| 同步滞后 | "有 N 个 CR 的外部同步已超过 24 小时——`/pace-sync push`" |
| 迭代时间紧迫 | "迭代即将结束但完成率不足 50%——`/pace-plan adjust` 调整范围或延长迭代" |
| 迭代健康度低 | "迭代健康度低于 0.5，进度落后于时间消耗——`/pace-plan adjust` 缩减范围" |
| 目标回顾逾期 | "距上次业务回顾已超 2 周，有未达成的业务指标——`/pace-retro`" |
| Snooze 唤醒 | "之前因「[搁置原因]」延后的变更「[描述]」（[CR编号]/[PF名]），触发条件「[条件]」已满足。需要现在评估吗？" |
| PF 验收漂移 | "功能「[PF名]」的实际变更范围偏离验收标准超 30%——`/pace-change modify`" |
| 重复失败模式 | "功能「[PF名]」下连续 N 个变更在「[检查项]」失败，建议审查检查策略。需要我分析共性原因吗？" |
| 需求冲突 | "有 N 个进行中变更修改同一区域（[文件]），注意合并冲突风险。需要我检查冲突详情吗？" |
| 运维关键词 | "检测到可能的生产问题描述——`/pace-feedback report` 系统化处理" |

## 信号分组与关联诊断

当同组 2+ 信号同时触发时，输出综合诊断而非独立建议。

| 组 | 信号 | 综合诊断模板 |
|----|------|------------|
| 效率 | CR 滞留 + 会话高产出 + 上下文密集 | "开发节奏紧凑但当前变更停滞，建议拆分 CR 或 /compact 释放上下文" |
| 质量 | 检查项重复失败 + PF 验收漂移 + 重复失败模式 | "质量信号集中，「[PF名]」可能存在系统性问题，建议 `/pace-guard report` 全面审查" |
| 计划 | 迭代接近尾声 + 迭代时间紧迫 + 迭代健康度低 | "迭代压力信号集中——`/pace-plan health` 查看健康度，考虑 `/pace-plan adjust` 调整范围" |
| 风险 | 风险积压 + 同步滞后 + 需求冲突 | "多维度风险叠加——`/pace-guard report` 了解全貌" |

**规则**：综合诊断替代组内独立建议，仍计为 1 条配额。无综合诊断触发时按独立信号处理。

## 动态信号权重

信号评估表的基础优先级（从上到下）可根据迭代阶段动态调整：

| 迭代阶段 | 判定方式 | 权重提升信号 |
|---------|---------|------------|
| 早期（完成率 <30%） | current.md PF 完成率 | CR 滞留、需求冲突——早期问题影响大 |
| 中期（30%-70%） | current.md PF 完成率 | 默认权重，无调整 |
| 冲刺期（>70% 或距结束 <3 天） | current.md 完成率或剩余时间 | 迭代时间紧迫、迭代健康度低、Review 积压——冲刺期效率优先 |
| 审查密集期 | in_review CR > 2 | Review 积压——审查瓶颈优先处理 |

**权重调整规则**：提升的信号在优先级排序中上移 3 位。无 current.md 时使用默认权重。

## 输出规则

- 无信号触发时完全静默（但见"正向反馈"规则）
- 有信号时：先检查是否有综合诊断可触发，再选取最高优先级的 1-2 条（应用动态权重后排序）
- 输出不含 ID、状态机术语——使用自然语言

## 正向反馈（沉默透明度）

当 pace-pulse 触发但无异常信号时，在以下特定节点输出轻量正向确认：

| 节点 | 条件 | 输出 |
|------|------|------|
| CR merged 后 | 本次 pulse 无异常信号 | `"✓ 节奏健康"` |
| 迭代完成 50% | current.md 完成率首次越过 50% + 无异常信号 | `"✓ 迭代过半，节奏稳定"` |
| 会话结束 | 本会话 pulse 从未触发异常信号 | 见"会话结束节奏摘要"章节 |

**规则**：
- 正向反馈每会话最多 1 次（不含会话结束摘要），避免变成噪音
- 不计入 §10 的"每会话 ≤3 条"配额
- 格式前缀：`✓`（区别于建议的 `💡` 和摘要的 `📊`）

## 严重度升级与去重优化

### 同信号去重（基础规则）
- 同一信号每会话只提醒一次（现有规则保留）

### 严重度升级（增强）
- 同一信号在后续检测中如果严重度显著加剧，允许再次提醒并标注"升级"
- **升级判定**：数值类信号当前值 ≥ 上次触发值的 2 倍（如审查积压从 3 涨到 6+）
- **输出格式**：`"⚠️ 升级：[信号描述，含更新数值]"` — 使用 `⚠️` 区别于首次的 `💡`
- 升级提醒计入 §10 配额

### 动态配额
- 短会话（< 30 分钟）：维持 ≤3 条
- 标准会话（30 分钟 - 2 小时）：≤3 条（默认）
- 长会话（> 2 小时）：≤5 条
- 会话时长从 state.md 会话开始时间或首个 checkpoint 时间推算

### 信号冷却期
- 被忽略的信号（用户未执行建议命令）设置 3 个 checkpoint 冷却期
- 冷却期内不再触发同类信号（除非严重度升级）

## 会话开始节奏检测

> 由 devpace-rules.md §1 在会话开始时调用。与推进模式脉搏检查（每 5 checkpoint）互补。

### 数据源

- `.devpace/metrics/dashboard.md` 的"最近更新"日期（**目录不存在时跳过**）
- `.devpace/backlog/` 中状态为 `in_review` 的 CR 数量
- `.devpace/iterations/current.md`（**文件不存在时跳过迭代相关检测**）

### 信号优先级（分层摘要，≤3 行）

**分层输出规则**：最高优先级信号以完整建议呈现（1 行），其余触发的信号合并为精简列表（1 行，格式：`另外注意：[信号A]、[信号B]`），总输出不超过 3 行。与 /pace-status 的关系：session-start 是"推送式"快照，/pace-status 是"拉取式"详情，两者互补。

| 优先级 | 条件 | 提醒 |
|--------|------|------|
| 1 | in_review CR > 0 | "有 N 个变更等待 review——`/pace-review`" |
| 2 | deployed 未 verified Release | "有未验证的 Release——`/pace-release verify`" |
| 3 | 迭代完成率 > 80% | "迭代接近完成——`/pace-plan next`" |
| 4 | 距 retro > 7 天 + merged CR | "已有交付成果可回顾——`/pace-retro`" |
| 5 | defect 占比 > 30% | "缺陷占比偏高，关注质量改进——`/pace-guard report`" |
| 6 | MoS 达成率 > 80% | "业务指标接近达成，建议回顾目标——`/pace-retro`" |
| 7 | 风险积压（`.devpace/risks/` 存在且 open 风险 > 3 或 High > 0） | "有 [N] 个未处理风险——`/pace-guard report`" |
| 8 | sync-mapping.md 存在 + 关联 CR 最后同步 > 24h | "有 N 个 CR 外部同步已滞后——`/pace-sync push`" |
| 9 | dashboard.md 最近更新 > 14 天 + MoS 有未勾选项 | "距上次回顾已超 2 周——`/pace-retro`" |
| 10 | Snooze 条目触发条件满足（CR 事件表/迭代变更记录） | "之前延后的变更触发条件已满足（详情见推进模式脉搏建议模板）" |
| 11 | 用户对话含运维关键词 + pace-feedback 未在本会话使用 | "检测到生产问题描述——`/pace-feedback report`" |

### 与推进模式脉搏检查的关系

| 维度 | 会话开始检测 | 推进模式脉搏 |
|------|------------|------------|
| 触发时机 | 会话开始，state.md 存在时 | 每 5 checkpoint 或 30 分钟 |
| 信号范围 | 项目级（跨 CR 的积压和进度） | CR 级（当前 CR 的健康度） |
| 输出位置 | 附加到 §1 报告 | 附加到 checkpoint 后 |
| 信号重叠 | in_review 积压、迭代完成率 | Review 积压、迭代接近尾声 |

## 会话结束节奏摘要

> 由 devpace-rules.md §6 在会话结束时调用。与推进模式脉搏检查和会话开始检测形成"开始-过程-结束"完整节奏感知。

### 触发条件

会话结束时（§6 会话结束协议），如果本会话处于推进模式（state.md 有"进行中"工作）。

### 摘要内容

输出 1-2 行会话级节奏总结，附加在 §6 会话结束总结之后：

| 维度 | 内容 | 条件 |
|------|------|------|
| 变更完成 | "完成 N 个变更（M merged, K in_review）" | 本会话有 CR 状态变化时 |
| 节奏信号 | "节奏信号：[最显著的 1 条信号摘要]" | 本会话 pulse 有触发时 |
| 迭代进度 | "迭代进度：X% → Y%（+Z%）" | iterations/current.md 存在且有进度变化时 |

### 输出规则

- 无推进模式活动时（纯探索会话）：输出 `"本次会话节奏健康，未检测到异常信号"` 或完全静默（如 .devpace/ 无任何修改）
- 有推进模式活动时：输出上表中符合条件的维度，合并为 1-2 行
- 不计入 §10 的"每会话 ≤3 条"配额（会话已结束，不影响工作流）
- 格式前缀：`📊`（区别于推进模式脉搏的 `💡`）

### 示例

```
📊 本次会话：完成 2 个变更（1 merged, 1 in_review），迭代进度 65% → 73%（+8%）
```

```
📊 本次会话节奏健康，未检测到异常信号
```

## Snooze 唤醒检测

> 由 OPT-06 引入。检测 Snooze 变更请求的触发条件是否已满足。

### 数据源

- `.devpace/backlog/` 中 CR 事件表含 Snooze 记录的条目
- `.devpace/iterations/current.md` 变更记录表中 Snooze 条目
- 触发条件类型：迭代边界（"下一迭代开始时"）、CR 完成（"X 功能完成后"）、时间（"N 天后"）

### 检测时机

| 时机 | 匹配的触发条件类型 |
|------|------------------|
| 会话开始节奏检测（§1） | 时间型（"N 天后"）、迭代型（"下一迭代开始时"——检测 current.md 是否为新迭代） |
| 新迭代创建（/pace-plan next） | 迭代型（"下一迭代开始时"） |
| CR merged（§11 连锁更新） | CR 完成型（"X 功能完成后"——检测 merged CR 关联的 PF 名称） |
| /pace-change 执行后 | 所有类型（作为兜底扫描，防止遗漏） |

### 检测流程

1. 扫描数据源中所有 Snooze 条目
2. 逐条评估触发条件是否满足
3. 已满足 → 输出非打断式提醒（格式见建议模板表）
4. **每个 Snooze 条目只提醒 1 次**——提醒后在原记录追加 `<!-- reminded: [日期] -->`
5. 未满足 → 静默跳过

### 提醒格式

```
💡 之前因「[搁置原因]」延后的变更「[变更描述]」（[CR编号]/[PF名]），触发条件「[条件描述]」已满足。需要现在评估吗？
```

**数据来源**：CR 文件事件表中的 Snooze 记录包含搁置原因（`操作` 列）和触发条件（`备注` 列），CR 头部包含关联 PF。读取时直接提取这些字段填充模板。

### 规则

- 非打断式：不暂停当前工作，仅附加在其他输出之后
- 不计入 §10 的"每会话 ≤3 条"限制——Snooze 唤醒是用户主动设置的延后，不是系统主动建议
- 无 Snooze 条目时整个检测步骤静默跳过
