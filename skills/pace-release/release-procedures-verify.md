# 发布管理执行规程——Verify

> **职责**：部署后验证流程，含自动健康检查和问题处理。
>
> **加载 common.md**：本文件与 `release-procedures-common.md` 配合使用。

## §0 速查卡片

- **自动验证**：读取 integrations/config.md 验证命令 → 结果预填到验证清单
- **手动验证**：逐项引导确认 + 可动态追加
- **健康检查**：可选持续监控（30s 间隔，连续 3 次成功/失败判定）
- **验证通过**：单环境→deployed→verified · 多环境→提示晋升或 verified

## 自动验证（如有配置）

读取 `integrations/config.md` 的"发布验证"段：

1. 执行验证命令（如 `curl -sf https://api.example.com/health`）
   - 在配置的超时时间内成功 → ✅ 预填到验证清单
   - 超时或失败 → ❌ 标记失败，提示人工确认
2. 执行额外验证命令（如有）
3. 自动验证结果展示在验证清单中

无验证命令配置时，跳过自动验证，完全手动。

## 验证清单

1. 展示 Release 的验证清单（自动验证结果预填）
2. 逐项引导确认（可批量确认）
3. 用户自定义验证项可动态追加

## 验证通过

全部验证项勾选：
- 单环境模式 → Release 状态 deployed → verified
- 多环境模式 → 如果不是最终环境 → 提示 `/pace-release deploy` 晋升到下一环境
- 多环境模式 → 最终环境 → Release 状态 deployed → verified

## 自动健康检查与回滚建议（可选）

当 `integrations/config.md` 配置了验证命令时，verify 阶段可选启用持续健康检查：

1. **触发条件**：用户在 verify 时选择"持续监控"选项（非默认）
2. **检查循环**：
   - 以配置的验证命令（如 `curl -sf URL`）为健康探针
   - 每 30 秒执行一次（间隔可通过验证超时推算）
   - 最多执行 10 次（5 分钟窗口）
3. **成功阈值**：连续 3 次成功 → 自动标记"健康检查通过" → 追加到验证清单
4. **失败阈值**：连续 3 次失败 → 弹出回滚建议（非自动执行）：
   ```
   ⚠️ 健康检查连续 3 次失败：
   - 失败时间：HH:MM - HH:MM
   - 错误信息：{最后一次失败的输出}

   建议操作：
   1. 运行 /pace-release rollback 回滚到上一版本
   2. 或手动排查问题后重新启动验证
   ```
5. **中途退出**：用户可随时中断健康检查循环，继续手动验证

**健康检查时间线**：检查结果记录到 Release 文件的"部署记录"表中：

```markdown
| 日期 | 环境 | 操作 | 结果 | 备注 |
|------|------|------|------|------|
| 2026-02-21 14:00 | production | 健康检查 | 5/5 通过 | 自动探针 30s 间隔 |
```

**规则**：
- 健康检查是 verify 的增强，不是前置——用户可完全跳过
- 仅在有验证命令配置时可用
- 不自动执行 rollback——始终保持人类确认原则
- 这是 v2.x Ops 执行层集成的先导能力

## 验证发现问题

1. 在 Release "部署后问题"表追加记录
2. 引导用户创建 defect/hotfix CR（调用 /pace-feedback report 逻辑）
3. 新创建的 defect CR 自动关联当前 Release
4. Release 状态保持 deployed（不回退到 staging）
