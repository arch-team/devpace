## 回顾报告格式

### 第一层：行动摘要（必输出）

```
## 迭代回顾摘要：[迭代名称]

📊 交付 M/N（X%）| 质量 Y% [趋势] | 周期 Z 天 [趋势]
⚠️ 关注：[最需要关注的 1-2 个问题，含数据]
✅ 亮点：[最突出的 1-2 个正面信号，含数据]
💡 建议：[下迭代最核心的 1 条改进建议]

📌 **待确认操作**（N 项）：
- [MoS 达标更新/其他建议操作]（`/pace-retro accept` 确认）

详情见下方各段...
```

**规则**：
- 摘要严格控制在 10 行以内
- 关注项选取标准：趋势恶化的指标 > 未达标的基准 > 新发现的问题
- 亮点选取标准：趋势持续改善 > 达到 Elite 级 > 首次达标
- 待确认操作为空时省略该段
- 人类用户看完摘要应能获得 80% 的关键信息

### 第二层：维度详情（按角色排序）

```
**交付**：计划 N 个产品功能，完成 M 个
**质量**：质量检查一次通过率 X%，打回率 Y%
**缺陷**：新增 N 个 defect（critical:A / major:B / minor:C），已修复 M 个，逃逸率 X%
**价值**：成效指标达成 A/B
**周期**：平均变更周期 Z 天，缺陷修复周期 W 天

**做得好的**：[从数据中识别正面趋势]
**需改进的**：[从数据中识别问题]
**缺陷分析**：[缺陷引入追溯 + 根因分类 + 预防建议]
**下个迭代建议**：[基于数据的改进建议]
```

### 条件分析段

维度详情之后，按数据可用性渐进暴露条件分析段（风险趋势、技术债务、缺陷分析、MoS、DORA、知识管理等）。详见 `retro-procedures-dimensions.md`。

## 角色适配输出

当 pace-role 设置了非 Dev 角色（或自动推断为非 Dev）时，回顾报告的摘要段按角色关注点重新排序和强调。

### 适配规则

基础数据收集不变（所有角色使用相同数据源），仅调整报告**行动摘要和维度详情的排序和措辞**：

| 角色 | 摘要首段关注 | 措辞风格 |
|------|------------|---------|
| Biz Owner | MoS 达成率 + 价值交付进展 + ROI 相关指标 | 业务术语，关联目标编号 |
| PM | 迭代完成率 + 交付效率（CR 周期）+ 范围变更次数 | 功能维度，关注节奏和交付速度 |
| Dev（默认） | 质量检查通过率 + 技术债 + 代码质量 | 技术术语，关注实现质量 |
| Tester | 缺陷分布 + 逃逸率 + 打回率 + 覆盖率趋势 | 质量维度，关注缺陷预防 |
| Ops | DORA 代理度量 + 部署频率 + MTTR + 稳定性 | 运维术语，关注交付可靠性 |

### 实现方式

1. 在 Step 3 生成报告前，读取当前角色（通过 devpace-rules.md §13 的会话上下文）
2-4. 通用适配原则见 `knowledge/role-adaptations.md`（权威源）

### 示例

**Dev 视角（默认）**——行动摘要：
```
📊 交付 3/5（60%）| 质量 85% ↑ | 周期 1.5 天 ↑
⚠️ 关注：打回率 15% ↑（上迭代 8%），2 个 major defect 未闭合
✅ 亮点：一次通过率 85% 创新高，DORA 前置时间 Elite 级
💡 建议：下迭代聚焦缺陷预防（边界条件遗漏占 60%）
```

**PM 视角**——行动摘要：
```
📊 完成 3/5 功能（60%）| 平均 CR 周期 1.5 天 ↑ | 范围变更 2 次
⚠️ 关注：迭代完成率偏低，1 个 P0 功能延期
✅ 亮点：CR 周期持续缩短（3 天→2.5→1.5），交付节奏加快
💡 建议：下迭代控制 PF 数量在 4 个以内，优先处理延期项
```

**Biz Owner 视角**——行动摘要：
```
📊 MoS 达成 2/5 | 3 个功能已上线 | OBJ-1 进度 60%
⚠️ 关注：MoS "用户注册率 > 80%" 尚无数据，需安排验证
✅ 亮点：本迭代新增 2 个 MoS 达标项，价值交付持续推进
💡 建议：下迭代优先完成 OBJ-1 剩余功能，启动 MoS 量化验证
```

## 迭代传递清单

完整回顾（Step 5）生成结构化的"迭代传递清单"，写入 `iterations/current.md` 的回顾 section，供 `/pace-plan next` 结构化消费。

### 输出格式

```markdown
## 迭代传递清单

### 继承到下个迭代的项目
- ⚠️ 改进：[问题描述]（数据支撑）→ 建议：[具体行动]
- 📊 度量基线：迭代速度 N PF/迭代、平均周期 X 天
- 🎯 未完成 PF：[PF 编号] [标题]——延期原因：[原因]

### 下个迭代建议关注
- 质量：[具体目标，如"打回率需从 15% 降至 <10%"]
- 交付：[容量建议，如"保持当前速度，不建议增加 PF 数量"]
- 风险：[需关注的风险项]
```

### 规则

- 继承项从回顾数据自动提取：趋势恶化的指标、未完成的 PF、高频缺陷模式
- 度量基线从 dashboard.md 提取核心指标当前值
- 建议关注维度从"需改进的"和"下个迭代建议"段落结构化提取
- pace-plan next 在规划时读取此 section，结构化地继承上个迭代的经验
- 不在 `mid` 模式和 `update` 模式中生成

## 报告质量自评

完整回顾（Step 6）在报告末尾输出质量自评，告诉用户报告的可信度：

```
### 报告质量

| 维度 | 评分 | 说明 |
|------|------|------|
| 数据充分度 | ⬛⬛⬛⬜⬜ 3/5 | [缺失数据说明，如"2 个 CR 缺少事件表"] |
| 趋势可信度 | ⬛⬛⬛⬛⬜ 4/5 | [趋势数据量说明，如"3 个迭代数据，方向可信"] |
| 建议可执行度 | ⬛⬛⬛⬜⬜ 3/5 | [建议质量说明，如"缺陷预防建议具体，交付建议偏泛"] |
```

### 评分规则

**数据充分度**：
- 5/5：所有 CR 有完整事件表 + 所有维度有数据
- 4/5：≥80% CR 事件完整 + 缺少 1 个条件维度
- 3/5：≥60% CR 事件完整 + 缺少 2 个条件维度
- 2/5：<60% CR 事件完整
- 1/5：仅有基础数据（状态分布），无事件详情

**趋势可信度**：
- 5/5：5+ 迭代历史数据
- 4/5：3-4 迭代历史数据
- 3/5：2 迭代历史数据
- 2/5：首次度量（无历史对比）
- 1/5：数据有明显缺失或异常

**建议可执行度**：
- 5/5：所有建议有具体行动 + 量化目标 + 责任方
- 4/5：多数建议具体可执行
- 3/5：部分建议较泛（如"加强测试"）
- 2/5：多数建议泛泛
- 1/5：数据不足以给出有意义的建议
