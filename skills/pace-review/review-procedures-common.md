# Review 通用规程

> **职责**：/pace-review 的通用规则，每次调用始终加载。其余规程按条件路由加载（见 SKILL.md Step 2 路由表）。

## §0 速查卡片

### 文件路由

| 文件 | 内容 | 加载条件 |
|------|------|---------|
| review-procedures-common.md（本文件） | 通用规则：速查+简化审批+交叉影响+S 格式+持久化+人类决策 | 始终加载 |
| review-procedures-gate.md | M+ 审查管线：意图一致性+对抗审查+累积 Diff+accept 消费+M/L-XL 格式 | M/L/XL 复杂度 |

### 核心流程速查

- **交叉影响扫描**：多活跃 CR 间的文件/模块重叠检测（仅重叠时显示）
- **摘要格式**：按复杂度分级——S 微型(3-5行)/M 标准/L-XL 完整+TL;DR
- **增量 Review**：打回后重审——读取打回原因→定向复查→未变更项标注"与上次一致"
- **人类决策处理**：approve/reject/revise/explore 四种决策的状态流转和后续动作
- **Review 摘要持久化**：写入 CR 验证证据 section，支持中断恢复和历史追溯

## 与简化审批的关系

简化审批（devpace-rules.md §2）适用于满足全部条件的简单 CR（S 复杂度 + 首次通过 + 0% 漂移），不经过本 Skill 的完整流程。

| 场景 | 处理方式 |
|------|---------|
| CR 满足简化审批条件 → 用户选"合并" | 由 /pace-dev 推进模式直接处理，不触发 /pace-review |
| CR 满足简化审批条件 → 用户选"看看" | 触发 /pace-review，使用 **S 级微型摘要**格式（已满足简化条件，不需要完整审查） |
| CR 不满足简化审批条件 | 正常走 /pace-review 完整流程，按复杂度选择摘要级别 |

## 交叉影响扫描

在开始意图一致性检查前，扫描当前项目中是否有多个活跃 CR 修改了相同文件或相关模块。

### 执行步骤

1. 列出当前所有活跃 CR（developing/verifying/in_review 状态）及其变更文件
2. 对当前待 review 的 CR 检测：
   - **文件重叠**：同一文件被多个 CR 修改
   - **模块重叠**：同一目录下的不同文件被不同 CR 修改
3. 在 Review 摘要中增加"交叉影响"section（仅检测到重叠时显示）

### 输出格式

```markdown
**交叉影响**：
  ⚠️ src/payment/retry.ts 同时被 CR-007（当前）和 CR-012（developing）修改
  ⚠️ src/auth/ 目录下 CR-007 改 login.ts，CR-015（verifying）改 session.ts
  建议：[先合并 CR-012 / 确认无语义冲突 / 关注合并顺序]
```

### 规则

- **仅在检测到重叠时显示**——无重叠时不输出此 section，不增加审批噪音
- **不阻断 Review 流程**——交叉影响是信息提示，不影响 Gate 2 通过/失败
- **简单 CR 也执行**——即使 S 复杂度，文件冲突检测仍有价值（成本低）

## 摘要格式

摘要深度按 CR 复杂度自适应分级（对齐 P2 渐进暴露 + P6 三层渐进透明）：

### S 级：微型摘要（3-5 行）

简单 CR 使用精简格式，降低审批认知负担。

```markdown
## [自然语言标题]

**变更**：[1 行变更描述] | **分支**：feature/xxx | **统计**：N 文件, +M/-K 行
**质量**：Gate 1 ✅ · Gate 2 ✅ · ⏳ 人类审批
```

### 格式选择规则

| CR 复杂度 | 摘要级别 | 特点 |
|-----------|---------|------|
| S | 微型 | 3-5 行，无意图匹配/对抗审查/累积 Diff |
| M | 标准 | 含意图匹配+对抗审查+累积 Diff，每条判断附推理后缀 |
| L/XL | 完整+TL;DR | M 级基础上前置执行摘要，用户先看核心结论再决定是否展开 |

简化审批入口的 CR（devpace-rules.md §2）触发 /pace-review 时，使用 S 级微型摘要格式。

M/L/XL 级摘要格式模板见 `review-procedures-gate.md`。

## Review 摘要持久化

Review 摘要生成后写入 CR 文件，支持中断恢复、增量 Review 和历史追溯。

### 写入步骤

1. 摘要生成完成后，将完整摘要文本写入 CR 文件的"验证证据"section
2. 格式标题：`### Review 摘要（第 N 轮，YYYY-MM-DD）`——N 为该 CR 的 review 轮次（首次为 1，每次打回后递增）
3. 与 accept 报告并列存放（不覆盖 accept 报告）
4. 多轮 review 时保留所有历史摘要（按轮次倒序排列，最新在前）

### 中断恢复

会话中断后再次触发 /pace-review 时：

1. 读取 CR 验证证据 section，检查是否有已生成的 Review 摘要
2. 如有 → 检查摘要生成后是否有新代码提交：
   - `git log --since="[摘要日期]" --oneline -- [CR 分支文件]` 为空 → 直接展示上次摘要 + 询问"上次的 Review 摘要仍然有效，是否直接审批？还是需要重新生成？"
   - 有新 commit → 提示"上次 review 后有 N 个新提交，建议重新生成摘要"，并启用增量 Review 模式
3. 如无 → 正常执行全量 Review

### 追溯价值

- 支持增量 Review：上次摘要作为 baseline，对比打回问题的修复情况
- 支持 /pace-learn：不仅有打回原因，还有完整审查发现
- 支持 /pace-trace：Review 判断作为决策轨迹的一部分

## 人类决策处理

| 用户回复 | Claude 动作 |
|---------|------------|
| "approved" / "通过" / "lgtm" | CR → approved → 执行 git merge → CR → merged → 连锁更新（project.md + state.md + iterations） |
| "打回" + 原因 | CR → developing，记录打回原因和时间到事件表，Claude 根据原因继续修改 |
| 具体修改意见 | Claude 自行修改代码 → 重新运行质量检查 → 再次提交 review（不重走完整流程，只更新摘要） |
| 探索性问题 | 暂停审批等待，进入探索模式（CR 保持 in_review），讨论结束后恢复审批流程 |

### 探索模式检测

检测信号与 devpace-rules.md 推进中探索连续模式一致：
- "让我想想""对比一下""如果…呢""还有更好方案吗""这个和之前的 B 方案比哪个好"
- 用户提问而非给出审批决策

行为：
1. CR 保持 in_review 状态不变
2. Claude 进入自由讨论，回应用户的探索问题
3. 用户回到审批话题（"好，通过""还是打回吧"）→ 恢复审批流程
4. 探索过程中如果用户决定修改 → 按"具体修改意见"流程处理
