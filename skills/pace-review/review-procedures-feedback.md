# 后决策处理规程

> **职责**：用户打回或给出修改意见后的处理流程。自包含——仅在人类决策触发后按需加载。

## 打回信息结构化

人类打回 CR 时，Claude 引导结构化记录打回信息，提升后续修复效率和 pace-learn 的 pattern 提取质量。

### 打回原因类别

| 类别 | 含义 | 典型场景 |
|------|------|---------|
| 范围 | 未满足验收条件或范围蔓延 | "验收条件 2 未实现"、"改了范围外的文件" |
| 质量 | 代码质量或测试覆盖不足 | "错误处理不充分"、"缺少边界测试" |
| 设计 | 架构或 API 设计需调整 | "接口设计不直观"、"应该用策略模式" |

### 打回记录格式

CR 事件表的打回事件应包含：
- **原因类别**：范围/质量/设计
- **具体描述**：1 句话说明问题
- **期望方向**：修复建议或期望结果

格式：`[类别]：[具体描述]，期望：[方向]`

示例：
| 日期 | 事件 | 操作者 | 备注 | 交接 |
|------|------|--------|------|------|
| 02-22 | in_review→developing | 用户 | 打回 | 质量：错误提示不清晰，期望：用户可理解的错误消息 |

### 处理规则

- 用户打回时未提供结构化原因 → Claude 主动追问："打回原因是范围/质量/设计？具体是哪个方面？"
- 追问不超过 1 次——用户回复模糊时，Claude 根据 review 上下文自行分类
- 打回信息同时记录到事件表的"交接"列，供 pace-learn 提取 pattern

## 修改意见处理规范

收到人类 review 修改意见时，按以下流程处理——不盲从、不表演性同意。

### 处理流程

1. **理解**：确认意见的真实意图（不假设字面意思 = 真实意图）
   - 如有歧义："你是希望 [解读 A] 还是 [解读 B]？"
2. **评估**：修改意见与 CR 范围/验收条件是否对齐？
   - **对齐** → 执行修改
   - **超出范围** → 记录到 CR 事件表（"Review 建议：[内容]，超出当前 CR 范围，建议作为新 CR"）→ 建议用户创建新 CR
   - **与已有决策冲突** → 引用 CR 事件表中的决策记录（"此前选择 X 而非 Y，因为 Z——是否需要调整这个决策？"）→ 等待用户确认
3. **执行 + 验证**：按修改后内容重跑受影响的质量检查 → 更新 Review 摘要

### 禁止行为

- 禁止 "You're absolutely right!" 式表演性回应——收到意见后先理解和评估，不盲目执行
- 禁止接受 YAGNI 违反的建议——如建议添加"以后可能需要"的功能，引用 CR 范围拒绝
- 禁止在未理解意图前就开始修改代码
