# M+ 审查管线规程

> **职责**：M/L/XL 复杂度 CR 的完整审查流程——意图一致性检查、对抗审查、累积 Diff 报告、accept 消费、M/L-XL 摘要格式。S 级 CR 不加载本文件。

## 意图一致性检查

当 CR 有范围和验收条件时（标准/复杂 CR），执行以下对比：

### 1. 加载意图数据

1. 读取 CR 文件的意图 section：
   - `范围`：做什么 / 不做什么
   - `验收条件`：怎样算完成
   - `约束`：边界条件、假设、限制
2. 如果意图 section 只有"用户原话"（简单 CR），跳过一致性检查

### 独立验证原则

Gate 2 执行时遵循"独立验证"原则——不信任 Gate 1 阶段的任何快照或结论，从零开始获取证据。

**执行顺序（强制）**：
1. **重新读取**验收条件（不假设 Gate 1 时的理解仍准确——上下文可能已推出窗口）
2. **独立获取** git diff（不复用 Gate 1 的快照——代码可能在 verifying 阶段有修改）
3. **先意图一致性 → 后集成质量**：意图不一致 → Gate 2 直接失败，不继续检查集成质量

**理由**：LLM 的上下文窗口有限，长会话后早期信息可能被推出。Gate 2 作为最后一道自动门禁，必须独立验证而非信任之前的记忆。

### 2. 获取实际变更

1. 读取 CR 的分支名
2. 运行 `git diff main...<branch> --stat` 获取变更统计
3. 运行 `git diff main...<branch>` 获取详细变更内容
4. 必要时读取关键变更文件理解完整上下文

### 3. 逐条对比

对每个验收条件，执行三层渐进透明输出（对齐 P6 + devpace-rules.md §5 推理后缀机制）：

**表面层**（默认输出）：
- **已满足**：实际变更覆盖该条件 → 标记 ✅ + ≤15 字推理后缀
- **未满足**：实际变更未覆盖 → 标记 ❌ + 说明缺失什么
- **部分满足**：部分覆盖 → 标记 ⚠️ + 说明哪些已做、哪些未做

推理后缀格式：`✅ 验收条件 N：[描述] — 已满足（[文件:行号] [关键实现]）`
示例：`✅ 验收条件 1：指数退避重试 — 已满足（RetryPolicy.ts:23 exponentialBackoff）`

**深入层**（用户追问"为什么"时展开）：
- 读了哪些文件 → 对比了什么 → 备选判断是什么 → 最终结论依据

### 4. 范围外改动检测

对比实际变更与"范围"字段：
- 列出不在范围内的文件或逻辑变更
- 为每项范围外改动附上原因（如：重构必要性、依赖关系）
- 判定为"合理扩展"或"范围蔓延"

### 5. 范围内遗漏检测

对比"范围"字段描述的工作与实际完成：
- 列出范围内声明要做但实际未做的项目
- 评估遗漏的严重程度

### 6. 业务追溯

为摘要的"为什么改"字段提供完整的价值链追溯：

1. 读取 CR 的`产品功能`字段 → 在 `project.md`（或 `features/PF-xxx.md`）中找到对应 PF
2. 从 PF 读取其关联的 BR → 输出链路：`BR-xxx(业务需求) → PF-yyy(功能) → CR-zzz(变更)`
3. 追溯不完整时的处理：
   - CR 无产品功能字段 → 输出"直接变更（无 PF 关联）"
   - PF 无 BR 关联 → 输出"PF-yyy → CR-zzz（BR 追溯不完整）"
   - 禁止编造不存在的关联——如实标注追溯深度

## 对抗审查（Adversarial Review）

意图一致性检查完成后，追加对抗审查——强制发现问题，对抗 Claude 的确认偏差。

### 核心原则

（对抗审查 vs 意图检查的区分和编写规则详见 `knowledge/_guides/checks-guide.md`）

**"必须找出问题"**——不是"检查是否有问题"，而是"假设存在问题，找出来"。零发现不可接受——如果 Claude 声称"完全没问题"，说明审查不够深入，需要重新分析。

### 执行步骤

1. **accept 报告联动**（如存在）：在执行对抗审查前，先读取 CR 验证证据中的 accept 报告
   - 弱覆盖/虚假覆盖的验收条件 → 标记为重点审查区域
   - Low 置信度的 L2 条目 → 对应代码投入更多维度检查
   - 将弱覆盖/低置信度区域作为"重点审查提示"聚焦对抗审查方向
   - 无 accept 报告 → 跳过此步，使用默认审查分配
2. **切换心态**：从"验证代码正确"切换到"假设代码有缺陷，找出证据"
3. **聚焦审查**：按以下维度逐一审查（每个维度至少思考 1 遍）：
   - **边界与错误路径**：空输入、极端值、并发、超时、重试等
   - **安全风险**：注入、越权、敏感数据暴露、输入未校验等
   - **性能隐患**：N+1 查询、大内存分配、无限循环风险、阻塞操作等
   - **集成风险**：与现有代码的一致性、API 契约变更、向后兼容性等
   - 如 accept 联动标记了重点区域 → 在相关维度投入更深检查（如弱覆盖的验收条件对应的代码路径）
4. **输出发现**：必须输出至少 1 项发现。每项包含：
   - **问题描述**：具体位置和内容
   - **严重程度**：`🔴 建议修复` / `🟡 建议改进` / `🟢 可选优化`
   - **假阳性声明**：每项发现后标注"⚠️ 对抗审查可能包含假阳性，请根据判断过滤"
5. **零发现兜底**：如果 4 个维度审查后仍无发现 → 输出 1 条`🟢 可选优化`建议（如命名改进、注释完善、测试补充等）

### 维度可配置化

默认使用上述 4 个固定维度。如果项目 `checks.md` 配置了自定义对抗审查维度，则替换默认维度。

```markdown
<!-- checks.md 可选配置示例 -->
## 对抗审查维度
- 数据一致性：事务边界、幂等性、并发冲突
- 可访问性：WCAG 2.1 AA 合规、屏幕阅读器兼容
- 向后兼容：API 契约、数据库 schema 迁移
```

- 未配置时使用默认 4 维度（向后兼容）
- 配置后完全替换默认维度——项目明确知道自己关注什么
- "零发现兜底"规则同样适用于自定义维度

### 与 Gate 2 的关系

- **意图一致性检查**先于对抗审查——意图不一致时 Gate 2 直接失败，不执行对抗审查
- **对抗审查不影响 Gate 2 通过/失败**——发现的问题记录到 Review 摘要供人类审批参考，不阻断流程
- 理由：对抗审查预期产生假阳性，如果让它影响 Gate 2 判定，会导致无法通过

### 简单 CR 跳过

复杂度为 S 的 CR 跳过对抗审查——变更范围小，确认式检查已足够。

## 累积 Diff 报告

Gate 3 人类审批前，生成变更到验收条件的映射报告，降低审批认知负担。

### 生成步骤

1. 执行 `git diff $(git merge-base main HEAD)..HEAD --stat` 获取 CR 分支全部累积变更统计
2. 执行 `git diff $(git merge-base main HEAD)..HEAD --name-only` 获取变更文件列表
3. 按目录/模块分组，统计每组变更行数和变更类型（新增/修改/删除）
4. **验收条件映射**：将每个变更文件/模块关联到 CR 验收条件，输出"验收条件 X → 由文件 Y 的变更实现"
5. 标注未被任何验收条件覆盖的变更文件（可能为合理扩展或范围蔓延）
6. 标注未被任何变更文件覆盖的验收条件（可能为遗漏实现）

### 输出格式

```
**累积变更报告**：
  模块 A（+N/-M 行）：
    - file1.ts（新增）→ 验收条件 1
    - file2.ts（修改）→ 验收条件 2
  模块 B（+N/-M 行）：
    - file3.ts（修改）→ 验收条件 1, 3
  ⚠️ 未覆盖的验收条件：[列表]
  ⚠️ 范围外变更：[文件列表 + 原因]
```

### 规则

- **简单 CR 跳过**：复杂度为 S 的 CR 不生成累积 Diff 报告（变更文件少，意图一致性检查已足够）
- **与 Gate 2 互补**：Gate 2 逐条验收比对检查"做没做到"，累积 Diff 报告展示"怎么做到的"——两者互补
- **嵌入摘要**：报告作为 Review 摘要的一部分呈现给人类审批者，不独立输出

## 语义一致性评分

Review 摘要中自动计算代码变更与验收标准的语义对齐度。

### 计算步骤

1. 读取 CR 关联 PF 的验收标准列表
2. 对每个验收标准，评估当前 CR 的代码变更是否有直接实现：
   - 有对应代码变更（函数/文件/模块可关联到标准描述）→ 计为"已覆盖"
   - 代码变更方向与标准描述不一致 → 标记为"偏离"
3. 计算一致性评分：已覆盖 / 总标准数

### 评分分级

| 比例 | 等级 | 说明 |
|------|------|------|
| ≥ 80% 已覆盖且无偏离 | 🟢 高 | 代码实现与需求高度对齐 |
| 50-79% 已覆盖或有少量偏离 | 🟡 中 | 部分对齐，需关注未覆盖项 |
| < 50% 已覆盖或有多项偏离 | 🔴 低 | 可能存在需求理解偏差 |

### 规则

- **S CR 简化**：仅检查意图 section 的验收条件（通常 1-3 条），不需要 PF 级标准
- **无 PF 关联时跳过**：不显示此段
- **与 Gate 2 意图匹配互补**：意图匹配检查"做没做到"，语义一致性量化"做对了多少"
- **偏离项突出**：有偏离时在摘要中列出具体标准和实际实现方向

## accept 报告结构化消费

当 CR 验证证据 section 包含 `/pace-test accept` 的验收验证报告时，Review 摘要自动提取关键信号。

**数据契约**：accept 报告的格式定义见 `knowledge/_schema/auxiliary/accept-report-contract.md`（生产方和消费方的共享接口）。

### 提取步骤

按 `accept-report-contract.md` "消费方提取规则"执行：

1. 读取 CR 文件的"验证证据"section
2. 如果包含"验收验证报告"标题 → 按契约定义的提取项提取汇总数据
3. 如果"验证证据"不包含 accept 报告 → 跳过此 section（不显示、不提示）

## M 级：标准摘要格式

中等复杂度 CR 使用标准格式，包含意图匹配和对抗审查。

```markdown
## [自然语言标题]

**改了什么**：[变更文件和内容描述，用自然语言]
**为什么改**：[关联的产品功能 → 业务需求追溯]
**影响范围**：[是否影响其他组件，模块依赖分析]

**意图匹配**：
  ✅ 验收条件 1：[描述] — 已满足（[≤15 字证据后缀]）
  ✅ 验收条件 2：[描述] — 已满足（[≤15 字证据后缀]）
  ⚠️ 范围外改动：[描述] — [原因/合理性]
  ❌ 范围内遗漏：[描述] — [缺失内容]

**AI 验收验证**：（仅当 CR 有 /pace-test accept 验证证据时显示）
  通过率：N/M | 未通过：[列表] | 验证级别分布：L1 动态 X · L2 静态 Y · L3 需人工 Z
  预言审查：[有效覆盖 A · 弱覆盖 B · 虚假覆盖 C]（仅有 test-strategy.md 时显示）
  置信度分布：🟢High P · 🟡Medium Q · 🟠Low R（L2 条目，仅有置信度数据时显示）
  自审计：[M 条命中合理化模式（已修正）]（仅有自审计数据时显示）

**对抗审查**：
  🔍 发现 N 项：
  🔴 [问题 1 描述]
  🟡 [问题 2 描述]
  ⚠️ 对抗审查可能包含假阳性，请根据判断过滤

**累积变更报告**：
  [模块分组 + 验收条件映射，格式见上方]

**语义一致性**：[🟢高/🟡中/🔴低] N/M 验收标准有直接代码实现 · 偏离项：[列表或"无"]

**质量检查状态**：
  ✅ [已通过的自动质量检查项]
  ⏳ 人类审批

**分支**：feature/xxx
**变更统计**：N 个文件, +M/-K 行
```
