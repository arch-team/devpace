# Review 摘要生成规程

> **职责**：/pace-review 的详细操作规程。SKILL.md 定义"做什么"，本文件定义"怎么做"。

## 意图一致性检查

当 CR 有范围和验收条件时（标准/复杂 CR），执行以下对比：

### 1. 加载意图数据

1. 读取 CR 文件的意图 section：
   - `范围`：做什么 / 不做什么
   - `验收条件`：怎样算完成
   - `约束`：边界条件、假设、限制
2. 如果意图 section 只有"用户原话"（简单 CR），跳过一致性检查

### 独立验证原则

Gate 2 执行时遵循"独立验证"原则——不信任 Gate 1 阶段的任何快照或结论，从零开始获取证据。

**执行顺序（强制）**：
1. **重新读取**验收条件（不假设 Gate 1 时的理解仍准确——上下文可能已推出窗口）
2. **独立获取** git diff（不复用 Gate 1 的快照——代码可能在 verifying 阶段有修改）
3. **先意图一致性 → 后集成质量**：意图不一致 → Gate 2 直接失败，不继续检查集成质量

**理由**：LLM 的上下文窗口有限，长会话后早期信息可能被推出。Gate 2 作为最后一道自动门禁，必须独立验证而非信任之前的记忆。

### 2. 获取实际变更

1. 读取 CR 的分支名
2. 运行 `git diff main...<branch> --stat` 获取变更统计
3. 运行 `git diff main...<branch>` 获取详细变更内容
4. 必要时读取关键变更文件理解完整上下文

### 3. 逐条对比

对每个验收条件：
- **已满足**：实际变更覆盖该条件 → 标记 ✅
- **未满足**：实际变更未覆盖 → 标记 ❌ + 说明缺失什么
- **部分满足**：部分覆盖 → 标记 ⚠️ + 说明哪些已做、哪些未做

### 4. 范围外改动检测

对比实际变更与"范围"字段：
- 列出不在范围内的文件或逻辑变更
- 为每项范围外改动附上原因（如：重构必要性、依赖关系）
- 判定为"合理扩展"或"范围蔓延"

### 5. 范围内遗漏检测

对比"范围"字段描述的工作与实际完成：
- 列出范围内声明要做但实际未做的项目
- 评估遗漏的严重程度

## 对抗审查（Adversarial Review）

意图一致性检查完成后，追加对抗审查——强制发现问题，对抗 Claude 的确认偏差。

### 核心原则

**"必须找出问题"**——不是"检查是否有问题"，而是"假设存在问题，找出来"。零发现不可接受——如果 Claude 声称"完全没问题"，说明审查不够深入，需要重新分析。

### 执行步骤

1. **切换心态**：从"验证代码正确"切换到"假设代码有缺陷，找出证据"
2. **聚焦审查**：按以下维度逐一审查（每个维度至少思考 1 遍）：
   - **边界与错误路径**：空输入、极端值、并发、超时、重试等
   - **安全风险**：注入、越权、敏感数据暴露、输入未校验等
   - **性能隐患**：N+1 查询、大内存分配、无限循环风险、阻塞操作等
   - **集成风险**：与现有代码的一致性、API 契约变更、向后兼容性等
3. **输出发现**：必须输出至少 1 项发现。每项包含：
   - **问题描述**：具体位置和内容
   - **严重程度**：`🔴 建议修复` / `🟡 建议改进` / `🟢 可选优化`
   - **假阳性声明**：每项发现后标注"⚠️ 对抗审查可能包含假阳性，请根据判断过滤"
4. **零发现兜底**：如果 4 个维度审查后仍无发现 → 输出 1 条`🟢 可选优化`建议（如命名改进、注释完善、测试补充等）

### 与 Gate 2 的关系

- **意图一致性检查**先于对抗审查——意图不一致时 Gate 2 直接失败，不执行对抗审查
- **对抗审查不影响 Gate 2 通过/失败**——发现的问题记录到 Review 摘要供人类审批参考，不阻断流程
- 理由：对抗审查预期产生假阳性，如果让它影响 Gate 2 判定，会导致无法通过

### 简单 CR 跳过

复杂度为 S 的 CR 跳过对抗审查——变更范围小，确认式检查已足够。

## 累积 Diff 报告

Gate 3 人类审批前，生成变更到验收条件的映射报告，降低审批认知负担。

### 生成步骤

1. 执行 `git diff $(git merge-base main HEAD)..HEAD --stat` 获取 CR 分支全部累积变更统计
2. 执行 `git diff $(git merge-base main HEAD)..HEAD --name-only` 获取变更文件列表
3. 按目录/模块分组，统计每组变更行数和变更类型（新增/修改/删除）
4. **验收条件映射**：将每个变更文件/模块关联到 CR 验收条件，输出"验收条件 X → 由文件 Y 的变更实现"
5. 标注未被任何验收条件覆盖的变更文件（可能为合理扩展或范围蔓延）
6. 标注未被任何变更文件覆盖的验收条件（可能为遗漏实现）

### 输出格式

```
**累积变更报告**：
  模块 A（+N/-M 行）：
    - file1.ts（新增）→ 验收条件 1
    - file2.ts（修改）→ 验收条件 2
  模块 B（+N/-M 行）：
    - file3.ts（修改）→ 验收条件 1, 3
  ⚠️ 未覆盖的验收条件：[列表]
  ⚠️ 范围外变更：[文件列表 + 原因]
```

### 规则

- **简单 CR 跳过**：复杂度为 S 的 CR 不生成累积 Diff 报告（变更文件少，意图一致性检查已足够）
- **与 Gate 2 互补**：Gate 2 逐条验收比对检查"做没做到"，累积 Diff 报告展示"怎么做到的"——两者互补
- **嵌入摘要**：报告作为 Review 摘要的一部分呈现给人类审批者，不独立输出

## 摘要格式

```markdown
## [自然语言标题]

**改了什么**：[变更文件和内容描述，用自然语言]
**为什么改**：[关联的产品功能 → 业务需求追溯]
**影响范围**：[是否影响其他组件，模块依赖分析]

**意图匹配**：（仅标准/复杂 CR 显示此 section）
  ✅ 验收条件 1：[描述] — 已满足
  ✅ 验收条件 2：[描述] — 已满足
  ⚠️ 范围外改动：[描述] — [原因/合理性]
  ❌ 范围内遗漏：[描述] — [缺失内容]

**对抗审查**：（仅 M/L/XL CR 显示此 section）
  🔍 发现 N 项：
  🔴 [问题 1 描述]
  🟡 [问题 2 描述]
  ⚠️ 对抗审查可能包含假阳性，请根据判断过滤

**累积变更报告**：（仅 M/L/XL CR 显示此 section）
  [模块分组 + 验收条件映射，格式见上方]

**质量检查状态**：
  ✅ [已通过的自动质量检查项]
  ⏳ 人类审批

**分支**：feature/xxx
**变更统计**：N 个文件, +M/-K 行
```

## 人类决策处理

| 用户回复 | Claude 动作 |
|---------|------------|
| "approved" / "通过" / "lgtm" | CR → approved → 执行 git merge → CR → merged → 连锁更新（project.md + state.md + iterations） |
| "打回" + 原因 | CR → developing，记录打回原因和时间到事件表，Claude 根据原因继续修改 |
| 具体修改意见 | Claude 自行修改代码 → 重新运行质量检查 → 再次提交 review（不重走完整流程，只更新摘要） |

## 打回信息结构化

人类打回 CR 时，Claude 引导结构化记录打回信息，提升后续修复效率和 pace-learn 的 pattern 提取质量。

### 打回原因类别

| 类别 | 含义 | 典型场景 |
|------|------|---------|
| 范围 | 未满足验收条件或范围蔓延 | "验收条件 2 未实现"、"改了范围外的文件" |
| 质量 | 代码质量或测试覆盖不足 | "错误处理不充分"、"缺少边界测试" |
| 设计 | 架构或 API 设计需调整 | "接口设计不直观"、"应该用策略模式" |

### 打回记录格式

CR 事件表的打回事件应包含：
- **原因类别**：范围/质量/设计
- **具体描述**：1 句话说明问题
- **期望方向**：修复建议或期望结果

格式：`[类别]：[具体描述]，期望：[方向]`

示例：
| 日期 | 事件 | 操作者 | 备注 | 交接 |
|------|------|--------|------|------|
| 02-22 | in_review→developing | 用户 | 打回 | 质量：错误提示不清晰，期望：用户可理解的错误消息 |

### 处理规则

- 用户打回时未提供结构化原因 → Claude 主动追问："打回原因是范围/质量/设计？具体是哪个方面？"
- 追问不超过 1 次——用户回复模糊时，Claude 根据 review 上下文自行分类
- 打回信息同时记录到事件表的"交接"列，供 pace-learn 提取 pattern

## 修改意见处理规范

收到人类 review 修改意见时，按以下流程处理——不盲从、不表演性同意。

### 处理流程

1. **理解**：确认意见的真实意图（不假设字面意思 = 真实意图）
   - 如有歧义："你是希望 [解读 A] 还是 [解读 B]？"
2. **评估**：修改意见与 CR 范围/验收条件是否对齐？
   - **对齐** → 执行修改
   - **超出范围** → 记录到 CR 事件表（"Review 建议：[内容]，超出当前 CR 范围，建议作为新 CR"）→ 建议用户创建新 CR
   - **与已有决策冲突** → 引用 CR 事件表中的决策记录（"此前选择 X 而非 Y，因为 Z——是否需要调整这个决策？"）→ 等待用户确认
3. **执行 + 验证**：按修改后内容重跑受影响的质量检查 → 更新 Review 摘要

### 禁止行为

- 禁止 "You're absolutely right!" 式表演性回应——收到意见后先理解和评估，不盲目执行
- 禁止接受 YAGNI 违反的建议——如建议添加"以后可能需要"的功能，引用 CR 范围拒绝
- 禁止在未理解意图前就开始修改代码
