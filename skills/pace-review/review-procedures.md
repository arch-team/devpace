# Review 摘要生成规程

> **职责**：/pace-review 的详细操作规程。SKILL.md 定义"做什么"，本文件定义"怎么做"。

## §0 速查卡片

- **交叉影响扫描**：多活跃 CR 间的文件/模块重叠检测（仅重叠时显示）
- **意图一致性检查**：验收条件 vs 实际变更的逐条比对，含独立验证+推理后缀
- **对抗审查**：站在"刁钻 Reviewer"视角审查——安全/性能/边界/可维护性
- **累积 Diff 报告**：多 CR 合并视角的变更汇总，含冲突检测
- **摘要格式**：按复杂度分级——S 微型(3-5行)/M 标准/L-XL 完整+TL;DR
- **增量 Review**：打回后重审——读取打回原因→定向复查→未变更项标注"与上次一致"
- **人类决策处理**：approve/reject/revise/explore 四种决策的状态流转和后续动作
- **修改意见处理规范**：结构化记录打回原因，防止信息丢失
- **Review 摘要持久化**：写入 CR 验证证据 section，支持中断恢复和历史追溯

## 与简化审批的关系

简化审批（devpace-rules.md §2）适用于满足全部条件的简单 CR（S 复杂度 + 首次通过 + 0% 漂移），不经过本 Skill 的完整流程。

| 场景 | 处理方式 |
|------|---------|
| CR 满足简化审批条件 → 用户选"合并" | 由 /pace-dev 推进模式直接处理，不触发 /pace-review |
| CR 满足简化审批条件 → 用户选"看看" | 触发 /pace-review，使用 **S 级微型摘要**格式（已满足简化条件，不需要完整审查） |
| CR 不满足简化审批条件 | 正常走 /pace-review 完整流程，按复杂度选择摘要级别 |

## 交叉影响扫描

在开始意图一致性检查前，扫描当前项目中是否有多个活跃 CR 修改了相同文件或相关模块。

### 执行步骤

1. 列出当前所有活跃 CR（developing/verifying/in_review 状态）及其变更文件
2. 对当前待 review 的 CR 检测：
   - **文件重叠**：同一文件被多个 CR 修改
   - **模块重叠**：同一目录下的不同文件被不同 CR 修改
3. 在 Review 摘要中增加"交叉影响"section（仅检测到重叠时显示）

### 输出格式

```markdown
**交叉影响**：
  ⚠️ src/payment/retry.ts 同时被 CR-007（当前）和 CR-012（developing）修改
  ⚠️ src/auth/ 目录下 CR-007 改 login.ts，CR-015（verifying）改 session.ts
  建议：[先合并 CR-012 / 确认无语义冲突 / 关注合并顺序]
```

### 规则

- **仅在检测到重叠时显示**——无重叠时不输出此 section，不增加审批噪音
- **不阻断 Review 流程**——交叉影响是信息提示，不影响 Gate 2 通过/失败
- **简单 CR 也执行**——即使 S 复杂度，文件冲突检测仍有价值（成本低）

## 意图一致性检查

当 CR 有范围和验收条件时（标准/复杂 CR），执行以下对比：

### 1. 加载意图数据

1. 读取 CR 文件的意图 section：
   - `范围`：做什么 / 不做什么
   - `验收条件`：怎样算完成
   - `约束`：边界条件、假设、限制
2. 如果意图 section 只有"用户原话"（简单 CR），跳过一致性检查

### 独立验证原则

Gate 2 执行时遵循"独立验证"原则——不信任 Gate 1 阶段的任何快照或结论，从零开始获取证据。

**执行顺序（强制）**：
1. **重新读取**验收条件（不假设 Gate 1 时的理解仍准确——上下文可能已推出窗口）
2. **独立获取** git diff（不复用 Gate 1 的快照——代码可能在 verifying 阶段有修改）
3. **先意图一致性 → 后集成质量**：意图不一致 → Gate 2 直接失败，不继续检查集成质量

**理由**：LLM 的上下文窗口有限，长会话后早期信息可能被推出。Gate 2 作为最后一道自动门禁，必须独立验证而非信任之前的记忆。

### 2. 获取实际变更

1. 读取 CR 的分支名
2. 运行 `git diff main...<branch> --stat` 获取变更统计
3. 运行 `git diff main...<branch>` 获取详细变更内容
4. 必要时读取关键变更文件理解完整上下文

### 3. 逐条对比

对每个验收条件，执行三层渐进透明输出（对齐 P6 + devpace-rules.md §5 推理后缀机制）：

**表面层**（默认输出）：
- **已满足**：实际变更覆盖该条件 → 标记 ✅ + ≤15 字推理后缀
- **未满足**：实际变更未覆盖 → 标记 ❌ + 说明缺失什么
- **部分满足**：部分覆盖 → 标记 ⚠️ + 说明哪些已做、哪些未做

推理后缀格式：`✅ 验收条件 N：[描述] — 已满足（[文件:行号] [关键实现]）`
示例：`✅ 验收条件 1：指数退避重试 — 已满足（RetryPolicy.ts:23 exponentialBackoff）`

**深入层**（用户追问"为什么"时展开）：
- 读了哪些文件 → 对比了什么 → 备选判断是什么 → 最终结论依据

### 4. 范围外改动检测

对比实际变更与"范围"字段：
- 列出不在范围内的文件或逻辑变更
- 为每项范围外改动附上原因（如：重构必要性、依赖关系）
- 判定为"合理扩展"或"范围蔓延"

### 5. 范围内遗漏检测

对比"范围"字段描述的工作与实际完成：
- 列出范围内声明要做但实际未做的项目
- 评估遗漏的严重程度

### 6. 业务追溯

为摘要的"为什么改"字段提供完整的价值链追溯：

1. 读取 CR 的`产品功能`字段 → 在 `project.md`（或 `features/PF-xxx.md`）中找到对应 PF
2. 从 PF 读取其关联的 BR → 输出链路：`BR-xxx(业务需求) → PF-yyy(功能) → CR-zzz(变更)`
3. 追溯不完整时的处理：
   - CR 无产品功能字段 → 输出"直接变更（无 PF 关联）"
   - PF 无 BR 关联 → 输出"PF-yyy → CR-zzz（BR 追溯不完整）"
   - 禁止编造不存在的关联——如实标注追溯深度

## 对抗审查（Adversarial Review）

意图一致性检查完成后，追加对抗审查——强制发现问题，对抗 Claude 的确认偏差。

### 核心原则

**"必须找出问题"**——不是"检查是否有问题"，而是"假设存在问题，找出来"。零发现不可接受——如果 Claude 声称"完全没问题"，说明审查不够深入，需要重新分析。

### 执行步骤

1. **accept 报告联动**（如存在）：在执行对抗审查前，先读取 CR 验证证据中的 accept 报告
   - 弱覆盖/虚假覆盖的验收条件 → 标记为重点审查区域
   - Low 置信度的 L2 条目 → 对应代码投入更多维度检查
   - 将弱覆盖/低置信度区域作为"重点审查提示"聚焦对抗审查方向
   - 无 accept 报告 → 跳过此步，使用默认审查分配
2. **切换心态**：从"验证代码正确"切换到"假设代码有缺陷，找出证据"
3. **聚焦审查**：按以下维度逐一审查（每个维度至少思考 1 遍）：
   - **边界与错误路径**：空输入、极端值、并发、超时、重试等
   - **安全风险**：注入、越权、敏感数据暴露、输入未校验等
   - **性能隐患**：N+1 查询、大内存分配、无限循环风险、阻塞操作等
   - **集成风险**：与现有代码的一致性、API 契约变更、向后兼容性等
   - 如 accept 联动标记了重点区域 → 在相关维度投入更深检查（如弱覆盖的验收条件对应的代码路径）
4. **输出发现**：必须输出至少 1 项发现。每项包含：
   - **问题描述**：具体位置和内容
   - **严重程度**：`🔴 建议修复` / `🟡 建议改进` / `🟢 可选优化`
   - **假阳性声明**：每项发现后标注"⚠️ 对抗审查可能包含假阳性，请根据判断过滤"
5. **零发现兜底**：如果 4 个维度审查后仍无发现 → 输出 1 条`🟢 可选优化`建议（如命名改进、注释完善、测试补充等）

### 维度可配置化

默认使用上述 4 个固定维度。如果项目 `checks.md` 配置了自定义对抗审查维度，则替换默认维度。

```markdown
<!-- checks.md 可选配置示例 -->
## 对抗审查维度
- 数据一致性：事务边界、幂等性、并发冲突
- 可访问性：WCAG 2.1 AA 合规、屏幕阅读器兼容
- 向后兼容：API 契约、数据库 schema 迁移
```

- 未配置时使用默认 4 维度（向后兼容）
- 配置后完全替换默认维度——项目明确知道自己关注什么
- "零发现兜底"规则同样适用于自定义维度

### 与 Gate 2 的关系

- **意图一致性检查**先于对抗审查——意图不一致时 Gate 2 直接失败，不执行对抗审查
- **对抗审查不影响 Gate 2 通过/失败**——发现的问题记录到 Review 摘要供人类审批参考，不阻断流程
- 理由：对抗审查预期产生假阳性，如果让它影响 Gate 2 判定，会导致无法通过

### 简单 CR 跳过

复杂度为 S 的 CR 跳过对抗审查——变更范围小，确认式检查已足够。

## 累积 Diff 报告

Gate 3 人类审批前，生成变更到验收条件的映射报告，降低审批认知负担。

### 生成步骤

1. 执行 `git diff $(git merge-base main HEAD)..HEAD --stat` 获取 CR 分支全部累积变更统计
2. 执行 `git diff $(git merge-base main HEAD)..HEAD --name-only` 获取变更文件列表
3. 按目录/模块分组，统计每组变更行数和变更类型（新增/修改/删除）
4. **验收条件映射**：将每个变更文件/模块关联到 CR 验收条件，输出"验收条件 X → 由文件 Y 的变更实现"
5. 标注未被任何验收条件覆盖的变更文件（可能为合理扩展或范围蔓延）
6. 标注未被任何变更文件覆盖的验收条件（可能为遗漏实现）

### 输出格式

```
**累积变更报告**：
  模块 A（+N/-M 行）：
    - file1.ts（新增）→ 验收条件 1
    - file2.ts（修改）→ 验收条件 2
  模块 B（+N/-M 行）：
    - file3.ts（修改）→ 验收条件 1, 3
  ⚠️ 未覆盖的验收条件：[列表]
  ⚠️ 范围外变更：[文件列表 + 原因]
```

### 规则

- **简单 CR 跳过**：复杂度为 S 的 CR 不生成累积 Diff 报告（变更文件少，意图一致性检查已足够）
- **与 Gate 2 互补**：Gate 2 逐条验收比对检查"做没做到"，累积 Diff 报告展示"怎么做到的"——两者互补
- **嵌入摘要**：报告作为 Review 摘要的一部分呈现给人类审批者，不独立输出

## accept 报告结构化消费

当 CR 验证证据 section 包含 `/pace-test accept` 的验收验证报告时，Review 摘要自动提取关键信号。

### 提取步骤

1. 读取 CR 文件的"验证证据"section
2. 如果包含"验收验证报告"标题 → 提取以下数据：
   - **逐条验证结果表**：统计通过/未通过/需人工数量 + 验证级别分布
   - **测试预言审查表**（如存在）：统计有效覆盖/弱覆盖/虚假覆盖数量
   - **合理化自审计**（如存在）：提取命中合理化模式的数量
   - **置信度数据**（如存在）：统计 L2 条目的 High/Medium/Low 分布
3. 如果"验证证据"不包含 accept 报告 → 跳过此 section（不显示、不提示）

### 消费原则

- **紧凑优先**：Review 摘要中仅显示汇总数据（1-3 行），不复制完整 accept 报告
- **风险聚焦**：如有未通过项或弱覆盖 → 在"风险点"段追加相关信息
- **不重复验证**：Review 消费已有的 accept 报告，不重新执行验收验证

## 增量 Review（Delta Review）

当 CR 此前有过 review 记录时（事件表中有 in_review→developing 的历史），启用增量 Review 模式，避免全量重做造成的 token 和时间浪费。

### 触发条件

- CR 事件表中有近期 in_review→developing 的打回记录
- 打回记录包含结构化打回信息（类别+描述+期望方向）

### 增量流程

1. **读取上次打回原因**：从 CR 事件表"交接"列提取打回类别和描述
2. **定向复查**：只对打回原因涉及的维度做检查
   - 打回类别"范围" → 重新执行意图一致性检查（范围外改动 + 范围内遗漏）
   - 打回类别"质量" → 重新执行对抗审查的相关维度
   - 打回类别"设计" → 重新检查涉及的文件和 API 变更
3. **未变更维度标注**：对未涉及打回原因的验收条件，标注"与上次一致，未重新检查"
4. **读取上次 Review 摘要**：从 CR 验证证据 section 读取上次摘要作为 baseline
5. **输出增量摘要**：重点突出"上次问题的修复情况"

### 增量摘要格式

```markdown
## [自然语言标题]（增量 Review · 第 N 轮）

**上次打回原因**：[类别]：[描述]
**修复情况**：
  ✅ [已修复的问题] — [修复方式简述]
  ❌ [仍未修复的问题] — [仍存在的原因]

**定向复查**：
  [仅打回涉及维度的检查结果]

**未变更项**：以下验收条件与上次一致，未重新检查
  ✅ 验收条件 N：[描述]

**质量检查状态**：
  ✅ [已通过的自动质量检查项]
  ⏳ 人类审批

**分支**：feature/xxx
**变更统计**（自上次 review 后）：N 个文件, +M/-K 行
```

### 规则

- **全量兜底**：用户可说"完整 review"或"重新审查"强制执行全量 Review
- **无打回记录退化**：事件表有 in_review→developing 但无结构化打回信息 → 退化为全量 Review
- **代码无变更**：上次 review 后 `git diff` 为空 → 直接复用上次摘要，提示"代码未变更，是否确认通过？"

## 摘要格式

摘要深度按 CR 复杂度自适应分级（对齐 P2 渐进暴露 + P6 三层渐进透明）：

### S 级：微型摘要（3-5 行）

简单 CR 使用精简格式，降低审批认知负担。

```markdown
## [自然语言标题]

**变更**：[1 行变更描述] | **分支**：feature/xxx | **统计**：N 文件, +M/-K 行
**质量**：Gate 1 ✅ · Gate 2 ✅ · ⏳ 人类审批
```

### M 级：标准摘要

中等复杂度 CR 使用标准格式，包含意图匹配和对抗审查。

```markdown
## [自然语言标题]

**改了什么**：[变更文件和内容描述，用自然语言]
**为什么改**：[关联的产品功能 → 业务需求追溯]
**影响范围**：[是否影响其他组件，模块依赖分析]

**意图匹配**：
  ✅ 验收条件 1：[描述] — 已满足（[≤15 字证据后缀]）
  ✅ 验收条件 2：[描述] — 已满足（[≤15 字证据后缀]）
  ⚠️ 范围外改动：[描述] — [原因/合理性]
  ❌ 范围内遗漏：[描述] — [缺失内容]

**AI 验收验证**：（仅当 CR 有 /pace-test accept 验证证据时显示）
  通过率：N/M | 未通过：[列表] | 验证级别分布：L1 动态 X · L2 静态 Y · L3 需人工 Z
  预言审查：[有效覆盖 A · 弱覆盖 B · 虚假覆盖 C]（仅有 test-strategy.md 时显示）
  置信度分布：🟢High P · 🟡Medium Q · 🟠Low R（L2 条目，仅有置信度数据时显示）
  自审计：[M 条命中合理化模式（已修正）]（仅有自审计数据时显示）

**对抗审查**：
  🔍 发现 N 项：
  🔴 [问题 1 描述]
  🟡 [问题 2 描述]
  ⚠️ 对抗审查可能包含假阳性，请根据判断过滤

**累积变更报告**：
  [模块分组 + 验收条件映射，格式见上方]

**质量检查状态**：
  ✅ [已通过的自动质量检查项]
  ⏳ 人类审批

**分支**：feature/xxx
**变更统计**：N 个文件, +M/-K 行
```

### L/XL 级：完整摘要 + TL;DR

大型 CR 前置 2-3 行执行摘要，用户可按需展开细节。

```markdown
## [自然语言标题]

> **TL;DR**：[2-3 行核心结论——改了什么核心逻辑、关键风险点、审批建议（通过/需关注/建议打回）]

**改了什么**：[变更文件和内容描述，用自然语言]
**为什么改**：[关联的产品功能 → 业务需求追溯]
**影响范围**：[是否影响其他组件，模块依赖分析]

**意图匹配**：
  ✅ 验收条件 1：[描述] — 已满足（[≤15 字证据后缀]）
  ✅ 验收条件 2：[描述] — 已满足（[≤15 字证据后缀]）
  ⚠️ 范围外改动：[描述] — [原因/合理性]
  ❌ 范围内遗漏：[描述] — [缺失内容]

**AI 验收验证**：（仅当 CR 有 /pace-test accept 验证证据时显示）
  通过率：N/M | 未通过：[列表] | 验证级别分布：L1 动态 X · L2 静态 Y · L3 需人工 Z
  预言审查：[有效覆盖 A · 弱覆盖 B · 虚假覆盖 C]（仅有 test-strategy.md 时显示）
  置信度分布：🟢High P · 🟡Medium Q · 🟠Low R（L2 条目，仅有置信度数据时显示）
  自审计：[M 条命中合理化模式（已修正）]（仅有自审计数据时显示）

**对抗审查**：
  🔍 发现 N 项：
  🔴 [问题 1 描述]
  🟡 [问题 2 描述]
  ⚠️ 对抗审查可能包含假阳性，请根据判断过滤

**累积变更报告**：
  [模块分组 + 验收条件映射，格式见上方]

**质量检查状态**：
  ✅ [已通过的自动质量检查项]
  ⏳ 人类审批

**分支**：feature/xxx
**变更统计**：N 个文件, +M/-K 行
```

### 格式选择规则

| CR 复杂度 | 摘要级别 | 特点 |
|-----------|---------|------|
| S | 微型 | 3-5 行，无意图匹配/对抗审查/累积 Diff |
| M | 标准 | 含意图匹配+对抗审查+累积 Diff，每条判断附推理后缀 |
| L/XL | 完整+TL;DR | M 级基础上前置执行摘要，用户先看核心结论再决定是否展开 |

简化审批入口的 CR（devpace-rules.md §2）触发 /pace-review 时，使用 S 级微型摘要格式。

## Review 摘要持久化

Review 摘要生成后写入 CR 文件，支持中断恢复、增量 Review 和历史追溯。

### 写入步骤

1. 摘要生成完成后，将完整摘要文本写入 CR 文件的"验证证据"section
2. 格式标题：`### Review 摘要（第 N 轮，YYYY-MM-DD）`——N 为该 CR 的 review 轮次（首次为 1，每次打回后递增）
3. 与 accept 报告并列存放（不覆盖 accept 报告）
4. 多轮 review 时保留所有历史摘要（按轮次倒序排列，最新在前）

### 中断恢复

会话中断后再次触发 /pace-review 时：

1. 读取 CR 验证证据 section，检查是否有已生成的 Review 摘要
2. 如有 → 检查摘要生成后是否有新代码提交：
   - `git log --since="[摘要日期]" --oneline -- [CR 分支文件]` 为空 → 直接展示上次摘要 + 询问"上次的 Review 摘要仍然有效，是否直接审批？还是需要重新生成？"
   - 有新 commit → 提示"上次 review 后有 N 个新提交，建议重新生成摘要"，并启用增量 Review 模式
3. 如无 → 正常执行全量 Review

### 追溯价值

- 支持增量 Review：上次摘要作为 baseline，对比打回问题的修复情况
- 支持 /pace-learn：不仅有打回原因，还有完整审查发现
- 支持 /pace-trace：Review 判断作为决策轨迹的一部分

## 人类决策处理

| 用户回复 | Claude 动作 |
|---------|------------|
| "approved" / "通过" / "lgtm" | CR → approved → 执行 git merge → CR → merged → 连锁更新（project.md + state.md + iterations） |
| "打回" + 原因 | CR → developing，记录打回原因和时间到事件表，Claude 根据原因继续修改 |
| 具体修改意见 | Claude 自行修改代码 → 重新运行质量检查 → 再次提交 review（不重走完整流程，只更新摘要） |
| 探索性问题 | 暂停审批等待，进入探索模式（CR 保持 in_review），讨论结束后恢复审批流程 |

### 探索模式检测

检测信号与 devpace-rules.md 推进中探索连续模式一致：
- "让我想想""对比一下""如果…呢""还有更好方案吗""这个和之前的 B 方案比哪个好"
- 用户提问而非给出审批决策

行为：
1. CR 保持 in_review 状态不变
2. Claude 进入自由讨论，回应用户的探索问题
3. 用户回到审批话题（"好，通过""还是打回吧"）→ 恢复审批流程
4. 探索过程中如果用户决定修改 → 按"具体修改意见"流程处理

## 打回信息结构化

人类打回 CR 时，Claude 引导结构化记录打回信息，提升后续修复效率和 pace-learn 的 pattern 提取质量。

### 打回原因类别

| 类别 | 含义 | 典型场景 |
|------|------|---------|
| 范围 | 未满足验收条件或范围蔓延 | "验收条件 2 未实现"、"改了范围外的文件" |
| 质量 | 代码质量或测试覆盖不足 | "错误处理不充分"、"缺少边界测试" |
| 设计 | 架构或 API 设计需调整 | "接口设计不直观"、"应该用策略模式" |

### 打回记录格式

CR 事件表的打回事件应包含：
- **原因类别**：范围/质量/设计
- **具体描述**：1 句话说明问题
- **期望方向**：修复建议或期望结果

格式：`[类别]：[具体描述]，期望：[方向]`

示例：
| 日期 | 事件 | 操作者 | 备注 | 交接 |
|------|------|--------|------|------|
| 02-22 | in_review→developing | 用户 | 打回 | 质量：错误提示不清晰，期望：用户可理解的错误消息 |

### 处理规则

- 用户打回时未提供结构化原因 → Claude 主动追问："打回原因是范围/质量/设计？具体是哪个方面？"
- 追问不超过 1 次——用户回复模糊时，Claude 根据 review 上下文自行分类
- 打回信息同时记录到事件表的"交接"列，供 pace-learn 提取 pattern

## 修改意见处理规范

收到人类 review 修改意见时，按以下流程处理——不盲从、不表演性同意。

### 处理流程

1. **理解**：确认意见的真实意图（不假设字面意思 = 真实意图）
   - 如有歧义："你是希望 [解读 A] 还是 [解读 B]？"
2. **评估**：修改意见与 CR 范围/验收条件是否对齐？
   - **对齐** → 执行修改
   - **超出范围** → 记录到 CR 事件表（"Review 建议：[内容]，超出当前 CR 范围，建议作为新 CR"）→ 建议用户创建新 CR
   - **与已有决策冲突** → 引用 CR 事件表中的决策记录（"此前选择 X 而非 Y，因为 Z——是否需要调整这个决策？"）→ 等待用户确认
3. **执行 + 验证**：按修改后内容重跑受影响的质量检查 → 更新 Review 摘要

### 禁止行为

- 禁止 "You're absolutely right!" 式表演性回应——收到意见后先理解和评估，不盲目执行
- 禁止接受 YAGNI 违反的建议——如建议添加"以后可能需要"的功能，引用 CR 范围拒绝
- 禁止在未理解意图前就开始修改代码
