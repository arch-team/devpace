# push — 推送实体状态到外部（内部模块）

> **职责**：定义单个实体推送到外部的核心执行步骤（状态比对→适配器更新→语义 Comment→限流）。被 `sync-procedures-entity.md` 智能同步流程调用。高级特性（Gate 结果同步）见 `sync-procedures-push-advanced.md`。

## 调用方式

本文件不再作为独立子命令入口。由 `sync-procedures-entity.md` §2.4b 调用，传入目标实体信息。

## 核心推送步骤

对单个已关联实体执行推送：

1. 读取实体当前状态（从 `extract-entity-metadata.mjs` 输出或直接读文件）
2. 通过操作语义"获取实体类型状态映射"→ 查询 sync-mapping.md 中对应类型的状态映射表 → 获取目标外部状态
3. 执行适配器"获取状态"操作 → 查询外部当前状态
4. **外部实体状态预检查**：按适配器"状态预检查"规则验证外部实体可更新。不可更新 → 输出警告并跳过
5. 比较：一致 → 跳过（返回"已一致"），不一致 → 执行更新
6. 执行适配器"状态更新策略"中对应的操作
7. 生成语义 Comment（见下方 §1）并执行适配器"添加评论"操作
8. 更新 sync-mapping.md 关联记录的最后同步时间和内容摘要哈希
9. **限流保护**：按适配器限流规则执行（含等待、重试、跳过逻辑）

## §1 语义 Comment 生成规则

Claude 读取实体上下文后生成语义丰富的 Comment。Comment 内容根据实体类型有所不同。

### CR 的 Comment（信息最丰富）

**信息采集**：
1. CR 标题和意图描述
2. **状态转换上下文**：从哪个状态转换以及原因
3. 最近的 Gate 结果（如有）
4. 关键验收条件达成情况（如有）
5. **变更管理摘要**（如有）
6. **PR 链接**（如有）

### Epic/BR/PF 的 Comment（简洁）

**信息采集**：
1. 实体标题
2. **状态变化**：旧状态 → 新状态
3. **完成度变化**（如有）：如"BR-001: 2/3 PF 完成 → 3/3 PF 完成"
4. **下游实体摘要**：如"Epic-001: 4 个 BR, 12 个 PF, 8 个 CR 已完成"

### Comment 格式（通用）

```
🔄 [{状态}] {实体标题}

{1-2 句上下文说明}

{可选附加信息}
```

**约束**：
- 不超过 8 行
- 不包含 devpace 内部术语（如 checkpoint、state.md）
- 信息采集失败时回退到简单格式：`🔄 [{状态}] {实体标题}`

## §2 同步结果输出格式

由 `sync-procedures-entity.md` §2.5 控制整体输出。本模块返回单个实体的推送结果：

```json
{ "id": "CR-003", "type": "cr", "external": "#42", "action": "更新标签 → in-progress", "result": "✅" }
```

**外部链接要求**：外部实体编号附带完整 URL，便于终端点击跳转。URL 从 sync-mapping.md 平台配置的"连接"字段拼装。
