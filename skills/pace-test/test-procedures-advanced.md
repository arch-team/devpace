# 测试执行规程——高级功能

> **职责**：/pace-test 的高级子命令规则——不稳定测试分析（§7 flaky）、模拟门禁执行（§8 dryrun）、测试基准线（§9 baseline）。

## §7 不稳定测试分析（flaky）

当用户执行 `/pace-test flaky` 时，进行**不稳定测试分析**和**主动测试维护检测**——从响应式维护升级为主动式维护。

### 流程

1. **收集测试执行历史**：
   - 扫描 `.devpace/backlog/` 中所有 CR 文件的事件表 → 提取以下类型的测试记录：
     - `/pace-test 执行` 类型的事件
     - Gate 1/Gate 2 事件中包含 "passed"/"failed" 关键词的检查结果（通过 /pace-dev 执行的测试）
   - 从 `insights.md`（如存在）提取测试相关的 pattern（`category: 质量保障`）
   - 从 `git log` 提取最近 N 次测试相关的提交（包含 test/fix 关键词）
2. **识别不稳定模式**：
   - **间歇失败**：同一检查项在多次执行中状态不一致（有时 PASS 有时 FAIL）
   - **环境依赖**：失败集中在特定时间段或特定 CR 上下文
   - **顺序依赖**：仅在特定执行顺序下失败
   - **超时波动**：同一检查项耗时方差超过 50%
3. **主动测试维护检测**（在不稳定分析之后执行）：
   - **永远通过的测试**：扫描测试文件，检测可能的空断言模式——无 assert/expect 语句、仅 `pass`/`return true` 的测试函数、被注释掉的断言
   - **执行时间膨胀**：对比 test-baseline.md（如存在）的历史耗时数据，标记耗时增长 >50% 的检查项
   - **长期未更新的测试**：通过 `git log` 检查测试文件最后修改时间，标记 >30 次 CR 未修改的测试文件（可能与源代码已脱节）
   - **死测试**：扫描被 `skip`/`xfail`/`@Disabled`/`.skip` 标记的测试，统计数量和分布
4. **评估影响**：
   - 不稳定测试导致的 Gate 1 误拦截次数（从 CR 事件表的重试记录推断）
   - 受影响的 PF 范围
5. **生成建议**
6. **结果持久化**：
   - 将发现的不稳定测试 pattern 和维护问题追加到 `.devpace/metrics/insights.md`
   - 格式遵循 insights.md 的 pattern 格式（category: 质量保障），示例：
     ```
     ### [日期] 不稳定测试：[检查项名称]
     **观察**：[失败率、模式描述]
     **规律**：[识别的不稳定模式]
     **证据**：[受影响 CR 列表、执行历史]
     **建议**：[修复方向]
     ```
   - 无发现时不写入
7. **回写 test-strategy.md**（flaky→strategy 闭环）：
   - 如果 `.devpace/rules/test-strategy.md` 存在且 Step 2 识别到不稳定测试：
     - 将不稳定测试对应的验收条件状态从 `✅ 已有` 降级为 `⚠️ 不稳定`
     - 更新策略总览表的"已覆盖"计数（不稳定测试不计入有效覆盖）
     - 在降级条目的"测试文件/位置"列追加注释：`（不稳定：[模式]）`
   - test-strategy.md 不存在 → 跳过
   - **与 accept Step 5.5 的关系**：accept 检测弱覆盖/虚假覆盖 → 降级；flaky 检测不稳定 → 降级。两者独立触发，共同维护 strategy 数据准确性

### 输出格式

```markdown
## 测试健康度分析

**分析范围**：最近 N 个 CR 的测试执行记录

### 不稳定测试

| # | 检查项 | 失败率 | 模式 | 影响 CR |
|---|--------|--------|------|---------|
| 1 | [检查名] | 2/5 (40%) | 间歇失败 | CR-003, CR-005 |

### 测试维护问题

| # | 问题类型 | 测试文件/检查项 | 详情 | 建议 |
|---|---------|---------------|------|------|
| 1 | 空断言 | tests/test_utils.py::test_helper | 无 assert 语句 | 补充断言或删除 |
| 2 | 耗时膨胀 | lint 检查 | 1.2s → 3.8s (+217%) | 检查新增文件量或优化配置 |
| 3 | 长期未更新 | tests/test_auth.py | 最后修改 CR-002（30+ CR 前） | 审查是否与源代码脱节 |
| 4 | 已跳过 | tests/test_legacy.py::test_old | @skip 标记 | 修复或删除 |

### 建议

| # | 检查项/文件 | 建议操作 | 理由 |
|---|------------|---------|------|
| 1 | [检查名] | 修复：[具体方向] | [原因分析] |
| 2 | [检查名] | 隔离：移至 Gate 2 意图检查 | 无法自动化稳定验证 |

### 修复优先级

1. **立即修复**：影响 P0 功能的不稳定测试 + 空断言测试（虚假安全感）
2. **计划修复**：间歇失败但有明确根因 + 耗时膨胀（CI 瓶颈）
3. **可暂时隔离**：低优先级或环境相关的不稳定测试
4. **清理建议**：长期跳过的测试 + 长期未更新的测试
```

### 降级场景

| 条件 | 行为 |
|------|------|
| 无 CR 事件表记录 | 不稳定分析输出"无历史测试数据"，但主动维护检测仍执行（扫描测试文件不需要历史数据） |
| 仅 1 次执行记录 | 不稳定分析输出"数据不足"，主动维护检测仍执行 |
| 无不稳定测试且无维护问题 | 输出"测试健康度良好 ✅" |
| 无测试文件 | 输出"未检测到测试文件，建议先执行 /pace-test generate 生成测试框架" |

## §8 模拟门禁执行（dryrun）

当用户执行 `/pace-test dryrun [1|2|4]` 时，以 dry-run 模式执行指定 Gate 的完整检查流程，不触发状态转换。

### 流程

1. **确定目标 Gate**：
   - 有参数 → 执行指定 Gate（1 / 2 / 4）
   - 无参数 → 根据当前 CR 状态推断下一个 Gate：
     - developing → Gate 1
     - verifying → Gate 2
     - approved/merged → Gate 4（如有 Release）
   - 无活跃 CR → 提示用户指定 Gate 编号
2. **加载检查项**：
   - 读取 `.devpace/rules/checks.md` → 提取目标 Gate 的所有检查项
   - Gate 4：从 `skills/pace-release/release-procedures-lifecycle.md` Gate 4 段落提取发布检查项
3. **执行检查（dry-run 模式）**：
   - 命令检查：实际执行 bash 命令，记录结果
   - 意图检查：Claude 按规则判定，输出结论
   - 对抗审查（Gate 2）：执行对抗审查，输出发现
   - 浏览器验收（Gate 2，前端项目 + Playwright MCP 可用时）：按 `verify-procedures.md` L1+ 流程执行，标注"🖥️ 浏览器验收"
   - **不触发 CR 状态转换**——仅输出结果
4. **生成模拟报告**

### 与 /pace-dev Gate 的区别

| 维度 | /pace-test dryrun | /pace-dev Gate |
|------|----------------|----------------|
| 状态转换 | 不转换（dry-run） | 转换 CR 状态 |
| 自动修复 | 不修复（仅报告） | 失败时自动修复+重试 |
| 用途 | 预检/调试/确认 | 正式门禁流程 |
| CR 事件记录 | 不写入 | 写入事件表 |

### 输出格式

```markdown
## Gate [N] 模拟执行报告（Dry-Run）

**CR**：[CR 标题]（如有）
**模式**：🔍 仅检查，不转换状态

### 检查结果

| # | 检查项 | 类型 | 状态 | 备注 |
|---|--------|------|------|------|
| 1 | [名称] | 命令 | ✅ PASS | — |
| 2 | [名称] | 意图 | ❌ FAIL | [判定理由] |
| 3 | [名称] | 命令 | ⏭️ SKIP | 依赖"[前置名称]"未通过 |

### 汇总

- 通过：N/M
- **预判**：[Gate 将通过 ✅ / Gate 将失败 ❌]
- 需修复项：[列表]

*提示：修复后可再次执行 `/pace-test dryrun [N]` 确认，或执行 `/pace-dev` 进入正式 Gate 流程。*
```

### 降级场景

| 条件 | 行为 |
|------|------|
| 无 checks.md | 使用 `test-procedures-core.md` §1.2 自动检测逻辑获取测试命令，仅执行命令检查 |
| Gate 编号无效 | 输出"支持的 Gate：1（developing→verifying）、2（verifying→in_review）、4（Release 发布）"|
| 无活跃 CR 且无参数 | 提示"请指定 Gate 编号：/pace-test dryrun 1" |

## §9 测试基准线（baseline）

当用户执行 `/pace-test baseline` 时，建立或更新测试执行基准线，供 /pace-retro 度量使用。

### 流程

1. **执行完整测试**：运行 `test-procedures-core.md` §1 基础测试执行流程，获取当前测试结果
2. **收集基准数据**：
   - 每项检查的通过状态和执行耗时
   - 总通过率
   - 总执行时间
   - 如果 `test-strategy.md` 存在 → 需求覆盖率
3. **对比历史基准**（如存在）：
   - 读取 `.devpace/rules/test-baseline.md`
   - 对比当前结果与上一次基准：
     - 通过率变化（↑/↓/→）
     - 耗时变化（快/慢/持平）
     - 新增/删除的检查项
4. **写入基准文件**：生成或更新 `.devpace/rules/test-baseline.md`

### 基准文件格式

格式遵循 `knowledge/_schema/test-baseline-format.md`（当前基准表 + 检查项明细表 + 历史趋势表）。

### 输出格式

```markdown
## 测试基准线更新

**状态**：[新建基准 / 更新基准]

### 当前基准

- 检查项：N 项（通过 X / 失败 Y）
- 通过率：Z%
- 执行时间：W.Ws

### 与上次基准对比（如有）

| 指标 | 上次 | 本次 | 变化 |
|------|------|------|------|
| 通过率 | X% | Y% | ↑ +N% |
| 执行时间 | As | Bs | ↓ 快 Cs |
| 检查项数 | M | N | +K 新增 |

基准文件已更新：`.devpace/rules/test-baseline.md`
```

### 降级场景

| 条件 | 行为 |
|------|------|
| 无 checks.md 且无可检测测试 | 输出"无可执行的测试命令，无法建立基准" |
| 首次建立基准 | 不做对比，直接生成初始基准文件 |
| 测试全部失败 | 仍记录基准（基准反映实际状态，非理想状态）|
