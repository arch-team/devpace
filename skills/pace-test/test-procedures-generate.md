# 测试执行规程——测试用例生成

> **职责**：/pace-test generate 测试用例生成——骨架/完整模式、策略自动注册。自包含。

## §0 速查卡片

- **生成规则**：测试名从验收标准自然语言翻译，按验收条件编号对应测试编号
- **两种模式**：骨架（默认，TODO 占位）/ 完整（`--full`，含断言+边界+异常）
- **TDD 引导**：CR developing 阶段+无实现代码时，输出 Red-Green-Refactor 提示

## §2 测试用例生成（generate）

当用户执行 `/pace-test generate [PF标题] [--full]` 时，基于 PF 验收标准生成测试用例。

### 生成模式

| 模式 | 触发 | 行为 |
|------|------|------|
| 骨架模式（默认） | `generate [PF]` | 生成测试框架 + TODO 占位符 |
| 完整模式 | `generate [PF] --full` | 生成含实际断言、边界条件、异常路径的完整测试实现 |

### 流程

1. **定位 PF 验收标准**：
   - 有 PF 参数 → 在 `project.md` 功能树中匹配
   - 无参数 → 使用当前 CR 关联的 PF
   - 未找到 → 提示用户指定
2. **提取验收条件**：从 PF 的用户故事和验收标准中提取可测试的断言
3. **检测技术栈**：按 test-procedures-common.md「技术栈检测（SSOT）」
4. **生成测试**（按模式分支）：

   **骨架模式**（默认）：
   - 按技术栈生成符合项目风格的测试文件骨架
   - 每个验收条件 → 一个测试用例（describe + it/test）
   - 包含 setup/teardown 骨架
   - 标注 `// TODO: implement` 的具体断言

   **完整模式**（`--full`）：
   - 在骨架基础上，生成完整测试实现：
     - **正向断言**：验收条件描述的核心行为
     - **边界条件**：输入边界值、空值、极值测试
     - **异常路径**：错误输入、权限不足、超时等异常场景
     - **数据准备**：fixture/mock 数据和 setup 逻辑
   - 阅读已有源代码（如有）以确保断言与实际接口一致
   - 标注 `// REVIEW: AI-generated assertion` 提醒用户审查 AI 生成的断言

5. **输出**：
   - 将生成的测试文件内容展示给用户
   - 用户确认后写入项目测试目录
6. **自动注册到测试策略**：
   - `.devpace/rules/test-strategy.md` 存在 → 更新对应验收条件的"测试文件/位置"列和"状态"列：
     - 骨架模式：`❌ 待建` → `⚠️ 部分覆盖`
     - 完整模式：`❌ 待建` → `✅ 已有`
     - 更新策略总览表的覆盖数值
   - 不存在 → 跳过，附引导提示："建议先执行 `/pace-test strategy` 生成测试策略"

### 生成规则

- 测试名称从验收标准的自然语言直接翻译（如"用户能上传 CSV" → `test_user_can_upload_csv`）
- 按验收条件编号对应测试编号，方便后续 coverage 追踪
- 包含边界条件测试建议（骨架模式为注释形式，完整模式为实际测试）
- 骨架模式不生成具体实现——只生成框架和断言占位
- 完整模式生成可运行的测试——但标注 AI 生成标记，提醒用户审查

### TDD 工作流引导

当用户在 CR developing 阶段（尚未编写实现代码）执行 `generate` 时，输出末尾附加 TDD 流程提示：

```
💡 TDD 建议：先运行生成的测试确认失败（Red），再实现功能使测试通过（Green），最后重构（Refactor）。
```

**触发条件**：CR 状态为 developing + 对应 PF 的实现代码尚未提交（通过 git diff 判断 CR 关联文件是否为新增）。已有实现代码时不输出（测试后补场景，非 TDD）。

**定位**：提示而非强制——devpace 不强制 TDD 工作流（UX P1 零摩擦），但在合适的时机推荐最佳实践。

### 输出格式

```
生成了 N 个测试用例[框架|实现]，覆盖 PF "[PF标题]" 的 M 项验收条件：
1. [验收条件] → test_xxx [骨架 | 含 K 个断言]
2. [验收条件] → test_yyy [骨架 | 含 K 个断言 + J 个边界测试]
...
文件：[测试文件路径]
模式：[骨架 | 完整（含 AI 生成断言，请审查 REVIEW 标记）]
```

### --brief 输出格式

`生成：N 用例 · 模式 [骨架/完整] · PF [名称]`

### 降级场景

| 条件 | 行为 |
|------|------|
| 无 PF 参数且无活跃 CR | 提示用户指定 PF 标题 |
| 无可识别技术栈 | 使用通用测试模板（JavaScript 风格 describe/it），标注"请根据项目实际技术栈调整" |
| 无 test-strategy.md | 跳过 Step 6 自动注册，输出引导："建议先执行 `/pace-test strategy` 生成测试策略" |
