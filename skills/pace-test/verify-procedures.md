# AI 验收验证规程

> **职责**：/pace-test accept 验收验证——L1/L2/L3 三级验证、合理化审计、测试预言审查。按 SKILL.md 路由表按需加载。

## §0 速查卡片

| Step | 名称 | 说明 |
|------|------|------|
| 0 | 断点恢复检测 | 检测 PARTIAL 标记，支持从中断点继续 |
| 1 | 提取验收标准 | 从 PF 获取可验证的断言列表 |
| 1.5 | 跨 CR 上下文检查 | 多 CR 实现同一 PF 时标注验证范围 |
| 2 | 确定验证策略 | L1 动态/L2 静态(含置信度)/L3 人工三级路径 |
| 3 | 逐条验证 | 执行验证并收集证据（含增量写入） |
| 3a | 合理化自审计 | 回查验证结论，检测 8 种合理化模式 |
| 3b | 测试预言审查 | 已有测试是否真的验了声称覆盖的条件 |
| 4 | 生成验证报告 | 结构化输出（逐条判定+置信度+自审计） |
| 5 | 写入 CR | 持久化验证证据 |
| 5.5 | 反写测试策略 | 弱覆盖/虚假覆盖→降级策略状态 |

- **核心理念**：测试验"代码正确"，accept 验"需求满足"——二者不可替代，accept 弥合断层
- **断点恢复**：PARTIAL 标记机制，支持验证中断后从断点继续
- **合理化预防**：8 种模式+运行时自审计（Step 3.7），防止降低验证标准
- **降级策略**：验收标准缺失→链式引导（strategy/手动/CR 降级），代码变更缺失→明确拦截

## 前置条件

1. 当前 CR 必须关联到一个 PF（产品功能）
2. PF 必须有可验证的验收标准
3. CR 必须有代码变更（git diff 非空）

缺少条件 2 → 输出链式引导：
- "验收标准缺失。可选操作：1) 执行 `/pace-test strategy` 生成策略映射（含验收标准推导）2) 在 project.md 中为 PF 手动添加验收标准 3) 在 CR 意图 section 补充 CR 级验收条件（降级路径）"
- 如果 CR 有意图 section 但无验收条件 → 额外提示"或在 CR 意图 section 的验收条件段落添加条目，accept 将使用 CR 级条件进行降级验证"

缺少条件 3 → 输出："当前 CR 无代码变更（git diff 为空）。请在提交代码后再执行验收验证。"

两种情况均不降级执行。

### 降级模式

当 CR 无 PF 关联（缺少条件 1）但 CR 意图 section 有验收条件时：
- 使用 CR 意图 section 的验收条件作为验证基准（替代 PF 级验收标准）
- 输出报告标注"⚠️ 降级验证——基于 CR 意图验收条件，非 PF 级验收标准"
- 验证流程与正常模式一致（Step 1-5），仅验收断言来源不同

当 CR 既无 PF 关联也无意图验收条件 → 输出明确提示，不执行。

## 执行流程

### Step 0：断点恢复检测

1. 读取 CR 文件的"验证证据"section
2. 如果存在标记 `<!-- PARTIAL: 已验证 N/M 条，最后验证时间 [datetime] -->` → 检测到部分验证结果：
   - 展示已验证条目的摘要（编号 + 状态）
   - 询问用户："检测到上次验证的部分结果（N/M 条已完成）。继续验证剩余条目（A{N+1}-A{M}），还是重新开始？"
   - 用户选择继续 → 跳转到 Step 3，从第 N+1 条开始执行
   - 用户选择重新开始 → 清除部分结果，从 Step 1 正常执行
3. 无部分结果 → 正常执行 Step 1

**规则**：
- 断点恢复仅在检测到 PARTIAL 标记时触发
- 恢复前检查 `git diff` 是否有新变更——有新变更时提示"代码有更新，建议重新验证已完成的条目"
- 用户不响应或超时 → 默认重新开始（安全选择）

### Step 1：提取验收标准

1. 读取当前 CR 文件 → 获取关联 PF（`产品功能` 字段）
2. 读取 `project.md` → 定位 PF 条目 → 获取用户故事和验收标准
3. 如果 CR 意图 section 有更细粒度的验收条件 → 合并使用（CR 验收条件优先级高于 PF 级别）

输出：编号的验收断言列表。

```
验收断言：
A1. [验收条件 1]
A2. [验收条件 2]
...
```

### Step 1.5：跨 CR 上下文检查

1. 检查当前 PF 是否关联多个 CR（扫描 `.devpace/backlog/` 中关联同一 PF 的 CR）
2. 如有多个 CR：
   - 报告开头添加："该 PF 由 N 个 CR 共同实现（[列表]），本次验证仅覆盖当前 CR 的变更范围"
   - 对每条验收条件：如果实现代码不在当前 CR diff 中但在其他 CR 中 → 标注"由 [其他 CR] 实现，本次不验证"
3. 仅 1 个 CR → 跳过此步骤

### Step 2：确定验证策略

对每个验收断言，评估可用的验证方式：

| 验证级别 | 条件 | 方式 | 可信度 |
|---------|------|------|--------|
| L1：动态验证 | 项目可运行 + 有测试命令/可交互 | 执行测试/CLI/Playwright | 高 |
| L2：静态语义验证 | 有代码变更可分析 | 阅读 diff + 相关代码 + 推理 | 中 |
| L3：不可验证 | 验收标准指向外部系统/主观体验 | 标注"需人工验证" + 说明原因 | — |

**验证级别选择规则**：
- 优先 L1（动态验证）：如果 checks.md 中有测试命令且可执行
- 降级 L2（静态验证）：动态验证不可用或失败时
- 标注 L3（不可验证）：涉及外部 API、主观体验、物理操作等

**策略引导（可选）**：如果 `.devpace/rules/test-strategy.md` 存在 → 读取每条验收条件的推荐测试类型，作为 L1 测试识别的优先级指导。不存在时静默跳过，不影响现有流程。

**Playwright 浏览器验收（可选增强）**：以下 L1+ 段落仅在项目有前端框架时适用，纯后端项目直接跳至 Step 3。当检测到项目有前端时，自动建议通过 Playwright MCP 执行真实浏览器验收，将验收从"读代码猜测"升级为"看到真实界面判断"。

前端项目检测条件（满足任一）：
- `package.json` 的 dependencies 含 `react` / `vue` / `angular` / `next` / `nuxt` / `svelte`
- 项目存在 `src/App.tsx` / `src/App.vue` / `src/app/` 等前端入口文件
- `.devpace/context.md` 技术栈 section 标注含前端框架

检测到前端 → 在 Step 2 验证策略表中增加 L1+ 级别：

| 验证级别 | 条件 | 方式 | 可信度 |
|---------|------|------|--------|
| L1+：浏览器验收 | 前端项目 + Playwright MCP 可用 | 启动开发服务器 → 浏览器截图/交互 → 对比验收标准 | 最高 |

**L1+ 执行流程**：
1. 检查 Playwright MCP 是否可用（`.mcp.json` 或全局配置含 `playwright`）
2. 不可用 → 输出"建议安装 Playwright MCP 以获得浏览器验收能力"，退回 L1/L2
3. 可用 → 询问用户"是否启动开发服务器进行浏览器验收？"
4. 用户确认 → 启动 dev server（`npm run dev` / `npm start` 等）→ 等待就绪
5. 通过 Playwright 导航到相关页面 → 截图 → 交互操作 → 验证可见结果
6. 每条验收断言产出：截图证据 + 交互验证结果
7. 验证报告中标注"🖥️ 浏览器验收"（区别于"📝 静态验收"）

**降级**：Playwright MCP 不可用 / 用户拒绝启动服务器 / dev server 启动失败 → 退回 L1（测试命令）或 L2（静态分析），不阻断验收流程。

### Step 3：逐条验证

对每个验收断言执行验证：

#### L1+ 浏览器验收（前端项目 + Playwright MCP 可用时）

1. 通过 Playwright MCP 导航到验收条件相关的页面/组件
2. 执行交互操作（点击、输入、导航）模拟用户行为
3. 截图记录可见状态
4. 对比验收断言与实际界面行为，判定通过/不通过
5. 记录验证证据：截图路径 + 交互日志 + 判定结论

#### L1 动态验证

1. 识别与该断言相关的测试（优先匹配 strategy 推荐的测试类型，其次按名称/描述匹配）
2. 执行相关测试，捕获输出
3. 如果无直接对应测试 → 尝试通过 CLI 或脚本模拟验证
4. 记录验证证据：测试输出摘要或执行结果

#### L2 静态语义验证

1. 获取 CR 的 `git diff`
2. 阅读变更代码 + 相关上下文文件
3. 对照验收断言，逐项判断：
   - 代码逻辑是否实现了该断言描述的行为
   - 是否存在明显的遗漏（边界条件、错误处理）
4. **L2 判定标准**（含置信度评分）：
   - **✅ 通过**：找到直接实现代码（含行号引用）+ 无明显遗漏
     - 🟢 **High**：直接的输入→输出映射，断言明确（如简单返回值、配置项）
     - 🟡 **Medium**：实现逻辑清晰但涉及多步处理或条件分支
   - **⚠️ 需补充验证**：实现代码存在但逻辑复杂/间接（异步、状态机、并发），无法从静态分析确认完整覆盖
     - 🟠 **Low**：始终标注 Low（复杂逻辑的静态判断天然不确定）
   - **❌ 未通过**：未找到实现代码 或 找到的代码与验收条件不匹配
     - 置信度不适用（判定明确）

   **置信度用途**：帮助人类审阅者聚焦——Low 置信度条目优先人工复核，High 置信度条目可快速确认。
5. 记录验证证据：代码引用（文件:行号）+ 判断依据 + L2 判定分类

#### 增量写入（断点恢复支持）

每完成一条验收断言的验证后，立即将结果追加到 CR 验证证据 section：
- 格式：验收验证报告的单行条目（`| AN | [断言] | [状态] | [级别] | [证据] |`）
- 在 section 开头维护标记：`<!-- PARTIAL: 已验证 N/M 条，最后验证时间 [datetime] -->`
- 全部完成后 → Step 5 将 PARTIAL 标记替换为完整报告（移除标记）

**规则**：
- 增量写入不替代 Step 5 的最终写入——Step 5 生成完整格式化报告覆盖增量数据
- 如果 CR 文件写入失败 → 不阻断验证流程，在内存中继续

#### L3 人工验证指导

L3 不仅标注"需人工验证"，还为人类验证者生成可执行的验证检查单。

1. **标注不可验证原因**（"需要访问外部 API"/"涉及 UI 视觉体验"等）
2. **生成人工验证检查单**：

   对每个 L3 验收条件，生成结构化验证指导：

   ```markdown
   #### L3 人工验证：[验收条件编号] [条件摘要]

   **不可自动验证原因**：[原因]

   **验证步骤**：
   1. [具体操作步骤 1]
   2. [具体操作步骤 2]
   3. [观察预期结果]

   **预期结果**：[可观察的成功标志]

   **测试数据建议**：[如需特定账号/数据/环境配置]

   **验证状态**：⬜ 待验证
   ```

   **验证步骤生成规则**：
   - 从验收条件的自然语言推导具体操作序列
   - 步骤粒度：每步是一个独立的用户操作（点击/输入/观察）
   - 包含环境前置条件（如"确保登录为管理员角色"）
   - 对涉及外部系统的条件，注明所需的外部系统状态
   - 对主观体验类条件（"界面流畅""响应迅速"），转化为可度量的观察点（"页面加载 <3s""无卡顿感"）

3. **探索性测试章程**（当 L3 条目 ≥3 个时附加）：

   ```markdown
   #### 探索性测试建议

   **章程**：在 [时间范围] 内，以 [角色] 身份，探索 [功能区域]，重点关注 [风险区域]。

   **关注区域**：
   - [从 L3 条件提炼的关注维度 1]
   - [从 L3 条件提炼的关注维度 2]

   **边界场景**：
   - [从验收条件推导的边界情况]
   ```

   - 章程从 L3 条件聚类推导（相关条件归入同一探索区域）
   - L3 条目 <3 个时不生成章程（逐条验证即可）

### Step 3a：合理化自审计

在逐条验证完成后，Claude 对自身的验证结论进行显式审计——检查是否无意中降低了验证标准。

**流程**：

1. **逐条回查**：对每个判定为 ✅ 通过 的验收断言，检查是否命中以下 8 种合理化模式：

   | # | 合理化模式 | 检查方式 | 命中后动作 |
   |---|-----------|---------|-----------|
   | R1 | "代码逻辑正确，应该能工作" | 证据是否仅有推理，无代码引用或测试输出 | 降级为 ⚠️，补充代码引用 |
   | R2 | "验收标准太模糊" | 是否跳过了验证而非输出歧义报告 | 补充歧义报告，不跳过 |
   | R3 | "需要运行环境才能验证" | L2 降级是否有明确标注 | 确认已标注"未进行动态验证" |
   | R4 | "所有条件都满足了" | 每条是否附带了代码引用或测试输出证据 | 无证据的降级为 ⚠️ |
   | R5 | "这条不重要" | 是否对某条验收标准给予了更宽松的判定 | 恢复同等标准 |
   | R6 | "之前已经验证过了" | 是否引用了旧结果而非本次独立验证 | 重新执行本次验证 |
   | R7 | "太多条了，抽样验证" | 是否跳过了某些验收断言 | 补全未验证的条目 |
   | R8 | "代码存在就等于通过" | L2 判定是否仅检查代码存在性而非行为正确性 | 重新审查实现完整性 |

2. **输出自审计结论**：在验证报告中 Step 4 的"逐条验证结果"表之后追加：

   ```markdown
   ### 合理化自审计

   | # | 验收断言 | 审计结果 | 说明 |
   |---|---------|---------|------|
   | A1 | [断言] | ✅ 无风险 | — |
   | A3 | [断言] | ⚠️ R1 命中 | 原判定仅有推理，已补充代码引用 src/handler.py:45 |

   **审计结论**：N 条通过审计，M 条命中合理化模式（已修正）
   ```

3. **修正联动**：命中合理化模式的条目，在 Step 4 验证报告的"逐条验证结果"表中更新对应状态（如从 ✅ 降级为 ⚠️）

**规则**：
- 自审计是 Step 3 的延伸，不增加额外的验证执行——仅回查已有结论
- 全部通过审计时输出"自审计：0 风险"（1 行），不输出详细表格
- 有命中时必须输出详细表格 + 修正说明

### Step 3b：测试预言审查（Test Oracle Check）

在逐条验证之后、生成报告之前，审查已有测试是否**真正验证**了其声称覆盖的验收条件。

**背景**：一个测试文件名为 `test_user_upload_csv` 并不意味着它真的验证了"用户能上传 CSV"——测试可能只检查了文件存在性，未验证解析结果。

**流程**：

1. **定位映射关系**：
   - 如果 `test-strategy.md` 存在 → 读取验收条件→测试映射表中状态为 `✅ 已有` 的条目
   - 如果不存在 → 跳过此步骤
2. **逐条审查测试预言**：对每个"已覆盖"的验收条件-测试对：
   - 阅读测试代码中的断言（assert/expect 语句）
   - 判断断言是否**实质性**验证了验收条件描述的行为
   - 分类判定：
     - **✅ 有效覆盖**：断言确实验证了验收条件的核心行为
     - **⚠️ 弱覆盖**：测试存在但断言仅覆盖部分行为（如只检查返回码不检查数据）
     - **❌ 虚假覆盖**：测试名称匹配但断言与验收条件无关
3. **输出**：在验证报告中增加"测试预言审查"section（见 Step 4 报告格式）

**注意**：此步骤仅在 test-strategy.md 存在且有 `✅ 已有` 条目时执行。不存在时静默跳过，不影响验收验证流程。

### Step 4：生成验证报告

```markdown
## 验收验证报告

**CR**：[CR 标题]（[CR-ID]）
**PF**：[PF 标题]（[PF-ID]）
**验证时间**：[日期时间]

### 逐条验证结果

| # | 验收断言 | 状态 | 验证级别 | 证据 |
|---|---------|------|---------|------|
| A1 | [断言内容] | ✅ 通过 | L1 动态 | 测试 test_xxx 通过 |
| A2 | [断言内容] | ✅ 通过 | L2 静态 🟢 | src/handler.py:45 实现了 CSV 解析逻辑 |
| A3 | [断言内容] | ❌ 未通过 | L2 静态 | 未找到进度条相关实现代码 |
| A4 | [断言内容] | ⚠️ 需人工 | L3 | 涉及第三方支付 API 回调 |

### 汇总

- 通过：N / M
- 未通过：X（[未通过断言列表]）
- 需人工验证：Y
- 验证覆盖率：(N+X) / M = Z%（排除 L3 不可验证项）

### 测试预言审查（条件输出：有 test-strategy.md 且有已覆盖条目时）

表格格式同 Step 3.5 输出（验收条件 | 映射测试 | 预言判定 | 说明）。

### 人工验证检查单（条件输出：有 L3 条目时）

表格格式同 Step 3 L3 输出（验收条件 | 验证步骤 | 预期结果 | 状态⬜）。由人类审阅者在 Gate 3 阶段执行。

### 未覆盖的风险点

- [边界条件/错误路径/性能场景等 verify 过程中发现的额外风险]

### 验收标准改进建议（条件输出：有模糊/不可测试条目时）

- [验收标准模糊/不可测试的条目及改进建议]
```

### Step 5：写入 CR

1. 将验证报告写入 CR 文件的"验证证据"section
2. 在 CR 事件表添加记录：
   ```
   | [日期] | /pace-test accept | Claude | A: N/M passed, X failed, Y manual | — |
   ```
3. 如果全部通过（含 L3 需人工项标注）→ 输出"验收验证通过"
4. 如果有未通过项 → 输出未通过清单，建议修复后重新 verify
5. **人工验证结果回收**（Gate 3 阶段）：
   - 当人类审阅者在 Gate 3 阶段完成人工验证后，手动更新 CR 验证证据 section 中"人工验证检查单"的状态列
   - 下次执行 `/pace-test accept` 或 `/pace-test report` 时，读取已有的人工验证结果 → 合并到报告中
   - 人工验证状态标记：`⬜ 待验证` → `✅ 通过`（人工确认通过）/ `❌ 未通过`（人工发现问题）
   - report 子命令消费人工验证结果时，将 L3 状态更新为人工判定结果（不再显示"需人工验证"）

### Step 5.5：反写测试策略（如有 test-strategy.md）

在 accept 验证完成后，将结果反写到测试策略以形成双向闭环。

1. 检查 `.devpace/rules/test-strategy.md` 是否存在 → 不存在时跳过此步骤
2. 检查 accept 的 Test Oracle Check 结果（Step 3.5）是否执行过 → 未执行时跳过此步骤
3. 对 Step 3.5 中判定为"⚠️ 弱覆盖"的条目 → 将 strategy 中对应验收条件的状态降为"⚠️ 部分覆盖"
4. 对 Step 3.5 中判定为"❌ 虚假覆盖"的条目 → 将 strategy 中对应验收条件的状态降为"❌ 待建"
5. 更新策略总览表的覆盖数值（重新计算已覆盖数和覆盖率）
6. 输出："测试策略已更新 N 项（与 §2 Step 6 的 generate→strategy 注册形成双向闭环）"

**注意**：仅在 Step 3.5 实际执行（test-strategy.md 存在且有已覆盖条目）时触发。不存在时静默跳过，不影响验收验证流程。

## 合理化预防——快速直觉校准

**铁律：验证必须有证据，推理不等于验证。**

> 8 种合理化模式的完整定义、检查方式和命中动作见 Step 3a 自审计表（SSOT）。以下为快速直觉校准：

| # | 借口关键词 | 一句话回击 |
|---|-----------|-----------|
| R1 | "应该能工作" | 推理≠证据，附代码引用 |
| R2 | "标准太模糊" | 输出歧义报告，不跳过 |
| R3 | "需要环境" | 降级 L2 并标注 |
| R4 | "都满足了" | 每条要证据 |
| R5 | "不重要" | PF 定义者决定 |
| R6 | "验过了" | 每次独立验证 |
| R7 | "抽样吧" | 全量是底线 |
| R8 | "代码在就行" | 存在≠正确 |

## 与 Gate 2 的区别

accept 验"需求是否满足"（逐条证据），Gate 2 验"集成质量是否达标"（通过/不通过）。二者定位见 SKILL.md "与现有机制的关系"。

## 降级策略

```
L1 动态验证
  ├── 成功 → 记录证据
  └── 失败（环境/命令不可用）
       ↓
L2 静态语义验证
  ├── 成功 → 记录证据（标注"静态验证"）
  └── 失败（无法从代码判断）
       ↓
L3 标注不可验证
  └── 记录原因 + 建议人工验证方法
```

每次降级必须在报告中明确标注验证级别，不可静默降级。

## 触发方式

| 模式 | 触发时机 | 行为 |
|------|---------|------|
| 显式 | 用户 `/pace-test accept [CR]` | 完整执行上述流程 |
| Gate 2 消费 | Gate 2 检查时发现 CR 已有验证证据 | 引用已有报告，不重复执行 |
| 自动（未来） | developing→verifying 转换时 | Phase 3 集成后自动触发 |

**铁律：自动模式只消费已有结果或执行最轻量检查，不在 Gate 流程中触发重量级操作。**
