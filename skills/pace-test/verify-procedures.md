# AI 验收验证规程

> **职责**：`/pace-test accept`（验收验证）的详细执行规则——利用 Claude 的语义理解能力，将 PF 验收标准（自然语言）转化为可执行的验证，逐条产出验证证据。

## 核心理念

传统测试验证"代码是否正确"，AI 验收验证"需求是否被满足"。二者不可互相替代：
- 测试通过 ≠ 验收标准被满足（测试可能未覆盖验收标准）
- 验收标准满足 ≠ 测试通过（验收标准可能有无法自动测试的部分）

`/pace-test accept` 弥合二者之间的断层，是 devpace BizDevOps 价值链中**验收标准→自动验证**的最后一公里。

## 前置条件

1. 当前 CR 必须关联到一个 PF（产品功能）
2. PF 必须有可验证的验收标准
3. CR 必须有代码变更（git diff 非空）

缺少条件 2 或 3 → 输出明确提示，不降级执行。

### 降级模式

当 CR 无 PF 关联（缺少条件 1）但 CR 意图 section 有验收条件时：
- 使用 CR 意图 section 的验收条件作为验证基准（替代 PF 级验收标准）
- 输出报告标注"⚠️ 降级验证——基于 CR 意图验收条件，非 PF 级验收标准"
- 验证流程与正常模式一致（Step 1-5），仅验收断言来源不同

当 CR 既无 PF 关联也无意图验收条件 → 输出明确提示，不执行。

## 执行流程

### Step 1：提取验收标准

1. 读取当前 CR 文件 → 获取关联 PF（`产品功能` 字段）
2. 读取 `project.md` → 定位 PF 条目 → 获取用户故事和验收标准
3. 如果 CR 意图 section 有更细粒度的验收条件 → 合并使用（CR 验收条件优先级高于 PF 级别）

输出：编号的验收断言列表。

```
验收断言：
A1. [验收条件 1]
A2. [验收条件 2]
...
```

### Step 2：确定验证策略

对每个验收断言，评估可用的验证方式：

| 验证级别 | 条件 | 方式 | 可信度 |
|---------|------|------|--------|
| L1：动态验证 | 项目可运行 + 有测试命令/可交互 | 执行测试/CLI/Playwright | 高 |
| L2：静态语义验证 | 有代码变更可分析 | 阅读 diff + 相关代码 + 推理 | 中 |
| L3：不可验证 | 验收标准指向外部系统/主观体验 | 标注"需人工验证" + 说明原因 | — |

**验证级别选择规则**：
- 优先 L1（动态验证）：如果 checks.md 中有测试命令且可执行
- 降级 L2（静态验证）：动态验证不可用或失败时
- 标注 L3（不可验证）：涉及外部 API、主观体验、物理操作等

### Step 3：逐条验证

对每个验收断言执行验证：

#### L1 动态验证

1. 识别与该断言相关的测试（通过测试名称/描述匹配）
2. 执行相关测试，捕获输出
3. 如果无直接对应测试 → 尝试通过 CLI 或脚本模拟验证
4. 记录验证证据：测试输出摘要或执行结果

#### L2 静态语义验证

1. 获取 CR 的 `git diff`
2. 阅读变更代码 + 相关上下文文件
3. 对照验收断言，逐项判断：
   - 代码逻辑是否实现了该断言描述的行为
   - 是否存在明显的遗漏（边界条件、错误处理）
4. 记录验证证据：代码引用（文件:行号）+ 判断依据

#### L3 标注不可验证

1. 明确标注原因（"需要访问外部 API"/"涉及 UI 视觉体验"等）
2. 建议人工验证方法

### Step 4：生成验证报告

```markdown
## 验收验证报告

**CR**：[CR 标题]（[CR-ID]）
**PF**：[PF 标题]（[PF-ID]）
**验证时间**：[日期时间]

### 逐条验证结果

| # | 验收断言 | 状态 | 验证级别 | 证据 |
|---|---------|------|---------|------|
| A1 | [断言内容] | ✅ 通过 | L1 动态 | 测试 test_xxx 通过 |
| A2 | [断言内容] | ✅ 通过 | L2 静态 | src/handler.py:45 实现了 CSV 解析逻辑 |
| A3 | [断言内容] | ❌ 未通过 | L2 静态 | 未找到进度条相关实现代码 |
| A4 | [断言内容] | ⚠️ 需人工 | L3 | 涉及第三方支付 API 回调 |

### 汇总

- 通过：N / M
- 未通过：X（[未通过断言列表]）
- 需人工验证：Y
- 验证覆盖率：(N+X) / M = Z%（排除 L3 不可验证项）

### 未覆盖的风险点

- [边界条件/错误路径/性能场景等 verify 过程中发现的额外风险]

### 验收标准改进建议（如有）

- [验收标准模糊/不可测试的条目及改进建议]
```

### Step 5：写入 CR

1. 将验证报告写入 CR 文件的"验证证据"section
2. 在 CR 事件表添加记录：
   ```
   | [日期] | /pace-test accept | Claude | A: N/M passed, X failed, Y manual | — |
   ```
3. 如果全部通过（含 L3 需人工项标注）→ 输出"验收验证通过"
4. 如果有未通过项 → 输出未通过清单，建议修复后重新 verify

## 合理化预防表

**铁律：验证必须有证据，推理不等于验证。**

| Claude 可能的借口 | 现实回击 |
|-----------------|---------|
| "代码逻辑正确，应该能工作" | 推理不是证据。L1 能执行就执行，L2 必须引用具体代码行。声称"正确"时必须附带代码引用 |
| "验收标准太模糊，无法精确验证" | 输出"验收标准歧义报告"——列出模糊点和建议的明确化方案。不可以模糊为借口跳过验证 |
| "需要运行环境才能验证，跳过吧" | 降级为 L2 静态验证，明确标注"未进行动态验证"。跳过 ≠ 通过 |
| "所有条件都满足了" | 每条必须附带证据。"满足"是结论，证据是代码引用或测试输出。无证据的"满足"等于未验证 |
| "这条验收标准不重要" | 重要性由 PF 定义者决定，不由验证者决定。所有验收标准同等对待 |
| "之前已经验证过了" | 每次 verify 独立执行，不引用过往结果。代码可能已变更，旧证据无效 |
| "太多条了，抽样验证吧" | 全量验证是底线。验收标准数量 = PF 定义者认为必要的数量。抽样 = 未覆盖的风险 |

## 与 Gate 2 验收比对的区别

| 维度 | Gate 2 验收比对 | /pace-test accept |
|------|---------------|-------------------|
| 心态 | 审查式（对照意图 section 判断） | 验证式（尝试执行、收集证据） |
| 输入 | CR 意图 section 验收条件 | PF 级验收标准 + CR 验收条件 |
| 粒度 | 验收条件满足/不满足 | 逐条 + 验证级别 + 证据引用 |
| 证据 | 代码阅读判断 | 测试输出/代码引用/执行结果 |
| 产出 | Gate 2 通过/失败 | 验证报告（可被 Gate 2 消费） |
| 触发 | Gate 2 自动执行 | 用户显式调用或自动模式 |

## 降级策略

```
L1 动态验证
  ├── 成功 → 记录证据
  └── 失败（环境/命令不可用）
       ↓
L2 静态语义验证
  ├── 成功 → 记录证据（标注"静态验证"）
  └── 失败（无法从代码判断）
       ↓
L3 标注不可验证
  └── 记录原因 + 建议人工验证方法
```

每次降级必须在报告中明确标注验证级别，不可静默降级。

## 触发方式

| 模式 | 触发时机 | 行为 |
|------|---------|------|
| 显式 | 用户 `/pace-test accept [CR]` | 完整执行上述流程 |
| Gate 2 消费 | Gate 2 检查时发现 CR 已有验证证据 | 引用已有报告，不重复执行 |
| 自动（未来） | developing→verifying 转换时 | Phase 3 集成后自动触发 |

**铁律：自动模式只消费已有结果或执行最轻量检查，不在 Gate 流程中触发重量级操作。**
