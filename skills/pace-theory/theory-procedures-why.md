# theory-procedures-why — `why` 子命令输出规则

> 由 SKILL.md 路由表加载。仅 `why` 子命令时读取。自包含，不依赖其他 procedures。

## §0 速查卡片

```
why 输出：行为列表 → 设计依据（§11 交叉引用）→ 理论原文（≤3 行）
支持精准查询：why gate、why cr、why paused 等
末尾导航到 /pace-trace
```

## §1 无参数 `why` 三层输出

```
## devpace 为什么这样做

### 近期行为
[从 state.md + CR 事件表 + Gate 结果 + iterations/current.md 提取]
- 行为 1：[描述]
- 行为 2：[描述]
...

### 设计依据
| 行为 | 设计决策 | 理论依据 |
|------|---------|---------|
| 行为 1 | §11 #N：[决策名称] | [理论摘要] |
| 行为 2 | §11 #N：[决策名称] | [理论摘要] |

### 理论原文（摘录）
> [theory.md 对应段落，≤3 行]

---
想看某个具体决策的完整判定过程？试试 `/pace-trace [CR] [type]`
```

## §2 `why <行为关键词>` 精准查询

支持的行为关键词映射：

| 关键词 | 匹配行为 | §11 对应决策 |
|--------|---------|-------------|
| `gate` / `门禁` / `质量检查` | 为什么执行质量检查 | #2（状态机+门禁）、#14（门禁幂等可重跑） |
| `cr` / `变更` / `任务` | 为什么创建 CR | #1（三层追溯）、#13（只记分支名） |
| `paused` / `暂停` | 为什么暂停保留工作 | #15（paused 保留工作成果）、#4（变更管理一等公民） |
| `review` / `审批` | 为什么停下等人类审批 | #7（human_review 门禁停车）、#6（Claude 技术闭环自治） |
| `trace` / `追溯` | 为什么维护追溯树 | #1（三层追溯）、#9（价值追溯树可视化） |
| `mos` / `成效` / `指标` | 为什么用 MoS 而非固定范围 | #3（MoS checkbox 考核）、#10（迭代+变更记录表） |
| `markdown` / `状态文件` | 为什么用 Markdown 做状态文件 | #5（消费者是 LLM+人类） |
| `sdd` / `结构化` | 为什么借鉴 SDD | #16（渐进式结构化） |
| `state` / `project` | 为什么分 state.md 和 project.md | #12（产品线信息分层） |
| `explore` / `advance` | 为什么分双模式 | #11（探索/推进双模式） |

精准查询时只输出对应决策的详细解释，不列出完整行为列表。

## §3 数据源读取顺序

1. `.devpace/state.md` — 当前工作状态和最近行为
2. `.devpace/backlog/` 最近修改的 CR 文件 — 事件表中的近期行为
3. `.devpace/rules/checks.md` — Gate 1/2 检查内容（解释门禁行为时）
4. `.devpace/iterations/current.md` — 迭代上下文（如存在）
5. `knowledge/theory.md` §11 — 设计决策权威源

未初始化时跳过 1-4，仅输出 §11 的通用理论解释。
